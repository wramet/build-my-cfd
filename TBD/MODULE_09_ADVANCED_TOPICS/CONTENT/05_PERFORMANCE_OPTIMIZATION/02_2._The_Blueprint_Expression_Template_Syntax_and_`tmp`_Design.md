## 2. แบบแปลน: ไวยากรณ์ Expression Template และการออกแบบ `tmp`

### ส่วนประกอบทางสถาปัตยกรรมหลัก

#### ส่วนประกอบที่ 1: Smart Pointer `tmp<>` สำหรับการจัดการ Temporary

คลาส `tmp<>` ทำหน้าที่เป็นระบบ reference-counted smart pointer ขั้นสูงของ OpenFOAM ที่ออกแบบมาโดยเฉพาะสำหรับการจัดการออบเจกต์ชั่วคราวในการดำเนินการ field คลาส template นี้แก้ไขปัญหาด้านประสิทธิภาพที่สำคัญในการคำนวณ CFD: การสร้างและทำลายออบเจกต์ field ระหว่างทางบ่อยครั้งระหว่างการดำเนินการทางคณิตศาสตร์

การ implement ที่ทำให้ง่ายขึ้นเผยให้เห็นหลักการออกแบบ:

```cpp
template<class T>
class tmp
{
    enum type { REUSABLE_TMP, NON_REUSABLE_TMP, CONST_REF };
    type type_;
    mutable T* ptr_;

public:
    explicit tmp(T* = 0, bool nonReusable = false);
    
    // Reference counting operations
    inline void operator++();
    inline void operator--();
};
```

ระบบ `tmp<>` จำแนกประเภทออบเจกต์เป็นสามประเภทที่แตกต่างกัน:

1. **REUSABLE_TMP**: ออบเจกต์ที่สามารถกำหนดใหม่และแคชไว้เพื่อใช้ในอนาคตได้
2. **NON_REUSABLE_TMP**: temporaries ที่ใช้ครั้งเดียวซึ่งควรกำจัด
3. **CONST_REF**: การอ้างอิงถึงออบเจกต์ที่มีอยู่โดยไม่มีความรับผิดชอบในการเป็นเจ้าของ

ปัญหาพื้นฐานที่ expression templates แก้ไขคือการกำจัด overhead ที่เกี่ยวข้องกับการสร้างและทำลายออบเจกต์ `tmp<>` แม้จะมีการปรับให้เหมาะสมของ smart pointer แต่วงจรการจองและปล่อยหน่วยความจำระหว่างการดำเนินการ field ที่ซับซ้อนยังคงสร้างปัญหา performance bottleneck ที่สำคัญในการจำลอง CFD

#### ส่วนประกอบที่ 2: สถาปัตยกรรมพื้นฐาน Expression Template

ระบบ expression template ของ OpenFOAM ใช้ประโยชน์จาก **Curiously Recurring Template Pattern (CRTP)** เพื่อให้ได้ static polymorphism ซึ่งกำจัด runtime overhead ของ virtual function calls ทางเลือกทางสถาปัตยกรรมนี้เป็นการปรับให้เหมาะสมที่สำคัญสำหรับการคำนวณ CFD ที่ต้องการประสิทธิภาพสูง

การ implement ของ CRTP ให้ข้อมูลประเภทเวลา compile และ static polymorphism:

```cpp
template<class Derived>
class ExpressionTemplate
{
public:
    const Derived& derived() const
    {
        return static_cast<const Derived&>(*this);
    }

    using value_type = typename Derived::value_type;
    using mesh_type = typename Derived::mesh_type;
    static constexpr bool is_field = Derived::is_field;
    static constexpr dimensionSet dimensions = Derived::dimensions;

    template<class Mesh>
    value_type evaluate(label cellI, const Mesh& mesh) const
    {
        return derived().evaluate(cellI, mesh);
    }
};
```

รูปแบบ CRTP ช่วยให้ compiler แก้ไขการเรียกใช้ฟังก์ชันทั้งหมดเวลา compile ซึ่งกำจัด vtable lookups ที่จะเกิดขึ้นกับ runtime polymorphism แบบดั้งเดิม การปรับให้เหมาะสมนี้มีค่าเป็นพิเศษในแอปพลิเคชัน CFD ที่การดำเนินการ field ถูกดำเนินการหลายล้านครั้งระหว่างการจำลอง

#### ส่วนประกอบที่ 3: Binary Expression Trees

Binary expression templates เป็นกระดูกสันหลังทางคำนวณของระบบ lazy evaluation ของ OpenFOAM พวกมันสร้าง parse trees ที่แสดงถึงการดำเนินการทางคณิตศาสตร์โดยไม่ดำเนินการคำนวณทันที โดยเลื่อนการประเมินจริงไปจนกว่าจะต้องการผลลัพธ์

การ implement ของ binary expression template แสดงให้เห็นหลักการของ lazy evaluation:

```cpp
template<class LHS, class RHS, class Op>
class BinaryExpression : public ExpressionTemplate<BinaryExpression<LHS, RHS, Op>>
{
private:
    const LHS& lhs_;
    const RHS& rhs_;
    const Op& op_;

public:
    using value_type = typename Op::template result_type<LHS, RHS>;

    static_assert(LHS::dimensions == RHS::dimensions,
                  "Cannot combine fields with different dimensions");

    BinaryExpression(const LHS& lhs, const RHS& rhs, const Op& op)
        : lhs_(lhs), rhs_(rhs), op_(op) {}

    template<class Mesh>
    value_type evaluate(label cellI, const Mesh& mesh) const
    {
        return op_(lhs_.evaluate(cellI, mesh), rhs_.evaluate(cellI, mesh));
    }
};
```

การออกแบบนี้สร้าง parse tree ที่เวลา compile แทนที่จะดำเนินการคำนวณที่ runtime เมื่อคุณเขียน expression เช่น `fieldA + fieldB` compiler จะสร้าง binary expression tree ที่แสดงถึงการดำเนินการบวก แต่ไม่ได้ดำเนินการคำนวณจริงจนกว่าจะขอผลลัพธ์อย่างชัดเจน

`static_assert` ช่วยให้มั่นใจว่ามีความสม่ำเสมอทางมิติเวลา compile ซึ่งป้องกันการดำเนินการที่ไร้ความหมายทางกายภาพก่อน runtime การตรวจสอบเวลา compile นี้เป็นหัวใจสำคัญของความปลอดภัยประเภทและความสามารถในการวิเคราะห์มิติของ OpenFOAM

โครงสร้าง expression tree ช่วยให้มีการปรับให้เหมาะสมอย่างทรงประสิทธิภาพ:
- **Loop fusion**: หลายการดำเนินการสามารถรวมเป็น loop เดียวได้
- **Elimination of temporaries**: ไม่มีการสร้างออบเจกต์ field ระหว่างทาง
- **Memory access optimization**: compiler สามารถปรับให้เหมาะสมรูปแบบการเข้าถึงหน่วยความจำเพื่อประสิทธิภาพ cache
- **Vectorization**: compiler สมัยใหม่สามารถปรับให้เหมาะสม loops ผลลัพธ์สำหรับคำสั่ง SIMD ได้ดีขึ้น
