## Part 1: Pattern Identification

### The Problem — Who Writes the Files?

In a naive CFD solver, I/O might look like this:

```cpp
// Naive approach — explicit I/O everywhere
solver.solve();
if (writeTime) {
    writeField("U", velocityField);
    writeField("p", pressureField);
    writeField("T", temperatureField);
    writeField("k", turbKineticEnergy);
    writeField("epsilon", dissipationRate);
    // ...and every derived quantity that might be needed
}
```

This approach has several deep problems. The solver function must know about every field that could ever need writing. Adding a new field (say, a passive scalar) requires modifying the solver's write loop. Fields that are only needed for post-processing clutter the main write path. Conversely, fields that are always needed for restart may be forgotten.

OpenFOAM solves this with a radically different design: **objects declare their own I/O requirements at construction time**, and the registry collects and executes those requirements automatically.

### The RAII I/O Principle

RAII (Resource Acquisition Is Initialization) is a C++ pattern where a resource is acquired in the constructor and released in the destructor. OpenFOAM applies this to I/O:

- **At construction:** an object declares whether it should be read from disk (`readOpt`) and whether it should be written to disk on request (`writeOpt`).
- **During lifetime:** the object is registered with a central `objectRegistry` that tracks it by name.
- **At write time:** the registry iterates all registered objects and calls `write()` on those flagged `AUTO_WRITE`. No central list is maintained by the solver.
- **At destruction:** the object removes itself from the registry (`checkOut`).

The result: adding a new field to a solver requires only constructing it with the correct I/O options. The write infrastructure requires zero changes.

### The Two Control Enumerations

#### `readOption` — What Happens at Construction

⭐ Verified from `openfoam_temp/src/OpenFOAM/db/IOobject/IOobject.H`, lines 118-124:

```cpp
enum readOption
{
    MUST_READ,
    MUST_READ_IF_MODIFIED,
    READ_IF_PRESENT,
    NO_READ
};
```

| Option | Behavior |
|--------|----------|
| `MUST_READ` | File must exist and be readable. Fatal error if missing. No timestamp checking. |
| `MUST_READ_IF_MODIFIED` | Same as `MUST_READ`, but the file's timestamp is monitored. If the file changes during the run (e.g., `fvSolution` is edited), the object is automatically re-read at the next time step. |
| `READ_IF_PRESENT` | Read the file if it exists. Silently continue if missing. Fatal only if the file exists but cannot be parsed. |
| `NO_READ` | Do not attempt to read. Construct from scratch. |

The distinction between `MUST_READ` and `READ_IF_PRESENT` is critical. Pressure `p` is typically `MUST_READ` — its absence is a user error. A turbulence model's optional coefficient override might be `READ_IF_PRESENT` — its absence is fine, defaults will be used.

#### `writeOption` — What Happens at Write Time

⭐ Verified from `openfoam_temp/src/OpenFOAM/db/IOobject/IOobject.H`, lines 126-131:

```cpp
enum writeOption
{
    AUTO_WRITE = 0,
    NO_WRITE = 1
};
```

| Option | Behavior |
|--------|----------|
| `AUTO_WRITE` | The `objectRegistry::writeObject()` will call `write()` on this object when the simulation requests a write (i.e., at output time steps). |
| `NO_WRITE` | The object is registered but never automatically written. It can still be written explicitly with a direct `obj.write()` call. |

Note the integer values: `AUTO_WRITE = 0`, `NO_WRITE = 1`. The `writeObject()` loop in the registry checks `writeOpt() != NO_WRITE`, meaning the integer comparison is `writeOpt() != 1`, i.e., it writes everything with value 0 (AUTO_WRITE). This is a deliberate design: if additional write modes were added in the future, they would all write unless explicitly set to `NO_WRITE`.

### How This Eliminates Explicit I/O

Consider a standard OpenFOAM solver startup. The solver does not contain a list of fields to write. Instead:

```cpp
// Each field's constructor encodes its own I/O intent:
volScalarField p(
    IOobject("p", runTime.timeName(), mesh,
             IOobject::MUST_READ,    // read from disk at startup
             IOobject::AUTO_WRITE),  // write at every output time step
    mesh
);

volVectorField U(
    IOobject("U", runTime.timeName(), mesh,
             IOobject::MUST_READ,
             IOobject::AUTO_WRITE),
    mesh
);

// Internal temporary field — no disk representation needed:
volScalarField rho(
    IOobject("rho", runTime.timeName(), mesh,
             IOobject::NO_READ,
             IOobject::NO_WRITE),
    mesh
);
```

The solver's time loop is then simply:

```cpp
while (runTime.loop()) {
    // ... solve equations ...
    runTime.write();  // writes p and U automatically; rho is silently skipped
}
```

The critical insight is that `runTime.write()` traverses the registry and writes only `AUTO_WRITE` objects. `rho` exists in the registry (it is `checkIn()`-ed) but is invisible to the write loop because its `writeOpt()` returns `NO_WRITE`.

### Tracing a Field's Complete Lifecycle

Let us trace the expression:

```cpp
volScalarField p(
    IOobject("p", runTime.timeName(), mesh,
             IOobject::MUST_READ, IOobject::AUTO_WRITE),
    mesh
);
```

**Step 1 — IOobject construction:**
`IOobject` is constructed as a temporary. It stores: `name_ = "p"`, `instance_ = "0"` (from `runTime.timeName()`), `db_ = mesh` (a reference to `fvMesh`, which is an `objectRegistry`), `rOpt_ = MUST_READ`, `wOpt_ = AUTO_WRITE`, `registerObject_ = true`.

**Step 2 — Field constructor:**
`volScalarField` (a `GeometricField<scalar, fvPatchField, volMesh>`) is constructed. Its constructor calls `regIOobject(IOobject)`, which copies the metadata from the temporary `IOobject`. Because `registerObject_ = true`, the `regIOobject` constructor ends with `checkIn()`.

**Step 3 — `regIOobject::checkIn()` is called:**

⭐ Verified from `openfoam_temp/src/OpenFOAM/db/regIOobject/regIOobject.C`, lines 205-237:

```cpp
// File: openfoam_temp/src/OpenFOAM/db/regIOobject/regIOobject.C
// Lines: 205-237
bool Foam::regIOobject::checkIn()
{
    if (!registered_)
    {
        // multiple checkIn of same object is disallowed
        registered_ = db().checkIn(*this);
        // ...
    }
    return registered_;
}
```

The guard `if (!registered_)` prevents double-registration. The actual insertion happens in `db().checkIn(*this)`, which calls `objectRegistry::checkIn(regIOobject&)`.

**Step 4 — `objectRegistry::checkIn()` inserts into the hash table:**

⭐ Verified from `openfoam_temp/src/OpenFOAM/db/objectRegistry/objectRegistry.C`, lines 268-307:

```cpp
// File: openfoam_temp/src/OpenFOAM/db/objectRegistry/objectRegistry.C
// Lines: 268-307

{
    // ...debug output...

    // If an object with the same name already exists and is owned
    // by the registry, delete it before inserting the new one.
    // (handles re-read scenarios)

    return const_cast<objectRegistry&>(*this).insert(io.name(), &io);
}
```

After this, `mesh["p"]` returns a pointer to the `volScalarField p`.

**Step 5 — Field reads its data:**
Because `rOpt_ = MUST_READ`, the field constructor calls `regIOobject::readHeaderOk()` then `readData(Istream&)`. The field data is populated from `case/0/p`.

**Step 6 — At write time, `runTime.write()` is called:**
This propagates to `objectRegistry::writeObject()`:

⭐ Verified from `openfoam_temp/src/OpenFOAM/db/objectRegistry/objectRegistry.C`, lines 599-629:

```cpp
// File: openfoam_temp/src/OpenFOAM/db/objectRegistry/objectRegistry.C
// Lines: 609-626
forAllConstIter(HashTable<regIOobject*>, *this, iter)
{
    // ...debug output...

    if (iter()->writeOpt() != NO_WRITE)   // i.e., writeOpt() == AUTO_WRITE (value 0)
    {
        ok = iter()->writeObject(fmt, ver, cmp, write) && ok;
    }
}
```

The loop visits every registered object. `p` has `writeOpt() == AUTO_WRITE (0)`, so `0 != 1` is true, and `p.writeObject()` is called. `rho` with `NO_WRITE (1)` satisfies `1 != 1` as false, so it is skipped.

**Step 7 — `volScalarField::writeObject()` delegates to `writeData()`:**
`regIOobject::writeObject()` writes the FoamFile header (class name, version, format), then calls the pure virtual `writeData(os)`. `volScalarField::writeData()` writes the boundary field and internal field values.

**Step 8 — Destruction and `checkOut()`:**
When `p` goes out of scope:

⭐ Verified from `openfoam_temp/src/OpenFOAM/db/regIOobject/regIOobject.C`, lines 150-175:

```cpp

{

    }

```

Because `p` is stack-allocated (`ownedByRegistry_ = false`), it calls `checkOut()`, which removes `"p"` from the registry's hash table. The registry's pointer to `p` is erased before `p`'s memory is reclaimed.

### The `objectRegistry` Destructor

⭐ Verified from `openfoam_temp/src/OpenFOAM/db/objectRegistry/objectRegistry.C`, lines 156-160:

```cpp
// File: openfoam_temp/src/OpenFOAM/db/objectRegistry/objectRegistry.C
// Lines: 156-160
Foam::objectRegistry::~objectRegistry()
{
    cacheTemporaryObjects_.clear();
    clear();  // deletes all ownedByRegistry_ objects
}
```

`clear()` iterates the hash table and `delete`s any object where `ownedByRegistry_ == true`. Stack-allocated objects (like our `p`) have already called `checkOut()` in their own destructors, so they are gone from the table by the time `~objectRegistry()` runs. The order of destruction matters: fields must be destroyed before the registry they are registered in. C++ guarantees local variables are destroyed in reverse order of construction, so constructing `mesh` before `p` ensures `p` is destroyed first.

## Part 3: C++ Mechanics Explained

### Why Raw Pointers, Not `unique_ptr`, in the Registry

The `HashTable<regIOobject*>` stores raw, non-owning pointers. This is intentional and correct. Here is why `unique_ptr` would be wrong:

```cpp
// Hypothetical WRONG design:
HashTable<std::unique_ptr<regIOobject>> objects_;

// Problem 1: cannot insert stack-allocated objects
volScalarField p(...);      // lives on the stack
objects_.insert("p", std::make_unique<...>(&p));  // unique_ptr would try to delete p!

// Problem 2: double-delete
// When p's destructor runs, it would call delete on itself.
// Then unique_ptr would also delete it. Undefined behavior.
```

The correct semantics are:

- **Stack-allocated fields:** the registry holds a raw pointer. The field's destructor calls `checkOut()` to remove the pointer. The registry never calls `delete` on them. (`ownedByRegistry_ = false`)
- **Heap-allocated fields stored via `store()`:** `ownedByRegistry_` is set to `true`. The registry is responsible for deletion. The field's destructor checks `ownedByRegistry_` and skips `checkOut()` because the registry will handle cleanup in `clear()`.

This is a classical observer pattern with manual ownership tracking rather than pointer ownership semantics.

### The `checkIn`/`checkOut` Double-Registration Guard

The guard `if (!registered_)` in `regIOobject::checkIn()` is essential:

```cpp
// Scenario: copy constructor
regIOobject(const regIOobject& rio, bool registerCopy)
{
    if (registerCopy)
    {
        // Transfer registration from original to copy
        rio.checkOut();
        checkIn();
    }
    // If !registerCopy, neither checkIn nor checkOut is called
}
```

Without the guard, constructing a field and then calling `checkIn()` again (e.g., in an inadvertent double-initialization) would insert the pointer twice under the same name. The hash table `insert` would succeed but the second insertion would shadow the first, leaving the first unreachable but still alive — a resource leak.

### `lookupObject<T>()` — Dynamic Cast with Graceful Failure

⭐ Verified from `openfoam_temp/src/OpenFOAM/db/objectRegistry/objectRegistryTemplates.C`, lines 165-204:

```cpp
// File: openfoam_temp/src/OpenFOAM/db/objectRegistry/objectRegistryTemplates.C
// Lines: 165-184
template<class Type>
const Type& Foam::objectRegistry::lookupObject(const word& name) const
{
    const_iterator iter = find(name);

    if (iter != end())
    {
        const Type* vpsiPtr_ = dynamic_cast<const Type*>(iter());
        //                     ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
        // cast from regIOobject* to Type*
        // returns nullptr if the object is not a Type

        if (vpsiPtr_)
        {
            return *vpsiPtr_;
        }

        FatalErrorInFunction
            << "lookup of " << name << " successful but it is not a "
            << Type::typeName << ", it is a " << iter()->type()
            << abort(FatalError);
    }
    else
    {
        if (this->parentNotTime())
        {
            return parent_.lookupObject<Type>(name);  // search parent registry
        }

        FatalErrorInFunction
            << "request for " << Type::typeName << " " << name
            << " from objectRegistry " << this->name() << " failed"
            << abort(FatalError);
    }
}
```

Two failure modes, two different errors:
1. **Name not found:** tries the parent registry (hierarchical lookup). If not in any ancestor, `FatalError` with a list of available objects of that type.
2. **Name found but wrong type:** `FatalError` saying "it is not a `volScalarField`, it is a `volVectorField`."

The `dynamic_cast` returning `nullptr` (not throwing) is what enables the clean type-mismatch error message. If `dynamic_cast` threw, you would get a generic `std::bad_cast` with no context.

### Why `writeData` Is Pure Virtual

`regIOobject::writeData(Ostream&)` is declared `= 0` (pure virtual). This enforces a contract: **every concrete class that can be registered must know how to serialize itself**.

The alternative — a default implementation that does nothing — would silently fail when a class forgot to override it. The pure virtual makes forgetting a compile-time error.

For `objectRegistry` itself, `writeData` is overridden with `NotImplemented`:

⭐ Verified from `openfoam_temp/src/OpenFOAM/db/objectRegistry/objectRegistry.H`, lines 297-303:

```cpp
// File: openfoam_temp/src/OpenFOAM/db/objectRegistry/objectRegistry.H
// Lines: 297-303
virtual bool writeData(Ostream&) const
{
    NotImplemented;
    return false;
}
```

This satisfies the pure virtual requirement (the class is now concrete) while making it clear that you should call `writeObject()` instead of `writeData()` for registries.

### Thread Safety Concerns

⚠️ The following is based on design analysis, not direct source verification.

The registry's hash table is modified from every field constructor (via `checkIn`) and every field destructor (via `checkOut`). In a multi-threaded context, concurrent field construction — say, in a parallel loop initializing temporary fields — could cause data races on the hash table.

OpenFOAM's current architecture is single-threaded at the field level (MPI parallelism is at the process level, not thread level). Each MPI rank has its own `objectRegistry` and there is no sharing across threads. This is why no mutex protects the hash table operations.

If you were adapting this pattern for a threaded environment, you would need either:
- A mutex wrapping `checkIn`/`checkOut`, or
- Thread-local registries that merge into a global one after parallel construction.

## Part 4: Implementation Exercise

We now implement a minimal but complete object registry in standard C++ (no OpenFOAM dependencies). The design mirrors the real OpenFOAM architecture: non-owning raw pointers, RAII registration, virtual `write()`, automatic write-on-destruct.

```cpp
// File: mini_registry.cpp
// Compile: g++ -std=c++17 -Wall -o mini_registry mini_registry.cpp
// Run:     ./mini_registry

#include <iostream>
#include <string>
#include <unordered_map>
#include <vector>
#include <stdexcept>

// ============================================================
// Forward declaration
// ============================================================

class MiniRegistry;

// ============================================================
// IOBase — mirrors regIOobject
// Stores name + write option. Knows its owning registry.
// Registers itself in the registry on construction.
// Deregisters itself on destruction.
// ============================================================

class IOBase
{
protected:
    std::string   name_;
    bool          autoWrite_;
    MiniRegistry* registry_;   // non-owning pointer to the registry

public:
    IOBase(const std::string& name, MiniRegistry& reg, bool autoWrite);

    // Non-copyable, non-moveable (mirrors OpenFOAM's deleted operators)
    IOBase(const IOBase&) = delete;
    IOBase& operator=(const IOBase&) = delete;

    virtual ~IOBase();

    // The pure virtual write contract — every concrete type must implement this
    virtual void writeData(std::ostream& os) const = 0;

    const std::string& name()     const { return name_; }
    bool               autoWrite() const { return autoWrite_; }
};


// ============================================================
// MiniRegistry — mirrors objectRegistry (HashTable<regIOobject*>)
// Non-owning store of IOBase pointers keyed by name.
// On destruction: writes all autoWrite objects.
// ============================================================

class MiniRegistry
{
    // Raw (non-owning) pointer store — objects manage their own lifetime
    std::unordered_map<std::string, IOBase*> objects_;

public:
    MiniRegistry() = default;

    // Non-copyable (mirrors objectRegistry's deleted copy constructor)
    MiniRegistry(const MiniRegistry&) = delete;
    MiniRegistry& operator=(const MiniRegistry&) = delete;

    // checkIn: insert object into registry
    // Mirrors objectRegistry::checkIn(regIOobject&)
    void checkIn(IOBase* obj)
    {
        if (!obj) return;

        auto it = objects_.find(obj->name());
        if (it != objects_.end())
        {
            std::cerr << "[registry] WARNING: object '" << obj->name()
                      << "' already registered. Overwriting.\n";
        }
        objects_[obj->name()] = obj;
        std::cout << "[registry] checkIn: '" << obj->name() << "'\n";
    }

    // checkOut: remove object from registry
    // Mirrors objectRegistry::checkOut(regIOobject&)
    void checkOut(IOBase* obj)
    {
        if (!obj) return;

        auto it = objects_.find(obj->name());
        if (it != objects_.end() && it->second == obj)
        {
            objects_.erase(it);
            std::cout << "[registry] checkOut: '" << obj->name() << "'\n";
        }
        else
        {
            std::cerr << "[registry] WARNING: checkOut failed for '"
                      << obj->name() << "' — not found in registry.\n";
        }
    }

    // lookupObject<T>: find by name and dynamic_cast to T
    // Mirrors objectRegistry::lookupObject<Type>(word&)
    template<class T>
    T* lookupObject(const std::string& name)
    {
        auto it = objects_.find(name);
        if (it == objects_.end())
        {
            throw std::runtime_error(
                "lookupObject: '" + name + "' not found in registry"
            );
        }

        // dynamic_cast returns nullptr if the type is wrong
        T* typed = dynamic_cast<T*>(it->second);
        if (!typed)
        {
            throw std::runtime_error(
                "lookupObject: '" + name + "' found but is wrong type"
            );
        }
        return typed;
    }

    // writeAll: iterate all registered objects; write AUTO_WRITE ones
    // Mirrors objectRegistry::writeObject() iteration loop
    void writeAll(std::ostream& os) const
    {
        std::cout << "[registry] writeAll: scanning "
                  << objects_.size() << " registered object(s)\n";

        for (const auto& kv : objects_)
        {
            IOBase* obj = kv.second;
            if (obj->autoWrite())   // mirrors: writeOpt() != NO_WRITE
            {
                std::cout << "[registry] writing AUTO_WRITE object: '"
                          << obj->name() << "'\n";
                obj->writeData(os);
            }
            else
            {
                std::cout << "[registry] skipping NO_WRITE object: '"
                          << obj->name() << "'\n";
            }
        }
    }

    // Destructor — automatically writes all autoWrite objects
    // This is the RAII payoff: no explicit write call needed at the call site
    ~MiniRegistry()
    {
        std::cout << "\n[registry] ~MiniRegistry: triggering auto-write on destruction\n";
        writeAll(std::cout);
        // Note: we do NOT delete objects_ values — they are non-owning pointers.
        // Each IOBase destructor will call checkOut(), but by the time ~MiniRegistry
        // runs, all stack-allocated IOBase objects registered here should already
        // be destroyed (C++ LIFO destruction of locals). The map cleanup is benign.
        std::cout << "[registry] ~MiniRegistry: done\n";
    }
};


// ============================================================
// IOBase constructor / destructor — implement registration
// ============================================================

IOBase::IOBase(const std::string& name, MiniRegistry& reg, bool autoWrite)
    : name_(name), autoWrite_(autoWrite), registry_(&reg)
{
    // Self-registration at construction — mirrors regIOobject::checkIn()
    registry_->checkIn(this);
}

IOBase::~IOBase()
{
    // Self-deregistration at destruction — mirrors regIOobject::~regIOobject()
    if (registry_)
    {
        registry_->checkOut(this);
    }
}

class ScalarField : public IOBase
{
    std::vector<double> data_;

public:
    ScalarField(
        const std::string&    name,
        MiniRegistry&         reg,
        bool                  autoWrite,
        std::vector<double>   data
    )
    :   IOBase(name, reg, autoWrite),

                  << data_.size() << " values\n";
    }

    ~ScalarField()
    {
        // IOBase destructor runs after this, calling checkOut automatically
        std::cout << "[ScalarField] '" << name_ << "' destructing\n";
    }

    // Implementation of the pure virtual write contract
    void writeData(std::ostream& os) const override
    {
        os << "ScalarField '" << name_ << "' {\n";
        os << "    values: [";
        for (std::size_t i = 0; i < data_.size(); ++i)
        {
            os << data_[i];
            if (i + 1 < data_.size()) os << ", ";
        }
        os << "]\n";
        os << "}\n";
    }

    double operator[](std::size_t i) const { return data_[i]; }
    std::size_t size() const { return data_.size(); }

        std::cout << "\n--- Constructing fields ---\n";

        // Correct type lookup
        ScalarField* pPtr = registry.lookupObject<ScalarField>("p");

        try

        try
        {
            ScalarField* missingPtr = registry.lookupObject<ScalarField>("T");
            (void)missingPtr;
        }
        catch (const std::runtime_error& e)
        {
            std::cout << "Expected error: " << e.what() << "\n";
        }

        std::cout << "\n--- Simulating solver time loop ---\n";
        std::cout << "(no explicit write call — registry destructor handles it)\n";

        // Fields are destroyed in reverse construction order: velocity, rho, pressure
        // Each calls checkOut() in its IOBase destructor.

        std::cout << "\n--- Explicit writeAll (simulating runTime.write()) ---\n";
        registry.writeAll(std::cout);

        std::cout << "\n--- Leaving scope: fields destruct first, then registry ---\n";
```

### Key Observation About Destruction Order

Notice that when `~MiniRegistry()` runs, the registry has 0 objects. This is because C++ destroys local variables in reverse declaration order:

1. `velocity` (declared last) destructs first → calls `checkOut("U")`
2. `rho` destructs → calls `checkOut("rho")`
3. `pressure` destructs → calls `checkOut("p")`
4. `registry` destructs last → `writeAll` finds empty map

This is why the explicit `writeAll` call (step labeled "simulating `runTime.write()`") is needed to demonstrate the actual write behavior. In OpenFOAM, `runTime.write()` is called **within the time loop**, while all fields are still alive and registered. The registry destructor's `clear()` then handles owned objects.

If you want the registry destructor to actually write, declare fields inside a **nested scope** or heap-allocate them with `store()`.

### Variant: Registry Destructor Writes Active Objects

```cpp
int main()

    // Heap-allocated field — transferred to registry ownership

    // Here we simulate with a nested scope instead:
    {

        // But p destroys before registry due to scope ordering

        // This is exactly what objectRegistry::store() achieves in OpenFOAM
    }

    // When registry destructs here, p has already checked out.
    // store() transfers ownership: registry destructs p, so p outlives its
    // lexical scope. That is the purpose of ownedByRegistry_.
    return 0;
```

## Part 5: Exercises and Self-Check

### Exercise 1 — What Happens if `checkOut` Is Forgotten?

Consider this code:

```cpp
MiniRegistry registry;

IOBase* orphan = new ScalarField("T", registry, true, {300.0, 301.0});
// orphan is never deleted, and never calls checkOut

// registry.writeAll() will write T correctly, because the pointer is valid.

// BUT: when registry destructs, it does NOT delete the pointer
// (it is non-owning). The memory at orphan is leaked.
// Worse: if we later called delete orphan manually, the IOBase destructor
// would call checkOut on a registry that may already be destroyed.
// -> Undefined behavior (use-after-free on registry_).
```

**Answer:** Forgetting `checkOut` (i.e., forgetting to call `delete` on a heap-allocated field) causes:
1. Memory leak — the field data is never freed.
2. Dangling pointer in the registry — if the registry is destroyed before the field (unusual ordering), calling `checkOut` from the field's destructor will access the destroyed registry (UB).

In OpenFOAM, this is why `regIOobject::store()` exists: it transfers ownership to the registry, guaranteeing the registry will `delete` the object and that no dangling pointer scenario can occur.

### Exercise 2 — Why Raw Pointer, Not `unique_ptr`, in the Registry?

**Question:** Why does `objectRegistry` use `HashTable<regIOobject*>` instead of `HashTable<std::unique_ptr<regIOobject>>`?

**Answer:** Because most registered objects are stack-allocated. A `unique_ptr` implies ownership — when the `unique_ptr` is destroyed, it calls `delete`. Calling `delete` on a stack-allocated object is undefined behavior. The registry must support both owned (heap) and non-owned (stack) objects. The `ownedByRegistry_` flag tracks ownership, and only `objectRegistry::clear()` calls `delete` on owned objects. A `unique_ptr` would conflate "stored here" with "owned by this container," which is not the correct semantics.

### Exercise 3 — What Does `MUST_READ` Do at Construction?

**Question:** When a field is constructed with `MUST_READ`, what happens if the file does not exist?

**Answer:** The field constructor calls `regIOobject::readHeaderOk()`, which attempts to open the file at `objectPath()`. If the file does not exist, `readHeaderOk()` returns false (for `MUST_READ`), and the constructor issues a `FatalError` — the program terminates with an error message indicating the missing file path. This is intentional: `MUST_READ` means the file must exist. The user must provide initial conditions.

With `READ_IF_PRESENT`, the same missing file causes `readHeaderOk()` to return false, but the constructor does not treat this as fatal — it proceeds with whatever default values were provided programmatically.

### Exercise 4 — How Does `lookupObject` Fail Gracefully?

**Question:** What are the two failure modes of `lookupObject<T>(name)` and how does each manifest?

**Answer:**

**Mode 1 — Name not found:**
`find(name)` returns `end()`. The method then checks `parentNotTime()`. If the current registry has a parent that is not the time registry, it recurses into the parent. This implements hierarchical lookup: a field registered in a sub-mesh can be found when looking through the top-level mesh registry. If no ancestor has the object, `FatalError` is issued listing all available objects of type `T`.

**Mode 2 — Name found, wrong type:**
`find(name)` succeeds but `dynamic_cast<const T*>(iter())` returns `nullptr`. This means an object with that name exists but is of a different type. `FatalError` is issued with both the requested type and the actual type. This catches common errors like requesting `lookupObject<volScalarField>("U")` when `U` is actually a `volVectorField`.

Neither failure throws a C++ exception — both use OpenFOAM's `FatalError` mechanism, which calls `std::abort()` after printing the diagnostic.

### Exercise 5 — Why Is `writeData` Pure Virtual?

**Question:** `objectRegistry` satisfies the pure virtual `writeData(Ostream&) const = 0` from `regIOobject` by providing a body that calls `NotImplemented`. Why not just provide an empty implementation in `regIOobject` instead of making it pure virtual?

**Answer:** Making `writeData` pure virtual serves two purposes:

1. **Compile-time enforcement:** Any class that inherits `regIOobject` and does not implement `writeData` cannot be instantiated. This is a zero-cost compile-time check that prevents accidentally registering objects that have no serialization logic.

2. **Semantic clarity:** A default empty `writeData` would silently produce empty output files — hard to debug. The pure virtual makes the oversight impossible at compile time rather than causing a confusing runtime behavior.

`objectRegistry` provides `NotImplemented` (not an empty body) specifically because the correct write path for a registry is `writeObject()`, not `writeData()`. The `NotImplemented` macro typically prints an error and aborts if called at runtime, making the incorrect call path loudly obvious.

### Exercise 6 — Trace the Destruction Order

**Question:** In the following code, in what order do destructors run, and what does each one do?

```cpp
{
    MiniRegistry reg;
    ScalarField a("a", reg, true,  {1.0});
    ScalarField b("b", reg, false, {2.0});
    ScalarField c("c", reg, true,  {3.0});
}
```

**Answer:** C++ destroys local variables in reverse declaration order within a scope:

1. `~ScalarField c` — prints "[ScalarField] 'c' destructing", then `~IOBase` calls `reg.checkOut(c)`.
2. `~ScalarField b` — prints "[ScalarField] 'b' destructing", then `~IOBase` calls `reg.checkOut(b)`.
3. `~ScalarField a` — prints "[ScalarField] 'a' destructing", then `~IOBase` calls `reg.checkOut(a)`.
4. `~MiniRegistry reg` — `writeAll` finds 0 objects (all have checked out). Prints "[registry] writeAll: scanning 0 registered object(s)".

After step 3, the registry is empty. The registry's `writeAll` in step 4 does nothing. This is exactly why `runTime.write()` must be called inside the time loop while all fields are alive — not at the end of `main()` after fields have been destroyed.

## Summary

Today we studied OpenFOAM's most elegant design pattern: automatic I/O through RAII registration.

| Concept | OpenFOAM Implementation | Our Mini Implementation |
|---------|------------------------|------------------------|
| I/O metadata | `IOobject`: `name_`, `rOpt_`, `wOpt_` | `IOBase`: `name_`, `autoWrite_` |
| Self-registration | `regIOobject::checkIn()`/`checkOut()` | `IOBase` constructor/destructor |
| Central store | `HashTable<regIOobject*>` in `objectRegistry` | `unordered_map<string, IOBase*>` |
| Write loop | `writeObject()`: `writeOpt() != NO_WRITE` | `writeAll()`: `autoWrite()` |
| Type-safe lookup | `dynamic_cast` in `lookupObject<T>()` | Same pattern |
| Serialization contract | Pure virtual `writeData(Ostream&)` | Pure virtual `writeData(ostream&)` |
| Ownership tracking | `ownedByRegistry_` flag | Not implemented (extension exercise) |

**Key insight:** The registry does not need to know what fields exist. Fields announce themselves at construction. This is the **Inversion of Control** principle applied to I/O: the infrastructure (registry) does not call fields; fields call the infrastructure (register themselves), and the infrastructure calls back at write time.

We will see how `runTime` is simultaneously a registry (holding the mesh, fields, dictionaries) and itself a `regIOobject` (it can write control dictionaries). The `Time::loop()` method and automatic `deltaT` adjustment will reveal the final layer of OpenFOAM's automatic I/O architecture.
