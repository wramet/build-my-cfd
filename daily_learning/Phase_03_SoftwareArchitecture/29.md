## Part 1: Pattern Identification / Problem Setup

### The Problem: Legacy Build Systems and Proprietary Build Tools

When you download the OpenFOAM source code and begin compiling it, you are introduced to `wmake` (short for *w*ithin *make*). If you look inside any OpenFOAM application or library directory, you won't find a `Makefile`, a `CMakeLists.txt`, or an IDE project file. Instead, you find a `Make/` directory containing exactly two files:

1. `files`: A list of `.C` source files and the name of the final executable or library to be produced.
2. `options`: A list of compiler flags and directories to search for `.H` headers.

**Why did OpenFOAM build `wmake`?**
OpenFOAM's architecture was laid down in the late 1980s and 1990s. At the time, writing portable, raw Makefiles for massive C++ projects across wildly different UNIX flavors, supercomputer architectures, and compiler vendors (GCC, Intel, PGI, Clang) was a chaotic nightmare. `wmake` was an incredibly impressive and necessary solution to a real problem at the time: it cleanly abstracted away the platform-specific compiler flags and the reliance on a single specific `make` syntax.

However, building software with `wmake` in 2026 has severe drawbacks for a modern C++ developer:

1. **Non-standard Ecosystem:** New developers must learn a proprietary syntax just to compile code. Knowledge of `wmake` is completely untransferable to any other C++ project in the industry.
2. **Limited IDE Tooling:** Modern IDEs (CLion, VS Code, Visual Studio, Neovim LSP) rely on a build system generator (like CMake or Meson) to construct the "compilation database" (the `compile_commands.json` file). Because IDEs don't natively understand `wmake`, you lose code completion, instantaneous jump-to-definition, and refactoring tools—the IDE simply doesn't know where the headers are or what macros are being defined during compilation.
3. **Hard to Integrate External Dependencies:** Importing a modern third-party C++ library (like `nlohmann/json`, `spdlog`, or Google Test) into a `wmake` project requires manually hardcoding paths in the `options` file, manually compiling the dependency, and hoping the ABI (Application Binary Interface) matches perfectly. It is fragile, error-prone, and labor-intensive.

### The Solution: Modern CMake (Target-Based Build Systems)

CMake is the undisputed industry standard build system generator for C++. Note the word *generator*: CMake itself does not invoke the compiler directly. It reads your configuration (`CMakeLists.txt`) and generates a build system (like GNU Make, Ninja, or Visual Studio Solutions) that actually drives the compiler.

#### Legacy CMake (Pre-3.0) vs. Modern CMake (3.0+)

If you have used CMake in the past and hated it, you likely experienced **Legacy CMake**. Older CMake code operated like a giant global shell script:

```cmake
# Legacy CMake (Anti-pattern — Do not use!)
include_directories(/path/to/headers)
add_definitions(-O3 -Wall -std=c++11)
add_library(MyLib source1.cpp)
add_executable(MyApp main.cpp)
```

This is an anti-pattern because it modifies **global state**. If you add a second library to the same project (e.g., `MyApp`), it accidentally inherits those global flags and header paths. As a project grows, it becomes impossible to untangle which target requires which includes, leading to accidental dependencies and namespace collisions.

#### Modern Target-Based CMake

Modern CMake treats libraries and executables as object-oriented **targets**. A target encapsulates everything required to build it and, crucially, everything required to *use* it:
- Its source files.
- Its include directories.
- Its compiler flags and definitions.
- Its link dependencies.

```cmake
# Modern CMake (Target-based)
add_library(MyLib STATIC source1.cpp)

# We attach properties to the TARGET, not to the directory.
target_include_directories(MyLib PUBLIC include/)
target_compile_features(MyLib PUBLIC cxx_std_20)
```

When an executable links against `MyLib`, CMake automatically propagates the correct include directories and the C++ standard to the executable. The executable doesn't need to know *where* `MyLib`'s headers are; it only needs to know that it mathematically depends on `MyLib`.

## Part 2: Source Code Deep Dive

Let's look at exactly how an OpenFOAM library is built with `wmake`, and how we map that exact same structure onto Modern CMake. We will use the `Field<T>` classes we discussed in Phase 1 as our operational example.

### OpenFOAM `wmake` Setup

In a typical OpenFOAM library setup, the `Make/files` document tells the script which sources to compile and where to put the output.

**`Make/files`**:
```text
Field/Field.C
GeometricField/GeometricField.C

LIB = $(FOAM_USER_LIBBIN)/libmyCFDCore
```

This tells `wmake`: "Compile these two C++ files and link them into a shared library named `libmyCFDCore.so`, placing the result in the user's library binary directory defined by the environment variable."

**`Make/options`**:
```text
EXE_INC = \
    -I$(FOAM_SRC)/OpenFOAM/lnInclude \
    -I../common/lnInclude \
    -std=c++14

EXE_LIBS = \
    -lOpenFOAM \
    -L$(FOAM_USER_LIBBIN) \
    -lmyCommonLib
```

This tells `wmake`: "When compiling, look in these specific directories for headers, and when linking, link against these specific libraries."

To build this, you open the terminal in the directory containing the `Make` folder and simply run the `wmake` command. It handles dependency tracking (mostly) and invokes `g++`.

### The CMake Equivalent

In Modern CMake, the equivalent configuration is written in a file named `CMakeLists.txt` placed at the root of the project. This declarative file entirely replaces both `Make/files` and `Make/options`.

```cmake
# CMakeLists.txt
cmake_minimum_required(VERSION 3.20...3.28)

project(my_cfd_core 
        VERSION 1.0.0 
        DESCRIPTION "A Modern C++ CFD Core Framework"
        LANGUAGES CXX)

# Define the library target. 
# We explicitly state STATIC here, but leaving it blank allows 
# the BUILD_SHARED_LIBS variable to control it globally.
add_library(CFDCore STATIC)

# Inform CMake about the source files (equivalent to Make/files).
# We use PRIVATE here because the source files are internal to compiling the target.
target_sources(CFDCore PRIVATE
    src/Field/Field.cpp
    src/GeometricField/GeometricField.cpp
)

# Tell the target where its headers are (equivalent to Make/options EXE_INC).
# PUBLIC means: use this directory to compile CFDCore AND pass this directory 
# to any target that links to CFDCore (so they can #include the headers too).
target_include_directories(CFDCore PUBLIC
    $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
    $<INSTALL_INTERFACE:include>
)

# Require C++20 standard. This replaces raw flags like -std=c++20.
# The compiler flag is handled completely abstractly by CMake across MSVC/GCC/Clang.
target_compile_features(CFDCore PUBLIC cxx_std_20)

# Apply warning flags, but ONLY for the compilers we know support them.
# This ensures we don't pass MSVC flags to GCC or vice versa.
if(CMAKE_CXX_COMPILER_ID MATCHES "GNU|Clang")
    target_compile_options(CFDCore PRIVATE -Wall -Wextra -Wpedantic)
endif()
```

Look at how explicit the CMake version is. It explicitly documents the required CMake version, the language expected, the visibility of the includes, the specific C++ standard required across platforms, and conditionally adds compiler flags.

Let's break down the C++ and build system mechanics of how these modifiers actually work in practice.

## Part 3: CMake Visibility Mechanics Explained

### The `PUBLIC`, `PRIVATE`, `INTERFACE` Keywords

The single most important concept in Modern CMake is understanding visibility modifiers. Every `target_*` command (`target_link_libraries`, `target_include_directories`, `target_compile_options`, `target_compile_features`) requires one of these three keywords. They control exactly how properties propagate (or "flow") through your build graph from dependencies to dependents.

#### 1. `PRIVATE` (Self only)
The property applies **only** to the target itself.
- **Usage:** If `CFDCore` uses `src/internal_memory_pool.hpp` to implement its internals in a `.cpp` file, but downstream users of `CFDCore` shouldn't see or use this header, include the path as `PRIVATE`.
- **Source files:** `target_sources` are almost always `PRIVATE`. The downstream executable linking to `CFDCore` doesn't need to compile `Field.cpp` again; it just needs the compiled binary `.a`/`.so` library and the public headers.

#### 2. `INTERFACE` (Dependents only)
The property applies **only** to downstream targets that link against this target.
- **Usage:** If you have a header-only library (like our template-heavy `Field<T>`), there are no `.cpp` files to compile into a static or shared library artifact. The library itself executes NO compilation steps (nothing to compile `PRIVATE`ly). Therefore, the include paths, C++ standard features, and macro definitions are purely an `INTERFACE` for *other* targets to consume.
- **Keywords:** `add_library(MyHeaderLib INTERFACE)` creates a virtual target that produces no binary output but carries properties (like include paths or C++ standards) to anything that `target_link_libraries(myapp PRIVATE MyHeaderLib)`.

#### 3. `PUBLIC` (Self + Dependents)
The property applies to the target itself **and** anything that links against it. It is effectively `PRIVATE` + `INTERFACE`.
- **Usage:** If `CFDCore` needs `<cmath>` to compile its `.cpp` files, and its public `GeometricField.hpp` header *also* includes `<cmath>` or custom headers from `include/`, those include paths must be `PUBLIC`. Both the library itself while compiling, and the downstream user while compiling, need to find those files.

### Generator Expressions (`$<...:...>`)

In the previous block, you saw this bizarre syntax inside the `target_include_directories` command:
```cmake
target_include_directories(CFDCore PUBLIC
    $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
    $<INSTALL_INTERFACE:include>
)
```

These are **Generator Expressions**. They are evaluated very late in the CMake generation phase. They solve a critical problem regarding pathing: *where are the headers located depending on the context?*

- While building from source in your development workspace, the headers are physically sitting in `my_cfd_project/include` on your hard drive. But if another project downloads your source code, the absolute path to `my_cfd_project/include` on *your* computer is meaningless to *their* computer.
- `$<BUILD_INTERFACE:...>` tells CMake: "When compiling this project from source in my local workspace, use the exact path `${CMAKE_CURRENT_SOURCE_DIR}/include`."
- `$<INSTALL_INTERFACE:...>` tells CMake: "If someone officially installs this library to their system (e.g., placing the `.so` inside `/usr/local/lib` and headers inside `/usr/local/include/CFDCore`), the headers will be located simply in the `include` folder relative to the installation prefix."

This ensures that your project functions correctly both as a local sub-project (via `add_subdirectory`) and as an installed system-wide dependency (via `find_package`).

### The Two-Step Configure and Build Process

CMake operates in two completely distinct phases: **Generation/Configuration** and **Building**.

This maps somewhat to how compilers work under the hood.

1. **Configuration (Generation) Phase:**
   CMake reads your `CMakeLists.txt`, calculates the dependency graph (which target `PUBLIC` flags flow into which other targets), searches your system for requested dependencies (like checking if you have `g++` or `clang++`), and writes a complete build map. 
   - By default on Linux, this output format is a giant `Makefile`.
   - We highly recommend appending `-G Ninja` to configure with the Ninja build system, which is significantly faster than GNU Make.

2. **Build Phase:**
   CMake steps back completely. It invokes the underlying build tool (Make, Ninja, MSBuild) to actually execute the compilation of the C++ files according to the generated map.

### Out-of-Source Builds and Workspace Hygiene

With `wmake`, object files (`.o`) are scattered directly into the source tree inside an architecture-specific directory, usually heavily nested like `Make/linux64GccDPInt32Opt`. If you compile for multiple environments, or just switch compilers, the source tree becomes dangerously polluted with object files. You rely heavily on `wclean` to wipe them.

CMake violently opposes this pattern and heavily enforces **out-of-source builds**. You never run CMake directly in the source directory. Instead, you create an isolated build directory.

```bash
# 1. Create a dedicated folder for all generated files
mkdir build
cd build

# 2. Configure/Generate the build system (pointing back to the source dir `..`)
cmake -G Ninja ..

# 3. Actually compile the code (CMake delegates to Ninja automatically)
cmake --build .
```

All generated files, intermediate objects (`.o`), compilation databases, and final binaries (`.a`, `.so`, executables) live entirely inside the `build/` directory. Your source code directory remains pristine. If the build state ever becomes corrupt or confusing, you do not run a complicated "clean" command; you simply `rm -rf build/` and start over perfectly clean in less than a second.

## Part 4: Implementation Exercise

Let's put this into practice by writing an end-to-end, robust `CMakeLists.txt` to build the `Field<T>` library you created in Phase 1. 

Since `Field<T>` uses aggressive C++20 templates (notably the `std::is_arithmetic_v` or `std::integral` concepts) and expression templates, all of the implementation logic is inside the `.hpp` header files themselves. The compiler has no `.cpp` files to statically compile into a library blob beforehand. 

Therefore, we will structure the CFD Core as an `INTERFACE` library. We will then build a small application executable to link against it and test it.

### Step 1: Project Structure Setup

Create the following file structure on your hard drive to represent a standard modern C++ repository format. Notice how cleanly the headers (`include/`) are separated from the implementations (`src/`).

```text
modern_cfd/
├── CMakeLists.txt
├── include/
│   └── Field.hpp
└── src/
    └── main.cpp
```

### Step 2: Source Code Components

Create the files. First, we write a highly simplified version of our generic `Field<T>`, using a modern C++20 concept to guarantee type safety at compile time.

**`include/Field.hpp`**
```cpp
// include/Field.hpp
#pragma once

#include <vector>
#include <numeric>
#include <stdexcept>
#include <concepts>

// A simple C++20 concept to ensure numeric safety.
// This prevents anyone from making a Field<std::string>.
template<typename T>
concept NumericFieldType = std::is_arithmetic_v<T>;

template<NumericFieldType T>
class Field {
private:
    std::vector<T> data_;

public:
    // Construct initialized to a specific value
    explicit Field(std::size_t n, const T& val = T{}) : data_(n, val) {}
    
    // Bounds-checked access
    std::size_t size() const { return data_.size(); }
    
    T& operator[](std::size_t i) { return data_.at(i); }
    const T& operator[](std::size_t i) const { return data_.at(i); }
    
    // Vectorized reduction using std::reduce from <numeric>
    T sum() const {
        // std::reduce is heavily optimizable and parallelizable
        return std::reduce(data_.begin(), data_.end(), T{});
    }

    // Parallel-safe field arithmetic addition operator
    Field operator+(const Field& rhs) const {
        if (size() != rhs.size()) {
            throw std::invalid_argument("Field addition size mismatch");
        }
        Field result(size());
        for (std::size_t i = 0; i < size(); ++i) {
            result[i] = data_[i] + rhs[i];
        }
        return result;
    }
};

// OpenFOAM heavily relies on type aliases for common components.
// We replace 'typedef Field<double> scalarField;' with the 'using' syntax.
using scalarField = Field<double>;
```

Next, the test executable to verify the library compiles and links properly, utilizing our aliases and operators.

**`src/main.cpp`**
```cpp
// src/main.cpp
#include "Field.hpp"
#include <iostream>

int main() {
    std::cout << "--- Modern CMake Field Test ---\n";
    
    try {
        // Construct a Pressure field of size 100 cells, at 1 atm (101325 Pa)
        scalarField pressure(100, 101325.0);
        
        // Construct a delta change field of 100.0 Pa everywhere
        scalarField dp(100, 100.0);
        
        // Execute field arithmetic (invokes Field::operator+)
        scalarField pNew = pressure + dp;

        std::cout << "Target element [50]: " << pNew[50] << " Pa\n";
        std::cout << "Field domain size:   " << pNew.size() << " Cells\n";
        std::cout << "Global domain sum:   " << pNew.sum() << "\n";
    } catch (const std::exception& e) {
        std::cerr << "Fatal Error: " << e.what() << "\n";
        return 1;
    }
    
    return 0;
}
```

### Step 3: The `CMakeLists.txt` Implementation

Write the following CMake configuration file precisely at the root `modern_cfd/` directory. Read the comments carefully, as they map back to the visibility rules discussed in Part 3.

```cmake
# CMakeLists.txt
cmake_minimum_required(VERSION 3.20...3.28)
project(CFDCore 
        VERSION 1.0.0 
        DESCRIPTION "Modern C++ Core Library replacing wmake"
        LANGUAGES CXX)

# ==========================================
# 1. Define the Field library Target
# ==========================================
# Since Field<T> is heavily templated, it's a header-only library.
# We declare it as an INTERFACE library because there are absolutely no 
# .cpp files to compile for this target.
add_library(FieldLib INTERFACE)

# Define its include directories.
# We use INTERFACE visibility because downstream targets need to know 
# where headers are to `#include "Field.hpp"`. Generator expressions 
# isolate the local build from an installed system build.
target_include_directories(FieldLib INTERFACE
    $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
)

# Require C++20 for this library (concepts, ranges, etc.)
# Any target linking to FieldLib will automatically upgrade to C++20!
target_compile_features(FieldLib INTERFACE cxx_std_20)


# ==========================================
# 2. Define the Executable Target
# ==========================================
# We define an executable target and provide its source files.
add_executable(TestField src/main.cpp)

# Enable aggressive warnings for our test executable to enforce code quality
if(CMAKE_CXX_COMPILER_ID MATCHES "GNU|Clang")
    # PRIVATE means these warning flags ONLY apply to compiling TestField itself.
    target_compile_options(TestField PRIVATE -Wall -Wextra -Wpedantic -Wconversion)
endif()

# Link the executable against the FieldLib interface library.
# This single command automatically injects the include paths
# and the C++20 standard compile feature requirement into the executable.
# Without this line, the executable would fail to find "Field.hpp".
target_link_libraries(TestField PRIVATE FieldLib)
```

Open your terminal in the `modern_cfd/` directory. Execute the CMake generation and build steps exactly like this. Notice how clean it is.

```bash
# You can append `-G Ninja` if you have Ninja installed for much faster speeds.
cmake -S . -B build

# 2. Compile the project. This invokes make or ninja automatically under the hood.
cmake --build build -j 4

# 3. Run the compiled executable from the build directory.

```

## Part 5: Summary and Trade-offs

In the late 1990s and 2000s, OpenFOAM's custom `wmake` system was absolutely essential to coordinate thousands of files across wildly different supercomputer architectures efficiently. But for a brand new CFD code built in 2026, using anything other than CMake is an architectural mistake that cripples developer velocity and limits recruitment to only those who already know your proprietary tools.

By abandoning `wmake` and adopting target-based Modern CMake, you gain massive advantages:

1. **Flawless IDE Tooling:** Because CMake generates industry-standard `compile_commands.json` files, any modern IDE (VS Code, CLion, Neovim LSP) will instantly understand your project structure. You automatically get perfect code completion, rapid jump-to-definition across huge libraries, and live inline warning analysis.
2. **Encapsulation of Dependencies:** In legacy systems (and `wmake`), dependencies are wildly unconstrained global strings. In Modern CMake, targets perfectly manage their own requirements via `PUBLIC`, `PRIVATE`, and `INTERFACE` modifiers. If an executable links against an interface library, all needed flags naturally traverse the dependency graph to the executable without developer intervention.
3. **Out-of-source Workspace Hygiene:** Your `src/` directory never gets dirtied with `.o` object files. Code lives absolutely isolated from compiler artifacts.
4. **Third-Party Integration:** Building external dependencies (which we will cover on Day 39) like `nlohmann/json` or `Google Test` becomes an integrated part of the build pipeline via `FetchContent`, completely automating the acquisition of heavily used tooling.

## Part 6: Imported Targets and Interface Libraries

### INTERFACE Libraries for Header-Only Code

Template libraries like `Field<T>` have all implementation in headers — nothing to compile. In Modern CMake, package these as `INTERFACE` targets:

```cmake
# Define a header-only library — no source files
add_library(field INTERFACE)

target_include_directories(field
    INTERFACE ${CMAKE_CURRENT_SOURCE_DIR}/include
)

target_compile_features(field
    INTERFACE cxx_std_20
)

# Consumer gets -I<include_dir> -std=c++20 automatically
add_executable(TestField tests/test_field.cpp)
target_link_libraries(TestField PRIVATE field)
```

`INTERFACE` scope means: "this requirement belongs to consumers, not to this target." Since there are no source files to compile, `PRIVATE` scope would have no effect.

### Imported Targets: `find_package` Integration

When depending on installed third-party libraries, Modern CMake uses **imported targets**:

```cmake
# Find Eigen3 (installed via apt, Homebrew, or vcpkg)
find_package(Eigen3 3.4 REQUIRED NO_MODULE)

add_executable(SolverApp src/main.cpp)
target_link_libraries(SolverApp
    PRIVATE
        field          # our INTERFACE target
        Eigen3::Eigen  # imported target from Eigen's CMake config
)
```

The `::` namespace convention signals an imported target. If the package was not found, CMake errors immediately with a clear message rather than passing an empty string to the linker.

`NO_MODULE` forces CMake to find Eigen's own `Eigen3Config.cmake` (accurate) rather than CMake's bundled `FindEigen3.cmake` (may be outdated).

### Generating `compile_commands.json`

This single line enables perfect IDE integration:

```cmake
set(CMAKE_EXPORT_COMPILE_COMMANDS ON)
```

After `cmake -S . -B build`, the file `build/compile_commands.json` contains the exact compiler invocation for every translation unit. Tools that read it: `clangd`, `ccls`, `clang-tidy`, VS Code C/C++ extension, CLion.

```bash
# Symlink to project root for IDE discovery
ln -sf build/compile_commands.json compile_commands.json
```

### Quick Reference: Key Modern CMake Commands

| Command | Scope | What it does |
|---------|-------|-------------|
| `target_include_directories(T S dirs)` | PUBLIC/PRIVATE/INTERFACE | Adds `-I` flags |
| `target_compile_features(T S features)` | PUBLIC/PRIVATE/INTERFACE | Enforces C++ standard |
| `target_compile_options(T S opts)` | PUBLIC/PRIVATE/INTERFACE | Passes raw compiler flags |
| `target_link_libraries(T S libs)` | PUBLIC/PRIVATE/INTERFACE | Links libraries + propagates deps |
| `target_compile_definitions(T S defs)` | PUBLIC/PRIVATE/INTERFACE | Adds `-D` macros |
| `add_library(T INTERFACE)` | — | Header-only target, no sources |
| `find_package(Pkg VER REQUIRED)` | — | Locates installed package |
| `FetchContent_MakeAvailable(Pkg)` | — | Downloads dep at configure time (Day 39) |

**Scope rules:** PRIVATE = only this target. PUBLIC = this target + all consumers. INTERFACE = consumers only (header-only targets always use INTERFACE).

### CMake Minimum Version Requirements

```cmake
# Required for C++20 module support
cmake_minimum_required(VERSION 3.28)

# Required for FetchContent (Day 39)
cmake_minimum_required(VERSION 3.14)

# Required for imported targets in find_package
cmake_minimum_required(VERSION 3.10)

# Recommended baseline for this course
cmake_minimum_required(VERSION 3.20)
```

Always use `cmake_minimum_required(VERSION 3.20)` as the project baseline. This version provides stable support for imported targets, `FetchContent`, `target_*` commands, and `compile_commands.json` export. Older versions have inconsistent behavior for scope propagation.

### Out-of-Source Build Workflow

```bash
# Configure (generates Makefiles or Ninja build files in build/)
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release

# Build (runs make or ninja automatically)
cmake --build build -j$(nproc)

# Run tests (if configured with enable_testing())
ctest --test-dir build

# Clean build
rm -rf build/   # No partial clean needed — just nuke and reconfigure
```

The `cmake -S . -B build` pattern keeps source and build directories completely separate. Contrast with OpenFOAM's `wmake` which writes `.o` files into the source tree, making `git status` noisy and forcing `wclean` before switching configurations.

> **Connection to Day 30:** Day 30 extends this foundation to shared library targets (`SHARED` vs `STATIC`) and install rules. The INTERFACE/PRIVATE/PUBLIC scope system introduced today becomes critical for correctly propagating flags across shared library boundaries.

### When to Use Each Library Type

| Type | Command | Compiled? | When to use |
|------|---------|-----------|-------------|
| INTERFACE | `add_library(T INTERFACE)` | No | Header-only (templates, concepts) |
| STATIC | `add_library(T STATIC src.cpp)` | Yes → `.a` | Fast link, single executable |
| SHARED | `add_library(T SHARED src.cpp)` | Yes → `.so` | Plugins, runtime linking, ABI stability |
| OBJECT | `add_library(T OBJECT src.cpp)` | Yes → `.o` | Shared object files between targets |

**Deliverable:** A complete `CMakeLists.txt` file that defines the template-based `Field<T>` as an `INTERFACE` library target. It enforces C++20 standard requirements automatically, builds an executable linking against it without manual include paths, and runs successfully out-of-source using standard `cmake` CLI commands, totally replacing proprietary `wmake` macros.
