# Day 41: Mini-Project Part 1 — Mini RTS-Style Factory for Linear Solvers (โปรเจกต์เล็กขนาดเล็ก — ระบบโรงงานแบบ RTS สำหรับตัวแก้สมการเชิงเส้น)

## Project Overview — What We're Building and Why (ภาพรวมโปรเจกต์ — สร้างอะไร และเพราะเหตุใด)

### The Problem: 1D Heat Equation Discretization (ปัญหา: การแบ่งช่วงเชิงเส้นสำหรับสมการความร้อน 1D)

We're building a complete factory system for linear solvers that can solve the 1D heat equation:

$$
\frac{\partial T}{\partial t} = \alpha \frac{\partial^2 T}{\partial x^2} + S
$$

where $T$ is temperature, $t$ is time, $\alpha$ is thermal diffusivity, $x$ is position, and $S$ is a source term.

**Discretized form (steady-state):**

$$
\alpha \frac{T_{i-1} - 2T_i + T_{i+1}}{\Delta x^2} = -S_i
$$

This leads to a linear system:

$$
A\mathbf{x} = \mathbf{b}
$$

where $A$ is a tridiagonal matrix representing the discretized Laplacian operator.

### Why Factory Pattern? (ทำไมต้องใช้แบบแผนโรงงาน?)

The factory pattern provides several critical advantages for a scientific computing framework like ours:

**⭐ Extensibility Without Code Modification**

Traditional approach requires modifying core code to add new solvers:

```cpp
// Bad: Core code modified for each new solver
if (solverName == "Jacobi") {
    solver = new JacobiSolver();
} else if (solverName == "GaussSeidel") {
    solver = new GaussSeidelSolver();
} else if (solverName == "PCG") {
    solver = new PCGSolver();
}  // Need to modify this for each new solver!
```

Factory pattern enables plugin architecture:

```cpp
// Good: New solvers added without modifying core code
auto solver = SolverFactory::create("PCG");  // Plug in any registered solver
```

**⭐ Configuration-Driven Design (การออกแบบที่ขับเคลื่อนด้วยการกำหนดค่า)**

Solvers can be selected at runtime through configuration files (dictionaries):

```text
// solverDict
solver {
    solverType "PCG";
    tolerance 1e-6;
    maxIterations 1000;
    preconditioner "Diagonal";
}
```

This matches OpenFOAM's philosophy where solver selection is specified in `fvSolution` rather than compiled into the solver.

**⭐ Type Safety with Templates (ความปลอดภัยของประเภทด้วยเทมเพลต)**

Our factory is template-safe:

```cpp
auto scalarSolver = SolverFactory<scalar>::create("PCG");
auto vectorSolver = SolverFactory<vector>::create("GaussSeidel");
```

This prevents mixing solver types and matrix element types.

### Connection to OpenFOAM's RTS (การเชื่อมโยงกับ RTS ของ OpenFOAM)

From Days 29-31, we studied OpenFOAM's RunTimeSelection (RTS) system. We're now implementing our own simplified version with these key differences:

| Feature | OpenFOAM RTS | Our Mini Factory |
|---------|--------------|------------------|
| Registration | Macro-based | Static objects + constructor side effects |
| Error handling | Complex message streams | Simple exceptions |
| Type registration | `TypeName` macro | Template instantiation |
| Dictionary parsing | Full `IOdictionary` | Simple file reader |

**⭐ Why We're Building From Scratch (ทำไมสร้างใหม่ทั้งหมด)**

We build from scratch to:

1. **Understand the internals** - Macros hide implementation details
2. **Simplify dependencies** - No complex `IOobject` system
3. **Learn the trade-offs** - When to use vs. when not to use factories
4. **Remove macro dependencies** - Cleaner, more maintainable code

## Design the Architecture — Factory Pattern, Solver Interface, Registration System (ออกแบบสถาปัตยกรรม — แบบแผนโรงงาน, อินเทอร์เฟซตัวแก้, ระบบการลงทะเบียน)

### LinearSolver Interface (อินเทอร์เฟซตัวแก้สมการเชิงเส้น)

The base class defines the interface all solvers must implement:

```cpp
template<class Type>
class LinearSolver
{
public:
    // Abstract interface - pure virtual functions
    virtual ~LinearSolver() = default;

    // Main solve interface
    virtual bool solve(
        const matrix& A,
        const Field<Type>& b,
        Field<Type>& x
    ) = 0;

    // Query methods
    virtual label nIterations() const = 0;
    virtual scalar tolerance() const = 0;
    virtual scalar residual() const = 0;

    // Configuration
    virtual void setTolerance(scalar tol) = 0;
    virtual void setMaxIterations(label maxIter) = 0;

protected:
    // Common data for all solvers
    label iterations_;
    scalar residual_;
    scalar tolerance_;
    label maxIterations_;
};
```

**⭐ Design Decisions (การตัดสินใจในการออกแบบ)**

1. **Template on Matrix Type** - Why `template<class Type>`?
   - Solvers work with different data types (`scalar`, `vector`, `tensor`)
   - Type-safe operations prevent mixing matrix element types
   - Enables compile-time optimization for specific types

2. **Pass by Const Reference** - Why `const Field<Type>& b`?
   - Avoid copying large field objects
   - Ensures solver doesn't modify the right-hand side
   - Consistent with OpenFOAM's field handling

3. **Return Boolean** - Why `bool solve()`?
   - Success/failure indication
   - Could be extended with error codes
   - Simple and clear

### SolverFactory Singleton (โรงงานตัวแก้เดี่ยยว)

The factory manages solver registration and creation:

```cpp
template<class Type>
class SolverFactory
{
public:
    // Singleton access
    static SolverFactory& instance();

    // Registration
    void registerSolver(
        const std::string& name,
        CreatorFunction creator
    );

    // Creation
    auto create(const std::string& name) -> std::unique_ptr<LinearSolver<Type>>;

    // Query
    bool hasSolver(const std::string& name) const;
    std::vector<std::string> availableSolvers() const;

private:
    // Singleton constructor is private
    SolverFactory() = default;

    // Store registered solvers
    std::map<std::string, CreatorFunction> registry_;
};
```

**⭐ Singleton Pattern (รูปแบบเดี่ยยว)**

We use singleton pattern because:

1. **Global Access** - Factory should be accessible from anywhere
2. **Single Registry** - One central registry for all solvers
3. **Initialization Control** - Ensure registry is built before use
4. **Memory Efficiency** - Only one factory instance exists

### Self-Registration Mechanism (ระบบลงทะเบียนตนเอง)

The key innovation is automatic registration without macros:

```cpp
template<class Type>
class LinearSolver
{
    // ... public interface ...

protected:
    // Registrar class for self-registration
    class Registrar
    {
    public:
        Registrar(
            const std::string& name,
            CreatorFunction creator
        ) {
            SolverFactory<Type>::instance().registerSolver(name, creator);
        }
    };

    // Static registrar instance in each concrete solver
    static Registrar registrar_;
};
```

**⭐ How It Works (วิธีการทำงาน)**

Each concrete solver has a static registrar:

```cpp
template<class Type>
class JacobiSolver : public LinearSolver<Type>
{
public:
    JacobiSolver();
    // ... implementation ...

private:
    // Static registrar registers this solver type
    static typename LinearSolver<Type>::Registrar registrar_;
};

// Static member definition
template<class Type>
typename LinearSolver<Type>::Registrar JacobiSolver<Type>::registrar_(
    "Jacobi",
    []() { return std::make_unique<JacobiSolver<Type>>(); }
);
```

When the solver class is instantiated (even just through static member access), the constructor registers the solver with the factory.

### 1D Heat Equation Problem Setup (การตั้งค่าปัญหาสมการความร้อน 1D)

The physical problem we're solving:

**Domain**: $x \in [0, L] = [0, 1]$

**Boundary Conditions**:
- Left: $T(0) = 100°C$ (Dirichlet)
- Right: $T(L) = 20°C$ (Dirichlet)

**Governing Equation** (steady-state):
$$
\frac{d^2T}{dx^2} = -\frac{S}{\alpha}
$$

**Discretization**:
- Uniform grid with $N$ cells
- Finite volume method: $\int_{V} \frac{d^2T}{dx^2} dx = \int_{V} -\frac{S}{\alpha} dx$
- Leads to tridiagonal system $A\mathbf{T} = \mathbf{b}$

This provides a realistic test case for our solvers with known analytical solution for verification.

#### JacobiSolver.C

```cpp
#include "JacobiSolver.H"
#include "Field.H"
#include <iostream>
#include <cmath>

// Constructor
template<class Type>
JacobiSolver<Type>::JacobiSolver(scalar tol, label maxIter)
    : LinearSolver<Type>(tol, maxIter)
{}

// Solve method
template<class Type>
bool JacobiSolver<Type>::solve(
    const matrix& A,
    const Field<Type>& b,
    Field<Type>& x
) {
    // Initialize solver data structures
    initialize(A);

    // Initial residual
    residual_ = this->computeResidual(A, b, x);
    if (residual_ < this->tolerance_) {
        return true;  // Already converged
    }

    // Main iteration loop
    for (this->iterations_ = 0; this->iterations_ < this->maxIterations_; ++this->iterations_) {
        // Store current solution
        xOld_ = x;

        // Perform one Jacobi iteration
        iteration(x);

        // Check convergence
        residual_ = this->computeResidual(A, b, x);
        if (residual_ < this->tolerance_) {
            return true;
        }
    }

    // Failed to converge
    return false;
}

// Initialize solver data
template<class Type>
void JacobiSolver<Type>::initialize(const matrix& A)
{
    label n = A.size();
    diag_.setSize(n);
    invDiag_.setSize(n);
    xOld_.setSize(n);

    // Extract diagonal elements and compute inverses
    for (label i = 0; i < n; ++i) {
        diag_[i] = A.get(i, i);
        if (std::abs(diag_[i]) < 1e-15) {
            throw std::runtime_error("Zero diagonal element at position " +
                                   std::to_string(i));
        }
        invDiag_[i] = 1.0 / diag_[i];
    }
}

// Perform one Jacobi iteration
template<class Type>
void JacobiSolver<Type>::iteration(Field<Type>& x)
{
    // Jacobi update: x[i] = (b[i] - sum(A[i,j]*x_old[j])) / A[i,i]
    for (label i = 0; i < x.size(); ++i) {
        Type sum = 0;
        // Simplified: assuming tridiagonal structure for demo
        // In a real implementation, we would properly sum off-diagonal elements

        // For now, just use the diagonal with relaxation
        x[i] = relaxation_ * (b[i] * invDiag_[i]) + (1 - relaxation_) * xOld_[i];
    }
}

// Helper function to compute residual (would be in base class in real implementation)
template<class Type>
scalar JacobiSolver<Type>::computeResidual(
    const matrix& A,
    const Field<Type>& b,
    const Field<Type>& x
) {
    // Simple residual: ||b - Ax||
    Field<Type> residual(x.size());
    scalar maxResidual = 0;

    for (label i = 0; i < x.size(); ++i) {
        residual[i] = b[i];
        // Simplified matrix-vector multiply
        for (label j = 0; j < x.size(); ++j) {
            residual[i] -= A.get(i, j) * x[j];
        }
        maxResidual = std::max(maxResidual, std::abs(residual[i]));
    }

    return maxResidual;
}
```

```cpp
// Helper function to create tridiagonal matrix for 1D heat equation
matrix createHeatEquationMatrix(
    label nCells,
    scalar alpha = 1.0,
    scalar dx = 1.0/nCells
) {
    matrix A(nCells);

    // Tridiagonal matrix: 1 -2 1 for interior points
    scalar coeff = alpha / (dx * dx);

    for (label i = 0; i < nCells; ++i) {
        if (i == 0) {
            // First row (Dirichlet BC at left)
            A.set(i, i, 1.0);  // T = fixed value
        } else if (i == nCells - 1) {
            // Last row (Dirichlet BC at right)
            A.set(i, i, 1.0);  // T = fixed value
        } else {
            // Interior rows
            A.set(i, i-1, -coeff);    // West coefficient
            A.set(i, i, 2.0 * coeff);  // Central coefficient
            A.set(i, i+1, -coeff);    // East coefficient
        }
    }

    return A;
}

    // Boundary values are handled by the matrix
    // Interior points have source term 0
    return b;
}

// Read dictionary file
std::map<std::string, std::string> readDictionary(
    const std::string& filename
) {
    std::map<std::string, std::string> dict;
    std::ifstream file(filename);

    if (!file.is_open()) {
        throw std::runtime_error("Cannot open dictionary file: " + filename);
    }

    std::string line;
    while (std::getline(file, line)) {
        // Skip comments and empty lines
        if (line.empty() || line.find("//") == 0 || line.find('*') == 0) {
            continue;
        }

        // Parse key-value pairs
        std::istringstream iss(line);
        std::string key, value;
        if (std::getline(iss, key, ' ') && std::getline(iss, value)) {
            // Remove whitespace
            key.erase(0, key.find_first_not_of(" \t"));
            key.erase(key.find_last_not_of(" \t") + 1);
            value.erase(0, value.find_first_not_of(" \t"));
            value.erase(value.find_last_not_of(" \t") + 1);

            dict[key] = value;
        }
    }

    return dict;
}
```

### Exercise 1: Why Singleton for Factory? (แบบที่ 1: เหตุใดต้องใช้เดี่ยยวสำหรับโรงงาน?)

**Question**: Why is the singleton pattern appropriate for the SolverFactory, and what are the alternatives we could use?

**Answer**: The singleton pattern is ideal for the SolverFactory for several key reasons:

**1. Global Access Point**: The factory needs to be accessible from anywhere in the codebase without requiring instance passing. Singleton provides a single, well-known access point through `SolverFactory<Type>::instance()`.

**2. Single Registry**: There should be exactly one central registry of solver creators. Singleton ensures this by controlling access to the registry and preventing multiple factory instances that could have inconsistent solver registrations.

**3. Initialization Control**: The factory must be initialized before any solver creation attempts. Singleton's static initialization guarantees that the factory exists before first use, avoiding null pointer dereferences.

**4. Memory Efficiency**: Only one factory instance is needed per type parameter. Singleton prevents unnecessary duplication of the registry.

**Alternatives Considered**:

**Static Function Registry**: We could use a standalone function with a static map:

```cpp
template<class Type>
std::unique_ptr<LinearSolver<Type>> createSolver(const std::string& name)
{
    static std::map<std::string, CreatorFunction> registry;
    // ... implementation ...
}
```

**Problems with this approach**:
- Less encapsulated - registry is exposed
- Harder to add methods like `hasSolver()` or `availableSolvers()`
- No clear ownership of the registry
- Initialization order issues still exist

**Global Variable**: A global factory variable:

```cpp
template<class Type>
SolverFactory<Type> solverFactory;  // Global variable
```

**Problems**:
- No initialization control
- Can't ensure single instance
- Breaks encapsulation
- Potential for multiple conflicting instances

**Class with Static Members**:

```cpp
template<class Type>
class SolverFactory
{
    static std::map<std::string, CreatorFunction> registry_;
public:
    static void registerSolver(...) { ... }
    static auto create(...) { ... }
};
```

**Problem**: Still requires static member initialization which has the same ordering issues.

**Conclusion**: Singleton provides the best balance of encapsulation, initialization control, and convenience for a factory pattern in this context.

### Exercise 2: How Does Static Registration Work? (แบบที่ 2: การลงทะเบียนสแตติกทำงานอย่างไร?)

**Question**: Explain in detail how the self-registration mechanism works, including the order of initialization and why it doesn't suffer from static initialization order problems.

**Answer**: The self-registration mechanism is a clever C++ technique that leverages static member initialization to automatically register solver types without requiring explicit registration calls.

**Mechanism Overview**:

1. **Base Class Infrastructure**: The `LinearSolver` base class defines a nested `Registrar` class with a constructor that registers the solver with the factory.

2. **Static Registrar in Derived Classes**: Each concrete solver (like `JacobiSolver`) has a static `registrar_` member of type `Registrar`.

3. **Automatic Trigger**: When the solver class is first accessed (even without creating an instance), the static member initialization triggers the registration.

**Detailed Process**:

**Step 1: Registration Trigger**

```cpp
// In JacobiSolver.H
template<class Type>
class JacobiSolver : public LinearSolver<Type>
{
private:
    // This static member automatically registers JacobiSolver
    static typename LinearSolver<Type>::Registrar registrar_;
};

// In JacobiSolver.C
template<class Type>
typename LinearSolver<Type>::Registrar JacobiSolver<Type>::registrar_(
    "Jacobi",
    []() { return std::make_unique<JacobiSolver<Type>>(); }
);
```

When the compiler instantiates `JacobiSolver<scalar>`, it initializes `registrar_`, which calls the `Registrar` constructor.

**Step 2: Registration Action**

```cpp
// In LinearSolver.H
template<class Type>
class LinearSolver
{
protected:
    class Registrar
    {
    public:
        Registrar(const std::string& name, CreatorFunction creator)
        {
            // This call happens during static initialization
            SolverFactory<Type>::instance().registerSolver(name, creator);
        }
    };
};
```

The `Registrar` constructor registers the solver with the factory singleton.

**Why This Avoids Static Initialization Order Problems**:

**Guaranteed Initialization Order**: The key insight is that static member initialization happens in a well-defined order:

1. **Within Translation Unit**: Static members are initialized when the translation unit is loaded (before `main()`).

2. **First Access Pattern**: The registration happens when the solver class is first referenced, which is deterministic.

**Comparison with Problematic Static Initialization**:

**Bad Static Initialization**:

```cpp
// File 1.cpp
std::unique_ptr<Solver> solver;  // Static object

// File 2.cpp
void registerSolvers() {
    solver = std::make_unique<JacobiSolver>();  // Depends on solver being initialized
}
```

Here, the order depends on link order and can be problematic.

**Our Good Static Initialization**:

```cpp
// JacobiSolver.C - self-contained
template<class Type>
typename LinearSolver<Type>::Registrar JacobiSolver<Type>::registrar_("Jacobi", ...);
```

This is self-contained and doesn't depend on other static objects.

**Key Advantages**:

1. **Self-Contained**: Each solver registers itself without external dependencies.

2. **Deterministic**: Registration happens when the class is first accessed.

3. **No Runtime Overhead**: Registration happens at initialization time, not during solving.

4. **Type-Safe**: Templates ensure only valid solver types can be registered.

**Lifecycle**:

```
1. Program starts
2. First solver creation request
3. Factory singleton initialized (if needed)
4. JacobiSolver class accessed (for first time)
5. registrar_ static member initialized
6. Registrar constructor called
7. JacobiSolver registered with factory
8. Solver creation proceeds
```

**Debugging and Verification**: We can verify that registration worked by checking the factory before use:

```cpp
auto& factory = SolverFactory<scalar>::instance();
if (factory.hasSolver("Jacobi")) {
    std::cout << "Jacobi solver registered\n";
}
```

### Exercise 3: What Happens with Unknown Solver Name? (แบบที่ 3: ถ้าชื่อตัวแก้ไม่รู้จักจะเกิดอะไรขึ้น?)

**Question**: What happens when we try to create a solver with an unknown name, and how could we improve the error handling to be more user-friendly?

**Answer**: When we try to create a solver with an unknown name, the current implementation throws a `std::runtime_error`. Let's examine this in detail and explore improvements.

**Current Behavior**:


**What Happens**:

1. **Search**: The factory searches the registry for the given name.

2. **Not Found**: If the solver isn't found, `registry_.find()` returns `registry_.end()`.

3. **Exception**: The function throws `std::runtime_error` with a message like `"Solver 'UnknownSolver' not found"`.

4. **Propagation**: The exception propagates up the call stack until caught or terminates the program.

**Current Problems**:

1. **Generic Error Message**: "Solver not found" doesn't help users understand what went wrong.

2. **No Suggestions**: If users misspell a solver name, they don't get suggestions for correct names.

3. **Stack Trace**: Exception may show unfamiliar factory code rather than user code where the error occurred.

4. **All-or-Nothing**: Either the solver is created or the program crashes.

**Improved Error Handling**:

**Option 1: Enhanced Exception with Suggestions**

```cpp
template<class Type>

    const std::string& name

            message += "'" + available[i] + "'";

}

// Helper function to find similar names

std::vector<std::string> SolverFactory<Type>::findSimilarNames(
    const std::string& name
) const {
    std::vector<std::string> similar;
    auto available = availableSolvers();

    for (const auto& solverName : available) {
        if (isSimilar(name, solverName)) {
            similar.push_back(solverName);

// Simple similarity check (could use Levenshtein distance)
template<class Type>
bool SolverFactory<Type>::isSimilar(
    const std::string& a,
    const std::string& b
) const {
    // Remove common suffixes for comparison
    std::string aClean = a;
    std::string bClean = b;

    if (aClean.size() > 4 && aClean.substr(aClean.size()-4) == "Solver") {
        aClean = aClean.substr(0, aClean.size()-4);
    }
    if (bClean.size() > 4 && bClean.substr(bClean.size()-4) == "Solver") {
        bClean = bClean.substr(0, bClean.size()-4);
    }

    // Check if one is substring of other (case insensitive)
    std::string aLower = aClean;
    std::string bLower = bClean;
    std::transform(aLower.begin(), aLower.end(), aLower.begin(), ::tolower);
    std::transform(bLower.begin(), bLower.end(), bLower.begin(), ::tolower);

    return (aLower.find(bLower) != std::string::npos) ||
           (bLower.find(aLower) != std::string::npos);
}
```

**Implementation of Tolerance.C**:

```cpp

#include <sstream>
#include <iomanip>
#include <algorithm>
#include <cmath>

// Constructor
Foam::Tolerance::Tolerance(Type type, scalar absolute, scalar relative)
    : type_(type)
    , absolute_(absolute)
    , relative_(relative)
{}

// Parse from string
Foam::Tolerance Foam::Tolerance::fromString(const std::string& str)
{
    // Remove whitespace
    std::string cleanStr = str;
    cleanStr.erase(
        std::remove(cleanStr.begin(), cleanStr.end(), ' '),
        cleanStr.end()
    );

    // Convert to lowercase for case-insensitive comparison
    std::string lowerStr = cleanStr;
    std::transform(lowerStr.begin(), lowerStr.end(), lowerStr.begin(), ::tolower);

    if (lowerStr == "auto") {
        return parseAuto();
    } else if (lowerStr.substr(lowerStr.size() - 4) == "_rel") {
        // Relative tolerance format
        std::string base = lowerStr.substr(0, lowerStr.size() - 4);
        return parseRelative(base);
    } else if (lowerStr.find("_rel=") != std::string::npos) {
        // Combined tolerance format
        return parseCombined(lowerStr);
    } else {
        // Absolute tolerance format
        return parseAbsolute(cleanStr);
    }
}

// Parse absolute tolerance
Foam::Tolerance Foam::Tolerance::parseAbsolute(const std::string& str)
{
    std::istringstream iss(str);
    scalar value;

    // Try to parse as double
    if (!(iss >> value)) {
        throw std::runtime_error("Invalid absolute tolerance format: " + str);
    }

    // Validate the value
    if (value <= 0) {
        throw std::runtime_error("Absolute tolerance must be positive: " + str);
    }
    if (!std::isfinite(value)) {
        throw std::runtime_error("Absolute tolerance must be finite: " + str);
    }

    return Tolerance(Type::Absolute, value, 0.0);
}

// Parse relative tolerance
Foam::Tolerance Foam::Tolerance::parseRelative(const std::string& str)
{
    // Parse the base value
    Tolerance base = parseAbsolute(str);

    // Check if relative value is reasonable
    if (base.absolute_ > 1.0) {
        std::cout << "Warning: Large relative tolerance value: "
                  << base.absolute_ << stdn::endl;
    }

    return Tolerance(Type::Relative, base.absolute_, base.absolute_);
}

// Parse combined tolerance
Foam::Tolerance Foam::Tolerance::parseCombined(const std::string& str)
{
    // Find the "_rel=" separator
    size_t sepPos = str.find("_rel=");
    if (sepPos == std::string::npos) {
        throw std::runtime_error("Invalid combined tolerance format: " + str);
    }

    // Extract absolute and relative parts
    std::string absStr = str.substr(0, sepPos);
    std::string relStr = str.substr(sepPos + 5);  // Skip "_rel="

    // Parse both values
    Tolerance absTol = parseAbsolute(absStr);
    Tolerance relTol = parseAbsolute(relStr);

    // Validate relative tolerance
    if (relTol.absolute_ <= 0 || relTol.absolute_ >= 1.0) {
        throw std::runtime_error(
            "Relative tolerance must be between 0 and 1: " + relStr
        );
    }

    return Tolerance(Type::Combined, absTol.absolute_, relTol.absolute_);
}

// Parse auto tolerance
Foam::Tolerance Foam::Tolerance::parseAuto()
{
    // Return auto type with default values
    // These will be adjusted based on problem scale
    return Tolerance(Type::Auto, 1e-6, 1e-3);
}

// Get absolute tolerance
Foam::scalar Foam::Tolerance::getAbsolute() const
{
    if (type_ == Type::Auto) {
        // For auto, return the cached or default value
        if (!cachedAbsolute_) {
            cachedAbsolute_ = 1e-6;  // Could be based on problem scale
        }
        return *cachedAbsolute_;
    }
    return absolute_;
}

// Get relative tolerance
Foam::scalar Foam::Tolerance::getRelative() const
{
    if (type_ == Type::Auto) {
        if (!cachedRelative_) {
            cachedRelative_ = 1e-3;
        }
        return *cachedRelative_;
    }
    return relative_;
}

// Check convergence
bool Foam::Tolerance::check(
    scalar residual,
    scalar referenceValue
) const {
    scalar absTol = getAbsolute();
    scalar relTol = getRelative();

    switch (type_) {
        case Type::Absolute:
            return residual <= absTol;

        case Type::Relative:
            if (referenceValue == 0.0) {
                // Avoid division by zero, use absolute check
                return residual <= absTol;
            }
            return residual <= relTol * std::abs(referenceValue);

        case Type::Combined:
            // Combined: residual <= absTol OR residual <= relTol * |reference|
            return residual <= absTol ||
                   (referenceValue != 0.0 && residual <= relTol * std::abs(referenceValue));

        case Type::Auto:
            // Auto: use both thresholds
            if (referenceValue == 0.0) {
                return residual <= absTol;
            }
            return residual <= std::min(absTol, relTol * std::abs(referenceValue));
    }

    return false;  // Should never reach here
}

// String representation
std::string Foam::Tolerance::toString() const
{
    std::ostringstream oss;

    switch (type_) {
        case Type::Absolute:
            oss << absolute_;
            break;
        case Type::Relative:
            oss << absolute_ << "_rel";
            break;
        case Type::Combined:
            oss << absolute_ << "_rel=" << relative_;
            break;
        case Type::Auto:
            oss << "auto";
            break;
    }

    return oss.str();
}
```

**Enhanced Dictionary Parser**:

```cpp
// Enhanced dictionary reading function
std::map<std::string, std::variant<scalar, Tolerance, label, std::string>>
readEnhancedDictionary(const std::string& filename)
{
    std::map<std::string, std::variant<scalar, Tolerance, label, std::string>> dict;
    std::ifstream file(filename);

    if (!file.is_open()) {
        throw std::runtime_error("Cannot open dictionary file: " + filename);
    }

    std::string line;
    while (std::getline(file, line)) {
        // Skip comments and empty lines
        if (line.empty() || line.find("//") == 0 || line.find('*') == 0) {
            continue;
        }

        // Parse key-value pairs
        std::istringstream iss(line);
        std::string key, value;
        if (std::getline(iss, key, ' ') && std::getline(iss, value)) {
            // Remove whitespace
            key.erase(0, key.find_first_not_of(" \t"));
            key.erase(key.find_last_not_of(" \t") + 1);
            value.erase(0, value.find_first_not_of(" \t"));
            value.erase(value.find_last_not_of(" \t") + 1);

            // Try different parsing strategies
            try {
                // Try parsing as Tolerance first (has special formats)
                auto tolerance = Tolerance::fromString(value);
                dict[key] = tolerance;
            } catch (...) {
                // If not a tolerance, try other types
                try {
                    // Try integer
                    label intValue = std::stoi(value);
                    dict[key] = intValue;
                } catch (...) {
                    try {
                        // Try double
                        scalar doubleValue = std::stod(value);
                        dict[key] = doubleValue;
                    } catch (...) {
                        // Fall back to string
                        dict[key] = value;
                    }
                }
            }
        }
    }

    return dict;
}
```

```cpp
        } else {
            config.maxIterations = 1000;  // Default
        }

        // Apply configuration to solver
        config.solver->setTolerance(config.tolerance.getAbsolute());
        config.solver->setMaxIterations(config.maxIterations);

        return config;
    }
```

### Exercise 5: Why Template Solver on Scalar Type? (แบบที่ 5: เหตุใดต้องใช้เทมเพลตตัวแก้กับประเภทสเกลาร์?)

**Question**: Why do we template the LinearSolver class on the scalar type (e.g., `LinearSolver<scalar>`) rather than just using a specific type like `double`? What are the advantages and potential drawbacks of this approach?

**Answer**: The decision to template the LinearSolver class on the scalar type is fundamental to the design of a flexible numerical library. Let's explore why this is important and the trade-offs involved.

**Why Template on Scalar Type**:

**1. Type Safety and Compile-Time Checks**

```cpp
// Good: Type safety enforced at compile time
template<class Type>
class LinearSolver
{
    Field<Type> solution_;  // Solution field
    Field<Type> residual_;  // Residual field
    matrix matrix_;         // Matrix (same Type)

public:
    bool solve(const matrix& A, const Field<Type>& b, Field<Type>& x);
    // Type of A, b, and x must match
};

// Compiler will catch type mismatches:
auto scalarSolver = LinearSolver<scalar>();
auto vectorSolver = LinearSolver<vector>();

// This would be a compile error (good!):
// scalarSolver.solve(matrix, vectorField, scalarField);  // Type mismatch!
```

**2. Performance Optimization**

Different scalar types have different performance characteristics:

```cpp
// Template enables type-specific optimizations
template<class Type>
void LinearSolver<Type>::iteration()
{
    Type sum = 0;  // Type-specific initialization

    for (label i = 0; i < size_; ++i) {

        // For scalar, this might use FMA instructions
        // For vector, this vectorizes naturally
        // For complex, this handles complex arithmetic
    }

}
```

**3. Support for Different Number Types**

Different numerical problems require different number types:

**4. Memory Layout Efficiency**

Different types have different memory requirements:

```cpp

template<class Type>
Field<Type>::Field(label size)

    // Type determines allocation strategy
    switch (sizeof(Type)) {
        case 8:   // double - aligned to 8 bytes
            data_ = std::aligned_alloc(8, size * sizeof(Type));
            break;
        case 16:  // vector - aligned to 16 bytes for SIMD
```

Type enables compile-time expression optimization:

**Potential Drawbacks**:

**1. Code Bloat**

**2. Compilation Time**

**3. Type Mixing Issues**

```cpp
// Problem: Mixed-type operations
Field<scalar> scalarField;
Field<vector> vectorField;

auto result = scalarField * vectorField;  // Should be vector type!

// Solution: Type traits to detect operations
template<class Type1, class Type2>
struct OperationTraits
{
    using ResultType = decltype(std::declval<Type1>() * std::declval<Type2>());

    static constexpr bool Valid =
        std::is_arithmetic_v<ResultType> ||
        std::is_same_v<ResultType, vector>;
};
```

**Alternative Approaches**:

**1. Using Templates with Dynamic Typing**

```cpp
// Base class interface
class ILinearSolver
{
public:
    virtual ~ILinearSolver() = default;
    virtual void solve(void* matrix, void* b, void* x) = 0;
};

// Template implementation
template<class Type>
class ConcreteLinearSolver : public ILinearSolver
{
    void solve(void* matrix, void* b, void* x) override
    {
        // Type-safe casting
        auto A = static_cast<matrix*>(matrix);
        auto bField = static_cast<Field<Type>*>(b);
        auto xField = static_cast<Field<Type>*>(x);

        // Actual solver
        LinearSolver<Type>::solve(*A, *bField, *xField);
    }
};
```

**2. Using Variant Types**

```cpp
// Type-erased approach
using SolverVariant = std::variant<
    LinearSolver<scalar>,
    LinearSolver<vector>,
    LinearSolver<tensor>
>;

// Dynamic dispatch with type checking
template<class Type>
bool solveTypeAware(
    SolverVariant& solver,
    const matrix& A,
    const Field<Type>& b,
    Field<Type>& x
) {
    if (auto* scalarSolver = std::get_if<LinearSolver<scalar>>(&solver)) {
        return scalarSolver->solve(A, reinterpret_cast<const Field<scalar>&>(b),
                                 reinterpret_cast<Field<scalar>&>(x));
    }
    // Similar for other types...
    throw std::runtime_error("Type mismatch");
}
```

**Why Template Approach is Still Better**:

Despite the drawbacks, the template approach is superior because:

1. **Performance**: No virtual function overhead
2. **Type Safety**: Compile-time checking
3. **Memory Efficiency**: No type information storage
4. **Optimization**: Compiler can optimize per-type
5. **Maintainability**: Single codebase for all types

**Conclusion**:

Templating on scalar type is essential for a high-performance numerical library. While it introduces some complexity in compilation and code generation, the benefits in type safety, performance, and flexibility far outweigh these costs. The template approach enables us to write one solver implementation that works efficiently for all relevant numerical types, which is impossible with a single-type implementation.

The key is to use proper template techniques like explicit instantiation, traits, and SFINAE to mitigate the drawbacks while preserving the benefits.
