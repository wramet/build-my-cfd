## Part 1: Pattern Identification / Problem Setup

### Static vs. Shared Libraries: The Core Trade-off

In Day 29, we built our C++20 `Field<T>` library as an `INTERFACE` target. Because it consisted entirely of templates meant to be instantiated by the client, there were no `.cpp` files to compile into binary objects.

However, as our CFD solver matures, we will write non-template classes (like the LDU sparse matrix assembler, time registries, JSON parsers, and linear solver sweeping algorithms) that have concrete `.cpp` files. These must be compiled into binary hardware instructions.

When packaging these binary objects from `.cpp` files into a library, you must make a critical architectural choice between two methods:

1. **Static Library (`.a` on Linux/macOS, `.lib` on Windows)**: 
   This is literally just a zip archive (an `ar` archive) of unlinked `.o` object files. When an executable (`my_solver`) links against a static library, the linker physically copies the required binary code straight out of the archive and pastes it directly into the final executable.
   - *Pros:* The final executable is completely self-contained. You can copy the single file to another computer and it will just work.
   - *Cons:* Binary bloat. If you build 10 different solver applications that all use the CFD Core static library, that core logic is duplicated 10 times on your hard drive. 

2. **Shared Library (`.so` on Linux, `.dylib` on macOS, `.dll` on Windows)**: 
   This is a standalone, fully-linked binary file. When an executable "links" against it, it doesn't copy the binary code. It merely records a hardcoded promise: *"When I am launched, please tell the operating system to load this shared library file from the hard drive into my memory space."*
   - *Pros:* Massive disk space savings. Ten different solver applications can all share the exact same 50MB `.so` file in memory simultaneously.
   - *Cons:* DLL Hell. If the `.so` file is deleted, updated improperly, or moved, the executable will crash instantly on startup with a devastating `Library not found` OS-level error.

**Why does OpenFOAM so heavily favor Shared Libraries?**
OpenFOAM's defining architectural feature is its extreme modularity. It uses advanced runtime-selection mechanisms (Plugin Architectures) to allow researchers to write custom turbulence models, boundary conditions, or solvers, compile them as independent `.so` library files, and dynamically load them into the running main solver *without recompiling the core framework*. 

This is fundamentally impossible with Static Libraries. You must use Shared Libraries in order to dynamically load plugins via the system `dlopen()` call, which we will build in Days 31–32.

### The Problem: Managing External Dependencies

Imagine an ambitious feature: our CFD Core needs a robust unit testing framework to guarantee conservation of mass, and we decide to use the industry-standard **Google Test (gtest)**.

In a legacy UNIX build system (including OpenFOAM's `wmake`), you would typically:
1. Figure out how to download Google Test's source code from GitHub manually.
2. Compile Google Test locally using its own specific instructions.
3. Use root permissions to install the compiled `.a` files to `/usr/local/lib` and the headers to `/usr/local/include`.
4. Hardcode those `/usr/local/` absolute paths into your `Make/options` file.

This completely destroys project reproducibility. If a colleague or continuous-integration server clones your CFD repository, compiling it will immediately fail because they haven't explicitly installed Google Test to their system root path. 

Modern CMake solves this elegantly with a built-in internet-aware module called `FetchContent`.

## Part 2: Source Code Deep Dive

Let's look at how we define and build a shared library in Modern CMake, configure its visibility correctly, and fetch an external dependency seamlessly over the network.

### Creating a Shared Library (CFD Core)

We will start by creating a very fundamental, non-templated core class: a `Timer` to track execution time during our intensive solver iterations.

**`include/Timer.hpp`**
```cpp
// include/Timer.hpp
#pragma once

#include <chrono>
#include <string>

// To export symbols properly from shared libraries on Windows, 
// complex __declspec macros are typically required. 
// However, on Linux/macOS, we can use simple compiler attributes.
// We will configure CMake to manage this cleanly across all platforms.
//
// Because we instruct CMake to hide all symbols by default (Best Practice),
// we must explicitly mark this class with a macro declaring it PUBLICly visible.

#ifndef CFD_CORE_API
#   if defined(_MSC_VER)
        // Windows export/import semantics
#       define CFD_CORE_API __declspec(dllexport)
#   else
        // GCC/Clang visibility attributes
#       define CFD_CORE_API __attribute__((visibility("default")))
#   endif
#endif

// The entire class is marked for export into the shared library interface
class CFD_CORE_API Timer {
private:
    std::chrono::high_resolution_clock::time_point start_time_;
    std::string name_;

public:
    explicit Timer(std::string name);
    ~Timer();
    
    // Disallow copying to prevent timing glitches
    Timer(const Timer&) = delete;
    Timer& operator=(const Timer&) = delete;

    double elapsed_seconds() const;
};
```

**`src/Timer.cpp`**
```cpp
// src/Timer.cpp
#include "Timer.hpp"
#include <iostream>

Timer::Timer(std::string name) 
    : start_time_(std::chrono::high_resolution_clock::now()), 
      name_(std::move(name)) {
    // Timer started upon construction
}

Timer::~Timer() {
    std::cout << "Timer [" << name_ << "] finished in " 
              << elapsed_seconds() << " seconds.\n";
}

double Timer::elapsed_seconds() const {
    auto end_time = std::chrono::high_resolution_clock::now();
    std::chrono::duration<double> diff = end_time - start_time_;
    return diff.count();
}
```

### The CMake Build Script

We now write the `CMakeLists.txt` to precisely orchestrate five things:
1. Compile `Timer.cpp` into a `.so` shared library.
2. Hide all internal C++ symbols by default to reduce binary size.
3. Reach out to GitHub and download the Google Test testing framework.
4. Compile a unit test.
5. Automatically link them all together perfectly.

```cmake
# CMakeLists.txt
cmake_minimum_required(VERSION 3.20...3.28)
project(CFDCore VERSION 1.0.0 LANGUAGES CXX)

# Require C++20 standard globally for the entire project
set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED True)

# Enable aggressive hidden symbol visibility (Best Practice for Shared Libs)
# This forces developers to explicitly state what is public using CFD_CORE_API
set(CMAKE_CXX_VISIBILITY_PRESET hidden)
set(CMAKE_VISIBILITY_INLINES_HIDDEN YES)

# ==========================================
# 1. Provide the CFD Core Shared Library
# ==========================================
# By explicitly specifying SHARED, we instruct CMake to build a .so / .dylib / .dll
add_library(CFDCoreLib SHARED)

# Add our source C++ files
target_sources(CFDCoreLib PRIVATE src/Timer.cpp)

# Tell the target where its headers are so consumers can find Timer.hpp.
target_include_directories(CFDCoreLib PUBLIC
    $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
    $<INSTALL_INTERFACE:include>
)

# ==========================================
# 2. Fetch External Dependencies (Google Test)
# ==========================================
# FetchContent is a native module built directly into modern CMake
include(FetchContent)

# Declare where Google Test lives on the internet
FetchContent_Declare(
  googletest
  GIT_REPOSITORY https://github.com/google/googletest.git
  GIT_TAG        v1.14.0 # Best Practice: ALWAYS pin a specific version tag!
)

# Make the dependency available for linking immediately.
# We explicitly set a cache variable to hide GoogleTest's own internal install 
# targets so they don't pollute our project's `make install` command.
set(INSTALL_GTEST OFF CACHE BOOL "" FORCE)
FetchContent_MakeAvailable(googletest)


# ==========================================
# 3. Build Unit Tests
# ==========================================
# We define our test executable target
add_executable(CFDCoreTests tests/test_timer.cpp)

# Link our test executable against BOTH our local CFD shared library 
# AND the Google Test target automatically downloaded by FetchContent!
target_link_libraries(CFDCoreTests PRIVATE 
    CFDCoreLib 
    GTest::gtest_main
)
```

## Part 3: C++ & CMake Mechanics Explained

### The Shared Library Paradigm

By typing the simple command `add_library(CFDCoreLib SHARED)`, CMake does an incredible amount of platform-specific orchestrating on your behalf:
1. On **Linux**, it invokes `g++ -shared -fPIC`. The `.so` file will be generated.
2. On **macOS**, it generates a `.dylib` and automatically manages the intensely complicated `RPATH` / `install_name_tool` linking mechanics so the executable actually knows where on the hard drive the `dylib` is when it launches.
3. On **Windows**, it generates both a `.dll` (for runtime execution) and a linking `.lib` stub (for compile-time resolving) for Microsoft MSVC.

**The `-fPIC` Flag (Position Independent Code)**
When compiling an executable binary, the compiler assumes it knows exactly where in RAM memory the application's code will be loaded. However, when compiling a heavily shared library, the Operating System dynamically loads the library into a totally arbitrary, randomized address in memory at runtime (a security feature called ASLR). 

Because the library doesn't mathematically know where it will land in RAM, the compiler must generate *Position Independent Code* using relative jump offsets instead of absolute memory pointers. The brilliant part is that CMake natively handles appending `-fPIC` to all targets that are part of a `SHARED` library across all compilers without you ever having to specify the raw compiler flag manually.

### Best Practice: Symbol Visibility

In the `CMakeLists.txt`, we added these two critical globally-applied lines:
```cmake
set(CMAKE_CXX_VISIBILITY_PRESET hidden)
set(CMAKE_VISIBILITY_INLINES_HIDDEN YES)
```

By strict default on Linux (unlike on Windows), every single class, helper function, template instantiation, and variable inside your shared library `.cpp` files is legally "exported" to the outside public world. Anyone can load your library and forcefully call internal mathematical functions you never intended to be public. 

This is terrible because it massively bloats the size of the `.so` file (often by 40-60%) and slows down OS load times.

By setting `VISIBILITY_PRESET hidden`, you tell the compiler to hide absolutely everything behind the firewall by default. You then have to explicitly opt-in the specific classes you want to expose to users (your API boundary). In GCC/Clang, this requires appending `__attribute__((visibility("default")))` to your public classes, which we wrapped nicely in the `CFD_CORE_API` macro inside `Timer.hpp`.

### Dependency Management via `FetchContent`

`FetchContent` is arguably the single most powerful feature introduced in Modern CMake for C++ software developers.
It completely removes the horrendous burden of managing dependencies manually via package managers (like `apt` or `brew`) or incredibly finicky Git Submodules.

1. **`FetchContent_Declare`**: Registers the network URI of the source code. This does *not* download the code yet.
2. **`FetchContent_MakeAvailable`**:
   - During the configure phase, it clones the Git repository precisely at the pinned tag `v1.14.0` directly into your isolated `build/_deps/googletest-src` folder.
   - It immediately executes Google Test's own native `CMakeLists.txt` using an internal `add_subdirectory()` command.
   - Google Test's exported interface targets (like `GTest::gtest_main`) instantly and magically become available in your current CMake configuration session as fully native targets.

When you run `cmake --build`, your compiler seamlessly builds the entire Google Test framework directly alongside your CFD Core project. This guarantees exact ABI matching, exact C++ standard compliance, and exact compiler flag compatibility because *you compiled it yourself on your exact machine.*

## Part 4: Implementation Exercise

Let's complete the final components of our CMake setup by actually implementing the unit test that heavily relies on the shared library.

### Step 1: Write the Test Code

Create the test file specifically requesting the Google Test framework headers `#include <gtest/gtest.h>` which was pulled automatically over the network from GitHub by `FetchContent`.

**`tests/test_timer.cpp`**
```cpp
// tests/test_timer.cpp
#include <gtest/gtest.h>
#include "Timer.hpp"
#include <thread>

// Define a Google Test suite for our CFD Core component
TEST(TimerTests, ElapsedSecondsTracksCorrectly) {
    // Instantiate our shared library object
    Timer timer("TestTimer");
    
    // Simulate CFD computation pausing for 100 milliseconds
    std::this_thread::sleep_for(std::chrono::milliseconds(100));

    double time_taken = timer.elapsed_seconds();
    
    // We expect the time taken to be slightly more than 0.1 seconds, 
    // but less than 0.2 seconds (accounting for OS thread-scheduler noise).
    EXPECT_GE(time_taken, 0.1);
    EXPECT_LT(time_taken, 0.2);
}

// NOTE: We absolutely do not need to write `int main() { RUN_ALL_TESTS(); }` 
// anywhere in this file because we safely linked against the `GTest::gtest_main`
// target in CMake, which automatically injects a pre-written async main() function!
```

## Part 5: Summary

You have successfully transitioned from header-only `INTERFACE` libraries to concrete native `SHARED` libraries relying on `.cpp` objects.

1. **Shared vs Static:** Shared libraries (`SHARED`) reduce compiled binary size on disk and massively enable dynamic module loading plugins (essential for Phase 3's ultimate plugin factory). CMake reliably automatically abstracts horrible platform differences (like `-fPIC` or `.so` vs `.dylib`).
2. **Visibility Flags:** Encapsulation in C++ happens not just at the simple class level (`public`/`private`) but critically at the binary exported symbol layer via `CMAKE_CXX_VISIBILITY_PRESET hidden`. Hiding internal implementation logic drastically shrinks library sizes, lowers ram usage, and fiercely limits API surface attack area.
3. **`FetchContent`:** This is the Modern C++ developer's ultimate secret networking weapon. It successfully transforms horrific dependency management from a painful, manual system-wide installation process into a highly localized, perfectly reproducible automated CI pipeline. The `CMakeLists.txt` essentially entirely self-documents exactly how to acquire, configure, build, and link any third-party open-source code on the internet.

## Part 6: Install Rules and Shared Library Distribution

### Why Install Rules Matter

The `cmake --build` step produces binaries inside your local `build/` directory. This is fine for development, but when you want to distribute `CFDCoreLib` to other projects — or deploy it to a production server — you need to place headers, the shared library file, and CMake config files in a well-known prefix location. CMake's `install()` command orchestrates this precisely.

### `GNUInstallDirs` and Standard Paths

The `GNUInstallDirs` module provides platform-correct directory variables so your project installs to the right locations on Linux (`/usr/local/...`), macOS (`/opt/...`), or any custom prefix:

```cmake
include(GNUInstallDirs)

# After including GNUInstallDirs, these variables are set:
# CMAKE_INSTALL_LIBDIR    → "lib" or "lib64" (platform dependent)
# CMAKE_INSTALL_INCLUDEDIR → "include"
# CMAKE_INSTALL_BINDIR     → "bin"
# CMAKE_INSTALL_DATADIR    → "share"
```

Never hardcode paths like `/usr/local/lib` in install rules. Always use these variables so the install is relocatable.

### The `install(TARGETS ...)` Command

Add the following block at the bottom of your `CMakeLists.txt`, after the `add_library` and `target_include_directories` calls:

```cmake
include(GNUInstallDirs)

install(TARGETS CFDCoreLib
    LIBRARY  DESTINATION ${CMAKE_INSTALL_LIBDIR}          # .so / .dylib
    ARCHIVE  DESTINATION ${CMAKE_INSTALL_LIBDIR}          # .a (if ever static)
    RUNTIME  DESTINATION ${CMAKE_INSTALL_BINDIR}          # .dll on Windows
    INCLUDES DESTINATION ${CMAKE_INSTALL_INCLUDEDIR}      # header search path hint
)

# Install public headers separately
install(DIRECTORY include/
    DESTINATION ${CMAKE_INSTALL_INCLUDEDIR}
    FILES_MATCHING PATTERN "*.hpp"
)
```

`LIBRARY`, `ARCHIVE`, and `RUNTIME` are artifact type selectors. CMake automatically picks the right one per platform: on Linux it routes the `.so` to `LIBRARY`; on Windows the `.dll` goes to `RUNTIME` and the link stub `.lib` goes to `ARCHIVE`.

### Versioning the Shared Library

When `CFDCoreLib` ships to downstream consumers, they need to know which binary-compatible version they are linking against. The two properties that control this are `VERSION` (full version string) and `SOVERSION` (ABI-compatible major version):

```cmake
set_target_properties(CFDCoreLib PROPERTIES
    VERSION   1.2.3    # Full version embedded in the filename
    SOVERSION 1        # Symlink target used by the dynamic linker
)
```

On Linux this produces three files in the install tree:

```text
lib/
├── libCFDCoreLib.so        → libCFDCoreLib.so.1   (symlink, used at link time)
├── libCFDCoreLib.so.1      → libCFDCoreLib.so.1.2.3 (symlink, used at runtime)
└── libCFDCoreLib.so.1.2.3  (actual binary)
```

The dynamic linker (`ld.so`) resolves `libCFDCoreLib.so.1` at runtime. If you release version `1.3.0` with the same ABI, you update `VERSION` to `1.3.0` and leave `SOVERSION` at `1`. Existing executables find the new file through the unchanged `.so.1` symlink without relinking.

### Complete CMakeLists.txt with Install Rules

```cmake
cmake_minimum_required(VERSION 3.20...3.28)
project(CFDCore VERSION 1.2.3 LANGUAGES CXX)

set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED True)

set(CMAKE_CXX_VISIBILITY_PRESET hidden)
set(CMAKE_VISIBILITY_INLINES_HIDDEN YES)

# ── Library target ───────────────────────────────────────────────
add_library(CFDCoreLib SHARED src/Timer.cpp)

set_target_properties(CFDCoreLib PROPERTIES
    VERSION   ${PROJECT_VERSION}   # 1.2.3
    SOVERSION ${PROJECT_VERSION_MAJOR}  # 1
)

target_include_directories(CFDCoreLib PUBLIC
    $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
    $<INSTALL_INTERFACE:include>
)

# ── FetchContent: Google Test ─────────────────────────────────────
include(FetchContent)
FetchContent_Declare(
    googletest
    GIT_REPOSITORY https://github.com/google/googletest.git
    GIT_TAG        v1.14.0
)
set(INSTALL_GTEST OFF CACHE BOOL "" FORCE)
FetchContent_MakeAvailable(googletest)

# ── Tests ─────────────────────────────────────────────────────────
add_executable(CFDCoreTests tests/test_timer.cpp)
target_link_libraries(CFDCoreTests PRIVATE CFDCoreLib GTest::gtest_main)

# ── Install rules ─────────────────────────────────────────────────
include(GNUInstallDirs)

install(TARGETS CFDCoreLib
    LIBRARY  DESTINATION ${CMAKE_INSTALL_LIBDIR}
    ARCHIVE  DESTINATION ${CMAKE_INSTALL_LIBDIR}
    RUNTIME  DESTINATION ${CMAKE_INSTALL_BINDIR}
    INCLUDES DESTINATION ${CMAKE_INSTALL_INCLUDEDIR}
)

install(DIRECTORY include/
    DESTINATION ${CMAKE_INSTALL_INCLUDEDIR}
    FILES_MATCHING PATTERN "*.hpp"
)

# ── CMake package config (for find_package support) ───────────────
install(EXPORT CFDCoreTargets
    FILE      CFDCoreTargets.cmake
    NAMESPACE CFDCore::
    DESTINATION ${CMAKE_INSTALL_LIBDIR}/cmake/CFDCore
)
```

The `install(EXPORT ...)` block generates a `CFDCoreTargets.cmake` file that allows downstream CMake projects to write `find_package(CFDCore)` and get the imported target `CFDCore::CFDCoreLib` with all include paths and link flags already populated.

### RPATH vs RUNPATH

When an executable is built on Linux, the dynamic linker needs to locate `.so` files at runtime. Two ELF header fields control the search order:

| Field | Search order | Overridable by `LD_LIBRARY_PATH`? |
|-------|-------------|-----------------------------------|
| `RPATH` | Before `LD_LIBRARY_PATH` | No — baked in, takes priority |
| `RUNPATH` | After `LD_LIBRARY_PATH` | Yes |

CMake sets `RPATH` automatically during the build step so that executables in `build/` can find `.so` files in `build/lib/`. When you `install`, CMake rewrites the `RPATH` to the final install prefix (`CMAKE_INSTALL_PREFIX/lib`). This is called **RPATH relocation** and happens transparently.

Two CMake variables control this behavior:

```cmake
# Keep RPATH in installed binaries (default: ON for most generators)
set(CMAKE_INSTALL_RPATH "${CMAKE_INSTALL_PREFIX}/${CMAKE_INSTALL_LIBDIR}")
set(CMAKE_INSTALL_RPATH_USE_LINK_PATH TRUE)
```

For plugins loaded via `dlopen()` at runtime (Day 32), the plugin `.so` file itself has no RPATH because it is not an executable — the calling process is responsible for having the correct library search paths established before calling `dlopen()`.

### Expected Install Layout

Run the install step with a custom prefix:

```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j 4
cmake --install build --prefix /opt/cfdcore
```

The resulting directory tree under `/opt/cfdcore`:

```text
/opt/cfdcore/
├── include/
│   └── Timer.hpp                          ← public header
└── lib/
    ├── libCFDCoreLib.so                   → libCFDCoreLib.so.1   (symlink)
    ├── libCFDCoreLib.so.1                 → libCFDCoreLib.so.1.2.3 (symlink)
    ├── libCFDCoreLib.so.1.2.3             ← actual binary
    └── cmake/
        └── CFDCore/
            └── CFDCoreTargets.cmake       ← for find_package(CFDCore)
```

A downstream project adds one line to its `CMakeLists.txt`:

```cmake

target_link_libraries(my_solver PRIVATE CFDCore::CFDCoreLib)
```

### Quick Reference: Install-Related CMake Variables

| Variable | Default | Purpose |
|----------|---------|---------|
| `CMAKE_INSTALL_PREFIX` | `/usr/local` | Root of the install tree |
| `CMAKE_INSTALL_LIBDIR` | `lib` | Library subdirectory (`lib64` on some Linux) |
| `CMAKE_INSTALL_INCLUDEDIR` | `include` | Header subdirectory |
| `CMAKE_INSTALL_BINDIR` | `bin` | Executable subdirectory |
| `CMAKE_INSTALL_RPATH` | `""` | RPATH embedded in installed binaries |
| `CMAKE_INSTALL_RPATH_USE_LINK_PATH` | `OFF` | Auto-populate RPATH from link directories |

Set `CMAKE_INSTALL_PREFIX` at configure time to override the default:

```bash
cmake -S . -B build -DCMAKE_INSTALL_PREFIX=/opt/cfdcore
cmake --install build
```

This one-line configure change redirects all `install()` destinations without touching the `CMakeLists.txt` source, which is critical for packaging (RPM, Debian, Homebrew formulae) where the package manager controls the prefix.
