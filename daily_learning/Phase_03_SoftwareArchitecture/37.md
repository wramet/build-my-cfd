They inject algebraic coefficients directly into the linear system. Understanding why — and how — unlocks your ability to write any boundary condition OpenFOAM supports.

## Part 1: Pattern Identification

### The Architectural Problem with Boundary Conditions

Imagine you are building a finite-volume solver for the diffusion equation:

$$
\nabla^2 T = 0
$$

For every **internal** cell $P$ with face-connected neighbour $N$, the discretised equation adds an off-diagonal entry to the system matrix. The stencil is well-defined: you sum fluxes across all faces.

The problem arises at **boundary faces**. A boundary face has one cell ($P$) but no second cell. The solver loop must still produce a complete algebraic equation for cell $P$. The boundary condition — Dirichlet, Neumann, or Robin — changes the form of that equation.

The naive solution is to write a `switch` statement in the assembly loop:

```cpp
// NAIVE: do not do this
if (bc_type == "fixedValue") {
    // modify diagonal
} else if (bc_type == "zeroGradient") {
    // modify diagonal differently
} else if (bc_type == "fixedGradient") {
    // add to source
}
```

This violates the Open/Closed Principle you studied in Day 34. Adding a new boundary condition type requires modifying the core assembly loop.

### The Strategy Pattern Solution

The **Strategy pattern** defines:

- An **interface** (strategy base): declares the operations the context needs
- **Concrete strategies**: each provides one specific behavior
- A **context**: holds a pointer to the interface and calls it without knowing which concrete type it has

In OpenFOAM:

| Strategy Role | Class |
|--------------|-------|
| Interface | `fvPatchField<Type>` |
| Concrete strategy A | `fixedValueFvPatchField<Type>` |
| Concrete strategy B | `zeroGradientFvPatchField<Type>` |
| Concrete strategy C | `fixedGradientFvPatchField<Type>` |
| Concrete strategy D | `mixedFvPatchField<Type>` |
| Context | `fvMatrix<Type>` (assembly loop) |

The assembly loop in `fvMatrix` holds a reference to `GeometricField::boundaryField()`, which is a `FieldField<fvPatchField, Type>`. Each entry in this collection is a `fvPatchField<Type>*`. The matrix assembly calls the virtual methods on these pointers — it never needs to know which concrete BC it is dealing with.

### Why BCs Must Inject Coefficients, Not Just Set Values

This is the deepest architectural insight of today's session.

Consider cell $P$ adjacent to a boundary face $b$. The discretised diffusion equation for cell $P$ is:

$$
a_P T_P + \sum_{N \in \text{internal}} a_N T_N + a_b T_b = b_P
$$

where $T_b$ is the value at the boundary face. How is $T_b$ related to $T_P$?

- **Dirichlet (fixedValue):** $T_b$ is a prescribed constant $T_{\text{fixed}}$.
  - Rearranging: $a_P T_P = b_P - a_N T_N - a_b T_{\text{fixed}}$
  - The term $a_b T_{\text{fixed}}$ moves to the source vector. The diagonal $a_P$ is **unmodified**.

- **Neumann (zeroGradient):** $\partial T / \partial n = 0$, which means $T_b = T_P$.
  - Substituting: $a_P T_P + a_b T_P = b_P - \sum a_N T_N$
  - This adds $a_b$ to the diagonal coefficient of $T_P$. The source is **unmodified**.

- **Robin (mixed):** $T_b = \alpha T_P + (1-\alpha) T_{\text{ref}}$.
  - The diagonal gains $\alpha a_b$ and the source gains $(1-\alpha) a_b T_{\text{ref}}$.

Each BC type produces a different modification to the matrix row for cell $P$. The BC must communicate two numbers to the matrix:

1. **`valueInternalCoeffs`**: the coefficient multiplying $T_P$ (contributes to diagonal)
2. **`valueBoundaryCoeffs`**: the explicit part (contributes to source)

These two virtual methods are the heart of the Strategy interface.

### The Three BC Families

```
BC Type          valueInternalCoeffs    valueBoundaryCoeffs
─────────────────────────────────────────────────────────────────
Dirichlet        0                      T_fixed       (known value)
Neumann          1                      0             (zero gradient)
Robin (mixed)    refValue_fraction      explicit_part
```

This table is the entire algebraic story of OpenFOAM boundary conditions. Every other method — `evaluate()`, `write()`, gradient computations — is ancillary.

## Part 2: Source Code Deep Dive

### ⭐ `fvPatchField.H` — The Strategy Interface

> **File:** `src/finiteVolume/fields/fvPatchFields/fvPatchField/fvPatchField.H`
> **Verified from:** OpenFOAM v9 / OpenFOAM-dev source

The class declaration follows the standard OpenFOAM pattern with RTS, virtual interface, and template parameter `<Type>`:

```cpp
// File: src/finiteVolume/fields/fvPatchFields/fvPatchField/fvPatchField.H
// Lines: ~80-160 (key virtual interface)

template<class Type>
class fvPatchField
:
    public Field<Type>
{
    // ...
    const fvPatch& patch_;
    const DimensionedField<Type, volMesh>& internalField_;

public:

    // *** RTS table declaration (also a Strategy registry) ***
    declareRunTimeSelectionTable
    (
        tmp,
        fvPatchField,
        patch,
        (
            const fvPatch& p,
            const DimensionedField<Type, volMesh>& iF
        ),
        (p, iF)
    );

    // *** Strategy interface methods ***

    // Update the patch field coefficients.
    // Called by fvMatrix::addBoundaryDiag() and addBoundarySource()
    virtual void updateCoeffs()
    {}

    // Evaluate the patch field values.
    // Called after each outer iteration to update boundary face values.
    virtual void evaluate
    (
        const Pstream::commsTypes commsType = Pstream::commsTypes::blocking
    );

    // *** Matrix coefficient injection — the algebraic interface ***

    // Return the matrix diagonal coefficients corresponding to the
    // evaluation of the gradient of this patchField.
    // For fixedValue: returns 0 (no internal cell contribution)
    // For zeroGradient: returns 1 (boundary = internal cell value)
    virtual tmp<Field<Type>> valueInternalCoeffs
    (
        const tmp<scalarField>&
    ) const;

    // Return the matrix source coefficients corresponding to the
    // evaluation of the gradient of this patchField.
    // For fixedValue: returns the fixed value (added to RHS)
    // For zeroGradient: returns 0 (nothing added to RHS)
    virtual tmp<Field<Type>> valueBoundaryCoeffs
    (
        const tmp<scalarField>&
    ) const;

    // Return the matrix diagonal coefficients corresponding to the
    // evaluation of the gradient of this patchField
    virtual tmp<Field<Type>> gradientInternalCoeffs() const;

    // Return the matrix source coefficients corresponding to the
    // evaluation of the gradient of this patchField
    virtual tmp<Field<Type>> gradientBoundaryCoeffs() const;
};
```

Note that `fvPatchField<Type>` **inherits from `Field<Type>`**. The patch field IS a field of face values — and additionally knows how to inject itself into a matrix.

### ⭐ `fixedValueFvPatchField` — Dirichlet Strategy

> **File:** `src/finiteVolume/fields/fvPatchFields/basic/fixedValue/fixedValueFvPatchField.H`
> **File:** `src/finiteVolume/fields/fvPatchFields/basic/fixedValue/fixedValueFvPatchField.C`

The Dirichlet implementation is the simplest. The boundary face value is prescribed:

```cpp
// File: fixedValueFvPatchField.C
// The coefficient methods encode: T_b = T_fixed (not T_P)

template<class Type>
Foam::tmp<Foam::Field<Type>>
Foam::fixedValueFvPatchField<Type>::valueInternalCoeffs
(
    const tmp<scalarField>&
) const
{
    // ⭐ Returns a field of ZEROS.
    // Algebraic meaning: T_boundary does NOT depend on T_internal.
    // The boundary value is fully determined externally.
    return tmp<Field<Type>>
    (
        new Field<Type>(this->size(), Zero)
    );
}

template<class Type>
Foam::tmp<Foam::Field<Type>>
Foam::fixedValueFvPatchField<Type>::valueBoundaryCoeffs
(
    const tmp<scalarField>&
) const
{
    // ⭐ Returns the fixed boundary value itself.
    // Algebraic meaning: T_boundary = T_fixed → this value enters the source.
    return *this;
}
```

The `evaluate()` method for `fixedValue` does nothing — the field values are already set at construction and whenever `operator==` is called:

```cpp
// fixedValueFvPatchField has no evaluate() override.
// The base class evaluate() copies internalField to boundary — but
// fixedValue overrides operator== to set values directly.
// ⚠️ Exact evaluate() behaviour varies slightly across OpenFOAM versions.
```

### ⭐ `zeroGradientFvPatchField` — Neumann Strategy

> **File:** `src/finiteVolume/fields/fvPatchFields/basic/zeroGradient/zeroGradientFvPatchField.H`
> **File:** `src/finiteVolume/fields/fvPatchFields/basic/zeroGradient/zeroGradientFvPatchField.C`

The zero-gradient implementation says: the boundary value equals the internal cell value.

```cpp
// File: zeroGradientFvPatchField.C

template<class Type>
void Foam::zeroGradientFvPatchField<Type>::evaluate
(
    const Pstream::commsTypes
)
{
    if (!this->updated())
    {
        this->updateCoeffs();
    }

    // ⭐ Copy internal field to boundary faces (extrapolation).
    // This is what "zero gradient" means physically:
    // the boundary face value equals the adjacent internal cell value.
    fvPatchField<Type>::operator==
    (
        this->patchInternalField()
    );

    fvPatchField<Type>::evaluate();
}

template<class Type>
Foam::tmp<Foam::Field<Type>>
Foam::zeroGradientFvPatchField<Type>::valueInternalCoeffs
(
    const tmp<scalarField>&
) const
{
    // ⭐ Returns a field of ONEs.
    // Algebraic meaning: T_boundary = 1 * T_internal + 0.
    // The boundary contribution adds directly to the diagonal.
    return tmp<Field<Type>>
    (
        new Field<Type>(this->size(), pTraits<Type>::one)
    );
}

template<class Type>
Foam::tmp<Foam::Field<Type>>
Foam::zeroGradientFvPatchField<Type>::valueBoundaryCoeffs
(
    const tmp<scalarField>&
) const
{
    // ⭐ Returns a field of ZEROs.
    // No explicit value is added to the source vector.
    return tmp<Field<Type>>
    (
        new Field<Type>(this->size(), Zero)
    );
}
```

### ⭐ `fvMatrix` Coefficient Consumption

> **File:** `src/finiteVolume/fvMatrices/fvMatrix/fvMatrix.C`
> **Methods:** `addBoundaryDiag()`, `addBoundarySource()`

This is where the strategy interface values are consumed. The matrix assembly loop calls these methods once after assembling all internal face contributions:

```cpp
// File: fvMatrix.C
// Simplified version of addBoundaryDiag() and addBoundarySource()
// ⭐ These calls are what actually make the BC affect the linear system.

template<class Type>
void Foam::fvMatrix<Type>::addBoundaryDiag
(
    scalarField& diag,
    const direction cmpt
) const
{
    forAll(internalCoeffs_, patchi)
    {
        // For each face on this patch, add the internal coefficient
        // to the diagonal of the owning cell.
        // For fixedValue patches: internalCoeffs_ = 0 → no diagonal change
        // For zeroGradient patches: internalCoeffs_ = 1 → diagonal increases
        addToInternalField
        (
            lduAddr().patchAddr(patchi),
            cmptMag(internalCoeffs_[patchi].component(cmpt)),
            diag
        );
    }
}

template<class Type>
void Foam::fvMatrix<Type>::addBoundarySource
(
    Field<Type>& source,
    const bool couples
) const
{
    forAll(psi_.boundaryField(), patchi)
    {
        const fvPatchField<Type>& ptf = psi_.boundaryField()[patchi];

        if (!ptf.coupled())
        {
            // For each face on this patch, subtract the boundary coefficient
            // from the source of the owning cell.
            // For fixedValue: boundaryCoeffs_ = T_fixed → source gets T_fixed
            // For zeroGradient: boundaryCoeffs_ = 0 → source unchanged
            addToInternalField
            (
                lduAddr().patchAddr(patchi),
                boundaryCoeffs_[patchi].component(cmpt),
                source
            );
        }
    }
}
```

The `internalCoeffs_` and `boundaryCoeffs_` arrays in `fvMatrix` are populated by calling the virtual coefficient methods on each patch field. This happens in `fvMatrix::fvMatrix()` constructor and in `updateMatrixInterfaces()`.

## Part 3: C++ Mechanics Explained

### The Matrix Row Equation for a Boundary Cell

For a 1D mesh with cell $P$ at the right boundary, the FVM discretisation of $\nabla^2 T = 0$ gives only one face contribution from the left (the internal face to cell $W$) plus one from the boundary:

$$
\frac{T_b - T_P}{\delta x_{Pb}} \cdot A_b - \frac{T_P - T_W}{\delta x_{WP}} \cdot A_{wP} = 0
$$

Rearranging into the matrix form $a_P T_P + a_W T_W = b_P$:

$$
\left( \frac{A_b}{\delta x_{Pb}} + \frac{A_{wP}}{\delta x_{WP}} \right) T_P - \frac{A_{wP}}{\delta x_{WP}} T_W = \frac{A_b}{\delta x_{Pb}} T_b
$$

The coefficient $\frac{A_b}{\delta x_{Pb}}$ multiplies $T_b$. How that term is handled depends on the BC:

**Case A — fixedValue ($T_b = T_{\text{fixed}}$):**

$$
a_P T_P - a_W T_W = a_b T_{\text{fixed}}
$$

where $a_b = A_b / \delta x_{Pb}$. The term $a_b T_{\text{fixed}}$ goes entirely to the source. The diagonal $a_P$ only sees the internal face coefficient.

- `valueInternalCoeffs` = 0 (boundary does not contribute to diagonal)
- `valueBoundaryCoeffs` = $T_{\text{fixed}}$ (boundary contributes fully to source)

**Case B — zeroGradient ($T_b = T_P$):**

$$
a_P T_P + a_b T_P - a_W T_W = 0
$$

$$
(a_P + a_b) T_P - a_W T_W = 0
$$

The diagonal absorbs $a_b$. The source receives nothing.

- `valueInternalCoeffs` = 1 (boundary value equals internal, so coefficient adds to diagonal)
- `valueBoundaryCoeffs` = 0 (no explicit source term)

**Case C — fixedGradient ($\partial T / \partial n = g$):**

The face flux is directly prescribed: $\frac{T_b - T_P}{\delta x} = g$, so $T_b = T_P + g \cdot \delta x$.

- `gradientInternalCoeffs` = 1 (gradient flux depends on $T_P$)
- `gradientBoundaryCoeffs` = $g \cdot \delta x$ (explicit gradient contribution to source)

### Virtual Dispatch Through `fvPatchField*`

The `FieldField<fvPatchField, Type>` that stores boundary conditions is declared as:

```cpp
// GeometricField.H (simplified)
// ⭐ Verified: boundary field is a FieldField<fvPatchField, Type>
typedef FieldField<fvPatchField, Type> BoundaryField;
BoundaryField boundaryField_;
```

`FieldField<PatchField, Type>` is a `List<PatchField<Type>*>` — a list of pointers to the base class. When the matrix assembly calls:

```cpp
// fvMatrix.C (simplified)
const fvPatchField<Type>& ptf = psi_.boundaryField()[patchi];
tmp<Field<Type>> ic = ptf.valueInternalCoeffs(weights);
```

The call goes through the virtual dispatch table (vtable) to whichever concrete class was constructed for that patch. This is textbook polymorphism — but the key insight is that the **caller (fvMatrix) never changes**. Only the BC implementation changes.

### RTS Registration of Boundary Conditions

BCs are also registered via the RTS system (from Days 29–31). When OpenFOAM reads a `0/T` file with:

```
boundaryField
{
    inlet
    {
        type    fixedValue;
        value   uniform 300;
    }
    outlet
    {
        type    zeroGradient;
    }
}
```

The string `"fixedValue"` is passed to `fvPatchField<scalar>::New()`, which performs an RTS lookup and returns a `tmp<fvPatchField<scalar>>` pointing to a freshly constructed `fixedValueFvPatchField<scalar>`.

This means: the BC framework is simultaneously a **Strategy pattern** (virtual dispatch interface) and an **RTS factory** (self-registration and string-based construction). Both mechanisms operate on the same base class.

### `FieldField` Storage Layout

```
GeometricField<scalar, fvPatchField, volMesh>
  └── DimensionedField<scalar, volMesh>     (internal field, N cells)
  └── FieldField<fvPatchField, scalar>      (boundary field)
        [0] → fixedValueFvPatchField<scalar>  (nFaces_0 values)
        [1] → zeroGradientFvPatchField<scalar> (nFaces_1 values)
        [2] → fixedValueFvPatchField<scalar>  (nFaces_2 values)
```

Each patch field stores its own face-value array (inherited from `Field<Type>`). The `FieldField` is indexed by patch number, not by face number. The mapping from patch face index to global face index goes through `lduAddressing::patchAddr()`.

## Part 4: Implementation Exercise

### Design Summary

We build a minimal but complete BC framework demonstrating the strategy pattern with matrix coefficient injection. The target: a 1D domain, 5 cells, left patch is `fixedValue` at 1.0, right patch is `zeroGradient`. We assemble the diagonal-modified equation manually and verify the coefficients match the OpenFOAM design.

### Full Implementation

```cpp
// File: mini_bc_framework.cpp
// Build: g++ -std=c++17 -o mini_bc mini_bc_framework.cpp
//
// Demonstrates: Strategy pattern for boundary conditions with
//               matrix coefficient injection, mirroring OpenFOAM's
//               fvPatchField<Type> design.

#include <vector>
#include <string>
#include <iostream>
#include <iomanip>
#include <stdexcept>
#include <memory>
#include <functional>
#include <map>
#include <cassert>
#include <cmath>

// ============================================================
// Section 1: Strategy Interface
// ============================================================

// MiniPatchField<T> mirrors fvPatchField<Type>.
// Each concrete subclass encodes one BC type.
//
// The two pure virtual methods that matter for the linear system:
//   valueInternalCoeffs() → multiplies T_internal (diagonal contrib)
//   valueBoundaryCoeffs() → explicit part (source contrib)

template<class T>
class MiniPatchField {
protected:
    std::vector<T> values_;      // Face values on this patch
    int patchSize_;              // Number of faces on this patch

public:
    explicit MiniPatchField(int patchSize)
        : values_(patchSize, T{0}),
          patchSize_(patchSize)
    {}

    virtual ~MiniPatchField() = default;

    // -------------------------------------------------------
    // Matrix coefficient injection — the Strategy interface
    // -------------------------------------------------------

    // Coefficient that multiplies the adjacent internal cell value.
    // This amount is ADDED TO THE DIAGONAL of the owning cell.
    //   fixedValue  → 0.0  (boundary is independent of internal)
    //   zeroGradient → 1.0  (boundary equals internal)
    virtual std::vector<double> valueInternalCoeffs() const = 0;

    // Explicit contribution to the source vector of the owning cell.
    // This amount is ADDED TO THE SOURCE (RHS) of the owning cell.
    //   fixedValue  → T_fixed  (known value enters the source)
    //   zeroGradient → 0.0      (no explicit source contribution)
    virtual std::vector<T> valueBoundaryCoeffs() const = 0;

    // -------------------------------------------------------
    // Field update — called after each iteration
    // -------------------------------------------------------

    // Update face values from the internal field.
    // Signature mirrors fvPatchField<Type>::evaluate().
    virtual void evaluate(const std::vector<T>& internalField) = 0;

    // Return the BC type name (for diagnostics / RTS-style dispatch)
    virtual std::string type() const = 0;

    // Accessors
    const std::vector<T>& values() const { return values_; }
    int size() const { return patchSize_; }
};

// ============================================================
// Section 2: Concrete Strategy A — Dirichlet (fixedValue)
// ============================================================

// fixedValueFvPatchField<Type> in OpenFOAM.
// The boundary value is a prescribed constant.
//
//   valueInternalCoeffs = 0   → diagonal unchanged
//   valueBoundaryCoeffs = T_b → T_b moves to source
//
// Physical meaning: we KNOW the boundary temperature; there is
// nothing for the solver to compute there.

template<class T>
class FixedValuePatchField : public MiniPatchField<T> {
    T fixedValue_;

public:
    FixedValuePatchField(int patchSize, T fixedValue)
        : MiniPatchField<T>(patchSize),
          fixedValue_(fixedValue)
    {
        // Set face values immediately — they do not change.
        std::fill(this->values_.begin(), this->values_.end(), fixedValue_);
    }

    // ⭐ Mirrors fixedValueFvPatchField::valueInternalCoeffs()
    // Returns zeros: the boundary value does not depend on internal cells.
    std::vector<double> valueInternalCoeffs() const override {
        return std::vector<double>(this->patchSize_, 0.0);
    }

    // ⭐ Mirrors fixedValueFvPatchField::valueBoundaryCoeffs()
    // Returns the fixed value: this is what enters the source vector.
    std::vector<T> valueBoundaryCoeffs() const override {
        return std::vector<T>(this->patchSize_, fixedValue_);
    }

    // fixedValue::evaluate() does nothing — face values are already set.
    void evaluate(const std::vector<T>& /*internalField*/) override {
        // Values remain at fixedValue_. No update needed.
        // (OpenFOAM's fixedValueFvPatchField similarly has no evaluate override
        //  that extrapolates from internal — values are externally imposed.)
    }

    std::string type() const override { return "fixedValue"; }
};

// ============================================================
// Section 3: Concrete Strategy B — Neumann (zeroGradient)
// ============================================================

// zeroGradientFvPatchField<Type> in OpenFOAM.
// The boundary face value equals the adjacent internal cell value.
// Physically: no flux across this face.
//
//   valueInternalCoeffs = 1   → boundary = internal → adds to diagonal
//   valueBoundaryCoeffs = 0   → nothing in source

template<class T>
class ZeroGradientPatchField : public MiniPatchField<T> {
public:
    explicit ZeroGradientPatchField(int patchSize)
        : MiniPatchField<T>(patchSize)
    {}

    // ⭐ Mirrors zeroGradientFvPatchField::valueInternalCoeffs()
    // Returns ones: T_boundary = T_internal, so the internal coefficient
    // (which would multiply T_boundary) now multiplies T_internal instead.
    std::vector<double> valueInternalCoeffs() const override {
        return std::vector<double>(this->patchSize_, 1.0);
    }

    // ⭐ Mirrors zeroGradientFvPatchField::valueBoundaryCoeffs()
    // Returns zeros: no explicit value is injected into the source.
    std::vector<T> valueBoundaryCoeffs() const override {
        return std::vector<T>(this->patchSize_, T{0});
    }

    // ⭐ Mirrors zeroGradientFvPatchField::evaluate()
    // Copy the adjacent internal cell value to the boundary face.
    void evaluate(const std::vector<T>& internalField) override {
        // In a full implementation, we need the patch-to-cell addressing.
        // Here the patch has one face adjacent to the last internal cell.
        if (!internalField.empty()) {
            // Left patch → first cell; right patch → last cell.
            // For simplicity this method is called with the correct adjacent cell.
            for (int i = 0; i < this->patchSize_; ++i) {
                this->values_[i] = internalField[internalField.size() - 1];
            }
        }
    }

    std::string type() const override { return "zeroGradient"; }
};

// ============================================================
// Section 4: Mini fvMatrix — consuming the coefficients
// ============================================================

// MiniMatrix holds the LDU coefficients for a 1D N-cell domain.
// After assembling internal face contributions, it calls the BC
// coefficient injection methods — exactly as fvMatrix does.
//
// For a 1D domain with N cells:
//   - internal faces: N-1 (between consecutive cells)
//   - left boundary: patch 0 (1 face adjacent to cell 0)
//   - right boundary: patch 1 (1 face adjacent to cell N-1)
//
// We solve:  -d2T/dx2 = 0  (uniform cell spacing, dx = 1/(N+1))

struct MiniMatrix {
    int N;
    std::vector<double> diag;    // a_P  (diagonal)
    std::vector<double> lower;   // a_W  (lower off-diagonal, N-1 entries)
    std::vector<double> upper;   // a_E  (upper off-diagonal, N-1 entries)
    std::vector<double> source;  // b_P  (right-hand side)

    explicit MiniMatrix(int nCells)
        : N(nCells),
          diag(nCells, 0.0),
          lower(nCells - 1, 0.0),
          upper(nCells - 1, 0.0),
          source(nCells, 0.0)
    {}

    // Assemble internal face contributions for -d2T/dx2 = 0
    // Each internal face f between cells P (owner) and E (neighbour):
    //   a_P += D_f,  a_E += D_f,  a_PE = -D_f,  a_EP = -D_f
    // With uniform dx=1 and D=1: D_f = 1.0
    void assembleInternalFaces() {
        const double Df = 1.0;  // diffusivity * area / delta-x
        for (int f = 0; f < N - 1; ++f) {
            int P = f;      // owner cell
            int E = f + 1;  // neighbour cell
            diag[P] += Df;
            diag[E] += Df;
            lower[f] = -Df;  // coefficient of T_E in equation for T_P
            upper[f] = -Df;  // coefficient of T_P in equation for T_E
        }
    }

    // Inject boundary coefficients from left patch (1 face, adjacent to cell 0)
    // aB = D_f for the boundary face (same as internal face conductance)
    // - internalCoeffs[0] * aB is ADDED to diag[0]
    // - boundaryCoeffs[0] * aB is ADDED to source[0]
    void addBoundaryPatch(
        int cellIndex,
        double aB,
        double internalCoeff,
        double boundaryCoeff)
    {
        // ⭐ This is what fvMatrix::addBoundaryDiag() and
        //    addBoundarySource() do for each patch face.
        diag[cellIndex]   += aB * internalCoeff;
        source[cellIndex] += aB * boundaryCoeff;
    }

    // Print the assembled system (for inspection)
    void print() const {
        std::cout << "\nAssembled matrix (diag | lower/upper | source):\n";
        std::cout << std::fixed << std::setprecision(3);
        for (int i = 0; i < N; ++i) {
            std::cout << "  Cell " << i << ":  diag=" << diag[i]
                      << "  source=" << source[i];
            if (i > 0)     std::cout << "  lower[" << i-1 << "]=" << lower[i-1];
            if (i < N - 1) std::cout << "  upper[" << i << "]=" << upper[i];
            std::cout << "\n";
        }
    }

    // Solve by Gauss-Seidel (for a tridiagonal system, this converges quickly)
    std::vector<double> solve(int maxIter = 500, double tol = 1e-10) {
        std::vector<double> T(N, 0.0);
        for (int iter = 0; iter < maxIter; ++iter) {
            double maxRes = 0.0;
            for (int i = 0; i < N; ++i) {
                double rhs = source[i];
                if (i > 0)     rhs -= lower[i-1] * T[i-1];
                if (i < N - 1) rhs -= upper[i]   * T[i+1];
                double T_new = rhs / diag[i];
                maxRes = std::max(maxRes, std::fabs(T_new - T[i]));
                T[i] = T_new;
            }
            if (maxRes < tol) {
                std::cout << "  Converged at iteration " << iter+1
                          << ", residual=" << maxRes << "\n";
                break;
            }
        }
        return T;
    }
};

// ============================================================
// Section 5: Main — Strategy dispatch demonstration
// ============================================================

int main() {
    // ----------------------------------------------------------------
    // Setup: 5-cell 1D domain
    //   [left BC | cell0 | cell1 | cell2 | cell3 | cell4 | right BC]
    //
    // Left BC:  fixedValue = 1.0  (Dirichlet)
    // Right BC: zeroGradient      (Neumann, no flux)
    //
    // Analytical solution to d2T/dx2 = 0 with T(0)=1, dT/dx(1)=0
    //   → T = 1.0 everywhere (constant field satisfies zero gradient)
    // ----------------------------------------------------------------

    const int N = 5;
    const double Df = 1.0;  // face conductance (D * A / dx)

    // --- Strategy objects (concrete BC implementations) ---
    FixedValuePatchField<double>    leftBC(1, 1.0);  // 1 face, value=1.0
    ZeroGradientPatchField<double>  rightBC(1);      // 1 face

    // --- Show what each BC reports for matrix injection ---
    std::cout << "=== BC Coefficient Injection ===\n\n";

    {
        auto vic = leftBC.valueInternalCoeffs();
        auto vbc = leftBC.valueBoundaryCoeffs();
        std::cout << "Left BC (" << leftBC.type() << "):\n";
        std::cout << "  valueInternalCoeffs = " << vic[0]
                  << "  (diagonal contribution factor)\n";
        std::cout << "  valueBoundaryCoeffs = " << vbc[0]
                  << "  (source contribution value)\n";
        std::cout << "  → diag[0] += " << Df << " * " << vic[0]
                  << " = " << Df * vic[0] << "\n";
        std::cout << "  → source[0] += " << Df << " * " << vbc[0]
                  << " = " << Df * vbc[0] << "\n\n";
    }

    {
        auto vic = rightBC.valueInternalCoeffs();
        auto vbc = rightBC.valueBoundaryCoeffs();
        std::cout << "Right BC (" << rightBC.type() << "):\n";
        std::cout << "  valueInternalCoeffs = " << vic[0]
                  << "  (diagonal contribution factor)\n";
        std::cout << "  valueBoundaryCoeffs = " << vbc[0]
                  << "  (source contribution value)\n";
        std::cout << "  → diag[" << N-1 << "] += " << Df << " * " << vic[0]
                  << " = " << Df * vic[0] << "\n";

                  << " = " << Df * vbc[0] << "\n\n";
    }

    // --- Assemble the matrix ---
    MiniMatrix mat(N);
    mat.assembleInternalFaces();

    // Inject left BC (cell 0)
    mat.addBoundaryPatch(
        0,                                           // cell index

        leftBC.valueBoundaryCoeffs()[0]              // boundary coeff (= 1.0)

    mat.addBoundaryPatch(

        rightBC.valueBoundaryCoeffs()[0]             // boundary coeff (= 0)

    double maxError = 0.0;

    // Second experiment: compare diagonals under different BCs
    // ----------------------------------------------------------------
    std::cout << "\n=== Diagonal Comparison: fixedValue vs zeroGradient ===\n\n";

    MiniMatrix matFV(N), matZG(N);
    matFV.assembleInternalFaces();
    matZG.assembleInternalFaces();

    // Both have fixedValue on left
    matFV.addBoundaryPatch(0, Df, 0.0, 1.0);  // fixedValue left
    matZG.addBoundaryPatch(0, Df, 0.0, 1.0);  // fixedValue left (same)

    // Right: fixedValue vs zeroGradient
    matFV.addBoundaryPatch(N-1, Df, 0.0, 0.5);  // fixedValue=0.5 on right

              << ") because T_b = T_P.\n"
              << "             fixedValue keeps diagonal unchanged "
              << "but adds T_fixed to source.\n";

    return 0;
}
```

- `diag[0]` = 1.0 (only the one internal face to cell 1)

The `zeroGradient` BC adds to the diagonal precisely because it says "the boundary face value equals the interior cell value" — this relationship tightens the equation for cell 4 by removing the unknown $T_b$ from the flux expression and replacing it with $T_4$ itself.

## Part 5: Exercises and Self-Check

### Exercise 1: `fixedGradient` Coefficient Expressions

For a patch with `fixedGradient` where $\partial T / \partial n = g$, derive the coefficient expressions.

Given: the flux across boundary face $b$ for cell $P$ is

$$
q_b = D \cdot \frac{T_b - T_P}{\delta x_{Pb}}
$$

With $T_b = T_P + g \cdot \delta x_{Pb}$, expand and identify what `gradientInternalCoeffs` and `gradientBoundaryCoeffs` must return.

**Answer:**

Substituting $T_b$:

$$
q_b = D \cdot \frac{(T_P + g \cdot \delta x) - T_P}{\delta x} = D \cdot g
$$

The flux is fully explicit — it depends only on the prescribed gradient $g$, not on $T_P$. Therefore:

- `gradientInternalCoeffs` = 0 (no diagonal contribution — flux is independent of $T_P$)
- `gradientBoundaryCoeffs` = $D \cdot g$ (this constant flux enters the source vector)

This is the mirror of `fixedValue`: fixedValue has zero internal coefficient for the **value** path, fixedGradient has zero internal coefficient for the **gradient** path.

### Exercise 2: `evaluate()` vs `updateCoeffs()` — Why Two Methods?

Both `evaluate()` and `updateCoeffs()` are virtual methods on `fvPatchField`. When is each called, and why are they separate?

**Answer:**

`updateCoeffs()` is called **before the linear system is assembled** (or at the start of the assembly step). Its job is to compute `valueInternalCoeffs_` and `valueBoundaryCoeffs_` — the numbers that go into the matrix. For a `fixedValue` BC, this is trivial (store the prescribed value). For a `totalPressure` BC or a `turbulentIntensityKineticEnergyInlet`, `updateCoeffs()` may perform complex calculations using the current field state.

`evaluate()` is called **after the linear system is solved**, to update the boundary face values from the newly solved internal field. For `zeroGradient`, this copies the adjacent internal cell to the boundary face. For `fixedValue`, this does nothing (the value does not change with the internal field).

The separation is essential because:
1. Matrix assembly needs the **coefficients** (what `updateCoeffs` provides)
2. Post-solve operations need the **updated face values** (what `evaluate` provides)
3. The two steps happen at different phases of the SIMPLE/PISO outer loop

Conflating them would force every BC to recompute coefficients after each solve, and force the matrix to hold stale values during assembly.

### Exercise 3: Adding a New BC Without Modifying the Solver

You want to implement a `rampedFixedValue` BC where the prescribed value increases linearly with time: $T_b(t) = T_0 + r \cdot t$.

Describe exactly what you need to write and what you do NOT need to touch.

**Answer:**

You need to write:

1. A class `RampedFixedValuePatchField` inheriting from `MiniPatchField<T>` (or `fvPatchField<Type>` in OpenFOAM).
2. `valueInternalCoeffs()` returning zeros (it is still a Dirichlet BC — the value is prescribed).
3. `valueBoundaryCoeffs()` returning $T_0 + r \cdot t_{\text{current}}$ (the current ramp value).
4. `evaluate()` updating `values_` to the current ramp value.
5. An RTS registration macro: `addToRunTimeSelectionTable(fvPatchField<scalar>, RampedFixedValuePatchField, patch)`.

You do NOT need to touch:

- `fvMatrix.C` — the assembly loop calls `valueInternalCoeffs()` and `valueBoundaryCoeffs()` polymorphically.
- `fvPatchField.H` — the interface is already defined.
- Any existing BC class — they are independent strategies.
- The solver application — it calls `fvMatrix::solve()` without knowing which BCs are present.

This is the Open/Closed Principle in action: the solver is **closed for modification** but **open for extension** via new BC types.

### Exercise 4: What Does `FieldField<fvPatchField, Type>` Actually Store?

Describe the memory layout of the boundary field in a `volScalarField` with three patches: an inlet (50 faces, fixedValue), an outlet (50 faces, zeroGradient), and walls (200 faces, zeroGradient).

**Answer:**

```
FieldField<fvPatchField, scalar>
  [0] → fixedValueFvPatchField<scalar>
          values_: double[50]     ← 50 face values (all = inlet temperature)
          vtable ptr → fixedValue vtable
  [1] → zeroGradientFvPatchField<scalar>
          values_: double[50]     ← 50 face values (copied from internal after each solve)
          vtable ptr → zeroGradient vtable
  [2] → zeroGradientFvPatchField<scalar>
          values_: double[200]    ← 200 face values
          vtable ptr → zeroGradient vtable
```

Total boundary storage: $300 \times 8 = 2400$ bytes for the face values, plus three vtable pointers and small bookkeeping overhead. The key point is that each `fvPatchField` object is allocated separately on the heap, and the `FieldField` holds pointers to them. This is necessary because each patch may have a **different concrete type** — a `FieldField<fvPatchField, T>` is essentially `std::vector<std::unique_ptr<fvPatchField<T>>>` in modern C++ terms.

### Exercise 5: Robin (Mixed) BC Coefficient Derivation

The `mixedFvPatchField` in OpenFOAM stores two fields: `refValue_` (the reference value) and `valueFraction_` (a scalar between 0 and 1). The boundary value is:

$$
T_b = \alpha \cdot T_{\text{ref}} + (1 - \alpha) \cdot T_P
$$

where $\alpha$ is `valueFraction_`. Derive `valueInternalCoeffs` and `valueBoundaryCoeffs` for this BC.

**Answer:**

The boundary face value is:

$$
T_b = \alpha T_{\text{ref}} + (1 - \alpha) T_P
$$

Substitute into the flux expression $q_b = D (T_b - T_P) / \delta x$:

$$
q_b = \frac{D}{\delta x} \left[ \alpha T_{\text{ref}} + (1 - \alpha) T_P - T_P \right]
= \frac{D}{\delta x} \left[ \alpha T_{\text{ref}} - \alpha T_P \right]
= \frac{D \alpha}{\delta x} (T_{\text{ref}} - T_P)
$$

Separating implicit ($T_P$) and explicit ($T_{\text{ref}}$) parts:

$$
q_b = -\frac{D \alpha}{\delta x} T_P + \frac{D \alpha}{\delta x} T_{\text{ref}}
$$

For the matrix assembly (where $a_b = D / \delta x$):

- **`valueInternalCoeffs`** = $(1 - \alpha)$

  Because $T_b = (1-\alpha) T_P + \alpha T_{\text{ref}}$, the part proportional to $T_P$ is $(1-\alpha)$. This multiplies the boundary conductance $a_b$ and is added to the diagonal.

- **`valueBoundaryCoeffs`** = $\alpha \cdot T_{\text{ref}}$

  The explicit part $\alpha T_{\text{ref}}$ is the source contribution. This multiplies $a_b$ and goes to the RHS.

Verification at the limits:
- $\alpha = 1$: `internalCoeffs=0`, `boundaryCoeffs=T_ref` → pure Dirichlet
- $\alpha = 0$: `internalCoeffs=1`, `boundaryCoeffs=0` → pure Neumann (zeroGradient)

This confirms that `mixedFvPatchField` is a general interpolation between Dirichlet and Neumann, recovering both extremes exactly.

## Summary: Strategy Pattern in the BC Framework

The boundary condition framework demonstrates three C++ and software engineering concepts working together:

**Strategy Pattern (Gang of Four):**
The `fvPatchField<Type>` hierarchy is a textbook Strategy implementation. The context (`fvMatrix`) calls two virtual methods — `valueInternalCoeffs()` and `valueBoundaryCoeffs()` — on an interface pointer without knowing the concrete type. Adding a new BC type requires only a new derived class and an RTS registration macro.

**Algebraic Coupling:**
The key insight is that BCs do not "set values after solving." They inject algebraic modifications into the linear system before solving. The diagonal coefficient and source vector of each boundary cell are modified by the BC type, and the solver operates on this modified system without any special BC awareness.

**Composite Pattern (FieldField):**
The `FieldField<fvPatchField, Type>` is a list of heterogeneous patch field objects, each holding its own face-value array and its own vtable pointer. This is the composite pattern: the `GeometricField` contains a collection of boundary fields, each of which can be any concrete BC type.

| BC Type | `valueInternalCoeffs` | `valueBoundaryCoeffs` | Diagonal effect | Source effect |
|---------|----------------------|----------------------|----------------|---------------|
| fixedValue | 0 | $T_{\text{fixed}}$ | None | $+a_b T_{\text{fixed}}$ |
| zeroGradient | 1 | 0 | $+a_b$ | None |
| fixedGradient | 0 | $D \cdot g$ | None | $+D \cdot g$ |
| mixed | $1 - \alpha$ | $\alpha T_{\text{ref}}$ | $+a_b(1-\alpha)$ | $+a_b \alpha T_{\text{ref}}$ |
