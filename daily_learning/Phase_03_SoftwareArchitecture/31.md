## Part 1: Pattern Identification / Problem Setup

### The Problem: Hardcoded Dependencies and the Switch Statement

Imagine we are building the central core loop for our brand new CFD application. Depending on the complex physics involved (e.g., incompressible vs. compressible, steady-state vs. transient), the user specifies which numerical solver they want to use in a text configuration file explicitly by string (e.g., `"GaussSeidel"`, `"PISO"`, `"SIMPLE"`).

A naive, procedural C++ implementation usually looks something like this:

```cpp
// Anti-pattern: The Hardcoded Switch Statement
std::unique_ptr<Solver> createSolver(const std::string& name, Mesh& mesh) {
    if (name == "GaussSeidel") {
        return std::make_unique<GaussSeidelSolver>(mesh);
    } else if (name == "PISO") {
        return std::make_unique<PISOSolver>(mesh);
    } else if (name == "SIMPLE") {
        return std::make_unique<SIMPLESolver>(mesh);
    } else if (name == "icoFoam") {
        return std::make_unique<IcoFoamSolver>(mesh);
    } else {
        throw std::invalid_argument("Unknown solver: " + name);
    }
}
```

This violates the **Open/Closed Principle (OCP)** from SOLID design principles:
> *Software entities (classes, modules, functions) should be open for extension, but closed for modification.*

**Why OCP Violation Matters:**
- Every new solver requires modifying core framework code
- Forces recompilation of entire application
- Prevents third-party binary plugins
- Breaks separation of concerns
- Makes testing difficult (cannot mock solvers)

### ⭐ Architecture Decision: Switch vs Factory vs Registry

| Approach | Extension Method | Modification Required | Plugin Support | Compile-Time Safety |
|----------|-----------------|----------------------|----------------|-------------------|
| **Switch Statement** | Edit function | ✅ Yes (recompile) | ❌ No | ✅ Full |
| **Factory Class** | Inherit + register | ✅ Yes (recompile) | ❌ No | ✅ Full |
| **Registry + dlopen** | Compile .so separately | ❌ No | ✅ Yes | ⚠️ Partial (runtime) |

**Decision:** We start with Factory (Day 31) → Registry + Static Initializers (Day 32) → Plugin loading (Day 41 mini-project)

### The OpenFOAM Solution: Run-Time Selection (RTS) Macros

OpenFOAM solves the OCP problem using **Run-Time Selection (RTS) Macros**:

```cpp
// OpenFOAM Legacy Pattern (DO NOT write this in 2026)
class myCustomSolver : public Solver {
    TypeName("myCustomSolver");  // Declares type info
};

defineTypeNameAndDebug(myCustomSolver, 0);
addToRunTimeSelectionTable(Solver, myCustomSolver, dictionary);
```

**Problems with OpenFOAM's Approach:**
- ❌ Macros hide code structure from IDEs
- ❌ Broken autocomplete and navigation
- ❌ Impossible-to-read compiler errors
- ❌ Type-unsafe (void* casting)
- ❌ Manual reference counting (error-prone)

### The Modern C++ Solution

With C++11/17, we achieve the same power with:
1. **`std::unordered_map`** — O(1) hash table lookup
2. **`std::function`** — Type-safe callable wrapper
3. **`std::unique_ptr`** — Automatic memory management
4. **Lambdas** — Inline registration functions

### Step 2: Modern Factory with Type Erasure

Create file `factory_pattern/include/SolverFactory.H`:

```cpp
#pragma once
#include "Solver.H"
#include <unordered_map>
#include <functional>
#include <memory>
#include <string>
#include <vector>
#include <stdexcept>
#include <algorithm>

// -[2] Type alias: Creator function signature
using SolverCreator = std::function<std::unique_ptr<Solver>()>;

class SolverFactory {
private:
    // -[3] Registry: String name → Creator function
    std::unordered_map<std::string, SolverCreator> registry_;

    // -[4] Private constructor (Singleton pattern)
    SolverFactory() = default;

    // Prevent copying
    SolverFactory(const SolverFactory&) = delete;
    SolverFactory& operator=(const SolverFactory&) = delete;

public:
    // -[5] Meyers Singleton (thread-safe in C++11+)
    static SolverFactory& instance() {
        static SolverFactory factory;
        return factory;
    }

    // -[6] Register a solver type
    void registerSolver(const std::string& name, SolverCreator creator) {
        if (registry_.find(name) != registry_.end()) {
            std::cerr << "Warning: Overwriting solver '" << name << "'\n";
        }
        registry_[name] = std::move(creator);
        std::cout << "  [Factory] Registered: '" << name << "'\n";
    }

    // -[7] Create solver instance by name
    std::unique_ptr<Solver> create(const std::string& name) const {
        auto it = registry_.find(name);

        if (it == registry_.end()) {
            auto available = listAvailable();
            throw std::invalid_argument(
                "Unknown solver: '" + name + "'. Available: " + available
            );
        }

        return it->second();
    }

    // -[8] Check if solver exists
    bool hasSolver(const std::string& name) const {
        return registry_.find(name) != registry_.end();
    }

    // -[9] List available solversers
    std::vector<std::string> availableSolvers() const {
        std::vector<std::string> keys;
        keys.reserve(registry_.size());

        for (const auto& pair : registry_) {
            keys.push_back(pair.first);
        }

        std::sort(keys.begin(), keys.end());
        return keys;
    }

    // -[10] Count registered solvers
    size_t count() const {
        return registry_.size();
    }

private:
    std::string listAvailable() const {
        auto solvers = availableSolvers();
        if (solvers.empty()) return "<none>";

        std::string result = solvers[0];
        for (size_t i = 1; i < solvers.size() && i < 3; ++i) {
            result += ", " + solvers[i];
        }

        if (solvers.size() > 3) {
            result += ", ... (" + std::to_string(solvers.size()) + " total)";
        }
        return result;
    }
};
```

## Part 3: Concrete Solver Implementations

### Step 3: Real CFD Solver Implementations

Create file `factory_pattern/src/concrete_solvers.C`:

```cpp
#include "SolverFactory.H"
#include <iostream>
#include <thread>
#include <chrono>

// -[1] Gauss-Seidel: Iterative linear solver
class GaussSeidelSolver : public Solver {
private:
    int maxIterations_;
    double tolerance_;

public:
    GaussSeidelSolver()
        : maxIterations_(1000), tolerance_(1e-6)
    {}

    void solve() override {
        std::cout << "  [GaussSeidel] Starting iterations (max=" << maxIterations_ << ")\n";

        // Simulate numerical work
        for (int iter = 0; iter < 10; ++iter) {
            std::this_thread::sleep_for(std::chrono::milliseconds(10));
            if (iter % 3 == 0) {
                std::cout << "    Iteration " << iter << ": residual = "
                          << (1.0 / (iter + 1)) << "\n";
            }
        }

        std::cout << "  [GaussSeidel] Converged\n";
    }

    std::string name() const override { return "GaussSeidel"; }

    void configure(const std::string& config) override {
        // Parse simple config like "maxIter:5000,tol:1e-8"
        size_t pos = config.find("maxIter:");
        if (pos != std::string::npos) {
            size_t end = config.find(",", pos);
            std::string val = config.substr(pos + 8, end - pos - 8);
            maxIterations_ = std::stoi(val);
        }
        Solver::configure(config);
    }
};

// -[2] PISO: Pressure-Implicit with Splitting of Operators
class PISOSolver : public Solver {
private:
    int nCorrectors_;

public:
    PISOSolver() : nCorrectors_(2) {}

    void solve() override {
        std::cout << "  [PISO] Starting pressure-velocity coupling\n";
        std::cout << "  [PISO] Non-orthogonal correction: " << nCorrectors_ << "\n";

        // Simulate PISO algorithm steps
        std::this_thread::sleep_for(std::chrono::milliseconds(50));
        std::cout << "  [PISO] Momentum equation solved\n";

        std::this_thread::sleep_for(std::chrono::milliseconds(50));
        std::cout << "  [PISO] Pressure correction applied\n";

        std::cout << "  [PISO] Converged\n";
    }

    std::string name() const override { return "PISO"; }
};

// -[3] SIMPLE: Semi-Implicit Method for Pressure-Linked Equations
class SIMPLESolver : public Solver {
private:
    double underRelaxation_;

public:
    SIMPLESolver() : underRelaxation_(0.7) {}

    void solve() override {
        std::cout << "  [SIMPLE] Starting SIMPLE algorithm\n";
        std::cout << "  [SIMPLE] Under-relaxation: " << underRelaxation_ << "\n";

        // Simulate SIMPLE steps
        for (int pass = 0; pass < 3; ++pass) {
            std::cout << "  [SIMPLE] Pass " << pass << ":\n";

            std::this_thread::sleep_for(std::chrono::milliseconds(30));
            std::cout << "    Momentum equations solved\n";

            std::this_thread::sleep_for(std::chrono::milliseconds(30));
            std::cout << "    Pressure equation solved\n";
        }

        std::cout << "  [SIMPLE] Converged\n";
    }

    std::string name() const override { return "SIMPLE"; }
};
```

## Part 5: Design Trade-offs & Analysis

### ⭐ Memory Layout and Performance

**Switch Statement (Naive):**
```assembly
; Assembly for switch statement (x86-64)
cmp     rdi, offset "GaussSeidel"[rip]
je      .L_GaussSeidel
cmp     rdi, offset "PISO"[rip]
je      .L_PISO
cmp     rdi, offset "SIMPLE"[rip]
je      .L_SIMPLE
jmp     .L_default
```
- **Branch prediction:** Poor (O(n) string compares)
- **Cache locality:** Poor (jump table scattered)
- **Instruction cache:** Misses on indirect jumps

**Factory with `std::unordered_map`:**
```cpp
// Hash table lookup (O(1) average)
auto it = registry_.find(name);  // Hash computation + single lookup
return it->second();              // Indirect function call
```
- **Branch prediction:** Excellent (no branching)
- **Cache locality:** Good (contiguous hash table)
- **Instruction cache:** Good (predictable indirection)

### ⭐ Type Erasure Deep Dive

**What `std::function` actually does:**

```cpp
// Conceptually (simplified)
template<typename Signature>
class function {
    struct Concept {
        virtual ~Concept() = default;
        virtual R invoke() = 0;
    };

    template<typename T>
    struct Model : Concept {
        T callable_;
        Model(T&& c) : callable_(std::forward<T>(c)) {}
        R invoke() override { return callable_(); }
    };

    std::unique_ptr<Concept> impl_;

public:
    template<typename T>
    function(T&& t) : impl_(new Model<T>(std::forward<T>(t))) {}

    R operator()() { return impl_->invoke(); }
};
```

**Cost:**
- **Memory:** 2 heap allocations (vtable + object)
- **Indirection:** 2 virtual calls (construction + invocation)
- **Benefit:** Type-erased storage in homogeneous container

### ⭐ Comparison: OpenFOAM vs Modern C++

| Aspect | OpenFOAM (C++98) | Modern C++ (C++17) |
|--------|-------------------|---------------------|
| **Registration** | Macro expansion | Lambda + `std::function` |
| **Storage** | Custom linked list | `std::unordered_map` |
| **Lookup complexity** | O(n) linear | O(1) hash table |
| **Type safety** | Void* casting | Compile-time templates |
| **Memory management** | Manual ref counting | `std::unique_ptr` |
| **Thread safety** | Manual locks | Meyers Singleton (thread-safe) |
| **Debugging** | Impossible (macro magic) | Full IDE support |
| **Binary plugins** | ✅ Yes (dlopen) | ⚠️ Requires additional work |

### Extension Path: Plugin System

For full plugin support (like OpenFOAM's `.so` loading):

```cpp
// Future enhancement: dlopen support
void loadPlugin(const std::string& path) {
    void* handle = dlopen(path.c_str(), RTLD_LAZY);
    using RegisterFunc = void(*)(SolverFactory&);
    RegisterFunc registerFunc = (RegisterFunc)dlsym(handle, "registerPlugin");

    if (registerFunc) {
        registerFunc(*this);  // Let plugin register itself
    }
}
```

This enables binary plugins without recompiling the main executable — exactly what OpenFOAM achieves with RTS macros, but using modern C++!
