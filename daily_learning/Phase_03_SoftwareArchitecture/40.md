## Pattern Identification — Exception vs Error Streams (รูปแบบการระบุสัญญาณผิดพลาด — ข้อยกเว้น ประกอบกับสตรีมข้อผิดพลาด)

### OpenFOAM's Error Handling Philosophy (ปรัชญาการจัดการข้อผิดพลาดของ OpenFOAM)

OpenFOAM takes a fundamentally different approach to error handling compared to traditional C++ applications. Instead of relying on C++ exceptions, OpenFOAM implements a stream-based error system that provides more control and better debugging capabilities in a scientific computing context.

**⭐ Why OpenFOAM Avoids C++ Exceptions**

The core reason OpenFOAM avoids traditional C++ exceptions lies in performance predictability and stack trace requirements:

1. **Performance Consistency**: In computational fluid dynamics, performance must be highly predictable. Exceptions introduce non-deterministic runtime costs that can vary dramatically between throw points, making it impossible to guarantee consistent execution times.

2. **Stack Trace Requirements**: CFD simulations often fail at very deep levels of nested function calls. Traditional exceptions only capture the call stack up to the throw point, but OpenFOAM needs complete diagnostic information including the originating function name, file, and line number.

3. **Parallel Execution**: In MPI-based parallel environments, exceptions become complex to propagate across process boundaries. OpenFOAM's error system can handle termination consistently across all processes.

4. **Resource Cleanup**: OpenFOAM must ensure proper cleanup of mesh data, field storage, and I/O resources. The stream-based approach allows for controlled resource cleanup before termination.

**⭐ The Stream-Based Approach**

OpenFOAM's error handling follows a stream-based pattern:

```cpp
// Traditional exception approach (avoided in OpenFOAM)
try {
    computeSomething();
} catch (const std::exception& e) {
    std::cerr << "Error: " << e.what() << std::endl;
}

// OpenFOAM stream-based approach
computeSomething();  // If this fails
FatalError << "Computation failed at timestep " << timeStep
          << " in function " << FUNCTION_NAME << exit(FatalError);
```

This approach provides several advantages:

- **Automatic Source Location**: The `FatalErrorIn` macro automatically inserts `__FILE__` and `__LINE__`
- **Formatted Output**: Errors are formatted consistently with proper indentation and structure
- **Flexible Severity Levels**: Info, Warning, SeriousError, and FatalError levels
- **Parallel Safety**: Works correctly in multi-process environments

**⭐ Error Hierarchy**

OpenFOAM implements a multi-level error hierarchy:

```
messageStream (base class)
├── Info (for informational messages)
├── Warning (for recoverable issues)
├── SeriousError (for problems that require intervention)
└── FatalError (for unrecoverable failures)
```

Each level has different behaviors:
- **Info**: Continues execution, logs to standard output
- **Warning**: Continues execution, logs to standard error with warning header
- **SeriousError**: Continues execution but logs prominently
- **FatalError**: Terminates the program after logging

This hierarchy allows developers to handle different types of errors appropriately while maintaining consistent formatting and behavior.

This automatically captures the location where the error occurred, making debugging much easier.

### Message Stream Hierarchy (ชั้นชั้นของสตรีมข้อความ)

**⭐ messageStream Architecture**

While we couldn't locate the exact `messageStream.H` file, we can infer its structure from usage patterns in `error.H`. The message stream system provides different severity levels:

```cpp
class messageStream
{
    // Basic stream functionality
    virtual messageStream& operator<<(const string& s);
    virtual messageStream& operator<<(int i);
    virtual messageStream& operator<<(double d);

    // Severity levels
    static int level;  // Controls output detail

    static const int WARNING = 1;
    static const int SERIOUS_ERROR = 2;
    static const int FATAL_ERROR = 3;

```

Each severity level has different characteristics:

**Info Stream** (สตรีมข้อมูล)
- Purpose: General information logging
- Output: Standard output
- Continuation: Execution continues normally
- Example: `Info << "Time step " << timeStep << " completed";`

**Warning Stream** (สตรีมคำเตือน)
- Purpose: Recoverable issues that don't stop execution
- Output: Standard error with " --> FOAM Warning:" prefix
- Continuation: Execution continues
- Example: `Warning << "Mesh quality is below recommended threshold";`

**SeriousError Stream** (สตรีมข้อผิดพลาดร้ายแรง)
- Purpose: Problems that require attention but don't stop execution
- Output: Standard error with " --> FOAM Serious Error:" prefix
- Continuation: Execution continues but may be in an inconsistent state
- Example: `SeriousError << "Boundary condition not properly initialized";`

**FatalError Stream** (สตรีมข้อผิดพลาดตายตัว)
- Purpose: Unrecoverable errors that must stop execution
- Output: Standard error with " --> FOAM FATAL ERROR:" prefix
- Continuation: Program termination
- Example: `FatalError << "Memory allocation failed in solver";`

## C++ Mechanics — Stream-Based Errors, Automatic Exit, Stack Traces (กลไก C++ — ข้อผิดพลาดระบบสตรีม, การออกโปรแกรมอัตโนมัติ, สแต็กเทรซ)

### Stream-Based Error Flow (การไหลของข้อผิดพลาดระบบสตรีม)

**⭐ The Error Stream Flow**

When an error occurs in OpenFOAM, it follows this flow:

1. **Error Macro Expansion**: `FatalErrorIn("function")` expands to `FatalError("function", __FILE__, __LINE__)`

2. **Stream Initialization**: The `operator()` method sets up the error context:
   ```cpp
   OSstream& Foam::error::operator()(const char* functionName,
                                     const char* sourceFileName,
                                     const int sourceFileLineNumber)
   {
       functionName_ = functionName;
       sourceFileName_ = sourceFileName;
       sourceFileLineNumber_ = sourceFileLineNumber;
       return operator OSstream&();
   }
   ```

3. **Stream Operations**: Data is streamed to the error stream:
   ```cpp
   FatalErrorIn("readMesh")
       << "Failed to open file: " << fileName
       << " at line " << lineNo << exit(FatalError);
   ```

4. **Termination Manipulator**: The `exit(FatalError)` manipulator triggers termination:
   ```cpp
   void Foam::error::exit(const int errNo)
   {
       if (throwExceptions_)
       {
           error errorException(*this);
           messageStream_.rewind();
           throw errorException;
       }
       else
       {
           Perr << endl << *this << endl
                << "\nFOAM exiting\n" << endl;
           ::exit(errNo);
       }
   }
   ```

**⭐ Exit vs Abort Behavior**

The `error` class provides two termination methods with different behaviors:

**exit() Method** (เมธอด exit)
- Purpose: Clean program termination
- Behavior: Cleans up resources, logs error, calls `::exit()`
- Stack trace: May or may not show stack depending on configuration
- Use case: Expected errors, configuration issues, file not found

```cpp
void Foam::error::exit(const int errNo)
{
    if (Pstream::parRun())
    {
        Pstream::exit(errNo);
    }
    else
    {
        if (throwExceptions_)
        {
            error errorException(*this);
            messageStream_.rewind();
            throw errorException;
        }
        else
        {
            Perr << endl << *this << endl
                 << "\nFOAM exiting\n" << endl;
            ::exit(errNo);
        }
    }
}
```

**abort() Method** (เมธอด abort)
- Purpose: Emergency stop for catastrophic failures
- Behavior: Immediate termination without cleanup, calls `::abort()`
- Stack trace: Always shows stack trace
- Use case: Memory corruption, segmentation faults, unrecoverable errors

```cpp
void Foam::error::abort()
{
    if (!throwExceptions_ && jobInfo::constructed)
    {
        jobInfo_.add("FatalError", operator dictionary());
        jobInfo_.abort();
    }

    if (abort_)  // FOAM_ABORT environment variable set
    {
        Perr << endl << *this << endl
             << "\nFOAM aborting (FOAM_ABORT set)\n" << endl;
        printStack(Perr);
        ::abort();
    }

    // ... parallel and exception handling cases
}
```

**⭐ Stack Trace Generation**

OpenFOAM provides sophisticated stack trace generation:


The stack trace includes:

**⭐ Parallel Error Handling**

In MPI environments, OpenFOAM handles errors consistently:

```cpp

{
```

The `Pstream::exit()` ensures all processes terminate together when one encounters a fatal error.

### Error Message Formatting (รูปแบบการจัดข้อความข้อผิดพลาด)

**⭐ Formatted Error Output**

The error formatting in `operator<<(Ostream&, const error&)` creates structured, readable output:

```cpp
Foam::Ostream& Foam::operator<<(Ostream& os, const error& fErr)
{
    os << endl
       << fErr.title().c_str() << endl
       << fErr.message().c_str();

    if (error::level >= 2 && fErr.sourceFileLineNumber())
    {
        os << endl << endl
           << "    From function " << fErr.functionName().c_str() << endl
           << "    in file " << fErr.sourceFileName().c_str()
           << " at line " << fErr.sourceFileLineNumber() << '.';
    }

    return os;
}
```

This produces output like:

```

    From function pEqn.H
    in file /path/to/file/pEqn.H at line 67.
```

**⭐ Error Dictionary Conversion**

The error can be converted to a dictionary for programmatic access:

```cpp
Foam::error::operator Foam::dictionary() const
{

    string oneLineMessage(message());
    oneLineMessage.replaceAll('\n', ' ');

    errDict.add("type", word("Foam::error"));
    errDict.add("message", oneLineMessage);
    errDict.add("function", functionName());
```

This allows error information to be processed programmatically, stored in log files, or sent to monitoring systems.

```cpp
#include <iostream>
#include <execinfo.h> // For stack traces

namespace MyCFD
{

// Static member initialization
ErrorLevel ErrorStream::currentLevel_ = ErrorLevel::INFO;

ErrorStream::ErrorStream(const std::string& title, ErrorLevel level)

{
}

template<typename T>
ErrorStream& ErrorStream::operator<<(const T& value)
{
    stream_ << value;
    return *this;
}

ErrorStream& ErrorStream::inFunction(const std::string& funcName,
                                     const std::string& fileName,
                                     int lineNum)
{
    functionName_ = funcName;
    sourceFileName_ = fileName;
    sourceFileLineNumber_ = lineNum;
    return *this;
}

std::string ErrorStream::getMessage() const
{
    return stream_.str();
}

void ErrorStream::exit(int errorCode)
{
    printError();
    terminate();
    _exit(errorCode); // Never reached
}

void ErrorStream::abort()
{
    std::cerr << "\n*** ABORTING ***\n" << std::endl;
    printStack(std::cerr);
    std::abort();
}

void ErrorStream::printError() const
{
    std::cerr << std::endl;

    // Print title based on severity
    switch(level_)
    {
        case ErrorLevel::INFO:
            std::cerr << " --> INFO: ";
            break;
        case ErrorLevel::WARNING:
            std::cerr << " --> WARNING: ";
            break;
        case ErrorLevel::SERIOUS_ERROR:
            std::cerr << " --> SERIOUS ERROR: ";
            break;
        case ErrorLevel::FATAL_ERROR:
            std::cerr << " --> FATAL ERROR: ";
            break;
    }

    std::cerr << title_ << std::endl;
    std::cerr << getMessage();

    if (!functionName_.empty())
    {
        std::cerr << std::endl << std::endl
                  << "    From function " << functionName_ << std::endl
                  << "    in file " << sourceFileName_
                  << " at line " << sourceFileLineNumber_ << '.';
    }

    std::cerr << std::endl;
}

{
    if (level_ == ErrorLevel::FATAL_ERROR)
    {
        std::cerr << "\nTERMINATING\n" << std::endl;
    }
}

// Stack trace utility
void ErrorStream::printStack(std::ostream& os)
{
    const int max_frames = 64;
    void* buffer[max_frames];
    int nptrs = backtrace(buffer, max_frames);

    char** strings = backtrace_symbols(buffer, nptrs);

    if (strings == nullptr)
    {
        os << "No stack trace available" << std::endl;
        return;
    }

    os << "Stack trace:" << std::endl;
    for (int i = 0; i < nptrs; ++i)
    {
        os << "  #" << i << " " << strings[i] << std::endl;
    }

    free(strings);
}

    if (!ioFileName_.empty())

            if (ioEndLine_ >= 0 && ioEndLine_ != ioStartLine_)
```

### Usage Examples (ตัวอย่างการใช้งาน)

```cpp
// File: main.C

#include "myErrorSystem.H"
#include <fstream>

class MeshReader
{
private:
    std::string fileName_;

public:
    MeshReader(const std::string& name) : fileName_(name) {}

    void readMesh()
    {
        std::ifstream file(fileName_);

        if (!file.is_open())
        {
            // Using our error system
            IOErrorIn("readMesh", *this)
                << "Failed to open mesh file: " << fileName_;
        }

        // Read mesh data
        int cellCount;
        file >> cellCount;

        if (file.fail())
        {
            FatalIOError.inFunction("readMesh", __FILE__, __LINE__)
                       .inFile(fileName_)
                       << "Failed to read cell count from mesh file";
        }

        if (cellCount <= 0)
        {
            Warning << "Mesh has " << cellCount << " cells (should be positive)";
        }

        Info << "Successfully read mesh with " << cellCount << " cells";

        file.close();

        if (!file.good())
        {
            SeriousError << "Error encountered while closing mesh file";
        }
    }

    std::string getFileName() const { return fileName_; }
};

int main()
{
    try
    {
        MeshReader reader("mesh.dat");

        // Normal operation
        Info << "Starting mesh reading process";

        reader.readMesh();

        // Example of intentional error
        Warning << "This is a warning message";

        Info << "Program completed successfully";

    }
    catch (const std::exception& e)
    {
        std::cerr << "Caught exception: " << e.what() << std::endl;
        return 1;
    }

    return 0;
}
```

## Exercises (แบบฝึกหัด)

### Exercise 1: Error Message Formatting (แบบที่ 1: การจัดรูปแบบข้อความผิดพลาด)

**Question**: Why does OpenFOAM use a stream-based approach instead of traditional C++ exceptions? What are the advantages and disadvantages of this design?

**Answer**:

OpenFOAM uses a stream-based approach to error handling for several key reasons:

**Advantages**:
1. **Performance Predictability**: Exceptions introduce non-deterministic runtime costs that are unacceptable in CFD where execution time must be predictable for performance analysis and optimization.

2. **Complete Diagnostic Information**: Traditional exceptions only capture the call stack up to the throw point, but OpenFOAM needs complete diagnostic information including function names, file locations, and line numbers from the original error source.

3. **Parallel Execution Support**: In MPI environments, exceptions become complex to propagate across process boundaries. The stream-based approach allows for consistent termination across all processes.

4. **Resource Cleanup Control**: OpenFOAM must ensure proper cleanup of mesh data, field storage, and I/O resources. The stream-based approach allows for controlled resource cleanup before termination.

5. **Automatic Source Location**: The `FatalErrorIn` macro automatically inserts `__FILE__` and `__LINE__`, eliminating the need for manual error location tracking.

**Disadvantages**:
1. **Less Structured Error Handling**: Unlike exceptions, there's no automatic stack unwinding and resource cleanup mechanism.

2. **Manual Cleanup Required**: Developers must ensure proper cleanup before calling `exit()` or `abort()`.

3. **Limited Exception-Specific Features**: Cannot use standard exception features like `try-catch` blocks with specific exception types.

4. **Performance Overhead for Non-Fatal Errors**: Stream-based warnings and info messages always go through the I/O system, even when they might be ignored.

The design choice reflects OpenFOAM's priorities: deterministic performance in scientific computing over traditional exception convenience.

### Exercise 2: Macro Expansion (แบบที่ 2: การขยายแมโคร)

```cpp
::Foam::FatalIOError.inFunction("readBoundary", __FILE__, __LINE_, meshIO)
```

### Exercise 3: Error Level Comparison (แบบที่ 3: เปรียบเทียบระดับข้อผิดพลาด)

**Question**: Compare the behavior of `Info`, `Warning`, `SeriousError`, and `FatalError` streams. When would you use each one? Provide examples for each.

**Answer**:

**Info Stream**
- **Purpose**: General information logging and progress reporting
- **Behavior**: Continues execution normally, output to standard output
- **Use Case**: Normal operation progress, statistics, debugging information

Example:
```cpp
Info << "Time step " << timeStep << " completed in "
     << runTime.elapsedTime() << " seconds" << endl;
Info << "Max residuals: " << maxResiduals;
```

**Warning Stream**
- **Purpose**: Recoverable issues that don't prevent execution but indicate potential problems
- **Behavior**: Continues execution, output to standard error with warning header
- **Use Case**: Boundary conditions with default values, numerical warnings, mesh quality issues

Example:
```cpp
if (mesh.quality() < 0.1)
{
    Warning << "Mesh quality is below recommended threshold of 0.1. "
            << "Using coarser solver settings";
}
```

**SeriousError Stream**
- **Purpose**: Problems that require attention but don't immediately stop execution
- **Behavior**: Continues execution but logs prominently, may indicate inconsistent state
- **Use Case**: Missing optional features, fallback behavior warnings, initialization issues

Example:
```cpp
if (!boundaryField.found("inlet"))
{
    SeriousError << "Inlet boundary not found, using zero gradient condition";
    boundaryField.set("inlet", zeroGradientFvPatchField<Type>::typeName);
}
```

**FatalError Stream**
- **Purpose**: Unrecoverable errors that must stop execution
- **Behavior**: Terminates program after logging, may exit or abort depending on configuration
- **Use Case**: Memory allocation failures, file I/O errors, configuration validation failures

Example:
```cpp
if (!sparseMatrix.solve())
{
    FatalError << "Linear solver failed to converge" << endl
               << "Iterations: " << solver.nIterations()
               << "Final residual: " << solver.finalResidual()
               << exit(FatalError);
}
```

### Exercise 4: Parallel Error Handling (แบบที่ 4: การจัดการข้อผิดพลาดแบบขนาน)

**Question**: How does OpenFOAM handle errors in parallel (MPI) environments? What special considerations must be made for error handling in distributed memory systems?

**Answer**:

OpenFOAM handles parallel errors through several mechanisms:

**1. Consistent Termination Across Processes**
```cpp
if (Pstream::parRun())
{
    Perr << endl << *this << endl
         << "\nFOAM parallel run exiting\n" << endl;
    Pstream::exit(errNo);
}
else
{
    ::exit(errNo);
}
```

The `Pstream::exit()` ensures all processes terminate together when one encounters a fatal error.

**2. Error State Synchronization**
Before reporting an error, OpenFOAM often synchronizes error states across processes:
- Reduce operations to check if any process has encountered an error
- Broadcast error information to all processes
- Ensure consistent error reporting across all processes

**3. Resource Cleanup in Parallel**
In parallel environments, resources must be cleaned up consistently:
- Mesh data distributed across processes must be freed
- File handles must be closed on all processes
- Communication channels must be properly shut down

**4. Deadlock Prevention**
Special care is needed to avoid deadlocks during error handling:
- Don't attempt to communicate between processes after an error is detected
- Use non-blocking communication for error reporting
- Ensure all processes can reach the error handling code

**5. I/O Coordination**
File I/O errors must be handled carefully:
- Only the process that encountered the I/O error should report it
- Other processes should wait for termination
- Log files must be written consistently (avoid interleaved writes)

**6. Signal Handling**
Process termination signals must be handled in all processes:
- SIGSEGV (segmentation fault)
- SIGFPE (floating point exception)
- SIGABRT (abort signal)

These mechanisms ensure that parallel CFD simulations fail gracefully and provide complete diagnostic information without leaving processes in an inconsistent state.

```cpp
RecoveryAction FVErrorStream::suggestAction() const
{
    switch(severity_)
    {
        case FVSeverity::INFO:
            return RecoveryAction::CONTINUE;
        case FVSeverity::WARNING:
            return RecoveryAction::FALLBACK;
        case FVSeverity::RECOVERABLE:
            return RecoveryAction::RETRY;
        case FVSeverity::CRITICAL:
            return RecoveryAction::ABORT;
        case FVSeverity::FATAL:
            return RecoveryAction::ABORT;
        default:
            return RecoveryAction::NONE;
    }
}

void FVErrorStream::report() const
{
    printHeader();
    printDetails();
}

void FVErrorStream::handle()
{
    report();

    auto action = suggestAction();

    switch(action)
    {
        case RecoveryAction::CONTINUE:
            // Do nothing, continue execution
            break;
        case RecoveryAction::FALLBACK:
            // Try recovery strategy
            for (const auto& [key, recovery] : recoveryStrategies)
            {
                if (recovery(*this))
                {
                    std::cout << "Error recovered using strategy: " << key << std::endl;
                    return;
                }
            }
            break;
        case RecoveryAction::RETRY:
            // Could implement retry logic here
            std::cout << "Retrying operation..." << std::endl;
            break;
        case RecoveryAction::ABORT:
            std::cout << "Aborting due to critical error" << std::endl;
            std::exit(1);
        default:
            break;
    }
}
```
