## Part 1: The Problem ŌĆö Field Without Mesh

### A Silent Bug Hiding in Plain Sight

On Day 01 we wrote code like this:

```cpp
// Day 01 ŌĆö no mesh, no invariant
Field<double> pressure(200);   // 200 values
Field<double> velocity(200);   // 200 values

// ... somewhere far away ...
Mesh mesh(100);                // 100 cells
for (int i = 0; i < mesh.nCells(); ++i) {
    pressure[i] = computePressure(mesh, i);
    velocity[i] = computeVelocity(mesh, i);
}
```

This looks fine. But suppose a colleague changes the mesh size:

```cpp
Mesh mesh(300);    // resize: 300 cells now
// Fields were never resized ŌĆö still 200 entries
```

The loop now runs from `i = 0` to `i = 299`. At `i = 200` we write past the end of `pressure` and `velocity`. The program has **undefined behavior**: no exception, no assert, just silent memory corruption that may only manifest as a wrong answer hours later.

### Three Common Failure Modes

**Failure Mode 1 ŌĆö Wrong size at construction:**

```cpp
Field<double> p(200);   // user guesses 200
// Real mesh has 100 cells. p has 2x too many values.
// Nothing catches this. Loops iterate over garbage.
```

**Failure Mode 2 ŌĆö Boundary condition loop goes out of bounds:**

```cpp
void applyBC(Field<double>& f, const Mesh& mesh, double wallValue) {
    // Assume last cell is adjacent to wall
    f[mesh.nCells()]     = wallValue;   // OUT OF BOUNDS if f.size() != mesh.nCells()+1
    f[mesh.nCells() - 1] = wallValue;
}
```

With a 100-cell mesh and a 200-element field, `f[100]` is valid memory but is the wrong conceptual location. The bug is logical, not a crash ŌĆö even harder to catch.

**Failure Mode 3 ŌĆö Field outlives its mesh:**

```cpp
Field<double>* pField = nullptr;
{
    Mesh localMesh(50);
    Field<double> f(localMesh.nCells());
    pField = &f;
}   // localMesh destroyed here. f also destroyed.
// pField is a dangling pointer ŌĆö use-after-free.
```

No invariant between `Field` and `Mesh` means there is nothing in the type system to prevent this.

### The Root Cause

The root cause is simple: **`Field<T>` has no knowledge of the mesh it belongs to.** Its size is set by an integer at construction and then drifts independently of any mesh object. This violates the principle that related invariants should be co-located and enforced by the type.

> **Design goal:** Make it impossible to create a `GeometricField<T>` without a `Mesh`, and make its size permanently bound to `mesh.nCells()`. The bug cannot exist if the type won't allow it.

## Part 2: The Concept ŌĆö Topology vs. Values

### Two Distinct Concerns

A CFD simulation has two fundamentally different kinds of data:

**Topology (Mesh):**
- How many cells are there? How many faces?
- Where are the cell centres?
- Which faces connect which cells?
- What is the volume of each cell? The area of each face?

This data describes the **shape** of the computational domain. It is read once from a mesh file (or generated by a mesher) and does not change during a simulation run (for fixed-mesh solvers). It is the same for every physical field.

**Values (Field):**
- What is the pressure at cell $i$?
- What is the velocity vector at face $j$?
- What is the temperature at cell $k$?

This data changes every time step. Each physical quantity has its own array, but all arrays live on the **same mesh**.

### Why Separate Them?

**Memory efficiency.** For a 1-million-cell mesh with 20 scalar fields, the mesh topology data (cell centres, face areas, connectivity) is approximately 8 MB. If each field stored a private copy of the mesh, that would be $20 \times 8 = 160$ MB wasted on duplicate geometry. Shared topology costs 8 MB once.

$$
\text{Memory savings} = (N_\text{fields} - 1) \times M_\text{mesh}
$$

where $M_\text{mesh}$ is the mesh memory footprint. At 20 fields and 8 MB per mesh copy, savings are 152 MB.

**Consistency.** A field that holds a `const Mesh&` reference cannot diverge from the mesh ŌĆö they are the same object. An operation that modifies the mesh (e.g., mesh refinement) forces all fields to be rebuilt from the updated reference.

**Single responsibility.** The `Mesh` class is responsible only for geometry. The `GeometricField<T>` class is responsible only for storing and accessing one array of values on that mesh. Neither class needs to know what the other is doing internally.

### The OpenFOAM Design

ŌŁÉ In OpenFOAM, this pattern is expressed through the `GeometricField` template:

```cpp
// Simplified OpenFOAM signature (src/OpenFOAM/fields/GeometricFields/GeometricField/GeometricField.H)
template<class Type, template<class> class PatchField, class GeoMesh>
class GeometricField
    : public DimensionedField<Type, GeoMesh>
{
    // Internal field: DimensionedField<Type, GeoMesh>
    // Boundary field: GeometricBoundaryField<Type, PatchField, GeoMesh>
};
```

Three template parameters carry three concerns:

| Parameter | Role | Example |
|-----------|------|---------|
| `Type` | Element type of each value | `scalar`, `vector`, `tensor` |
| `PatchField` | How boundary patches store data | `fixedValueFvPatchField`, `zeroGradientFvPatchField` |
| `GeoMesh` | Which mesh type owns the field | `volMesh` (cell centres), `surfaceMesh` (face centres) |

Our simplified version collapses this to two concerns: an element type `T` and a `const Mesh&`. We omit patch fields and multiple mesh types to keep the lesson focused on the separation of topology from values.

The arrow is a dependency (dashed), not ownership. `GeometricField<T>` does not own the `Mesh` ŌĆö it holds a `const` reference. The `Mesh` must outlive all fields that reference it. This is a deliberate design choice that mirrors how OpenFOAM's `fvMesh` outlives all `volScalarField` and `volVectorField` objects in a solver.

## Part 3: Core Implementation

### 3a. The Mesh Struct

```cpp
// file: day03_mesh_field.cpp
#include <vector>
#include <stdexcept>
#include <iostream>
#include <iomanip>
#include <string>
#include <cassert>

// ---------------------------------------------------------------------------
// Mesh: stores topology and geometry for a 1D uniform mesh on [0, 1].
//
// Design notes:
//   - nCells_: number of control volumes (cells)
//   - nFaces_: number of cell interfaces (nCells + 1 for 1D)
//   - cellCentres_: x-coordinate of each cell centre
//
// The mesh is immutable after construction ŌĆö no setters.
// ---------------------------------------------------------------------------
class Mesh {
public:
    explicit Mesh(int nCells)
        : nCells_(nCells)
        , nFaces_(nCells + 1)
        , cellCentres_(nCells, 0.0)
    {
        if (nCells <= 0)
            throw std::invalid_argument("Mesh: nCells must be positive");

        // Uniform 1D mesh on [0, 1]: centre of cell i is at (i + 0.5) / nCells
        for (int i = 0; i < nCells_; ++i)
            cellCentres_[i] = (i + 0.5) / static_cast<double>(nCells_);
    }

    // Non-copyable: sharing is via const reference, never by value
    Mesh(const Mesh&)            = delete;
    Mesh& operator=(const Mesh&) = delete;
    Mesh(Mesh&&)                 = default;
    Mesh& operator=(Mesh&&)      = default;

    int nCells() const noexcept { return nCells_; }
    int nFaces() const noexcept { return nFaces_; }

    double cellCentre(int i) const {
        if (i < 0 || i >= nCells_)
            throw std::out_of_range("Mesh::cellCentre: index out of range");
        return cellCentres_[i];
    }

    const std::vector<double>& cellCentres() const noexcept {
        return cellCentres_;
    }

private:
    int                 nCells_;
    int                 nFaces_;
    std::vector<double> cellCentres_;
};
```

Key design decisions:

- The `Mesh` is **non-copyable by value** (copy constructor deleted). Sharing happens through `const Mesh&`. This prevents accidental deep copies when passing meshes to functions.
- The constructor validates `nCells > 0` and throws `std::invalid_argument` ŌĆö fail early, fail loudly.
- `cellCentre(int i)` does bounds checking and throws `std::out_of_range`. The raw `cellCentres()` accessor returns a `const` reference for read-only iteration.

### 3b. The GeometricField Template

```cpp
// ---------------------------------------------------------------------------
// GeometricField<T>: a field of values living on a Mesh.
//
// Invariant (established at construction, never violated):
//   values_.size() == static_cast<std::size_t>(mesh_.nCells())
//
// The invariant is maintained because:
//   1. values_ is sized at construction from mesh_.nCells()
//   2. No method changes values_.size() after construction
//   3. mesh_ is a const reference ŌĆö nCells() cannot change
// ---------------------------------------------------------------------------
template<typename T>
class GeometricField {
public:
    // Construction always requires a Mesh ŌĆö no default constructor.
    // This makes the invariant impossible to violate at the type level.
    explicit GeometricField(const Mesh& mesh, T initVal = T{})
        : mesh_(mesh)
        , values_(static_cast<std::size_t>(mesh.nCells()), initVal)
    {}

    // Allow constructing from a vector of the right size.
    GeometricField(const Mesh& mesh, std::vector<T> values)
        : mesh_(mesh)
        , values_(std::move(values))
    {
        if (static_cast<int>(values_.size()) != mesh_.nCells())
            throw std::invalid_argument(
                "GeometricField: values.size() != mesh.nCells()");
    }

    // Element access with bounds checking
    T& operator[](int i) {
        if (i < 0 || i >= mesh_.nCells())
            throw std::out_of_range("GeometricField: index out of range");
        return values_[static_cast<std::size_t>(i)];
    }

    const T& operator[](int i) const {
        if (i < 0 || i >= mesh_.nCells())
            throw std::out_of_range("GeometricField: index out of range");
        return values_[static_cast<std::size_t>(i)];
    }

    // Iterators ŌĆö compatible with range-for and std algorithms
    typename std::vector<T>::iterator       begin()       { return values_.begin(); }
    typename std::vector<T>::iterator       end()         { return values_.end(); }
    typename std::vector<T>::const_iterator begin() const { return values_.begin(); }
    typename std::vector<T>::const_iterator end()   const { return values_.end(); }

    // Metadata
    int         size()  const noexcept { return mesh_.nCells(); }
    const Mesh& mesh()  const noexcept { return mesh_; }

    // Field arithmetic: element-wise add (same mesh required)
    GeometricField operator+(const GeometricField& rhs) const {
        if (&mesh_ != &rhs.mesh_)
            throw std::invalid_argument(
                "GeometricField::operator+: fields must share the same Mesh");
        std::vector<T> result(values_.size());
        for (std::size_t i = 0; i < values_.size(); ++i)
            result[i] = values_[i] + rhs.values_[i];
        return GeometricField(mesh_, std::move(result));
    }

    // Scale by a scalar factor
    GeometricField operator*(T factor) const {
        std::vector<T> result(values_.size());
        for (std::size_t i = 0; i < values_.size(); ++i)
            result[i] = values_[i] * factor;
        return GeometricField(mesh_, std::move(result));
    }

private:
    const Mesh&     mesh_;    // non-owning reference ŌĆö Mesh must outlive this field
    std::vector<T>  values_;  // exactly mesh_.nCells() elements, always
};
```

Critical points:

1. **No default constructor.** A `GeometricField<T>` cannot exist without a `Mesh`. The compiler enforces this ŌĆö there is no code path that creates an unattached field.
2. **The `operator+` checks `&mesh_ == &rhs.mesh_`** ŌĆö same object in memory. Adding fields from different meshes is a logic error caught at runtime.
3. **The second constructor validates size** before accepting an externally-provided `std::vector<T>`. This is the only entry point where a mismatch could occur, and it is guarded with a throw.

### 3c. Factory Functions for Common Field Initializations

```cpp
// ---------------------------------------------------------------------------
// Factory functions: idiomatic C++ alternatives to constructors with
// complex initialization logic. Return by value ŌĆö NRVO eliminates copies.
// ---------------------------------------------------------------------------

// Create a uniform field (same value everywhere)
GeometricField<double> makeUniformField(const Mesh& mesh, double value) {
    return GeometricField<double>(mesh, value);
}

// Create a linear profile: value at cell i interpolates linearly from lo to hi
//   value(x) = lo + (hi - lo) * x,  x in [0, 1]
GeometricField<double> makeLinearProfile(
    const Mesh& mesh, double lo, double hi)
{
    std::vector<double> vals(static_cast<std::size_t>(mesh.nCells()));
    for (int i = 0; i < mesh.nCells(); ++i) {
        double x = mesh.cellCentre(i);   // x in (0, 1)
        vals[static_cast<std::size_t>(i)] = lo + (hi - lo) * x;
    }
    return GeometricField<double>(mesh, std::move(vals));
}

// Create a sinusoidal profile: value(x) = amplitude * sin(pi * x)
GeometricField<double> makeSineProfile(
    const Mesh& mesh, double amplitude)
{
    constexpr double pi = 3.14159265358979323846;
    std::vector<double> vals(static_cast<std::size_t>(mesh.nCells()));
    for (int i = 0; i < mesh.nCells(); ++i) {
        double x = mesh.cellCentre(i);
        vals[static_cast<std::size_t>(i)] = amplitude * std::sin(pi * x);
    }
    return GeometricField<double>(mesh, std::move(vals));
}
```

Factory functions keep the `GeometricField` constructor clean (it does not need to know about sine waves or linear ramps). The initialization logic is decoupled from the type ŌĆö a direct application of the Single Responsibility Principle.

### 3d. Complete main.cpp

```cpp
// ---------------------------------------------------------------------------
// main() ŌĆö demonstrates Mesh + GeometricField usage
// ---------------------------------------------------------------------------
void printField(
    const std::string&           name,
    const GeometricField<double>& f,
    const std::string&           unit)
{
    const Mesh& mesh = f.mesh();
    std::cout << "\n" << name << " (" << unit << "):\n";
    std::cout << std::fixed << std::setprecision(4);
    for (int i = 0; i < f.size(); ++i) {
        std::cout << "  [" << std::setw(2) << i << "]"
                  << "  x = " << std::setw(6) << mesh.cellCentre(i)
                  << "   " << name << " = " << std::setw(10) << f[i]
                  << " " << unit << "\n";
    }
}

int main() {
    // -----------------------------------------------------------------------
    // 1. Create a 10-cell 1D mesh on [0, 1]
    // -----------------------------------------------------------------------
    Mesh mesh(10);
    std::cout << "Mesh: " << mesh.nCells() << " cells, "
              << mesh.nFaces() << " faces\n";

    // -----------------------------------------------------------------------
    // 2. Create a uniform pressure field (101325 Pa everywhere)
    //    Both fields share the SAME Mesh object ŌĆö no copy of geometry data.
    // -----------------------------------------------------------------------
    auto pressure    = makeUniformField(mesh, 101325.0);
    auto temperature = makeLinearProfile(mesh, 300.0, 400.0);
    auto velocity    = makeSineProfile(mesh, 1.0);

    // -----------------------------------------------------------------------
    // 3. Print fields
    // -----------------------------------------------------------------------
    printField("pressure",    pressure,    "Pa");
    printField("temperature", temperature, "K");
    printField("velocity",    velocity,    "m/s");

    // -----------------------------------------------------------------------
    // 4. Demonstrate shared mesh (same address, no copy)
    // -----------------------------------------------------------------------
    std::cout << "\nMesh address:              " << &mesh << "\n";
    std::cout << "pressure.mesh() address:   " << &pressure.mesh() << "\n";
    std::cout << "temperature.mesh() address:" << &temperature.mesh() << "\n";
    std::cout << "velocity.mesh() address:   " << &velocity.mesh() << "\n";
    bool allSame = (&pressure.mesh() == &mesh)
                && (&temperature.mesh() == &mesh)
                && (&velocity.mesh() == &mesh);
    std::cout << "All fields share one Mesh (no copy): "
              << (allSame ? "true" : "false") << "\n";

    // -----------------------------------------------------------------------
    // 5. Field arithmetic on the shared mesh
    //    Create a "combined" field = pressure * scale + temperature offset
    // -----------------------------------------------------------------------
    auto scaled = pressure * 0.5;    // half the pressure everywhere
    std::cout << "\nscaled pressure at cell 0: " << scaled[0] << " Pa\n";
    std::cout << "scaled pressure at cell 9: " << scaled[9] << " Pa\n";

    // -----------------------------------------------------------------------
    // 6. Demonstrate the invariant being enforced
    // -----------------------------------------------------------------------
    try {
        std::vector<double> wrongSizeVec(5, 0.0);  // 5 != 10
        GeometricField<double> badField(mesh, wrongSizeVec);
    } catch (const std::invalid_argument& e) {
        std::cout << "\nCaught expected error: " << e.what() << "\n";
    }

    return 0;
}
```

## Part 4: Profiling / Benchmark ŌĆö Shared Mesh vs. Copied Mesh

### The Memory Cost of Not Sharing

Consider a 1D mesh with $N$ cells. The mesh stores:
- `nCells_`: 4 bytes (int)
- `nFaces_`: 4 bytes (int)
- `cellCentres_`: $N \times 8$ bytes (double per cell)
- `std::vector` overhead: 24 bytes (pointer + size + capacity)

For large $N$, the dominant cost is `cellCentres_`: approximately $8N$ bytes. In a realistic 3D mesh, the topology also includes face owner/neighbour arrays, face area vectors, and cell volumes ŌĆö easily $40N$ to $80N$ bytes for a production mesh. We use $8N$ bytes as a conservative lower bound for this analysis.

For $F$ fields on the same mesh:

$$
M_\text{shared} = 8N + F \times 8N_\text{values}
$$

$$
M_\text{copied} = F \times (8N + 8N_\text{values}) = F \times 16N
$$

Since each field has exactly $N$ values (one per cell), $N_\text{values} = N$:

$$
M_\text{shared} = 8N + 8FN = 8N(1 + F)
$$

$$
M_\text{copied} = 16FN
$$

$$
\text{Savings} = M_\text{copied} - M_\text{shared} = 16FN - 8N(1 + F) = 8N(F - 1)
$$

### Memory Comparison Table

The table below shows memory usage for 10 fields (`F = 10`) at various mesh sizes. "Shared" means one `GeometricField<T>` holding `const Mesh&`. "Copied" means each field holds its own `Mesh` by value.

| N (cells) | Mesh size (MB) | 10 fields ŌĆö shared (MB) | 10 fields ŌĆö copied (MB) | Savings (MB) |
|----------:|---------------:|------------------------:|------------------------:|-------------:|
| 10,000    | 0.08           | 0.88                    | 1.60                    | 0.72         |
| 100,000   | 0.76           | 8.36                    | 15.26                   | 6.90         |
| 1,000,000 | 7.63           | 83.92                   | 152.59                  | 68.67        |
| 5,000,000 | 38.15          | 419.43                  | 762.94                  | 343.51       |

At 1 million cells and 10 fields, the shared-mesh design saves approximately **69 MB** of RAM compared to value-by-copy. At 5 million cells (a modest industrial-scale mesh), savings exceed **343 MB**.

The `sizeof` of `GeometricField<double>` is 32 bytes: one 8-byte reference (which the compiler stores as a pointer), plus one 24-byte `std::vector<double>` (pointer + size + capacity). This is constant ŌĆö the actual data lives on the heap and is **not** part of the object's in-place size.

### Performance: Construction Time

For large meshes, the dominant construction cost is the `std::vector` allocation in `values_`. Mesh construction (the `cellCentres_` fill) is a one-time cost paid once regardless of how many fields are created.

| N (cells) | Mesh construction | Field construction (each) | 10 fields total |
|----------:|------------------:|--------------------------:|----------------:|
| 100,000   | ~0.3 ms           | ~0.4 ms                   | ~4.3 ms         |
| 1,000,000 | ~3.1 ms           | ~4.2 ms                   | ~45.1 ms        |
| 5,000,000 | ~15.4 ms          | ~21.0 ms                  | ~225.4 ms       |

ŌÜĀ’ĖÅ These timings are approximate estimates based on typical `std::vector` allocation benchmarks on a modern x86-64 CPU. Actual results depend on allocator, CPU cache state, and compiler flags. Measure on your hardware with `-O2`.

The key observation: mesh construction time is paid **once**. Creating 10 fields on the same mesh costs 10 field allocations, not 10 mesh allocations plus 10 field allocations.

**Exercise 2 ŌĆö Field difference.**
Add an `operator-` to `GeometricField<T>` that subtracts two fields element-wise. Verify the same `&mesh_ == &rhs.mesh_` guard applies. Create two fields and print their difference.

**Solution sketch:**
```cpp
GeometricField operator-(const GeometricField& rhs) const {
    if (&mesh_ != &rhs.mesh_)
        throw std::invalid_argument("fields must share the same Mesh");
    std::vector<T> result(values_.size());
    for (std::size_t i = 0; i < values_.size(); ++i)
        result[i] = values_[i] - rhs.values_[i];
    return GeometricField(mesh_, std::move(result));
}
```

**Exercise 3 ŌĆö L2 norm.**
Write a free function `double l2norm(const GeometricField<double>& f)` that computes:

$$
\|f\|_2 = \sqrt{\frac{1}{N} \sum_{i=0}^{N-1} f[i]^2}
$$

Test it on the uniform pressure field (expected: 101325.0) and the sine velocity field.

**Solution sketch:**
```cpp
#include <cmath>
#include <numeric>

double l2norm(const GeometricField<double>& f) {
    double sum = 0.0;
    for (int i = 0; i < f.size(); ++i)
        sum += f[i] * f[i];
    return std::sqrt(sum / f.size());
}
```

## Summary

Today's core lesson in three sentences:

1. A raw `Field<T>` with no mesh attachment allows silent size mismatches ŌĆö bugs that manifest as wrong answers, not crashes.
2. Separating topology (`Mesh`) from values (`GeometricField<T>`) and binding them through a `const Mesh&` enforces the invariant `field.size() == mesh.nCells()` structurally ŌĆö the bug cannot be written.
3. Shared topology (by reference) saves significant memory at scale: ~69 MB for 10 fields on a 1M-cell mesh, growing linearly with field count.

OpenFOAM's `GeometricField<Type, PatchField, GeoMesh>` is a production-scale extension of this same design: three template parameters separate element type, boundary behavior, and mesh type ŌĆö all grounded in the same fundamental principle of keeping topology and values cleanly distinct.
