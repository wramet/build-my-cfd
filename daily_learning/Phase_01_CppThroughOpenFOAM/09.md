# Day 09: Expression Templates — Field Arithmetic Without Temporaries

## Part 1: Pattern Identification

### ⭐ What are Expression Templates?

**Expression templates** are a C++ technique that **builds expression trees at compile time** to avoid temporary objects.

```cpp
// Naive: Creates temporary
result = (a + b) + c;  // Creates tmp for (a+b), then adds c

// Expression templates: No temporary
result = (a + b) + c;  // Builds tree, evaluates directly into result
```

> **⭐ Verified Fact:** OpenFOAM uses expression templates in `FieldFunctions.H` (see `src/OpenFOAM/fields/Fields/Field/FieldFunctions.H`).

### ⭐ The Temporary Problem

**Naive field arithmetic:**
```cpp
// Field addition (naive implementation)
template<typename T>
Field<T> operator+(const Field<T>& a, const Field<T>& b)
{
    Field<T> result(a.size());  // Allocate
    for (std::size_t i = 0; i < a.size(); ++i)
    {
        result[i] = a[i] + b[i];
    }
    return result;  // Return by value (copy!)
}
```

**Problem with `result = a + b + c`:**
1. Compute `tmp1 = a + b` → Allocate + copy
2. Compute `tmp2 = tmp1 + c` → Allocate + copy
3. Assign `result = tmp2` → Copy again

**Total: 3 allocations + 3 copies for 1 expression!**

### ⭐ Expression Template Solution

```cpp
// Expression template (no temporary)
template<typename Expr1, typename Expr2, typename Op>
class BinaryExpr
{
    const Expr1& left_;
    const Expr2& right_;

public:
    BinaryExpr(const Expr1& e1, const Expr2& e2) : left_(e1), right_(e2) {}

    T operator[](std::size_t i) const { return Op::apply(left_[i], right_[i]); }
};

// Build expression tree (lazy evaluation)
auto expr = BinaryExpr{a, b, Add{}} + c;
// When accessing expr[i]: computes a[i] + b[i] + c[i] on the fly
```

## Part 2: Source Code Reading

### ⭐ FieldFunctions.H Expression Templates

```cpp
// File: src/OpenFOAM/fields/Fields/Field/FieldFunctions.H

template<class Type>
class Field
:
    public List<Type>
{
public:
    // -[1] Field addition with expression templates ⭐
    template<class Type2>
    void operator=(const tmp<Field<Type2>>& tf)
    {
        const Field<Type2>& f = tf();
        forAll(*this, i)
        {
            this->operator[](i) = f[i];  // Direct assignment
        }
        tf.clear();  // Mark tmp as empty
    }

    // -[2] Addition operator
    template<class Type2>
    tmp<Field<typename typeOfSum<Type, Type2>::type>>
    operator+(const Field<Type2>& f) const
    {
        tmp<Field<typename typeOfSum<Type, Type2>::type>> tRes
        (
            new Field<typename typeOfSum<Type, Type2>::type>(this->size())
        );
        Field<typename typeOfSum<Type, Type2>::type>& res = tRes.ref();

        forAll(*this, i)
        {
            res[i] = this->operator[](i) + f[i];  // Element-wise add
        }

        return tRes;  // ⭐ Move (not copy)
    }
};
```

### ⭐ How OpenFOAM Avoids Temporaries

**Strategy 1: `tmp<>` move semantics**
```cpp
// Return tmp, not Field
tmp<Field<scalar>> operator+(const Field<scalar>& a, const Field<scalar>& b);

// Usage
auto result = a + b;  // result is tmp<>, moved (not copied)
```

**Strategy 2: Direct assignment**
```cpp
// tmp assignment bypasses copy
void operator=(const tmp<Field<scalar>>& tResult)
{
    // Write directly into this field
    for (label i = 0; i < size(); ++i)
    {
        this->v_[i] = tResult()[i];
    }
    tResult.clear();  // Prevent deletion
}
```

## Part 3: Implementation Exercise

### Mini-Implementation: Expression Templates

Create file `expression_templates.H`:

```cpp
#ifndef expression_templates_H
#define expression_templates_H

#include <vector>
#include <iostream>
#include <type_traits>

// -[1] Expression tag base
struct Expr {};

// -[2] Literal (leaf node)
template<typename T>
class Literal : public Expr
{
    T val_;

public:
    Literal(T val) : val_(val) {}

    T eval(std::size_t) const { return val_; }
};

// -[3] Variable reference (leaf node)
template<typename T>
class Variable : public Expr
{
    const std::vector<T>& data_;

public:
    Variable(const std::vector<T>& data) : data_(data) {}

    T eval(std::size_t i) const { return data_[i]; }
};

// -[4] Binary operation (internal node)
template<typename Left, typename Right, typename Op>
class BinaryOp : public Expr
{
    Left left_;
    Right right_;
    Op op_;

public:
    BinaryOp(const Left& l, const Right& r)
    : left_(l), right_(r), op_() {}

    auto eval(std::size_t i) const
    -> decltype(op_(left_.eval(i), right_.eval(i)))
    {
        return op_(left_.eval(i), right_.eval(i));
    }
};

// -[5] Operations
struct Add
{
    template<typename T>
    auto operator()(T a, T b) const -> decltype(a + b) { return a + b; }
};

struct Multiply
{
    template<typename T>
    auto operator()(T a, T b) const -> decltype(a * b) { return a * b; }
};

struct Subtract
{
    template<typename T>
    auto operator()(T a, T b) const -> decltype(a - b) { return a - b; }
};

// -[6] Operator overloads (build expression tree)
template<typename T>
Literal<T> lit(T val) { return Literal<T>(val); }

template<typename T>
Variable<T> var(const std::vector<T>& data) { return Variable<T>(data); }

template<typename Left, typename Right>
auto operator+(const Left& l, const Right& r) -> BinaryOp<Left, Right, Add>
{
    return BinaryOp<Left, Right, Add>(l, r);
}

template<typename Left, typename Right>
auto operator*(const Left& l, const Right& r) -> BinaryOp<Left, Right, Multiply>
{
    return BinaryOp<Left, Right, Multiply>(l, r);
}

// -[7] Evaluator
template<typename Expr, typename T = void>
class Evaluator
{
    const Expr& expr_;
    std::vector<T> result_;

public:
    Evaluator(const Expr& expr, std::size_t n)
    : expr_(expr), result_(n)
    {}

    const std::vector<T>& eval()
    {
        for (std::size_t i = 0; i < result_.size(); ++i)
        {
            result_[i] = expr_.eval(i);
        }
        return result_;
    }
};

#endif
```

## Part 4: Design Trade-offs

### Expression Template Trade-offs

| Aspect | Expression Templates | Naive Evaluation |
|--------|---------------------|------------------|
| **Temporaries** | Zero (lazy eval) | N-1 for N operators |
| **Compilation time** | Longer | Shorter |
| **Binary size** | Larger | Smaller |
| **Debugging** | Harder (complex types) | Easier |
| **Runtime performance** | Faster | Slower |

### When to Use Expression Templates

**✅ Use when:**
- Operating on **large objects** (fields, matrices)
- Expressions have **many operations**
- Performance is critical (CFD kernels)

**❌ Avoid when:**
- Objects are small (scalars)
- Compilation time is critical
- Debugging complex bugs

### OpenFOAM's Strategy

OpenFOAM uses a **hybrid approach**:

1. **Expression templates** for field operations
2. **`tmp<>`** for intermediate results
3. **Lazy evaluation** in some places
4. **Eager evaluation** in others (for simplicity)

**Example:**
```cpp
// Complex expression (uses expression templates + tmp)
auto result = fvc::div(phi, U) + fvc::laplacian(nu, U) - fvc::grad(p);

// Efficiently evaluates as:
// for (cell i):
//     result[i] = div(phi,U)[i] + laplacian(nu,U)[i] - grad(p)[i]
```

## Summary

**⭐ Key Takeaways:**

1. **Expression templates** = build trees, evaluate lazily
2. **Eliminates temporaries** in complex expressions
3. **Lazy evaluation** = compute on-demand
4. **OpenFOAM uses expr templates** in field operations
5. **Enables efficient operator overloading** for large objects
