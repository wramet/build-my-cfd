## Part 1: The Problem — Unconstrained Templates

### What Goes Wrong with "Any Type"?

On Day 01, we built `Field<T>` that works with any type `T`. But what if someone instantiates it with the wrong type?

```cpp
Field<std::string> stringField(10);  // What does this even mean?
Field<std::vector<int>> nestedField(5);  // Field of vectors?
```

These compile successfully but make no sense mathematically. What does "magnitude of a string" mean? What's the "zero vector" for a `vector<int>`?

**The OpenFOAM approach (C++98/03):** Use `pTraits<T>` + `static_assert` to catch errors at compile time.

**The Modern C++ approach (C++20):** Use **Concepts** to express requirements directly in the template signature.

---

### Why `static_assert` Isn't Enough

Before Concepts, we could write:

```cpp
template<class T>
class Field {
    static_assert(std::is_arithmetic<T>::value,
        "Field<T> requires an arithmetic type (scalar, vector, or tensor)");
    // ...
};
```

**Problems:**
1. **Too late** — Error fires inside the class definition, not at the call site
2. **Cryptic** — Error message shows 40+ lines of template backtrace before reaching your `static_assert`
3. **Inflexible** — Can't say "has `operator+`" without complex SFINAE

**Enter C++20 Concepts:**
```cpp
template<FieldElement T>  // ← Concept constraint
class Field { /* ... */ };

Field<double> f1;    // ✅ OK
Field<std::string> f2;  // ❌ Clear error: "std::string does not satisfy FieldElement"
```

The error appears **at the instantiation line** with a **3-line message** instead of 40 lines of template gibberish.

## Part 2: C++20 Concepts — The Modern Approach

### What is a Concept?

A **Concept** is a named predicate that evaluates to `true` or `false` at compile time. It's a compile-time check for "does this type support what we need?"

```cpp
#include <concepts>

// Define a Concept: "T must be a scalar or vector type"
template<typename T>
concept FieldElement = std::is_arithmetic_v<T> || requires(T t) {
    { t.x } -> std::convertible_to<double>;
    { t.y } -> std::convertible_to<double>;
    { t.z } -> std::convertible_to<double>;
    { t + t } -> std::same_as<T>;
    { t * double{} } -> std::same_as<T>;
};

// Use the Concept to constrain a template
template<FieldElement T>
class Field {
    std::vector<T> data_;
public:
    // Now guaranteed: T is arithmetic OR has x,y,z and operator+
    T sum() const {
        return std::reduce(data_.begin(), data_.end(), T{});
    }
};
```

### Standard Library Concepts (You Don't Need to Write These!)

C++20 provides ready-to-use Concepts in `<concepts>`:

| Concept | Meaning | Example Types |
|---------|---------|---------------|
| `std::integral<T>` | `T` is an integer type | `int`, `long`, `size_t` |
| `std::floating_point<T>` | `T` is a floating-point type | `float`, `double`, `long double` |
| `std::arithmetic<T>` | `T` is integral or floating-point | `int`, `double`, etc. |
| `std::signed_integral<T>` | Signed integer | `int`, `long` |
| `std::unsigned_integral<T>` | Unsigned integer | `unsigned int`, `size_t` |
| `std::convertible_to<T, U>` | `T` can convert to `U` | All derived→base, `double`→`int` |
| `std::same_as<T, U>` | `T` and `U` are the same type | Exactly `int` == `int` |
| `std::default_initializable<T>` | `T` can be default-constructed | Most types |
| `std::copyable<T>` | `T` can be copied | All value types |
| `std::semiregular<T>` | `T` is default-initializable + copyable | Plain old data types |
| `std::regular<T>` | `T` is semiregular + equality-comparable | `int`, `double`, `Vec3` |

**Most common use:** `std::arithmetic<T>` for scalar/vector/tensor types that support numeric operations.

```cpp
template<std::arithmetic T>  // ← One concept, clear intent
class Field { /* ... */ };
```

### Custom Concepts: Combining Requirements

You can combine multiple requirements using logical operators:

```cpp
// Concept: "T must be copyable and have operator+ and operator*"
template<typename T>
concept Numeric = std::copyable<T> && requires(T a, T b, double scalar) {
    { a + b } -> std::convertible_to<T>;
    { a * scalar } -> std::convertible_to<T>;
    { a += b } -> std::same_as<T&>;
};

// Concept: "T must be a 3D vector with x, y, z components"
template<typename T>
concept Vector3D = requires(T v) {
    { v.x } -> std::convertible_to<double>;
    { v.y } -> std::convertible_to<double>;
    { v.z } -> std::convertible_to<double>;
};

// Use both concepts together
template<Vector3D T>  // ← T must satisfy Vector3D
class VectorField { /* ... */ };
```

### `requires` Clause Syntax

There are **three equivalent ways** to write a constrained template:

```cpp
// Style 1: Template argument with Concept (preferred)
template<FieldElement T>
class Field1 { /* ... */ };

// Style 2: trailing requires clause
template<typename T>
requires FieldElement<T>
class Field2 { /* ... */ };

// Style 3: abbreviated function template (auto + Concept)
void process(const FieldElement auto& field) { /* ... */ }
```

**Recommendation:** Use **Style 1** for class templates (clearest), **Style 3** for function templates (concise).

## Part 3: Error Messages — Concepts vs SFINAE

### The Old Way: SFINAE Hell (C++98/11/14)

Before C++20, constraining templates required **SFINAE** (Substitution Failure Is Not An Error) — a technique so complex it became a meme:

```cpp
// Pre-C++20: Using SFINAE to constrain templates
#include <type_traits>

template<typename T,
    typename std::enable_if<std::is_arithmetic<T>::value, int>::type = 0>
class Field {
    std::vector<T> data_;
public:
    T sum() const { /* ... */ }
};

// What happens if you write:
Field<std::string> bad;  // Error message is 40+ lines of...
```

**Typical error message (g++ with SFINAE):**
```text
error: no type named 'type' in 'struct std::enable_if<false, int>'
note: in instantiation of template class 'Field<std::__cxx11::basic_string<char> >'
note:   required from here
note: the expression 'std::is_arithmetic<std::__cxx11::basic_string<char> >::value' evaluates to false
note: in expansion of macro 'ENABLE_IF_ARITHMETIC'
note:   candidate expects 1 argument, 2 provided
note:   template<class T, typename std::enable_if<std::is_arithmetic<T>::value, int>::type* = nullptr>
note:   no known conversion for argument 1 from 'std::__cxx11::basic_string<char>' to 'int'
note: (and 30 more lines of template instantiation trace...)
```

**The user sees:** 40 lines of template gibberish, buried in line 15 is "is_arithmetic evaluates to false".

---

### The Modern Way: C++20 Concepts (C++20)

```cpp
// C++20: Using Concepts
template<std::arithmetic T>
class Field {
    std::vector<T> data_;
public:
    T sum() const { /* ... */ }
};

// What happens if you write:
Field<std::string> bad;  // Error message is...
```

**Typical error message (g++ with Concepts):**
```text
error: 'std::string' does not satisfy 'arithmetic'
note: the concept 'std::arithmetic<T>' evaluates to 'false' for 'std::string'
note: see declaration of 'Field'
  17 | template<std::arithmetic T>
     |              ^~~~~~~~~~
note: required from here
  42 | Field<std::string> bad;
     |       ^~~~~~~~~~~~
```

**The user sees:** 3 lines of clear explanation: "`std::string` does not satisfy `arithmetic`".

---

### Comparison Table

| Aspect | SFINAE (C++98/11/14) | Concepts (C++20) |
|--------|---------------------|------------------|
| **Syntax** | `typename enable_if<condition, int>::type = 0` | `template<Concept T>` |
| **Error message length** | 40-80 lines | 3-5 lines |
| **Error clarity** | "no type named 'type' in enable_if" | "`T` does not satisfy `Concept`" |
| **Readability** | Requires metaprogramming expertise | Self-documenting |
| **Composition** | Complex (`enable_if<cond1 && cond2>`) | Simple (`Concept1 && Concept2`) |
| **Function overloading** | SFINAE-based overload sets | Constrained overloads |
| **Compilation speed** | Slow (many instantiations) | Faster (early rejection) |

**Bottom line:** Concepts make template errors **readable** for humans, not just compiler developers.

## Part 4: Implementation Exercise — Field with C++20 Concepts

### Building a Modern `Field<T>` with Concepts

```cpp
// Compile: g++ -std=c++20 -O2 -Wall concepts_demo.cpp -o concepts_demo

#include <iostream>
#include <vector>
#include <concepts>
#include <algorithm>
#include <numeric>
#include <ranges>

// =====================================================
// PART 1: Define Concepts
// =====================================================

// Concept: T must be a scalar or vector type with numeric operations
template<typename T>
concept FieldElement = std::arithmetic<T> || requires(T t, T u, double s) {
    // Must have x, y, z (for vector types)
    { t.x } -> std::convertible_to<double>;
    { t.y } -> std::convertible_to<double>;
    { t.z } -> std::convertible_to<double>;

    // Must support addition
    { t + u } -> std::same_as<T>;

    // Must support scalar multiplication
    { t * s } -> std::same_as<T>;
    { s * t } -> std::same_as<T>;

    // Must support compound assignment
    { t += u } -> std::same_as<T&>;
};

// =====================================================
// PART 2: Vec3 Type (satisfies FieldElement)
// =====================================================

struct Vec3 {
    double x{}, y{}, z{};

    Vec3() = default;
    Vec3(double x_, double y_, double z_) : x(x_), y(y_), z(z_) {}

    Vec3 operator+(const Vec3& r) const { return {x+r.x, y+r.y, z+r.z}; }
    Vec3& operator+=(const Vec3& r) { x+=r.x; y+=r.y; z+=r.z; return *this; }

    Vec3 operator*(double s) const { return {s*x, s*y, s*z}; }
    friend Vec3 operator*(double s, const Vec3& v) { return {s*v.x, s*v.y, s*v.z}; }

    double mag() const { return std::sqrt(x*x + y*y + z*z); }

    friend std::ostream& operator<<(std::ostream& os, const Vec3& v) {
        return os << "(" << v.x << ", " << v.y << ", " << v.z << ")";
    }
};

// =====================================================
// PART 3: Field Class Constrained by FieldElement
// =====================================================

template<FieldElement T>
class Field {
    std::vector<T> data_;

public:
    Field() = default;
    explicit Field(std::size_t n) : data_(n) {}

    std::size_t size() const { return data_.size(); }
    bool empty() const { return data_.empty(); }

    T& operator[](std::size_t i) { return data_[i]; }
    const T& operator[](std::size_t i) const { return data_[i]; }

    auto begin() { return data_.begin(); }
    auto end() { return data_.end(); }
    auto begin() const { return data_.begin(); }
    auto end() const { return data_.end(); }

    // Sum all elements (using std::reduce from C++17)
    T sum() const {
        if constexpr (std::arithmetic<T>) {
            // For scalars: start from T{} (zero)
            return std::reduce(data_.begin(), data_.end(), T{});
        } else {
            // For vectors: also start from T{} (default-constructed zero)
            return std::reduce(data_.begin(), data_.end(), T{});
        }
    }

    // Element-wise magnitude (only for vectors)
    auto magField() const requires (!std::arithmetic<T>) {
        Field<double> result(data_.size());
        std::ranges::transform(data_, result.begin(), [](const T& v) { return v.mag(); });
        return result;
    }

    // Scalar multiplication
    Field& operator*=(double s) {
        for (auto& val : data_) val = val * s;
        return *this;
    }

    friend Field operator*(Field f, double s) {
        f *= s;
        return f;
    }

    friend Field operator*(double s, Field f) {
        f *= s;
        return f;
    }
};

// Type aliases
using scalarField = Field<double>;
using vectorField = Field<Vec3>;

// =====================================================
// PART 4: Demonstration
// =====================================================

int main() {
    std::cout << "=== Day 02: C++20 Concepts & Constraints ===\n\n";

    // --- Scalar field demo ---
    scalarField sf(5);
    sf[0] = 1.0; sf[1] = 2.0; sf[2] = 3.0; sf[3] = 4.0; sf[4] = 5.0;

    std::cout << "Scalar field: ";
    for (auto val : sf) std::cout << val << " ";
    std::cout << "\n";

    std::cout << "Sum: " << sf.sum() << "\n";
    std::cout << "2 * field: ";
    for (auto val : 2.0 * sf) std::cout << val << " ";
    std::cout << "\n\n";

    // --- Vector field demo ---
    vectorField vf(3);
    vf[0] = {3, 0, 0};   // mag = 3
    vf[1] = {0, 4, 0};   // mag = 4
    vf[2] = {0, 0, 5};   // mag = 5

    std::cout << "Vector field:\n";
    for (const auto& v : vf) std::cout << "  " << v << "\n";

    std::cout << "\nMagnitudes: ";
    auto mags = vf.magField();
    for (auto m : mags) std::cout << m << " ";
    std::cout << "\n";

    std::cout << "Sum: " << vf.sum() << "\n";

    // --- Uncomment to see Concept error messages ---
    // Field<std::string> bad;  // ❌ Clear error: "std::string does not satisfy FieldElement"
    // Field<std::vector<int>> nested;  // ❌ Clear error: "vector<int> does not satisfy FieldElement"

    return 0;
}
```

**Output:**
```text
=== Day 02: C++20 Concepts & Constraints ===

Scalar field: 1 2 3 4 5
Sum: 15
2 * field: 2 4 6 8 10

Vector field:
  (3, 0, 0)
  (0, 4, 0)
  (0, 0, 5)

Magnitudes: 3 4 5
Sum: (3, 4, 5)
```

### Key Features in the Code

| Feature | Line(s) | Modern C++ Approach |
|---------|---------|---------------------|
| **Concept definition** | 14-27 | `requires` clause checks for `x`, `y`, `z`, `operator+`, `operator*` |
| **Constrained template** | 63 | `template<FieldElement T>` — only valid types can instantiate |
| **`if constexpr`** | 84-92 | Compile-time branch: `std::arithmetic<T>` check at compile time |
| **Constrained member function** | 95-97 | `requires (!std::arithmetic<T>)` — only for vector types |
| **`std::reduce`** | 87, 91 | C++17 algorithm for reduction (replaces manual loop) |
| **`std::ranges::transform`** | 96 | C++20 range-based transform (cleaner than iterator loop) |

## Part 5: OpenFOAM Historical Comparison

### OpenFOAM's Approach: `pTraits` + `static_assert`

OpenFOAM (C++98/03) couldn't use Concepts, so it used:

```cpp
// OpenFOAM's way (C++98/03)
template<class Type>
class Field {
    // Require pTraits specialization
    static_assert(pTraits<Type>::nComponents > 0,
        "Field<Type> requires pTraits specialization");

    std::vector<Type> data_;

public:
    Type sum() const {
        Type result = pTraits<Type>::zero;
        for (const auto& val : data_) result += val;
        return result;
    }
};
```

**Problems with this approach:**
1. **Late error detection** — `static_assert` fires inside the class, not at instantiation
2. **Manual trait specialization** — Must write `pTraits<T>` for every type
3. **Boilerplate** — Each new type requires custom trait code
4. **Cryptic errors** — Error shows `static_assert` failure, not why `pTraits<T>` is missing

---

### Modern C++ Replacement: Concepts

```cpp
// Modern C++20 way
template<FieldElement T>
class Field {
    std::vector<T> data_;

public:
    T sum() const {
        return std::reduce(data_.begin(), data_.end(), T{});
    }
};
```

**Advantages:**
1. **Early error detection** — Error appears at instantiation site
2. **No manual traits** — Concept checks type properties directly
3. **Self-documenting** — `template<FieldElement T>` reads like English
4. **Clear errors** — Compiler tells you exactly which requirement failed

---

### Migration: OpenFOAM Patterns → Modern C++

| OpenFOAM Pattern | Modern C++ Replacement |
|------------------|------------------------|
| `pTraits<Type>::zero` | `T{}` (value initialization) |
| `pTraits<Type>::nComponents` | Concept check: `requires { t.x; t.y; t.z; }` |
| `pTraits<Type>::typeName` | `std::typeid(T).name()` or custom `name()` member |
| `static_assert(pTraits<T>::nComponents > 0)` | `template<FieldElement T>` |
| SFINAE with `enable_if` | `template<Concept T>` |
| Tag dispatch for overload resolution | Constrained function overloads with Concepts |

## Summary

**⭐ Key Takeaways:**

1. **C++20 Concepts** are named predicates that constrain template parameters at compile time
2. **Concepts replace SFINAE** — instead of 40-line error messages, you get 3-line clear explanations
3. **Standard library Concepts** (`std::arithmetic`, `std::integral`, etc.) cover most numeric needs
4. **Custom Concepts** combine multiple requirements using `requires` and logical operators (`&&`, `||`)
5. **Constrained templates** use syntax `template<Concept T>` which is self-documenting
6. **`if constexpr`** enables compile-time branching based on type properties
7. **OpenFOAM's `pTraits`** is replaced by Concepts — no manual trait specializations needed
