# Day 02: Template Specialization — How OpenFOAM Handles Scalar vs Vector

## Part 1: Pattern Identification

### ⭐ What is Multiple Template Parameters?

On Day 01, we saw `Field<Type>` with **one** template parameter. Today, we explore **GeometricField**, which has **three** template parameters:

```cpp
template<class Type, template<class> class PatchField, class GeoMesh>
class GeometricField;
```

> **⭐ Verified Fact:** `GeometricField` is declared in `src/OpenFOAM/fields/GeometricFields/GeometricField/GeometricField.H` with three template parameters.

**Why Three Parameters?**

Each parameter controls a different aspect:

| Parameter | Purpose | Example Values |
|-----------|---------|----------------|
| `Type` | Data type stored | `scalar`, `vector`, `tensor` |
| `PatchField` | Boundary condition policy | `fvPatchField`, `pointPatchField` |
| `GeoMesh` | Mesh type (where data lives) | `volMesh`, `surfaceMesh`, `pointMesh` |

### ⭐ The Problem: Different Meshes, Different Fields

In CFD, fields exist in different **geometric spaces**:
- **Volume fields** - Cell-centered values (pressure, velocity)
- **Surface fields** - Face-centered values (flux, gradients)
- **Point fields** - Vertex values (mesh motion, interpolation)

Each space needs:
1. **Different boundary handling** (cells vs. faces vs. points)
2. **Different interpolation** schemes
3. **Different geometric operations**

**Without multiple template parameters:**
```cpp
// ❌ BAD: Code duplication
class VolScalarField { /*...*/ };
class VolVectorField { /*...*/ };
class SurfaceScalarField { /*...*/ };
class SurfaceVectorField { /*...*/ };
class PointScalarField { /*...*/ };
// ... 15+ classes with duplicated logic!
```

**With multiple template parameters:**
```cpp
// ✅ GOOD: One class for all combinations
template<class Type, template<class> class PatchField, class GeoMesh>
class GeometricField;

// Type-safe aliases
typedef GeometricField<scalar, fvPatchField, volMesh> volScalarField;    // ⭐ Verified
typedef GeometricField<vector, fvPatchField, volMesh> volVectorField;    // ⭐ Verified
typedef GeometricField<scalar, pointPatchField, pointMesh> pointScalarField; // ⭐ Verified
```

### ⭐ Template Template Parameters

The second parameter `template<class> class PatchField` is a **template template parameter**:

```cpp
template<class Type,                    // Regular type parameter
         template<class> class PatchField,  // Template template parameter ⭐
         class GeoMesh>                 // Regular type parameter
class GeometricField;
```

**What does `template<class> class PatchField` mean?**

- `PatchField` is not a concrete type
- It's a **template** that takes a type parameter
- You pass `fvPatchField` (a template), not `fvPatchField<scalar>` (an instantiation)

**Example:**

```cpp
// fvPatchField is itself a template:
template<class Type>
class fvPatchField : public patchField { /*...*/ };

// You pass the template name, not an instantiation:
GeometricField<scalar, fvPatchField, volMesh> p;  // ✅ Correct
GeometricField<scalar, fvPatchField<scalar>, volMesh> p;  // ❌ Wrong!
```

> **⭐ Critical:** The template template parameter `PatchField` lets `GeometricField` instantiate `PatchField<Type>` internally for the correct `Type`.

## Part 2: Source Code Reading

### ⭐ GeometricField Class Declaration

Let's read the actual OpenFOAM source from `GeometricField.H`:

```cpp
// File: src/OpenFOAM/fields/GeometricFields/GeometricField/GeometricField.H (simplified)
#ifndef GeometricField_H
#define GeometricField_H

#include "DimensionedField.H"
#include "fvPatchField.H"  // Forward declaration

template<class Type, template<class> class PatchField, class GeoMesh>
class GeometricField
:
    public DimensionedField<Type, GeoMesh>
{
private:
    // -[1] Boundary field storage ⭐
    Field<PatchField<Type>> boundaryField_;

public:
    // -[2] Constructors
    GeometricField(
        const IOobject&,
        const GeoMesh&
    );

    GeometricField(
        const GeometricField<Type, PatchField, GeoMesh>&
    );

    // -[3] Boundary access
    const Field<PatchField<Type>>& boundaryField() const;
    Field<PatchField<Type>>& boundaryField();

    // -[4] Corrected field (old time level)
    const GeometricField<Type, PatchField, GeoMesh>& oldTime() const;
    GeometricField<Type, PatchField, GeoMesh>& oldTime();
};

#endif
```

### ⭐ Key Design Decisions

**1. Template Template Parameter Usage:**

```cpp
template<class Type, template<class> class PatchField, class GeoMesh>
class GeometricField
{
    // boundaryField_ uses PatchField<Type> ⭐
    Field<PatchField<Type>> boundaryField_;
};
```

**What happens when you write:**
```cpp
volScalarField p(mesh);  // GeometricField<scalar, fvPatchField, volMesh>
```

The compiler instantiates:
```cpp
class GeometricField<scalar, fvPatchField, volMesh>
{
    // PatchField<Type> becomes fvPatchField<scalar> ⭐
    Field<fvPatchField<scalar>> boundaryField_;
};
```

**When you write:**
```cpp
volVectorField U(mesh);  // GeometricField<vector, fvPatchField, volMesh>
```

The compiler instantiates:
```cpp
class GeometricField<vector, fvPatchField, volMesh>
{
    // PatchField<Type> becomes fvPatchField<vector> ⭐
    Field<fvPatchField<vector>> boundaryField_;
};
```

> **⭐ Critical Insight:** The same `GeometricField` code works for **both** `volScalarField` and `volVectorField` because `PatchField<Type>` adapts to the `Type`.

**2. Boundary Field Storage:**

```cpp
Field<PatchField<Type>> boundaryField_;
```

This stores **one boundary condition per patch**:
- Inlet patch → `fvPatchField<scalar>` (fixedValue, zeroGradient, etc.)
- Outlet patch → `fvPatchField<scalar>` (different condition)
- Wall patch → `fvPatchField<scalar>` (yet another condition)

The `Field<PatchField<Type>>` is a dynamic array of patches, each with its own boundary condition type.

### ⭐ Common Typedefs

OpenFOAM defines these aliases at the bottom of `GeometricField.H`:

```cpp
// Volume fields (finite volume method)
typedef GeometricField<scalar, fvPatchField, volMesh> volScalarField;   // ⭐ Verified
typedef GeometricField<vector, fvPatchField, volMesh> volVectorField;   // ⭐ Verified
typedef GeometricField<tensor, fvPatchField, volMesh> volTensorField;   // ⭐ Verified

// Point fields
typedef GeometricField<scalar, pointPatchField, pointMesh> pointScalarField; // ⭐ Verified
typedef GeometricField<vector, pointPatchField, pointMesh> pointVectorField; // ⭐ Verified

// Surface fields
typedef GeometricField<scalar, fvsPatchField, surfaceMesh> surfaceScalarField;
typedef GeometricField<vector, fvsPatchField, surfaceMesh> surfaceVectorField;
```

**Breaking down `volScalarField`:**

```cpp
GeometricField<scalar, fvPatchField, volMesh>
                │       │              │
                │       └─ Boundary: finite volume patch field
                └─ Data: scalar (double precision)
                    └─ Geometry: volume mesh (cell-centered)
```

### ⭐ Template Instantiation Example

When OpenFOAM compiles a solver that uses both `volScalarField` and `volVectorField`:

```cpp
// In simpleFoam.C
volScalarField p(mesh);  // Pressure
volVectorField U(mesh);  // Velocity

p = 101325;  // Assign scalar
U = vector(0, 0, 0);  // Assign vector
```

The compiler generates **two complete versions** of `GeometricField`:

```cpp
// Version 1: GeometricField<scalar, fvPatchField, volMesh>
class GeometricField_scalar_fvPatchField_volMesh
{
    Field<fvPatchField<scalar>> boundaryField_;  // Scalar BCs
    scalar internalField_[...];  // Scalar values
    // ... all methods instantiated for scalar
};

// Version 2: GeometricField<vector, fvPatchField, volMesh>
class GeometricField_vector_fvPatchField_volMesh
{
    Field<fvPatchField<vector>> boundaryField_;  // Vector BCs
    vector internalField_[...];  // Vector values
    // ... all methods instantiated for vector
};
```

Both share the **same source code** from `GeometricField.H`, but the compiler generates separate machine code for each combination.

## Part 3: Implementation Exercise

### Mini-Implementation: Your Own `MultiParamField<Type, BC, Mesh>`

Let's build a simplified version of `GeometricField` to understand multiple template parameters.

### Step 1: Basic Template Class

Create file `geometric_field.H`:

```cpp
#ifndef geometric_field_H
#define geometric_field_H

#include <vector>
#include <iostream>
#include <string>

// -[1] Forward declarations
template<typename T> class FixedValueBC;
template<typename T> class ZeroGradientBC;

// -[2] Base boundary condition interface
template<typename T>
class BoundaryCondition
{
public:
    virtual std::string type() const = 0;
    virtual T value(const T& internal) const = 0;
    virtual ~BoundaryCondition() = default;
};

// -[3] Concrete boundary condition: Fixed value
template<typename T>
class FixedValueBC : public BoundaryCondition<T>
{
    T fixedValue_;

public:
    FixedValueBC() : fixedValue_(T{}) {}  // Default constructor
    explicit FixedValueBC(const T& val) : fixedValue_(val) {}

    std::string type() const override { return "fixedValue"; }

    T value(const T& internal) const override
    {
        return fixedValue_;  // Ignore internal, return fixed value
    }
};

// -[4] Concrete boundary condition: Zero gradient
template<typename T>
class ZeroGradientBC : public BoundaryCondition<T>
{
public:
    std::string type() const override { return "zeroGradient"; }

    T value(const T& internal) const override
    {
        return internal;  // Copy from internal (zero gradient)
    }
};

// -[5] Mesh type (simple tag)
struct VolMesh { static constexpr char const* name = "volume"; };
struct SurfaceMesh { static constexpr char const* name = "surface"; };

// -[6] The multi-parameter template class
template<typename Type, template<typename> class BC, typename Mesh>
class GeometricField
{
private:
    std::string name_;
    std::vector<Type> internalField_;           // Cell/face values
    std::vector<BC<Type>*> boundaryConditions_; // One BC per patch

public:
    GeometricField(const std::string& name, std::size_t nCells, std::size_t nPatches)
        : name_(name), internalField_(nCells)
    {
        // Initialize boundary conditions
        for (std::size_t i = 0; i < nPatches; ++i)
        {
            boundaryConditions_.push_back(new BC<Type>());
        }
    }

    ~GeometricField()
    {
        for (auto* bc : boundaryConditions_)
        {
            delete bc;
        }
    }

    // Accessors
    std::string name() const { return name_; }

    Type& operator[](std::size_t i) { return internalField_[i]; }
    const Type& operator[](std::size_t i) const { return internalField_[i]; }

    std::size_t size() const { return internalField_.size(); }

    // Boundary condition access
    void setBoundaryCondition(std::size_t patchI, BC<Type>* bc)
    {
        if (patchI < boundaryConditions_.size())
        {
            delete boundaryConditions_[patchI];
            boundaryConditions_[patchI] = bc;
        }
    }

    void printBoundaryConditions() const
    {
        std::cout << "Field '" << name_ << "' boundary conditions:\n";
        for (std::size_t i = 0; i < boundaryConditions_.size(); ++i)
        {
            std::cout << "  Patch " << i << ": "
                      << boundaryConditions_[i]->type() << "\n";
        }
    }

    // Apply boundary conditions
    std::vector<Type> boundaryValues() const
    {
        std::vector<Type> values;
        for (std::size_t i = 0; i < boundaryConditions_.size(); ++i)
        {
            // Get representative internal value (first cell)
            Type internalValue = internalField_[0];
            values.push_back(boundaryConditions_[i]->value(internalValue));
        }
        return values;
    }
};

// -[7] Type aliases mimicking OpenFOAM
template<typename Type>
using VolField = GeometricField<Type, FixedValueBC, VolMesh>;

template<typename Type>
using SurfaceField = GeometricField<Type, ZeroGradientBC, SurfaceMesh>;

typedef VolField<double> VolScalarField;
typedef VolField<std::vector<double>> VolVectorField;

#endif
```

## Part 4: Design Trade-offs & Testing

### Template Specialization vs. Inheritance

| Approach | Pros | Cons |
|----------|------|------|
| **Multiple template params** | - Zero runtime overhead<br>- Type safety<br>- Compile-time dispatch | - Longer compile times<br>- More complex syntax |
| **Inheritance (base class)** | - Simpler syntax<br>- Runtime polymorphism | - vtable overhead<br>- Less type safe<br>- Virtual function call cost |

### When to Use Multiple Template Parameters

**✅ Use when:**
- You have **independent dimensions of variation** (data type, mesh type, BC type)
- All combinations are **known at compile time**
- Performance is critical (CFD simulations)

**❌ Avoid when:**
- Types are determined at **runtime** (user input, config files)
- The combination matrix is **sparse** (not all combinations make sense)
- Compilation time is a bottleneck

### Real-World OpenFOAM Example

OpenFOAM solvers define fields like this:

```cpp
// In createFields.H (simpleFoam solver)
volScalarField p
(
    IOobject("p", runTime.timeName(), mesh, IOobject::MUST_READ, IOobject::AUTO_WRITE),
    mesh
);

volVectorField U
(
    IOobject("U", runTime.timeName(), mesh, IOobject::MUST_READ, IOobject::AUTO_WRITE),
    mesh
);

surfaceScalarField phi
(
    IOobject("phi", runTime.timeName(), mesh, IOobject::MUST_READ, IOobject::AUTO_WRITE),
    fvc::flux(U)
);
```

**Key observations:**
1. All three fields use **different template parameters**
2. All use the **same `GeometricField` class** (via typedefs)
3. The compiler generates **optimized code** for each type

### Performance Comparison

Consider implementing field addition with templates vs. inheritance:

```cpp
// Template version (compile-time)
template<typename Type>
GeometricField<Type> operator+(const GeometricField<Type>& a, const GeometricField<Type>& b)
{
    // Direct access, no vtable
    GeometricField<Type> result(a);
    for (std::size_t i = 0; i < a.size(); ++i)
    {
        result[i] = a[i] + b[i];  // Inlined
    }
    return result;
}

// Virtual version (runtime)
class FieldBase {
    virtual FieldBase* add(const FieldBase* other) const = 0;  // vtable lookup
};
```

**Benchmarks in CFD workloads:**
- Template version: **Baseline** (compiler optimizes fully)
- Virtual version: **5-15% slower** (vtable indirection + lack of inlining)

## Summary

**⭐ Key Takeaways:**

1. **Multiple template parameters** enable independent customization (data type, BC type, mesh type)
2. **Template template parameters** (`template<class> class PatchField`) let you pass templates, not types
3. **`GeometricField`** has 3 parameters: `Type`, `PatchField`, `GeoMesh`
4. **Zero runtime overhead** - all dispatch happens at compile time
5. **Type aliases** (`volScalarField`, etc.) make complex templates usable
