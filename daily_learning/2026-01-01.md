## 1. Theory: Core Equations & Physics

### 1.1 Conservation of Mass (Continuity Equation)

**สมการการอนุรักษ์มวล (Continuity Equation)**

$$
\frac{\partial \rho}{\partial t} + \nabla \cdot (\rho \mathbf{U}) = 0
$$

**คำอธิบายพจน์:**
- $\rho$ (rho) = **ความหนาแน่น** (density) [kg/m³]
- $\mathbf{U}$ = **สนามความเร็ว** (velocity field) [m/s]
- $\nabla \cdot$ = **ตัวดำเนินการไดเวอร์เจนซ์** (divergence operator)
- $t$ = **เวลา** (time) [s]

สำหรับการไหลแบบ **ไม่อัดตัว** (incompressible flow): $\nabla \cdot \mathbf{U} = 0$

---

### 1.2 Conservation of Momentum (Navier-Stokes Equation)

**สมการการอนุรักษ์โมเมนตัม (Momentum Equation)**

$$
\frac{\partial (\rho \mathbf{U})}{\partial t} + \nabla \cdot (\rho \mathbf{U} \mathbf{U}) = -\nabla p + \nabla \cdot \boldsymbol{\tau} + \rho \mathbf{g}
$$

**คำอธิบายพจน์:**
- $p$ = **ความดัน** (pressure) [Pa]
- $\boldsymbol{\tau}$ (tau) = **เทนเซอร์ความเค้น** (stress tensor) [Pa]
- $\mathbf{g}$ = **ความเร่งเนื่องจากแรงโน้มถ่วง** (gravitational acceleration) [m/s²]
- $\rho \mathbf{U} \mathbf{U}$ = **การพาโมเมนตัม** (momentum convection)

**สำหรับของไหลนิวตัน (Newtonian Fluid):**

$$
\boldsymbol{\tau} = \mu \left[ \nabla \mathbf{U} + (\nabla \mathbf{U})^T \right] - \frac{2}{3}\mu (\nabla \cdot \mathbf{U})\mathbf{I}
$$

- $\mu$ (mu) = **ความหนืด** (dynamic viscosity) [Pa·s]
- $\mathbf{I}$ = **เทนเซอร์เอกลักษณ์** (identity tensor)

---

### 1.3 Conservation of Energy

**สมการการอนุรักษ์พลังงาน (Energy Equation)**

$$
\frac{\partial (\rho e)}{\partial t} + \nabla \cdot (\rho e \mathbf{U}) = -\nabla \cdot \mathbf{q} + \boldsymbol{\tau} : \nabla \mathbf{U} + S_h
$$

**คำอธิบายพจน์:**
- $e$ = **พลังงานภายในต่อหน่วยมวล** (specific internal energy) [J/kg]
- $\mathbf{q}$ = **ฟลักซ์ความร้อน** (heat flux) [W/m²]
- $S_h$ = **แหล่งกำเนิดความร้อน** (heat source) [W/m³]
- $\boldsymbol{\tau} : \nabla \mathbf{U}$ = **การกระจายความร้อนจากความเค้น** (viscous dissipation)

**กฎของฟูริเยร์ (Fourier's Law):**
$$
\mathbf{q} = -k \nabla T
$$

- $k$ = **สัมประสิทธิ์การนำความร้อน** (thermal conductivity) [W/(m·K)]
- $T$ = **อุณหภูมิ** (temperature) [K]

---

### 1.4 Equation of State

**สมการสถานะ (Equation of State)**

สำหรับ **แก๊สอุดมคติ (Ideal Gas):**

$$
p = \rho R T
$$

- $R$ = **ค่าคงที่ของแก๊สเฉพาะ** (specific gas constant) [J/(kg·K)]

สำหรับ **ของไหลที่ไม่อัดตัว (Incompressible Fluid):**

$$
\rho = \text{constant}
$$

---

### 1.5 Transport Equations (General Form)

**สมการขนส่งทั่วไป (General Transport Equation)**

$$
\frac{\partial (\rho \phi)}{\partial t} + \nabla \cdot (\rho \phi \mathbf{U}) = \nabla \cdot (\Gamma_\phi \nabla \phi) + S_\phi
$$

**คำอธิบายพจน์:**
- $\phi$ (phi) = **ตัวแปรขนส่ง** (transported variable) - สามารถเป็นอุณหภูมิ ความเข้มของสาร ฯลฯ
- $\Gamma_\phi$ (Gamma) = **สัมประสิทธิ์การแพร่** (diffusion coefficient)
- $S_\phi$ = **เทอมแหล่งกำเนิด** (source term)

**แต่ละเทอมแทน:**
1. $\frac{\partial (\rho \phi)}{\partial t}$ = **การเปลี่ยนแปลงตามเวลา** (transient term)
2. $\nabla \cdot (\rho \phi \mathbf{U})$ = **การพา** (convection term)
3. $\nabla \cdot (\Gamma_\phi \nabla \phi)$ = **การแพร่** (diffusion term)
4. $S_\phi$ = **แหล่งกำเนิด/ตัวรับ** (source/sink term)

## 2. OpenFOAM Class Hierarchy & Implementation

OpenFOAM represents governing equations through field classes that store and manipulate data at mesh points/cells.

```

├── volScalarField  (scalar field on cell centers)

│
```

## 3. Code Walkthrough

### 3.1 UEqn.H

ไฟล์ `UEqn.H` ใช้สำหรับสร้างสมการโมเมนตัม (Momentum Equation) ใน OpenFOAM solver ต่างๆ เช่น `simpleFoam`, `pimpleFoam` ฯลฯ

**โครงสร้างพื้นฐาน:**

```cpp
// สร้างสมการเมทริกซ์สำหรับสนามความเร็ว U
fvVectorMatrix UEqn
(
    fvm::ddt(U)                              // เทอมอนุพันธ์เวลา (transient)
  + fvm::div(phi, U)                         // เทอมการพา (convection)
  + turbulence->divDevReff(U)                // เทอมการแพร่ของความเค้น (viscous term)
 ==
    fvOptions(U)                             // เทอมแหล่งกำเนิด (source terms)
);

// ผ่อนคลายสมการ (relaxation) เพื่อเสถียรภาพเชิงตัวเลข
UEqn.relax();

// แก้สมการโดยไม่รวมเทอมความดัน (pressure-free solution)
solve(UEqn == -fvc::grad(p));
```

**คำอธิบายเทอมสำคัญ:**

| เทอม | ความหมาย | การดำเนินการ |
|-------|-----------|----------------|
| `fvm::ddt(U)` | ∂U/∂t | Implicit - ใช้เมทริกซ์ |
| `fvm::div(phi, U)` | ∇·(UU) | Implicit - ใช้เมทริกซ์ |
| `turbulence->divDevReff(U)` | ∇·τ_eff | Implicit - ความเค้นจากความปั่น |
| `fvc::grad(p)` | -∇p | Explicit - คำนวณโดยตรง |
| `fvOptions(U)` | S_U | Explicit/Implicit - แหล่งกำเนิด |

**การแยกสมการ (Splitting):**

OpenFOAM ใช้ **PISO/SIMPLE algorithm** โดยแยกสมการโมเมนตัมและความดัน:

1. **Predictor Step:** แก้ UEqn โดยไม่รวม ∇p
2. **Pressure Equation:** แก้สมการความดันจาก ∇·U = 0
3. **Corrector Step:** แก้ไข U ด้วย ∇p ใหม่

**ตัวอย่างเพิ่มเติม (สำหรับ compressible flow):**

```cpp
// กรณีของไหลที่อัดตัว (compressible)
fvVectorMatrix UEqn
(
    fvm::ddt(rho, U)                         // รวมความหนาแน่น
  + fvm::div(rhoPhi, U)                      // flux ถ่วงมวล
  + turbulence->divDevRhoReff(U)             // viscous term (rho-weighted)
 ==
    fvOptions(rho, U)
);
```

**Reference Files:**
- `$FOAM_APP/solvers/incompressible/simpleFoam/UEqn.H`
- `$FOAM_APP/solvers/incompressible/pimpleFoam/UEqn.H`
- `$FOAM_SRC/finiteVolume/fields/fvPatchFields/`

### 3.2 pEqn.H

ไฟล์ `pEqn.H` ใช้สำหรับสร้างและแก้สมการความดัน (Pressure Equation) ซึ่งได้มาจากการนำสมการต่อเนื่อง (continuity) มาใช้แก้ไขความดันและแก้ไขสนามความเร็ว

**โครงสร้างพื้นฐาน (สำหรับ incompressible flow):**

```cpp
// สร้างสมการความดันจากสมการโมเมนตัมและเงื่อนไขการไหลต่อเนื่อง
volScalarField rAU(1.0/UEqn.A());           // 逆ของเมทริกซ์สัมประสิทธิ์
volVectorField HbyA(constrainHbyA(rAU*UEqn.H(), U, p));  // เทอม H/A
surfaceScalarField phiHbyA
(
    "phiHbyA",
    fvc::flux(HbyA)                         // คำนวณ flux จาก HbyA
  + fvc::interpolate(rAU)*fvc::ddtCorr(U, phi)
);

// แก้ไข pressure boundary conditions
adjustPhi(phiHbyA, U, p);

// สร้างสมการความดัน (Poisson equation)
while (simple.correctNonOrthogonal())
{
    fvScalarMatrix pEqn
    (
        fvm::laplacian(rAU, p) == fvc::div(phiHbyA)
    );

    pEqn.setReference(pRefCell, pRefValue);  // ตั้งค่า reference pressure
    pEqn.solve();                            // แก้สมการ

    if (simple.finalNonOrthogonalIter())
    {
        phi = phiHbyA - pEqn.flux();         // แก้ไข flux ด้วยความดันใหม่
    }
}

// แก้ไขสนามความเร็วด้วย gradient ของความดัน
U = HbyA - rAU*fvc::grad(p);
U.correctBoundaryConditions();
```

**คำอธิบายเทอมสำคัญ:**

| เทอม | ความหมาย | การดำเนินการ |
|-------|-----------|----------------|
| `UEqn.A()` | สัมประสิทธิ์เชิงเส้น | Diagonal of matrix |
| `UEqn.H()` | เทอมที่ไม่ขึ้นกับ U | Source term (explicit) |
| `fvm::laplacian(rAU, p)` | ∇·(rAU∇p) | Implicit Poisson |
| `fvc::div(phiHbyA)` | ∇·(H/A) | Explicit divergence |
| `pEqn.flux()` | -rAU∇p·Sf | Pressure flux correction |

**ขั้นตอนอัลกอริทึม (SIMPLE/PISO):**

1. **Predict:** คำนวณ `HbyA` จาก UEqn (โดยไม่รวม ∇p)
2. **Pressure Eq:** แก้ ∇·(rAU∇p) = ∇·(HbyA) สำหรับ p
3. **Correct Flux:** อัปเดต φ = φ_HbyA - φ_p
4. **Correct Velocity:** อัปเดต U = HbyA - rAU∇p

**ตัวอย่างเพิ่มเติม (สำหรับ compressible flow):**

```cpp
// กรณีของไหลที่อัดตัว (compressible)
volScalarField rAU(1.0/UEqn.A());
volVectorField HbyA("HbyA", U);
HbyA = rAU*UEqn.H();

surfaceScalarField phiHbyA("phiHbyA", fvc::flux(HbyA));
phiHbyA += fvc::interpolate(rho*rAU)*fvc::ddtCorr(U, rhoPhi);

// แก้สมการความดันแบบ compressible
while (pimple.correctNonOrthogonal())
{
    fvScalarMatrix pEqn
    (
        fvm::ddt(psi, p)
      + fvm::div(phiHbyA, p)
      - fvm::laplacian(rho*rAU, p)
     ==
        fvOptions(p)
    );

    pEqn.solve();
    
    if (pimple.finalNonOrthogonalIter())
    {
        phi = phiHbyA + pEqn.flux();
    }
}
```

### 3.3 createFields.H

ไฟล์ `createFields.H` ใช้สำหรับสร้างและเริ่มต้นสนามต่างๆ (fields) ที่จำเป็นสำหรับการคำนวณ รวมถึงอ่านค่าจากไฟล์คอนฟิกูเรชันและตั้งค่าเริ่มต้น

**โครงสร้างพื้นฐาน:**

```cpp
// สร้างสนามความเร็ว (velocity field)
Info<< "Reading field U\n" << endl;
volVectorField U
(
    IOobject
    (
        "U",
        runTime.timeName(),
        mesh,
        IOobject::MUST_READ,
        IOobject::AUTO_WRITE
    ),
    mesh
);

// สร้างสนามความดัน (pressure field)
Info<< "Reading field p\n" << endl;
volScalarField p
(
    IOobject
    (
        "p",
        runTime.timeName(),
        mesh,
        IOobject::MUST_READ,
        IOobject::AUTO_WRITE
    ),
    mesh
);

// สร้างสนาม flux บนผิวเซลล์ (surface flux field)
Info<< "Reading/calculating face flux field phi\n" << endl;
surfaceScalarField phi
(
    IOobject
    (
        "phi",
        runTime.timeName(),
        mesh,
        IOobject::READ_IF_PRESENT,
        IOobject::AUTO_WRITE
    ),
    fvc::flux(U)
);

// สร้างโมเดลความปั่น (turbulence model)

(
    incompressible::turbulenceModel::New(U, phi, laminarTransport)
);

// ตั้งค่า reference pressure สำหรับ incompressible flow
label pRefCell = 0;
scalar pRefValue = 0.0;
setRefCell(p, mesh.solutionDict().subDict("SIMPLE"), pRefCell, pRefValue);
```

**คำอธิบายเทอมสำคัญ:**

| เทอม | ความหมาย | การดำเนินการ |

#### 4.1.1 ddtSchemes (Time Derivative Schemes)

**สมการอนุพันธ์เวลา (Temporal Discretization)**

รูปแบบ: `ddt(phi)` หรือ `ddt(rho, phi)`

| Scheme | คำอธิบาย | ความแม่นยำ | เสถียรภาพ | การใช้งาน |
|--------|-----------|-------------|-----------|-----------|
| **Euler** | Implicit Euler 1st order | ต่ำ | สูง | Transient ทั่วไป |
| **backward** | Backward 2nd order | ปานกลาง | สูง | Transient ที่ต้องการความแม่นยำ |
| **CrankNicolson** | Trapezoidal 2nd order | สูง | ปานกลาง | Transient ที่ต้องการความแม่นยำสูง |
| **localEuler** | Local time-stepping | ต่ำ | สูง | Steady-state แบบ pseudo-transient |
| **steadyState** | ไม่มีเทอมเวลา | - | - | การไหลแบบ steady |

**ตัวอย่างการตั้งค่า:**

```cpp
ddtSchemes
{
    default         Euler;              // 1st order implicit
    // หรือใช้ backward สำหรับความแม่นยำสูงกว่า
    // default         backward;
    
    // สำหรับ steady-state solver
    // default         steadyState;
}
```

**คำอธิบาย:**
- **Euler**: ใช้ค่าที่ time step ปัจจุบันเพื่อคำนวณ derivative → เสถียรแต่ความแม่นยำต่ำ (O(Δt))
- **backward**: ใช้ค่าจาก 2 time step ก่อนหน้า → แม่นยำกว่า (O(Δt²))
- **CrankNicolson**: เฉลี่ยระหว่าง current และ new time → แม่นยำสูงสุด (O(Δt²)) แต่อาจไม่เสถียร

#### 4.1.2 gradSchemes (Gradient Calculation Schemes)

**สมการการคำนวณกราดิเอนต์ (Gradient Reconstruction)**

รูปแบบ: `grad(phi)` หรือ `grad(U)`

| Scheme | คำอธิบาย | ความแม่นยำ | ข้อจำกัด |
|--------|-----------|-------------|----------|
| **Gauss linear** | Central differencing ผ่าน cell faces | 2nd order | ต้องการ mesh ที่มีคุณภาพ |
| **Gauss linear corrected** | Linear + non-orthogonal correction | 2nd order | สำหรับ mesh ที่ไม่ orthogonal |
| **Gauss leastSquares** | Least-squares reconstruction | 2nd order | เสถียรกว่าบน mesh ที่ไม่ดี |
| **fourth** | 4th order central differencing | 4th order | ต้องการ mesh สมมาตรมาก |
| **leastSquares** | Pure least-squares (no Gauss) | 2nd order | ใช้กับ polyhedral mesh |

**ตัวอย่างการตั้งค่า:**

```cpp
gradSchemes
{
    default         Gauss linear;
    
    // สำหรับ mesh ที่มี non-orthogonality สูง
    // default         Gauss linear corrected;
    
    // สำหรับ mesh ที่มีคุณภาพต่ำ
    // default         Gauss leastSquares;
}
```

**คำอธิบาย:**
- **Gauss linear**: ใช้ Green-Gauss theorem → ∇φ = Σ(φ_f Sf) / V
- **corrected**: เพิ่มเทอมแก้ไขสำหรับ mesh ที่ไม่ orthogonal → เหมาะสำหรับ complex geometry
- **leastSquares**: ใช้ least-squares method หา gradient → เสถียรกว่าบน bad mesh

---

#### 4.1.3 divSchemes (Divergence/Convection Schemes)

**สมการการพา (Convection Terms)**

รูปแบบ: `div(phi, U)` หรือ `div(phi, T)`

| Scheme | คำอธิบาย | ความแม่นยำ | เสถียรภาพ | การใช้งาน |
|--------|-----------|-------------|-----------|-----------|
| **Gauss upwind** | Upwind differencing | 1st order | สูงมาก | Initial guess, high Re |
| **Gauss linear** | Central differencing | 2nd order | ต่ำ | Low Re, laminar |
| **Gauss linearUpwind** | Linear upwind | 2nd order | ปานกลาง | General purpose |
| **Gauss limitedLinearV** | TVD limiter | 2nd order | ปานกลาง | High Re, shocks |
| **Gauss vanLeer** | Van Leer limiter | 2nd order | ปานกลาง | Compressible flow |
| **Gauss QUICK** | QUICK scheme | 3rd order | ปานกลาง | Structured mesh |
| **Gauss MUSCL** | MUSCL scheme | 2nd order | ปานกลาง | Compressible |

**ตัวอย่างการตั้งค่า:**

```cpp
divSchemes
{
    default         none;
    
    // สำหรับ velocity (high Reynolds number)
    div(phi,U)      Gauss limitedLinearV 1;
    
    // สำหรับ turbulent kinetic energy
    div(phi,k)      Gauss upwind;
    div(phi,epsilon) Gauss upwind;
    
    // สำหรับ temperature/energy
    div(phi,T)      Gauss limitedLinearV 1;
    
    // สำหรับ compressible flow
    div(phi,e)      Gauss vanLeer;
}
```

**คำอธิบาย:**
- **upwind**: ใช้ค่าจาก upstream cell → ไม่ oscillate แต่ diffuse มาก (numerical diffusion)
- **linear**: Central differencing → แม่นยำแต่ oscillate ได้บน coarse mesh
- **limitedLinear**: ผสม linear + upwind ด้วย limiter → ลด oscillation
- **vanLeer**: Flux limiter scheme → เหมาะกับ compressible shocks

**Limiter Values:**
- `0` = pure upwind (most stable)
- `1` = full linear (most accurate)
- `0.5` = blended (compromise)

#### 4.1.4 laplacianSchemes (Diffusion Schemes)

**สมการการแพร่ (Diffusion Terms)**

รูปแบบ: `laplacian(nu, U)` หรือ `laplacian(Dk, k)`

| Scheme | คำอธิบาย | ความแม่นยำ | การใช้งาน |
|--------|-----------|-------------|-----------|
| **Gauss linear corrected** | Linear + non-orthogonal correction | 2nd order | General purpose |
| **Gauss linear uncorrected** | Linear ธรรมดา | 2nd order | Orthogonal mesh |
| **Gauss limited** | Limited gradient | 2nd order | High non-orthogonality |
| **Gauss fourth** | 4th order | 4th order | High-quality mesh |

**ตัวอย่างการตั้งค่า:**

```cpp
laplacianSchemes
{
    default         Gauss linear corrected;
    
    // สำหรับ orthogonal mesh (เช่น blockMesh)
    // default         Gauss linear uncorrected;
    
    // สำหรับ mesh ที่มี non-orthogonality สูงมาก (> 70)
    // default         Gauss limited 0.5;
}
```

**คำอธิบาย:**
- **corrected**: เพิ่มเทอมแก้ไขสำหรับ non-orthogonal faces → แม่นยำกว่าบน complex mesh
- **uncorrected**: เร็วกว่าแต่ error สูงบน non-orthogonal mesh
- **limited**: จำกัด gradient correction → เสถียรกว่าบน highly skewed mesh

**Non-orthogonality Check:**
```bash
checkMesh -allGeometry -allTopology
```
- 0-25°: Excellent → `uncorrected`
- 25-50°: Good → `corrected`
- 50-70°: Fair → `corrected` + `nNonOrthogonalCorrectors`
- >70°: Poor → `limited` หรือ remesh

#### 4.2.1 Solvers Section

**Linear Solvers สำหรับแก้ระบบสมการเชิงเส้น**

รูปแบบ: `solver <solver_name>;`

| Solver | คำอธิบาย | เหมาะกับ | ความเร็ว | หน่วยความจำ |
|--------|-----------|-----------|----------|-------------|
| **GAMG** | Geometric-Algebraic Multi-Grid | Symmetric matrices (pressure, Poisson) | เร็วมาก | ปานกลาง |
| **PCG** | Preconditioned Conjugate Gradient | Symmetric positive-definite | เร็ว | ต่ำ |
| **PBiCG** | Preconditioned Bi-Conjugate Gradient | Non-symmetric matrices | ปานกลาง | ต่ำ |
| **smoothSolver** | Jacobi/Gauss-Seidel smoothing | General purpose | ช้า | ต่ำมาก |
| **PBiCGStab** | BiCGStab with preconditioning | Non-symmetric, ill-conditioned | เร็ว | ปานกลาง |

**Preconditioners:**
```cpp
// สำหรับ PCG/PBiCG
preconditioner  DIC;     // Diagonal Incomplete Cholesky (symmetric)
preconditioner  DILU;    // Diagonal Incomplete LU (non-symmetric)
preconditioner  GAMG;    // Geometric-Algebraic Multi-Grid
```

**ตัวอย่างการตั้งค่า:**

```cpp
solvers
{
    // Pressure equation (symmetric, Poisson-type)
    p
    {
        solver          GAMG;
        tolerance       1e-06;      // Absolute tolerance
        relTol          0.01;       // Relative tolerance (1%)
        smoother        GaussSeidel;
        nPreSweeps      0;
        nPostSweeps     2;
        cacheAgglomeration on;
        agglomerator    faceAreaPair;
        mergeLevels     1;
    }
    
    // Velocity equation (non-symmetric)
    U
    {
        solver          smoothSolver;
        smoother        GaussSeidel;
        tolerance       1e-05;
        relTol          0.1;
        nSweeps         2;
    }
    
    // Turbulence equations (non-symmetric)
    k
    {
        solver          PBiCGStab;
        preconditioner  DILU;
        tolerance       1e-06;
        relTol          0.1;
    }
    
    epsilon
    {
        solver          PBiCGStab;
        preconditioner  DILU;
        tolerance       1e-06;
        relTol          0.1;
    }
}
```

**คำอธิบาย:**
- **tolerance**: ค่า error สัมบูรณ์สูงสุดที่ยอมรับ (เช่น 1e-06)
- **relTol**: ค่า error สัมพัทธ์ (0.1 = 10% ของ initial residual)
- **GAMG**: ใช้ multi-grid acceleration → เหมาะกับ large meshes (> 1M cells)
- **smoothSolver**: ใช้ iterative smoothing → เร็วสำหรับ small meshes

**Final Iteration:**
```cpp
pFinal
{
    $p;             // สืบทอดค่าจาก p
    relTol          0;      // บังคับ converge ถึง absolute tolerance
}
```

#### 4.2.2 Algorithm Settings

**SIMPLE Algorithm (Semi-Implicit Method for Pressure-Linked Equations)**

```cpp
SIMPLE
{
    nNonOrthogonalCorrectors 0;    //จำนวนครั้งที่แก้สมการความดันซ้ำสำหรับ non-orthogonal mesh
    
    consistent      yes;           // ใช้ consistent momentum interpolation
    
    residualControl                  // ค่า residual ที่ต้องการถึง
    {
        p               1e-4;       // Pressure residual < 1e-4
        U               1e-4;       // Velocity residual < 1e-4
        "(k|epsilon|omega)" 1e-3;   // Turbulence residual < 1e-3
    }
}
```

**คำอธิบาย:**
- **nNonOrthogonalCorrectors**: จำนวนรอบที่แก้ pEqn ซ้ำสำหรับ mesh ที่ไม่ orthogonal
  - `0`: Orthogonal mesh
  - `1-3`: Mild non-orthogonality (50-70°)
  - `>3`: Severe non-orthogonality (>70°)
- **consistent**: ใช้ momentum interpolation ที่ consistent กับ discrete continuity
- **residualControl**: กำหนด convergence criteria สำหรับ steady-state

**PISO Algorithm (Pressure-Implicit with Splitting of Operators)**

```cpp
PISO
{
    nCorrectors     2;             // จำนวน PISO loops ต่อ time step
    nNonOrthogonalCorrectors 0;    // Non-orthogonal correction loops
    nAlphaCorr      1;             // Correctors for volume fraction (multiphase)
    nAlphaSubCycles 2;             // Sub-cycles for alpha (VOF)
}
```

**คำอธิบาย:**
- **nCorrectors**: จำนวนรอบ pressure-velocity correction (มาตรฐาน = 2-3)
  - มากขึ้น = เสถียรขึ้น แต่ช้าลง
  - น้อยลง = เร็วขึ้น แต่อาจไม่เสถียร
- **nAlphaCorr**: สำหรับ multiphase flow (interFoam, interDyMFoam)

**PIMPLE Algorithm (Hybrid PISO-SIMPLE)**

```cpp
PIMPLE
{
    // PISO settings
    nCorrectors     2;
    nNonOrthogonalCorrectors 0;
    
    // SIMPLE settings (for outer iterations)
    nOuterCorrectors    1;         // จำนวน outer iterations ต่อ time step
    
    // Time control
    maxCo           0.5;           // Maximum Courant number
    maxDeltaT       1;             // Maximum time step [s]
    
    // Convergence control
    residualControl
    {
        p               1e-4;
        U               1e-4;
        "(k|epsilon|omega)" 1e-3;
    }
}
```

**คำอธิบาย:**
- **nOuterCorrectors**: จำนวน outer loops (SIMPLE-like) ภายในแต่ละ time step
  - `1`: Pure PISO (transient)
  - `>1`: PIMPLE (pseudo-transient หรือ large time step)
- **maxCo**: จำกัด Courant number สำหรับเสถียรภาพ
  - Incompressible: Co < 1
  - Compressible: Co < 0.3-0.5

#### 4.2.3 Relaxation Factors

**Under-Relaxation สำหรับเสถียรภาพเชิงตัวเลข**

```cpp
relaxationFactors
{
    fields
    {
        p               0.3;       // Pressure relaxation
        rho             1;         // Density (compressible)
        pFinal          1;         // Final iteration (no relaxation)
    }
    
    equations
    {
        U               0.7;       // Velocity relaxation
        k               0.7;       // Turbulent kinetic energy
        epsilon         0.7;       // Dissipation rate
        omega           0.7;       // Specific dissipation rate
        nuSgs           0.7;       // Eddy viscosity (LES)
    }
}
```

**คำอธิบาย:**
- **Relaxation factor (α)**: ค่าใหม่ = α·ค่าที่คำนวณ + (1-α)·ค่าเดิม
  - `1`: ไม่มี relaxation (ใช้ค่าใหม่ทั้งหมด)
  - `0.3-0.5`: สูง (conservative) → เสถียรมาก แต่ converge ช้า
  - `0.7-0.9`: ปานกลาง → สมดุลระหว่างเสถียรภาพและความเร็ว
  - `1`: ต่ำ (aggressive) → เร็วแต่อาจไม่เสถียร

**คำแนะนำการตั้งค่า:**

| สถานการณ์ | p | U | k/epsilon/omega |
|-----------|---|---|----------------|
| Initial guess | 0.2-0.3 | 0.5-0.7 | 0.5-0.7 |
| Converging | 0.5-0.7 | 0.8-0.9 | 0.8-0.9 |
| Final iteration | 1 | 1 | 1 |
| Bad mesh | 0.1-0.2 | 0.3-0.5 | 0.3-0.5 |

**Automatic Relaxation:**
```cpp
relaxationFactors
{
    fields
    {
        p               0.3;
    }
    
    equations
    {
        U               0.7;
        ".*"            0.7;       // ใช้ regex กำหนดค่า default
    }
}
```

#### 4.2.4 Convergence Criteria

**การตรวจสอบความลู่เข้า (Convergence Checking)**

```cpp
SIMPLE
{
    residualControl
    {
        p               1e-4;       // Initial residual < 1e-4
        U               1e-4;
        "(k|epsilon|omega)" 1e-3;
    }
}
```

**คำอธิบาย:**
- **Initial residual**: ||Ax - b|| / ||b|| (relative to source term)
- **Final residual**: ||Ax_new - b|| / ||Ax_old - b|| (relative to previous iteration)

**ตัวอย่างการตั้งค่า:**

```cpp
// Strict convergence (high accuracy)
residualControl
{
    p               1e-6;
    U               1e-6;
    "(k|epsilon)"   1e-5;
}

// Loose convergence (fast computation)
residualControl
{
    p               1e-3;
    U               1e-3;
    "(k|epsilon)"   1e-2;
}

// Field-specific convergence
residualControl
{
    p               1e-4;
    U               1e-4;
    T               1e-5;       // Temperature (if energy equation on)
    "(k|epsilon|omega)" 1e-3;
}
```

### 4.4 Troubleshooting Guide

#### 4.4.1 Divergence Issues

**อาการ:**
- Residual ไม่ลู่เข้า (oscillate หรือเพิ่มขึ้น)
- Solution แตก (NaN หรือ Inf)

**วิธีแก้:**
1. **ลด relaxation factors** (p: 0.2-0.3, U: 0.5-0.7)
2. **เปลี่ยนเป็น upwind scheme** สำหรับ div terms
3. **เพิ่ม nNonOrthogonalCorrectors** สำหรับ bad mesh
4. **ลด time step** (Co < 0.3)

---

#### 4.4.2 Slow Convergence

**อาการ:**
- Residual ลู่เข้าช้ามาก
- ใช้เวลาคำนวณนาน

**วิธีแก้:**
1. **เพิ่ม relaxation factors** หลังจาก solution เสถียรแล้ว
2. **ใช้ limitedLinear** แทน upwind
3. **ใช้ GAMG solver** สำหรับ pressure equation
4. **ตรวจสอบ mesh quality** → remesh ถ้าจำเป็น

---

#### 4.4.3 Mesh Quality Issues

**ตรวจสอบ mesh:**
```bash
checkMesh -allGeometry -allTopology
```

**ค่าที่สำคัญ:**
- **Non-orthogonality**: < 70° (ideal < 25°)
- **Skewness**: < 2 (ideal < 0.5)
- **Aspect ratio**: < 1000 (ideal < 10)

**การปรับ settings:**
- Non-orthogonality 50-70°: `nNonOrthogonalCorrectors 1-2`
- Non-orthogonality > 70°: `nNonOrthogonalCorrectors 3-5` หรือ remesh
- High skewness: ใช้ `Gauss limited` schemes

---

### 4.5 Best Practices

1. **เริ่มต้นด้วย conservative settings** (upwind, low relaxation)
2. **ค่อยๆ เพิ่มความแม่นยำ** หลังจาก solution เสถียร
3. **ตรวจสอบ mesh quality** ก่อนรัน
4. **ใช้ residualControl** เพื่อหยุดการคำนวณอัตโนมัติ
5. **บันทึก settings ที่ดี** สำหรับ test cases ที่คล้ายกัน

### Task 3: Implement Momentum Equation with Source Term

**Objective:** Modify the standard momentum equation to include a custom source term representing a porous media resistance:
$$

**Solution:** Implement the momentum equation with Darcy-Forchheimer source term

**File: `UEqn.H`**
```cpp
// Solve the Momentum equation

// Create temporary fields for source term calculation
volScalarField magU = mag(U);
volScalarField DarcyTerm = mu * alpha;  // Viscous resistance
volScalarField ForchheimerTerm = 0.5 * rho * beta * magU;  // Inertial resistance

// Momentum equation with porous media source term
fvVectorMatrix UEqn
(
    fvm::ddt(rho, U)
  + fvm::div(rhoPhi, U)
  + turbulence->divDevRhoReff(U)
 ==
    fvOptions(rho, U)
  - fvm::Sp(DarcyTerm + ForchheimerTerm, U)  // Porous media source: -S_U
);

UEqn.relax();

fvOptions.constrain(UEqn);

if (pimple.momentumPredictor())
{
    solve(UEqn == -fvc::grad(p));

}
```

## 6. Concept Checks

### Concept Check 1: Pressure-Velocity Coupling

ในอัลกอริทึม SIMPLE สำหรับการไหลแบบไม่อัดตัว (incompressible flow) เหตุใดเราจึงต้องแยกสมการโมเมนตัมและสมการความดันออกจากกัน และเราใช้เงื่อนไขอะไรในการอนุพันธ์สมการความดัน (Poisson equation) จากสมการต่อเนื่อง (continuity equation)?

<details><summary>คำตอบ</summary>

**การแยกสมการ:**
สมการโมเมนตัมและสมการความดันถูกแยกกันเพราะ:
1. **ไม่มีสมการชัดเจนสำหรับความดัน**: ใน incompressible flow ความดันไม่ปรากฏใน equation of state แต่ทำหน้าที่เป็น Lagrange multiplier ที่บังคับให้สนามความเร็วเป็นไปตาม continuity equation
2. **ความยากในการแก้ระบบ coupled**: ถ้าแก้พร้อมกันโดยตรงจะได้ระบบสมการขนาดใหญ่มาก (4 variables per cell: p, Ux, Uy, Uz) ซึ่งแก้ได้ยากและใช้หน่วยความจำสูง

**การอนุพันธ์สมการความดัน:**
เราเริ่มจากสมการต่อเนื่อง: ∇·U = 0

จากสมการโมเมนตัมที่ discrete แล้ว:
```
A_P U_P + Σ_N A_N U_N = -∇p + H
```
โดยที่ H คือเทอมที่ไม่ขึ้นกับความดัน

จัดรูปใหม่:
```
U_P = (H - ∇p) / A_P
```

แทนกลับเข้าใน continuity equation:
```
∇·[(H - ∇p) / A_P] = 0
```

ซึ่งนำไปสู่ Poisson equation สำหรับความดัน:
```
∇·(1/A_P ∇p) = ∇·(H/A_P)
```

ใน OpenFOAM นี้คือ:
```cpp
fvm::laplacian(rAU, p) == fvc::div(phiHbyA)
```
โดยที่ rAU = 1/A_P และ phiHbyA = flux จาก H/A_P

</details>

### Concept Check 2: Implicit vs Explicit Discretization

เมื่อใดควรใช้ `fvm::` (implicit) และเมื่อใดควรใช้ `fvc::` (explicit) ใน OpenFOAM? อธิบายผลต่อเสถียรภาพเชิงตัวเลข (numerical stability) และโครงสร้างเมทริกซ์ที่เกิดขึ้น

<details><summary>คำตอบ</summary>

**Implicit (`fvm::` - Finite Volume Method):**
- **ใช้เมื่อ**: เทอมที่มีตัวแปรที่เราแก้ (unknown) และต้องการเสถียรภาพสูง
- **ตัวอย่าง**: 
  - `fvm::ddt(U)` - เทอมเวลา
  - `fvm::div(phi, U)` - เทอมการพา
  - `fvm::laplacian(nu, U)` - เทอมการแพร่
- **ผลต่อเมทริกซ์**: สร้างเมทริกซ์สัมประสิทธิ์ (coefficient matrix) → ต้องแก้ระบบสมการเชิงเส้น
- **เสถียรภาพ**: สูงกว่า (unconditionally stable สำหรับ linear problems)
- **ค่าใช้จ่าย**: แก้ระบบสมการช้ากว่า แต่สามารถใช้ time step ที่ใหญ่กว่าได้

**Explicit (`fvc::` - Finite Volume Calculus):**
- **ใช้เมื่อ**: เทอมที่ไม่มี unknown หรือคำนวณจากค่าเดิม
- **ตัวอย่าง**:
  - `fvc::grad(p)` - คำนวณ gradient จากความดันที่รู้แล้ว
  - `fvc::div(phi)` - คำนวณ divergence ของ flux
  - `fvc::laplacian(sigma, U)` - ถ้า sigma เป็นค่าคงที่
- **ผลต่อเมทริกซ์**: ไม่สร้างเมทริกซ์ → คำนวณโดยตรง
- **เสถียรภาพ**: ต่ำกว่า (conditionally stable - มีขีดจำกัด time step)
- **ค่าใช้จ่าย**: คำนวณเร็ว แต่ต้องใช้ time step เล็ก

**ตัวอย่างใน UEqn.H:**
```cpp
fvVectorMatrix UEqn
(
    fvm::ddt(U)              // Implicit: สร้างเมทริกซ์
  + fvm::div(phi, U)         // Implicit: สร้างเมทริกซ์
  + turbulence->divDevReff(U)  // Implicit: สร้างเมทริกซ์
 ==
    fvOptions(U)             // Explicit/Implicit: ขึ้นกับประเภท source term
);

solve(UEqn == -fvc::grad(p)); // Explicit: คำนวณ gradient โดยตรง
```

**กฎพื้นฐาน:**
- ถ้าเทอมมี unknown → ใช้ `fvm::` (implicit)
- ถ้าเทอมเป็น known quantity → ใช้ `fvc::` (explicit)
- Pressure gradient ใน momentum equation → ใช้ `fvc::` เพราะแก้แยกจากสมการความดัน

</details>

### Concept Check 3: Non-Orthogonality Correction

เหตุใด mesh ที่มี non-orthogonality สูงจึงทำให้เกิดความผิดพลาดในการคำนวณ laplacian และเราแก้ไขปัญหานี้ด้วยวิธีการใดใน OpenFOAM?

<details><summary>คำตอบ</summary>

**ปัญหาของ Non-Orthogonal Mesh:**

ใน finite volume method การคำนวณ laplacian ใช้ Gauss theorem:
```
∇·(Γ∇φ) ≈ Σ_f (Γ_f ∇φ_f · S_f)
```

โดยปกติเราประมาณ:
```
∇φ_f · S_f ≈ Γ_f (φ_N - φ_P) / |d| · |S_f|
```

ซึ่งถูกต้อง **เฉพาะกรณีที่**:
- Face normal vector (S_f) ขนานกับ vector ระหว่าง cell centers (d)
- หรือ mesh เป็น orthogonal

**ถ้า mesh ไม่ orthogonal:**
- S_f และ d ไม่ขนานกัน
- เกิด error จากการที่ projection ไม่ตรงกัน
- Error แปรผันตามมุมระหว่าง S_f และ d

**วิธีแก้ไขใน OpenFOAM:**

1. **Non-Orthogonal Correction**:
```cpp
laplacianSchemes
{
    default Gauss linear corrected;
}
```

Correction term:
```
∇φ_f · S_f = Γ_f [(φ_N - φ_P)/|d| + k · (∇φ_f - (∇φ_f · d/|d|)d/|d|)] · S_f
```
โดยที่ k คือ correction factor

2. **การแก้สมการซ้ำ (nNonOrthogonalCorrectors)**:
```cpp
SIMPLE
{
    nNonOrthogonalCorrectors 2;
}
```

แก้ pEqn ซ้ำหลายรอบเพื่อปรับปรุงค่า:
- รอบที่ 1: ใช้ค่า p เดิมคำนวณ ∇p
- รอบที่ 2+: ใช้ค่า p ใหม่คำนวณ ∇p ที่แม่นยำขึ้น

3. **Limited Scheme** (สำหรับกรณีรุนแรง):
```cpp
laplacianSchemes
{
    default Gauss limited 0.5;
}
```

จำกัด correction term เพื่อป้องกัน instability

**แนวทางการใช้งาน:**
- Non-orthogonality < 25°: `uncorrected` (เร็ว)
- Non-orthogonality 25-70°: `corrected` + `nNonOrthogonalCorrectors 1-2`
- Non-orthogonality > 70°: `limited` หรือ remesh

**ตรวจสอบ mesh:**
```bash
checkMesh -allGeometry -allTopology
```

</details>

### Concept Check 4: Turbulence Modeling - Wall Functions

เหตุใดเราจึงต้องใช้ wall functions สำหรับ turbulence modeling และเงื่อนไข y+ มีความสำคัญอย่างไรในการเลือกใช้ wall functions หรือ resolving viscous sublayer?

<details><summary>คำตอบ</summary>

**ที่มาของปัญหา:**
ในการไหลแบบ turbulent ใกล้ผนัง:
- Velocity gradient สูงมาก (∂U/∂y ใหญ่)
- Viscous sublayer บางมาก (y+ < 5)
- ต้องการ mesh ละเอียดมากใกล้ผนัง (Δy ≈ 1e-6 m สำหรับ Re = 1e6)

**ถ้า resolve ทั้งหมด (Low Reynolds Number Model):**
- ต้องใช้ mesh ละเอียดมาก → cell count สูง
- y+ < 1 ทุกจุดบนผนัง
- คำนวณช้าแต่แม่นยำ

**ทางเลือก: Wall Functions**
- ไม่ resolve viscous sublayer
- ใช้ semi-empirical formulas ผสมกับ law-of-the-wall
- ใช้ mesh หยาบกว่าได้ (y+ ≈ 30-300)

**เงื่อนไข y+:**

| y+ Range | วิธีการ | Mesh Requirement | ความแม่นยำ |
|----------|----------|------------------|--------------|
| **y+ < 1** | Low-Re Model | ละเอียดมาก | สูง |
| **1 < y+ < 5** | Transition zone | ละเอียดปานกลาง | ปานกลาง |
| **30 < y+ < 300** | Standard Wall Functions | หยาบ | ปานกลาง-ต่ำ |
| **y+ > 300** | Invalid | หยาบเกินไป | ผิดพลาด |

**Wall Functions ใน OpenFOAM:**

1. **Standard Wall Functions** (k-epsilon, k-omega):
```cpp
boundaryField
{
    wall
    {
        type            kqRWallFunction;
        value           uniform 0;
    }
}
```
ใช้ได้เมื่อ: 30 < y+ < 300

2. **Low-Re Wall Functions** (k-omega SST):
```cpp
boundaryField
{
    wall
    {
        type            omegaWallFunction;
        value           uniform 0;
    }
}
```
ใช้ได้เมื่อ: y+ < 1

**การคำนวณ y+:**
```
y+ = (u* y) / ν
```
โดยที่:
- u* = friction velocity = √(τ_w / ρ)
- y = ระยะห่างจากผนัง
- ν = kinematic viscosity

**ตรวจสอบ y+ ใน OpenFOAM:**
```bash
# รัน simulation แล้วตรวจสอบ
postProcess -func yPlus

# หรือใช้ paraFoam  visualize
```

**แนวทางการเลือก:**
- **High Re, ไม่สนใจ viscous sublayer**: Wall functions (y+ ≈ 50-100)
- **Low Re, ต้องการความแม่นยำ**: Low-Re model (y+ < 1)
- **Heat transfer สำคัญ**: Resolve viscous sublayer (y+ < 1)

### Concept Check 5: Relaxation Factors และ Convergence

ใน steady-state solver (เช่น simpleFoam) เหตุใดเราจึงต้องใช้ under-relaxation? อธิบายความสัมพันธ์ระหว่าง relaxation factor และความเสถียรของการคำนวณ รวมถึงผลกระทบต่ออัตราการลู่เข้า (convergence rate)

<details><summary>คำตอบ</summary>

**ที่มาของปัญหา:**

ใน SIMPLE algorithm สำหรับ steady-state:
1. แก้สมการโมเมนตัม → ได้ U*
2. แก้สมการความดัน → ได้ p'
3. แก้ไข U และ p → ได้ U_new, p_new

ปัญหา: ค่า U_new และ p_new อาจแตกต่างจากค่าเดิมมากเกินไป → ทำให้ solution oscillate หรือ diverge

**Under-Relaxation:**

อัปเดตค่าด้วยการถ่วงน้ำหนัก:
```
φ_new = α φ_calculated + (1 - α) φ_old
```

โดยที่:
- α = relaxation factor (0 < α ≤ 1)
- φ_calculated = ค่าที่คำนวณได้จากสมการ
- φ_old = ค่าจาก iteration ก่อนหน้า

**ผลของค่า α:**

| α | ผลต่อเสถียรภาพ | อัตราการลู่เข้า | การใช้งาน |
|---|---|---|---|
| **α → 0** | สูงมาก (เกือบไม่เปลี่ยนแปลง) | ช้ามาก | Initial guess หรือ bad mesh |
| **α = 0.2-0.3** | สูง | ช้า | Starting phase |
| **α = 0.5-0.7** | ปานกลาง | ปานกลาง | Standard operation |
| **α = 0.8-0.9** | ต่ำ | เร็ว | Near convergence |
| **α = 1** | ต่ำสุด (ไม่มี relaxation) | เร็วสุด | Final iteration |

**ความสัมพันธ์ระหว่าง α และ Convergence:**

```
Convergence Rate ∝ α  (เมื่อใกล้ลู่เข้า)
Stability ∝ 1/α      (ยิ่ง α น้อยยิ่งเสถียร)
```

**Trade-off:**
- **α น้อย**: เสถียรแต่ converge ช้า → ใช้เวลาคำนวณนาน
- **α มาก**: converge เร็วแต่อาจ diverge → เสี่ยงต่อการแตก

**กลยุทธ์การปรับค่า:**

1. **Adaptive Relaxation** (แนะนำ):
```cpp
relaxationFactors
{
    fields
    {
        p               0.3;       // เริ่มต้นด้วยค่าต่ำ
    }
    equations
    {
        U               0.7;
        "(k|epsilon)"   0.7;
    }
}
```

2. **การปรับค่าแบบ Manual**:
   - **Iteration 1-100**: α_p = 0.2-0.3 (build stability)
   - **Iteration 100-500**: α_p = 0.5-0.7 (accelerate)
   - **Iteration 500+**: α_p = 0.8-0.9 (final convergence)
   - **Final**: α_p = 1.0 (no relaxation)

3. **Residual-Based Relaxation**:
```cpp
SIMPLE
{
    residualControl
    {
        p               1e-4;
        U               1e-4;
    }
    
    // ลด relaxation เมื่อ residual สูง
    // เพิ่ม relaxation เมื่อ residual ต่ำ
}
```

**ตัวอย่างใน fvSolution:**
```cpp
relaxationFactors
{
    fields
    {
        p               0.3;       // Pressure: ไวต่อการเปลี่ยนแปลง
        rho             1;         // Density: compressible flow
    }
    
    equations
    {
        U               0.7;       // Velocity: ปานกลาง
        k               0.7;       // Turbulence: ปานกลาง
        epsilon         0.7;       
    }
}
```

**เหตุผลที่ Pressure ต้องการ α ต่ำกว่า:**
- Pressure เป็นตัวแปร global (เปลี่ยนแปลงทั่วทั้ง domain ทันที)
- Velocity เป็นตัวแปร local (เปลี่ยนแปลงเฉพาะที่)
- ดังนั้น pressure ไวต่อการเปลี่ยนแปลงมากกว่า → ต้องใช้ α ต่ำ

**การตรวจสอบ Convergence:**
```bash
# ตรวจสอบ residual history
grep "Initial residual" log.simpleFoam | awk '{print $4}' > residuals.dat

# Plot ดูว่า converge หรือยัง
python plot_residuals.py
```

**แนวทางการ Troubleshooting:**
- **Oscillating residuals**: ลด α (p: 0.2, U: 0.5)
- **Diverging (NaN)**: ลด α อย่างมาก (p: 0.1, U: 0.3)
- **Stuck at high residual**: เพิ่ม α หรือตรวจสอบ BC
- **Very slow convergence**: เพิ่ม α หลังจาก solution เสถียร

</details>

ในอัลกอริทึม SIMPLE สำหรับการไหลแบบไม่อัดตัว (incompressible flow) เหตุใดเราจึงต้องแยกสมการโมเมนตัมและสมการความดันออกจากกัน และเราใช้เงื่อนไขอะไรในการอนุพันธ์สมการความดัน (Poisson equation) จากสมการต่อเนื่อง (continuity equation)?

<details><summary>คำตอบ</summary>

**การแยกสมการ:**
สมการโมเมนตัมและสมการความดันถูกแยกกันเพราะ:
1. **ไม่มีสมการชัดเจนสำหรับความดัน**: ใน incompressible flow ความดันไม่ปรากฏใน equation of state แต่ทำหน้าที่เป็น Lagrange multiplier ที่บังคับให้สนามความเร็วเป็นไปตาม continuity equation
2. **ความยากในการแก้ระบบ coupled**: ถ้าแก้พร้อมกันโดยตรงจะได้ระบบสมการขนาดใหญ่มาก (4 variables per cell: p, Ux, Uy, Uz) ซึ่งแก้ได้ยากและใช้หน่วยความจำสูง

**การอนุพันธ์สมการความดัน:**
เราเริ่มจากสมการต่อเนื่อง: ∇·U = 0

จากสมการโมเมนตัมที่ discrete แล้ว:
```
A_P U_P + Σ_N A_N U_N = -∇p + H
```
โดยที่ H คือเทอมที่ไม่ขึ้นกับความดัน

จัดรูปใหม่:
```
U_P = (H - ∇p) / A_P
```

แทนกลับเข้าใน continuity equation:
```
∇·[(H - ∇p) / A_P] = 0
```

ซึ่งนำไปสู่ Poisson equation สำหรับความดัน:
```
∇·(1/A_P ∇p) = ∇·(H/A_P)
```

ใน OpenFOAM นี้คือ:
```cpp
fvm::laplacian(rAU, p) == fvc::div(phiHbyA)
```
โดยที่ rAU = 1/A_P และ phiHbyA = flux จาก H/A_P

</details>

### Concept Check 2: Implicit vs Explicit Discretization

เมื่อใดควรใช้ `fvm::` (implicit) และเมื่อใดควรใช้ `fvc::` (explicit) ใน OpenFOAM? อธิบายผลต่อเสถียรภาพเชิงตัวเลข (numerical stability) และโครงสร้างเมทริกซ์ที่เกิดขึ้น

<details><summary>คำตอบ</summary>

**Implicit (`fvm::` - Finite Volume Method):**
- **ใช้เมื่อ**: เทอมที่มีตัวแปรที่เราแก้ (unknown) และต้องการเสถียรภาพสูง
- **ตัวอย่าง**: 
  - `fvm::ddt(U)` - เทอมเวลา
  - `fvm::div(phi, U)` - เทอมการพา
  - `fvm::laplacian(nu, U)` - เทอมการแพร่
- **ผลต่อเมทริกซ์**: สร้างเมทริกซ์สัมประสิทธิ์ (coefficient matrix) → ต้องแก้ระบบสมการเชิงเส้น
- **เสถียรภาพ**: สูงกว่า (unconditionally stable สำหรับ linear problems)
- **ค่าใช้จ่าย**: แก้ระบบสมการช้ากว่า แต่สามารถใช้ time step ที่ใหญ่กว่าได้

**Explicit (`fvc::` - Finite Volume Calculus):**
- **ใช้เมื่อ**: เทอมที่ไม่มี unknown หรือคำนวณจากค่าเดิม
- **ตัวอย่าง**:
  - `fvc::grad(p)` - คำนวณ gradient จากความดันที่รู้แล้ว
  - `fvc::div(phi)` - คำนวณ divergence ของ flux
  - `fvc::laplacian(sigma, U)` - ถ้า sigma เป็นค่าคงที่
- **ผลต่อเมทริกซ์**: ไม่สร้างเมทริกซ์ → คำนวณโดยตรง
- **เสถียรภาพ**: ต่ำกว่า (conditionally stable - มีขีดจำกัด time step)
- **ค่าใช้จ่าย**: คำนวณเร็ว แต่ต้องใช้ time step เล็ก

**ตัวอย่างใน UEqn.H:**
```cpp
fvVectorMatrix UEqn
(
    fvm::ddt(U)              // Implicit: สร้างเมทริกซ์
  + fvm::div(phi, U)         // Implicit: สร้างเมทริกซ์
  + turbulence->divDevReff(U)  // Implicit: สร้างเมทริกซ์
 ==
    fvOptions(U)             // Explicit/Implicit: ขึ้นกับประเภท source term
);

solve(UEqn == -fvc::grad(p)); // Explicit: คำนวณ gradient โดยตรง
```

**กฎพื้นฐาน:**
- ถ้าเทอมมี unknown → ใช้ `fvm::` (implicit)
- ถ้าเทอมเป็น known quantity → ใช้ `fvc::` (explicit)
- Pressure gradient ใน momentum equation → ใช้ `fvc::` เพราะแก้แยกจากสมการความดัน

</details>

### Concept Check 3: Non-Orthogonality Correction

เหตุใด mesh ที่มี non-orthogonality สูงจึงทำให้เกิดความผิดพลาดในการคำนวณ laplacian และเราแก้ไขปัญหานี้ด้วยวิธีการใดใน OpenFOAM?

<details><summary>คำตอบ</summary>

**ปัญหาของ Non-Orthogonal Mesh:**

ใน finite volume method การคำนวณ laplacian ใช้ Gauss theorem:
```
∇·(Γ∇φ) ≈ Σ_f (Γ_f ∇φ_f · S_f)
```

โดยปกติเราประมาณ:
```
∇φ_f · S_f ≈ Γ_f (φ_N - φ_P) / |d| · |S_f|
```

ซึ่งถูกต้อง **เฉพาะกรณีที่**:
- Face normal vector (S_f) ขนานกับ vector ระหว่าง cell centers (d)
- หรือ mesh เป็น orthogonal

**ถ้า mesh ไม่ orthogonal:**
- S_f และ d ไม่ขนานกัน
- เกิด error จากการที่ projection ไม่ตรงกัน
- Error แปรผันตามมุมระหว่าง S_f และ d

**วิธีแก้ไขใน OpenFOAM:**

1. **Non-Orthogonal Correction**:
```cpp
laplacianSchemes
{
    default Gauss linear corrected;
}
```

Correction term:
```
∇φ_f · S_f = Γ_f [(φ_N - φ_P)/|d| + k · (∇φ_f - (∇φ_f · d/|d|)d/|d|)] · S_f
```
โดยที่ k คือ correction factor

2. **การแก้สมการซ้ำ (nNonOrthogonalCorrectors)**:
```cpp
SIMPLE
{
    nNonOrthogonalCorrectors 2;
}
```

แก้ pEqn ซ้ำหลายรอบเพื่อปรับปรุงค่า:
- รอบที่ 1: ใช้ค่า p เดิมคำนวณ ∇p
- รอบที่ 2+: ใช้ค่า p ใหม่คำนวณ ∇p ที่แม่นยำขึ้น

3. **Limited Scheme** (สำหรับกรณีรุนแรง):
```cpp
laplacianSchemes
{
    default Gauss limited 0.5;
}
```

จำกัด correction term เพื่อป้องกัน instability

**แนวทางการใช้งาน:**
- Non-orthogonality < 25°: `uncorrected` (เร็ว)
- Non-orthogonality 25-70°: `corrected` + `nNonOrthogonalCorrectors 1-2`
- Non-orthogonality > 70°: `limited` หรือ remesh

**ตรวจสอบ mesh:**
```bash
checkMesh -allGeometry -allTopology
```

</details>

## 6. Concept Checks

### Concept Check 1: Pressure-Velocity Coupling

ในอัลกอริทึม SIMPLE สำหรับการไหลแบบไม่อัดตัว (incompressible flow) เหตุใดเราจึงต้องแยกสมการโมเมนตัมและสมการความดันออกจากกัน และเราใช้เงื่อนไขอะไรในการอนุพันธ์สมการความดัน (Poisson equation) จากสมการต่อเนื่อง (continuity equation)?

<details><summary>คำตอบ</summary>

**การแยกสมการ:**
สมการโมเมนตัมและสมการความดันถูกแยกกันเพราะ:
1. **ไม่มีสมการชัดเจนสำหรับความดัน**: ใน incompressible flow ความดันไม่ปรากฏใน equation of state แต่ทำหน้าที่เป็น Lagrange multiplier ที่บังคับให้สนามความเร็วเป็นไปตาม continuity equation
2. **ความยากในการแก้ระบบ coupled**: ถ้าแก้พร้อมกันโดยตรงจะได้ระบบสมการขนาดใหญ่มาก (4 variables per cell: p, Ux, Uy, Uz) ซึ่งแก้ได้ยากและใช้หน่วยความจำสูง

**การอนุพันธ์สมการความดัน:**
เราเริ่มจากสมการต่อเนื่อง: ∇·U = 0

จากสมการโมเมนตัมที่ discrete แล้ว:
```
A_P U_P + Σ_N A_N U_N = -∇p + H
```
โดยที่ H คือเทอมที่ไม่ขึ้นกับความดัน

จัดรูปใหม่:
```
U_P = (H - ∇p) / A_P
```

แทนกลับเข้าใน continuity equation:
```
∇·[(H - ∇p) / A_P] = 0
```

ซึ่งนำไปสู่ Poisson equation สำหรับความดัน:
```
∇·(1/A_P ∇p) = ∇·(H/A_P)
```

ใน OpenFOAM นี้คือ:
```cpp
fvm::laplacian(rAU, p) == fvc::div(phiHbyA)
```
โดยที่ rAU = 1/A_P และ phiHbyA = flux จาก H/A_P

</details>

### Concept Check 2: Implicit vs Explicit Discretization

เมื่อใดควรใช้ `fvm::` (implicit) และเมื่อใดควรใช้ `fvc::` (explicit) ใน OpenFOAM? อธิบายผลต่อเสถียรภาพเชิงตัวเลข (numerical stability) และโครงสร้างเมทริกซ์ที่เกิดขึ้น

<details><summary>คำตอบ</summary>

**Implicit (`fvm::` - Finite Volume Method):**
- **ใช้เมื่อ**: เทอมที่มีตัวแปรที่เราแก้ (unknown) และต้องการเสถียรภาพสูง
- **ตัวอย่าง**: 
  - `fvm::ddt(U)` - เทอมเวลา
  - `fvm::div(phi, U)` - เทอมการพา
  - `fvm::laplacian(nu, U)` - เทอมการแพร่
- **ผลต่อเมทริกซ์**: สร้างเมทริกซ์สัมประสิทธิ์ (coefficient matrix) → ต้องแก้ระบบสมการเชิงเส้น
- **เสถียรภาพ**: สูงกว่า (unconditionally stable สำหรับ linear problems)
- **ค่าใช้จ่าย**: แก้ระบบสมการช้ากว่า แต่สามารถใช้ time step ที่ใหญ่กว่าได้

**Explicit (`fvc::` - Finite Volume Calculus):**
- **ใช้เมื่อ**: เทอมที่ไม่มี unknown หรือคำนวณจากค่าเดิม
- **ตัวอย่าง**:
  - `fvc::grad(p)` - คำนวณ gradient จากความดันที่รู้แล้ว
  - `fvc::div(phi)` - คำนวณ divergence ของ flux
  - `fvc::laplacian(sigma, U)` - ถ้า sigma เป็นค่าคงที่
- **ผลต่อเมทริกซ์**: ไม่สร้างเมทริกซ์ → คำนวณโดยตรง
- **เสถียรภาพ**: ต่ำกว่า (conditionally stable - มีขีดจำกัด time step)
- **ค่าใช้จ่าย**: คำนวณเร็ว แต่ต้องใช้ time step เล็ก

**ตัวอย่างใน UEqn.H:**
```cpp
fvVectorMatrix UEqn
(
    fvm::ddt(U)              // Implicit: สร้างเมทริกซ์
  + fvm::div(phi, U)         // Implicit: สร้างเมทริกซ์
  + turbulence->divDevReff(U)  // Implicit: สร้างเมทริกซ์
 ==
    fvOptions(U)             // Explicit/Implicit: ขึ้นกับประเภท source term
);

solve(UEqn == -fvc::grad(p)); // Explicit: คำนวณ gradient โดยตรง
```

**กฎพื้นฐาน:**
- ถ้าเทอมมี unknown → ใช้ `fvm::` (implicit)
- ถ้าเทอมเป็น known quantity → ใช้ `fvc::` (explicit)
- Pressure gradient ใน momentum equation → ใช้ `fvc::` เพราะแก้แยกจากสมการความดัน

</details>

### Concept Check 3: Non-Orthogonality Correction

เหตุใด mesh ที่มี non-orthogonality สูงจึงทำให้เกิดความผิดพลาดในการคำนวณ laplacian และเราแก้ไขปัญหานี้ด้วยวิธีการใดใน OpenFOAM?

<details><summary>คำตอบ</summary>

**ปัญหาของ Non-Orthogonal Mesh:**

ใน finite volume method การคำนวณ laplacian ใช้ Gauss theorem:
```
∇·(Γ∇φ) ≈ Σ_f (Γ_f ∇φ_f · S_f)
```

โดยปกติเราประมาณ:
```
∇φ_f · S_f ≈ Γ_f (φ_N - φ_P) / |d| · |S_f|
```

ซึ่งถูกต้อง **เฉพาะกรณีที่**:
- Face normal vector (S_f) ขนานกับ vector ระหว่าง cell centers (d)
- หรือ mesh เป็น orthogonal

**ถ้า mesh ไม่ orthogonal:**
- S_f และ d ไม่ขนานกัน
- เกิด error จากการที่ projection ไม่ตรงกัน
- Error แปรผันตามมุมระหว่าง S_f และ d

**วิธีแก้ไขใน OpenFOAM:**

1. **Non-Orthogonal Correction**:
```cpp
laplacianSchemes
{
    default Gauss linear corrected;
}
```

Correction term:
```
∇φ_f · S_f = Γ_f [(φ_N - φ_P)/|d| + k · (∇φ_f - (∇φ_f · d/|d|)d/|d|)] · S_f
```
โดยที่ k คือ correction factor

2. **การแก้สมการซ้ำ (nNonOrthogonalCorrectors)**:
```cpp
SIMPLE
{
    nNonOrthogonalCorrectors 2;
}
```

แก้ pEqn ซ้ำหลายรอบเพื่อปรับปรุงค่า:
- รอบที่ 1: ใช้ค่า p เดิมคำนวณ ∇p
- รอบที่ 2+: ใช้ค่า p ใหม่คำนวณ ∇p ที่แม่นยำขึ้น

3. **Limited Scheme** (สำหรับกรณีรุนแรง):
```cpp
laplacianSchemes
{
    default Gauss limited 0.5;
}
```

จำกัด correction term เพื่อป้องกัน instability

**แนวทางการใช้งาน:**
- Non-orthogonality < 25°: `uncorrected` (เร็ว)
- Non-orthogonality 25-70°: `corrected` + `nNonOrthogonalCorrectors 1-2`
- Non-orthogonality > 70°: `limited` หรือ remesh

**ตรวจสอบ mesh:**
```bash
checkMesh -allGeometry -allTopology
```

</details>

### Concept Check 4: Turbulence Modeling - Wall Functions

เหตุใดเราจึงต้องใช้ wall functions สำหรับ turbulence modeling และเงื่อนไข y+ มีความสำคัญอย่างไรในการเลือกใช้ wall functions หรือ resolving viscous sublayer?

<details><summary>คำตอบ</summary>

**ที่มาของปัญหา:**
ในการไหลแบบ turbulent ใกล้ผนัง:
- Velocity gradient สูงมาก (∂U/∂y ใหญ่)
- Viscous sublayer บางมาก (y+ < 5)
- ต้องการ mesh ละเอียดมากใกล้ผนัง (Δy ≈ 1e-6 m สำหรับ Re = 1e6)

**ถ้า resolve ทั้งหมด (Low Reynolds Number Model):**
- ต้องใช้ mesh ละเอียดมาก → cell count สูง
- y+ < 1 ทุกจุดบนผนัง
- คำนวณช้าแต่แม่นยำ

**ทางเลือก: Wall Functions**
- ไม่ resolve viscous sublayer
- ใช้ semi-empirical formulas ผสมกับ law-of-the-wall
- ใช้ mesh หยาบกว่าได้ (y+ ≈ 30-300)

**เงื่อนไข y+:**

| y+ Range | วิธีการ | Mesh Requirement | ความแม่นยำ |
|----------|----------|------------------|--------------|
| **y+ < 1** | Low-Re Model | ละเอียดมาก | สูง |
| **1 < y+ < 5** | Transition zone | ละเอียดปานกลาง | ปานกลาง |
| **30 < y+ < 300** | Standard Wall Functions | หยาบ | ปานกลาง-ต่ำ |
| **y+ > 300** | Invalid | หยาบเกินไป | ผิดพลาด |

**Wall Functions ใน OpenFOAM:**

1. **Standard Wall Functions** (k-epsilon, k-omega):
```cpp
boundaryField
{
    wall
    {
        type            kqRWallFunction;
        value           uniform 0;
    }
}
```
ใช้ได้เมื่อ: 30 < y+ < 300

2. **Low-Re Wall Functions** (k-omega SST):
```cpp
boundaryField
{
    wall
    {
        type            omegaWallFunction;
        value           uniform 0;
    }
}
```
ใช้ได้เมื่อ: y+ < 1

**การคำนวณ y+:**
```
y+ = (u* y) / ν
```
โดยที่:
- u* = friction velocity = √(τ_w / ρ)
- y = ระยะห่างจากผนัง
- ν = kinematic viscosity

**ตรวจสอบ y+ ใน OpenFOAM:**
```bash
# รัน simulation แล้วตรวจสอบ
postProcess -func yPlus

# หรือใช้ paraFoam  visualize
```

**แนวทางการเลือก:**
- **High Re, ไม่สนใจ viscous sublayer**: Wall functions (y+ ≈ 50-100)
- **Low Re, ต้องการความแม่นยำ**: Low-Re model (y+ < 1)
- **Heat transfer สำคัญ**: Resolve viscous sublayer (y+ < 1)

</details>

### Concept Check 5: Relaxation Factors และ Convergence

ใน steady-state solver (เช่น simpleFoam) เหตุใดเราจึงต้องใช้ under-relaxation? อธิบายความสัมพันธ์ระหว่าง relaxation factor และความเสถียรของการคำนวณ รวมถึงผลกระทบต่ออัตราการลู่เข้า (convergence rate)

<details><summary>คำตอบ</summary>

**ที่มาของปัญหา:**

ใน SIMPLE algorithm สำหรับ steady-state:
1. แก้สมการโมเมนตัม → ได้ U*
2. แก้สมการความดัน → ได้ p'
3. แก้ไข U และ p → ได้ U_new, p_new

ปัญหา: ค่า U_new และ p_new อาจแตกต่างจากค่าเดิมมากเกินไป → ทำให้ solution oscillate หรือ diverge

**Under-Relaxation:**

อัปเดตค่าด้วยการถ่วงน้ำหนัก:
```
φ_new = α φ_calculated + (1 - α) φ_old
```

โดยที่:
- α = relaxation factor (0 < α ≤ 1)
- φ_calculated = ค่าที่คำนวณได้จากสมการ
- φ_old = ค่าจาก iteration ก่อนหน้า

**ผลของค่า α:**

| α | ผลต่อเสถียรภาพ | อัตราการลู่เข้า | การใช้งาน |เสถียรภาพ | อัตราการลู่เข้า | การใช้งาน |
|---|---|---|---|
| **α → 0** | สูงมาก (เกือบไม่เปลี่ยนแปลง) | ช้ามาก | Initial guess หรือ bad mesh |
| **α = 0.2-0.3** | สูง | ช้า | Starting phase |
| **α = 0.5-0.7** | ปานกลาง | ปานกลาง | Standard operation |
| **α = 0.8-0.9** | ต่ำ | เร็ว | Near convergence |
| **α = 1** | ต่ำสุด (ไม่มี relaxation) | เร็วสุด | Final iteration |

**ความสัมพันธ์ระหว่าง α และ Convergence:**

```
Convergence Rate ∝ α  (เมื่อใกล้ลู่เข้า)
Stability ∝ 1/α      (ยิ่ง α น้อยยิ่งเสถียร)
```

**Trade-off:**
- **α น้อย**: เสถียรแต่ converge ช้า → ใช้เวลาคำนวณนาน
- **α มาก**: converge เร็วแต่อาจ diverge → เสี่ยงต่อการแตก

**กลยุทธ์การปรับค่า:**

1. **Adaptive Relaxation** (แนะนำ):
```cpp
relaxationFactors
{
    fields
    {
        p               0.3;       // เริ่มต้นด้วยค่าต่ำ
    }
    equations
    {
        U               0.7;
        "(k|epsilon)"   0.7;
    }
}
```

2. **การปรับค่าแบบ Manual**:
   - **Iteration 1-100**: α_p = 0.2-0.3 (build stability)
   - **Iteration 100-500**: α_p = 0.5-0.7 (accelerate)
   - **Iteration 500+**: α_p = 0.8-0.9 (final convergence)
   - **Final**: α_p = 1.0 (no relaxation)

3. **Residual-Based Relaxation**:
```cpp
SIMPLE
{
    residualControl
    {
        p               1e-4;
        U               1e-4;
    }
    
    // ลด relaxation เมื่อ residual สูง
    // เพิ่ม relaxation เมื่อ residual ต่ำ
}
```

**ตัวอย่างใน fvSolution:**
```cpp
relaxationFactors
{
    fields
    {
        p               0.3;       // Pressure: ไวต่อการเปลี่ยนแปลง
        rho             1;         // Density: compressible flow
    }
    
    equations
    {
        U               0.7;       // Velocity: ปานกลาง
        k               0.7;       // Turbulence: ปานกลาง
        epsilon         0.7;       
    }
}
```

**เหตุผลที่ Pressure ต้องการ α ต่ำกว่า:**
- Pressure เป็นตัวแปร global (เปลี่ยนแปลงทั่วทั้ง domain ทันที)
- Velocity เป็นตัวแปร local (เปลี่ยนแปลงเฉพาะที่)
- ดังนั้น pressure ไวต่อการเปลี่ยนแปลงมากกว่า → ต้องใช้ α ต่ำ

**การตรวจสอบ Convergence:**
```bash
# ตรวจสอบ residual history
grep "Initial residual" log.simpleFoam | awk '{print $4}' > residuals.dat

# Plot ดูว่า converge หรือยัง
python plot_residuals.py
```

**แนวทางการ Troubleshooting:**
- **Oscillating residuals**: ลด α (p: 0.2, U: 0.5)
- **Diverging (NaN)**: ลด α อย่างมาก (p: 0.1, U: 0.3)
- **Stuck at high residual**: เพิ่ม α หรือตรวจสอบ BC
- **Very slow convergence**: เพิ่ม α หลังจาก solution เสถียร

</details>
