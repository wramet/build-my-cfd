## 1. Theory: Core Equations & Physics

### 1.1 Conservation of Mass (Continuity Equation)

**สมการการอนุรักษ์มวล (Continuity Equation)**

$$
\frac{\partial \rho}{\partial t} + \nabla \cdot (\rho \mathbf{U}) = 0
$$

**คำอธิบายพจน์:**
- $\rho$ (rho) = **ความหนาแน่น** (density) [kg/m³]
- $\mathbf{U}$ = **สนามความเร็ว** (velocity field) [m/s]
- $\nabla \cdot$ = **ตัวดำเนินการไดเวอร์เจนซ์** (divergence operator)
- $t$ = **เวลา** (time) [s]

สำหรับการไหลแบบ **ไม่อัดตัว (incompressible flow)**: $\nabla \cdot \mathbf{U} = 0$

---

### 1.2 Conservation of Momentum (Navier-Stokes Equation)

**สมการการอนุรักษ์โมเมนตัม (Momentum Equation)**

$$
\frac{\partial (\rho \mathbf{U})}{\partial t} + \nabla \cdot (\rho \mathbf{U} \mathbf{U}) = -\nabla p + \nabla \cdot \boldsymbol{\tau} + \rho \mathbf{g}
$$

**คำอธิบายพจน์:**
- $p$ = **ความดัน** (pressure) [Pa]
- $\boldsymbol{\tau}$ (tau) = **เทนเซอร์ความเค้น** (stress tensor) [Pa]
- $\mathbf{g}$ = **ความเร่งเนื่องจากแรงโน้มถ่วง** (gravitational acceleration) [m/s²]
- $\nabla p$ = **แรงตามแนวปกติของความดัน** (pressure gradient force)
- $\nabla \cdot \boldsymbol{\tau}$ = **แรงเหนียว** (viscous force)

**สำหรับของไหลนิวตัน (Newtonian Fluid):**

$$
\boldsymbol{\tau} = \mu \left[ \nabla \mathbf{U} + (\nabla \mathbf{U})^T \right] - \frac{2}{3}\mu (\nabla \cdot \mathbf{U})\mathbf{I}
$$

- $\mu$ (mu) = **ความหนืดพลศาสตร์** (dynamic viscosity) [Pa·s]
- $\mathbf{I}$ = **เทนเซอร์เอกลักษณ์** (identity tensor)

---

### 1.3 Conservation of Energy

**สมการการอนุรักษ์พลังงาน (Energy Equation)**

$$
\frac{\partial (\rho e)}{\partial t} + \nabla \cdot (\rho e \mathbf{U}) = -\nabla \cdot \mathbf{q} + \boldsymbol{\tau} : \nabla \mathbf{U} + S
$$

**คำอธิบายพจน์:**
- $e$ = **พลังงานภายในต่อหน่วยมวล** (specific internal energy) [J/kg]
- $\mathbf{q}$ = **ฟลักซ์ความร้อน** (heat flux) [W/m²]
- $S$ = **แหล่งกำเนิดพลังงาน** (energy source) [W/m³]
- $\boldsymbol{\tau} : \nabla \mathbf{U}$ = **การกระจายพลังงานเนื่องจากความเค้น** (viscous dissipation)

**กฎของฟูริเยร์ (Fourier's Law):**

$$
\mathbf{q} = -k \nabla T
$$

- $k$ = **สัมประสิทธิภาพการนำความร้อน** (thermal conductivity) [W/(m·K)]
- $T$ = **อุณหภูมิ** (temperature) [K]

---

### 1.4 Equation of State

**สมการสถานะ (Equation of State)**

สำหรับ **แก๊สอุดมคติ (Ideal Gas)**:

$$
p = \rho R T
$$

- $R$ = **ค่าคงที่ของแก๊สเฉพาะ** (specific gas constant) [J/(kg·K)]

สำหรับ **ของไหลที่ไม่อัดตัว (Incompressible Fluid)**:

$$
\rho = \text{constant}
$$

---

### 1.5 Transport Equations (General Form)

**รูปแบบทั่วไปของสมการเคลื่อนที่ (General Transport Equation)**

$$
\frac{\partial (\rho \phi)}{\partial t} + \nabla \cdot (\rho \phi \mathbf{U}) = \nabla \cdot (\Gamma_\phi \nabla \phi) + S_\phi
$$

**คำอธิบายพจน์:**
- $\phi$ (phi) = **ตัวแปรขนส่ง** (transported variable) - สามารถเป็นสเกลาร์หรือเวกเตอร์
- $\Gamma_\phi$ (Gamma) = **สัมประสิทธิภาพการแพร่** (diffusion coefficient)
- $S_\phi$ = **เทอมแหล่งกำเนิด/เก็บกิน** (source/sink term)

**องค์ประกอบของสมการ:**
1. **เทอมไม่คงที่ (Unsteady term)**: $\frac{\partial (\rho \phi)}{\partial t}$
2. **เทอมเนื่องจากการนำ (Convection term)**: $\nabla \cdot (\rho \phi \mathbf{U})$
3. **เทอมเนื่องจากการแพร่ (Diffusion term)**: $\nabla \cdot (\Gamma_\phi \nabla \phi)$
4. **เทอมแหล่งกำเนิด (Source term)**: $S_\phi$

## 3. Code Walkthrough

### 3.1 UEqn.H

ไฟล์ `UEqn.H` ใช้สำหรับสร้างสมการโมเมนตัมใน OpenFOAM solver ต่างๆ (เช่น `simpleFoam`, `pimpleFoam`)

**โครงสร้างพื้นฐาน:**

```cpp
// สร้างสมการเมทริกซ์สำหรับสมการโมเมนตัม
tmp<fvVectorMatrix> UEqn
(
    fvm::div(phi, U)              // เทอมการนำ (Convection)
  + fvm::laplacian(nu, U)         // เทอมการแพร่ (Diffusion/Viscous)
  - fvc::div(nu*dev(T(fvc::grad(U)))) // เทอมความเค้นสำหรับของไหลที่ไม่อัดตัว
);

// แก้สมการโมเมนตัม
UEqn.solve();

// ปรับปรุงแรงดันจากสมการโมเมนตัม
if (simple.momentumPredictor())
{
    U = UEqn.H()/A(U);
    U.correctBoundaryConditions();
}
```

**คำอธิบาย:**
- `fvm::div(phi, U)` - เทอมการนำ (convection term) ใช้ implicit discretization
- `fvm::laplacian(nu, U)` - เทอมความหนืด (viscous term) ใช้ implicit discretization
- `fvc::div(...)` - เทอมการกระจายความเค้น (stress divergence) ใช้ explicit discretization
- `UEqn.solve()` - แก้ระบบสมการเชิงเส้น
- `UEqn.H()` - ดึงส่วน H (explicit part) ของเมทริกซ์
- `A(U)` - ดึงส่วน A (diagonal coefficients) ของเมทริกซ์

**ตัวอย่างเพิ่มเติมสำหรับแรงโน้มถ่วง:**

```cpp
// เพิ่มเทอมแรงโน้มถ่วง
tmp<fvVectorMatrix> UEqn
(
    fvm::div(phi, U)
  + fvm::laplacian(nu, U)
 ==
    fvOptions(U)  // แหล่งกำเนิดเพิ่มเติม (เช่น แรงต้าน)
);

UEqn.relax();  // การผ่อนคลายสมการ (under-relaxation)
```

### 3.2 pEqn.H

ไฟล์ `pEqn.H` ใช้สำหรับสร้างและแก้สมการความดัน (pressure equation) ใน OpenFOAM solver ต่างๆ (เช่น `simpleFoam`, `pimpleFoam`)

**โครงสร้างพื้นฐาน:**

```cpp
// สร้างสมการความดันจากสมการโมเมนตัม
volScalarField rAU(1.0/UEqn.A());
volVectorField HbyA(constrainHbyA(rAU*UEqn.H(), U, p));
surfaceScalarField phiHbyA("phiHbyA", fvc::flux(HbyA));

// ปรับปรุงเงื่อนไขขอบเขตสำหรับ flux
adjustPhi(phiHbyA, U, p);

// สร้างสมการความดัน
tmp<fvScalarMatrix> pEqn
(
    fvm::laplacian(rAU, p) == fvc::div(phiHbyA)
);

// แก้สมการความดัน
pEqn.solve();

// แก้ไข flux และความเร็วจากความดันใหม่
phi = phiHbyA - pEqn.flux();
U = HbyA - rAU*fvc::grad(p);
U.correctBoundaryConditions();
```

**คำอธิบาย:**
- `rAU` = ค่าผกผันของสัมประสิทธิ์เชิงแนวทแยง (inverse diagonal coefficients)
- `HbyA` = สนามความเร็วชั่วคราวจากสมการโมเมนตัม (intermediate velocity)
- `phiHbyA` = flux ชั่วคราวที่ผิวขอบเขต
- `fvm::laplacian(rAU, p)` = เทอมการแพร่ของความดัน (implicit)
- `fvc::div(phiHbyA)` = เทอมไดเวอร์เจนซ์ของ flux (explicit)
- `pEqn.solve()` = แก้สมการความดัน
- `pEqn.flux()` = ดึง flux จากสมการความดันที่แก้แล้ว

**ตัวอย่างสำหรับ SIMPLE algorithm:**

```cpp
// การผ่อนคลายสมการความดัน (under-relaxation)
pEqn.relax();

// แก้ไข flux ภายในโดเมน
phi = phiHbyA + pEqn.flux()/rAU;

// แก้ไขสนามความเร็ว
U = HbyA - rAU*fvc::grad(p);
U.correctBoundaryConditions();

// ทำให้ flux สอดคล้องกับเงื่อนไขการอนุรักษ์มวล
#include "continuityErrs.H"
```

### 3.3 createFields.H

ไฟล์ `createFields.H` ใช้สำหรับสร้างและเริ่มต้นสนามต่างๆ ที่จำเป็นสำหรับการคำนวณ CFD ใน OpenFOAM solver

**โครงสร้างพื้นฐาน:**

```cpp
// สร้างสนามความเร็ว (velocity field)
Info<< "Reading field U\n" << endl;
volVectorField U
(
    IOobject
    (
        "U",
        runTime.timeName(),
        mesh,
        IOobject::MUST_READ,
        IOobject::AUTO_WRITE
    ),
    mesh
);

// สร้างสนามความดัน (pressure field)
Info<< "Reading field p\n" << endl;
volScalarField p
(
    IOobject
    (
        "p",
        runTime.timeName(),
        mesh,
        IOobject::MUST_READ,
        IOobject::AUTO_WRITE
    ),
    mesh
);

// สร้างคุณสมบัติการขนส่ง (transport properties)
singlePhaseTransportModel laminarTransport(U, phi);

// สร้างแบบจำลองความปั่นป่วน (turbulence model)
autoPtr<incompressible::turbulenceModel> turbulence
(
    incompressible::turbulenceModel::New(U, phi, laminarTransport)
);
```

**คำอธิบาย:**
- `volVectorField U` = สนามเวกเตอร์สำหรับเก็บความเร็ว [m/s]
- `volScalarField p` = สนามสเกลาร์สำหรับเก็บความดัน [Pa]
- `IOobject` = คลาสสำหรับจัดการการอ่าน/เขียนไฟล์
- `MUST_READ` = บังคับให้อ่านค่าเริ่มต้นจากไฟล์
- `AUTO_WRITE` = เขียนผลลัพธ์อัตโนมัติเมื่อบันทึกข้อมูล
- `singlePhaseTransportModel` = แบบจำลองคุณสมบัติการขนส่งสำหรับของไหลเฟสเดียว
- `turbulenceModel::New()` = สร้างแบบจำลองความปั่นป่วนตามที่ระบุใน `constant/turbulenceProperties`

## 4. Dictionary Analysis & Configuration

### 4.1 `system/fvSchemes`

**Purpose:** Defines numerical discretization schemes for temporal, spatial, and interpolation operations.

**Key Subdictionaries:**

```cpp
// Time discretization
ddtSchemes
{
    default         Euler;           // First-order implicit
    // default         backward;      // Second-order implicit
}

// Gradient schemes
gradSchemes
{
    default         Gauss linear;    // Central differencing
    // default         Gauss cubic;    // Fourth-order (less stable)
}

// Divergence (convection) schemes
divSchemes
{
    default         none;
    div(phi,U)      Gauss linearUpwind grad(U);  // Bounded upwind
    div(phi,k)      Gauss upwind;                // First-order stable
    div(phi,epsilon) Gauss upwind;
}

// Laplacian (diffusion) schemes
laplacianSchemes
{
    default         Gauss linear corrected;
}

// Interpolation schemes
interpolationSchemes
{
    default         linear;
}

// SnGrad schemes (surface normal gradient)
snGradSchemes
{
    default         corrected;
}
```

**Scheme Selection Guidelines:**
- **Steady-state**: Use `Euler` or `backward` for time, `Gauss linear` for gradients
- **Transient**: Use `backward` (2nd order) or `CrankNicolson` (2nd order)
- **High Reynolds**: Use `linearUpwind` or `limitedLinear` for convection to avoid oscillations
- **Low Reynolds**: `Gauss linear` is sufficient and more accurate

---

### 4.2 `system/fvSolution`

**Purpose:** Controls solver algorithms, tolerances, and algorithm-specific settings.

**Key Subdictionaries:**

```cpp
// Solver settings for linear equation systems
solvers
{
    p
    {
        solver          GAMG;
        tolerance       1e-06;      // Absolute tolerance
        relTol          0.01;       // Relative tolerance (1%)
        smoother        GaussSeidel;
    }

    pFinal
    {
        $p;                        // Inherit from p
        relTol          0;         // Force absolute tolerance
    }

    U
    {
        solver          smoothSolver;
        smoother        GaussSeidel;
        tolerance       1e-05;
        relTol          0.1;
    }
}

// SIMPLE algorithm settings (steady-state)
SIMPLE
{
    nNonOrthogonalCorrectors 0;
    consistent      yes;        // Use consistent SIMPLE formulation
    
    residualControl
    {
        p               1e-4;    // Convergence criteria
        U               1e-4;
        "(k|epsilon)"   1e-4;
    }
}

// PIMPLE algorithm settings (transient)
PIMPLE
{
    nCorrectors     2;          // Pressure correctors per time step
    nNonOrthogonalCorrectors 1;
    nAlphaCorr      1;          // Volume fraction correctors (multiphase)
    nOuterCorrectors 1;         // Outer correctors (coupled problems)
    
    maxCo           0.5;        // Max Courant number
    maxAlphaCo      0.5;        // Max volume fraction Courant number
    
    rDeltaTSmoothingCoeff 0.1;  // Time step smoothing
    rDeltaTDampingCoeff 0.2;    // Time step damping
}

// Relaxation factors (under-relaxation)
relaxationFactors
{
    fields
    {
        p               0.3;    // Pressure relaxation
        rho             1;      // Density (no relaxation)
    }
    equations
    {
        U               0.7;    // Momentum relaxation
        "(k|epsilon)"   0.8;    // Turbulence relaxation
    }
}
```

**Solver Selection Guidelines:**
- **Pressure (p)**: Use `GAMG` for large meshes, `PCG` for small meshes
- **Velocity (U)**: Use `smoothSolver` with `GaussSeidel` for most cases
- **Turbulence**: `smoothSolver` is usually sufficient
- **Final tolerance**: Set `relTol` to 0 for final outer iteration in transient

## 5. Hands-on: Practical Tasks & Coding

### Task 1: Create a Custom Boundary Condition

**Objective:** Implement a simple time-varying velocity boundary condition.

**File:** `myTimeVaryingFixedValueFvPatchVectorField.C`

```cpp
#include "fvPatchField.H"
#include "volFields.H"
#include "addToRunTimeSelectionTable.H"

// * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * //

namespace Foam
{

// Constructor
myTimeVaryingFixedValueFvPatchVectorField::
myTimeVaryingFixedValueFvPatchVectorField
(
    const fvPatch& p,
    const DimensionedField<vector, volMesh>& iF
)
:
    fixedValueFvPatchVectorField(p, iF),
    amplitude_(vector::zero),
    frequency_(0.0)
{}


// Constructor from dictionary
myTimeVaryingFixedValueFvPatchVectorField::
myTimeVaryingFixedValueFvPatchVectorField
(
    const fvPatch& p,
    const DimensionedField<vector, volMesh>& iF,
    const dictionary& dict
)
:
    fixedValueFvPatchVectorField(p, iF, dict),
    amplitude_(dict.lookup("amplitude")),
    frequency_(dict.lookupOrDefault<scalar>("frequency", 1.0))
{
    // Evaluate at t = 0
    updateCoeffs();
}


// Update the patch value
void myTimeVaryingFixedValueFvPatchVectorField::updateCoeffs()
{
    if (this->updated())
    {
        return;
    }
    
    const scalar t = this->db().time().value();
    const scalar omega = 2.0 * constant::mathematical::pi * frequency_;
    
    // U(t) = U0 + A * sin(omega * t)
    vectorField::operator=(
        amplitude_ * Foam::sin(omega * t)
    );
    
    fixedValueFvPatchVectorField::updateCoeffs();
}


// Write
void myTimeVaryingFixedValueFvPatchVectorField::write(Ostream& os) const
{
    fixedValueFvPatchVectorField::write(os);
    os.writeKeyword("amplitude") << amplitude_ << token::END_STATEMENT << nl;

## 6. Concept Checks

### Check 1: Discretization Schemes

**Question:** For a high-Reynolds number turbulent flow simulation, which convection scheme would you choose and why?

<details>
<summary>Answer</summary>

For high-Reynolds number turbulent flows, I would use **`Gauss linearUpwind grad(U)`** or **`Gauss limitedLinear 1`**.

**Reasoning:**
1. **Stability**: Central differencing (`Gauss linear`) can cause unphysical oscillations (wiggles) in regions of high gradients (shocks, boundary layers) due to the lack of numerical diffusion
2. **Boundedness**: `linearUpwind` uses upwind-biased interpolation which adds controlled numerical diffusion, ensuring bounded solutions
3. **Accuracy**: `linearUpwind` is second-order accurate (unlike first-order `upwind`), maintaining reasonable accuracy
4. **Turbulence modeling**: High-Re flows have sharp velocity gradients; upwind schemes prevent non-physical oscillations that can destabilize turbulence models

**Alternative**: `Gauss limitedLinear 1` uses a limiter function to switch between central and upwind based on the local gradient, providing a good balance between accuracy and stability.
</details>

### Check 2: Pressure-Velocity Coupling

**Question:** Explain the role of `nNonOrthogonalCorrectors` in `fvSolution` and when you would increase it.

<details>
<summary>Answer</summary>

**Purpose:** `nNonOrthogonalCorrectors` controls the number of times the pressure equation is re-solved within a single time step (or outer iteration) to account for mesh non-orthogonality.

**How it works:**
1. The pressure equation contains a non-orthogonal correction term: `∇·(rAU ∇p)`
2. On highly non-orthogonal meshes, this term is significant and cannot be fully captured in a single solve
3. Multiple correctors iteratively improve the solution by updating the flux and re-solving

**When to increase:**
- **Mesh quality**: For meshes with maximum non-orthogonality > 70°, use 2-3 correctors
- **Convergence**: If pressure residuals plateau or oscillate, increasing correctors can help
- **Accuracy**: For steady-state simulations requiring high accuracy on poor quality meshes

**Trade-off**: Each additional corrector increases computational cost. Use the minimum number that gives stable convergence.

**Typical values:**
- Orthogonal/hex-dominant meshes: 0
- Moderate non-orthogonality (< 60°): 1
- High non-orthogonality (> 70°): 2-3
- Extreme cases: up to 5 (but consider improving mesh quality instead)
</details>

### Check 3: Under-Relaxation Factors

**Question:** You're running a steady-state RANS simulation that is diverging. How would you adjust the relaxation factors in `fvSolution`?

<details>
<summary>Answer</summary>

**Initial diagnosis:** Divergence in steady-state simulations is often caused by aggressive under-relaxation, leading to unstable updates.

**Adjustment strategy:**

1. **Reduce pressure relaxation** (most critical):
   ```cpp
   p               0.2;  // Reduce from 0.3 to 0.2
   ```

2. **Reduce momentum relaxation**:
   ```cpp
   U               0.5;  // Reduce from 0.7 to 0.5
   ```

3. **Reduce turbulence relaxation**:
   ```cpp
   "(k|epsilon|omega)"  0.5;  // Reduce from 0.8
   ```

4. **For severe divergence**, use very conservative values:
   ```cpp
   p               0.1;
   U               0.3;
   "(k|epsilon)"   0.3;
   ```

**Why this works:**
- Under-relaxation limits the change in field values between iterations: `φ_new = φ_old + α(φ_calculated - φ_old)`
- Smaller α means more gradual updates, improving stability
- Pressure-velocity coupling is most sensitive, hence `p` needs the strongest relaxation

**Recovery strategy:**
1. Reduce factors until simulation stabilizes
2. Run until residuals drop by 1-2 orders of magnitude
3. Gradually increase factors back to normal values
4. Monitor residuals for any sudden increases

**Note**: If you need α < 0.2 for stability, consider other issues (mesh quality, boundary conditions, initial conditions)
</details>

    // default         Gauss cubic;    // Fourth-order (less stable)
}

// Divergence (convection) schemes
divSchemes
{
    default         none;
    div(phi,U)      Gauss linearUpwind grad(U);  // Bounded upwind
    div(phi,k)      Gauss upwind;                // First-order stable
    div(phi,epsilon) Gauss upwind;
}

// Laplacian (diffusion) schemes
laplacianSchemes
{
    default         Gauss linear corrected;
}

// Interpolation schemes
interpolationSchemes
{
    default         linear;
}

// SnGrad schemes (surface normal gradient)
snGradSchemes
{
    default         corrected;
}
```

**Scheme Selection Guidelines:**
- **Steady-state**: Use `Euler` or `backward` for time, `Gauss linear` for gradients
- **Transient**: Use `backward` (2nd order) or `CrankNicolson` (2nd order)
- **High Reynolds**: Use `linearUpwind` or `limitedLinear` for convection to avoid oscillations
- **Low Reynolds**: `Gauss linear` is sufficient and more accurate

## 6. Concept Checks

### Check 1: Discretization Schemes

**Question:** For a high-Reynolds number turbulent flow simulation, which convection scheme would you choose and why?

<details>
<summary>Answer</summary>

For high-Reynolds number turbulent flows, I would use **`Gauss linearUpwind grad(U)`** or **`Gauss limitedLinear 1`**.

**Reasoning:**
1. **Stability**: Central differencing (`Gauss linear`) can cause unphysical oscillations (wiggles) in regions of high gradients (shocks, boundary layers) due to the lack of numerical diffusion
2. **Boundedness**: `linearUpwind` uses upwind-biased interpolation which adds controlled numerical diffusion, ensuring bounded solutions
3. **Accuracy**: `linearUpwind` is second-order accurate (unlike first-order `upwind`), maintaining reasonable accuracy
4. **Turbulence modeling**: High-Re flows have sharp velocity gradients; upwind schemes prevent non-physical oscillations that can destabilize turbulence models

**Alternative**: `Gauss limitedLinear 1` uses a limiter function to switch between central and upwind based on the local gradient, providing a good balance between accuracy and stability.
</details>

### Check 2: Pressure-Velocity Coupling

**Question:** Explain the role of `nNonOrthogonalCorrectors` in `fvSolution` and when you would increase it.

<details>
<summary>Answer</summary>

**Purpose:** `nNonOrthogonalCorrectors` controls the number of times the pressure equation is re-solved within a single time step (or outer iteration) to account for mesh non-orthogonality.

**How it works:**
1. The pressure equation contains a non-orthogonal correction term: `∇·(rAU ∇p)`
2. On highly non-orthogonal meshes, this term is significant and cannot be fully captured in a single solve
3. Multiple correctors iteratively improve the solution by updating the flux and re-solving

**When to increase:**
- **Mesh quality**: For meshes with maximum non-orthogonality > 70°, use 2-3 correctors
- **Convergence**: If pressure residuals plateau or oscillate, increasing correctors can help
- **Accuracy**: For steady-state simulations requiring high accuracy on poor quality meshes

**Trade-off**: Each additional corrector increases computational cost. Use the minimum number that gives stable convergence.

**Typical values:**
- Orthogonal/hex-dominant meshes: 0
- Moderate non-orthogonality (< 60°): 1
- High non-orthogonality (> 70°): 2-3
- Extreme cases: up to 5 (but consider improving mesh quality instead)
</details>

### Check 3: Under-Relaxation Factors

**Question:** You're running a steady-state RANS simulation that is diverging. How would you adjust the relaxation factors in `fvSolution`?

<details>
<summary>Answer</summary>

**Initial diagnosis:** Divergence in steady-state simulations is often caused by aggressive under-relaxation, leading to unstable updates.

**Adjustment strategy:**

1. **Reduce pressure relaxation** (most critical):
   ```cpp
   p               0.2;  // Reduce from 0.3 to 0.2
   ```

2. **Reduce momentum relaxation**:
   ```cpp
   U               0.5;  // Reduce from 0.7 to 0.5
   ```

3. **Reduce turbulence relaxation**:
   ```cpp
   "(k|epsilon|omega)"  0.5;  // Reduce from 0.8
   ```

4. **For severe divergence**, use very conservative values:
   ```cpp
   p               0.1;
   U               0.3;
   "(k|epsilon)"   0.3;
   ```

**Why this works:**
- Under-relaxation limits the change in field values between iterations: `φ_new = φ_old + α(φ_calculated - φ_old)`
- Smaller α means more gradual updates, improving stability
- Pressure-velocity coupling is most sensitive, hence `p` needs the strongest relaxation

**Recovery strategy:**
1. Reduce factors until simulation stabilizes
2. Run until residuals drop by 1-2 orders of magnitude
3. Gradually increase factors back to normal values
4. Monitor residuals for any sudden increases

**Note**: If you need α < 0.2 for stability, consider other issues (mesh quality, boundary conditions, initial conditions)
</details>

### Check 4: Solver Selection

**Question:** Compare `GAMG` and `PCG` solvers. When would you use each?

<details>
<summary>Answer</summary>

**GAMG (Geometric-Algebraic Multi-Grid):**

**Mechanism:**
- Uses a hierarchy of coarse grids to accelerate convergence
- Smooths error on different scales using different grid levels
- Reduces low-frequency errors efficiently

**Advantages:**
- **O(N) complexity**: Scales linearly with mesh size
- **Fast for large meshes**: Ideal for > 1M cells
- **Low memory**: Coarse grids add minimal overhead

**Disadvantages:**
- Setup cost for coarse grids
- Less efficient on small meshes
- May struggle with highly anisotropic cells

**Best for:**
- Large meshes (> 500K cells)
- Pressure equation (elliptic, smooth solution)
- 3D complex geometries
