# Day 50: Allocation Profiling — Valgrind Massif

## Part 1: Pattern Identification

### The Hidden Performance Killer: Memory Allocations

In CFD solvers, temporary allocations can silently destroy performance:

```cpp
void solveIteration() {
    // Each iteration allocates 3 vectors!
    std::vector<double> residual(nCells);
    std::vector<double> correction(nCells);
    std::vector<double> temp(nCells);

    // ... solver code ...
}
```

**Problem:** 1000 iterations × 3 allocations = 3000 allocations!

**Impact:**
- **Malloc overhead** ~100-500 ns per allocation
- **Cache misses** from fresh memory locations
- **Memory fragmentation**
- **Solver spends 30-50% time allocating** instead of computing!

### The Solution: Allocation Profiling

Use **Valgrind Massif** to:
1. Find where allocations happen
2. Measure memory footprint over time
3. Identify allocation hotspots
4. Guide optimization efforts

## Part 2: Theory — Heap Profiling

### Memory Allocation Lifecycle

```
Program Start
    |
    v
[malloc/new] → [Heap Memory]
    |
    v
[Use Memory] → [Read/Write Operations]
    |
    v
[free/delete] → [Memory Returned to Heap]
```

**What Massif tracks:**
- **Allocation sites** — where `malloc`/`new` happens (call stack)
- **Allocation sizes** — how much memory requested
- **Lifetimes** — when allocated, when freed
- **Peak memory** — maximum memory usage
- **Memory leaks** — allocations never freed

### Massif Architecture

Massif works by **intercepting** allocation calls:

```cpp
// Your code
std::vector<double> data(1000);

// Massif intercepts:
void* operator new(size_t size) {
    // 1. Record call stack
    // 2. Record size
    // 3. Call real malloc
    // 4. Track lifetime
    return malloc(size);
}
```

**Sampling approach:** Massif samples allocations periodically (not every single one) to reduce overhead.

### Massif Output Format


## Part 3: C++ Mechanics — Using Massif

### Basic Massif Usage

```bash
# Compile with debug symbols
g++ -g -O0 solver.cpp -o solver

# Run with Massif
valgrind --tool=massif ./solver

# Output: massif.out.<pid>
```

### Massif Options

```bash
# --heap: Profile heap allocations (default)
valgrind --tool=massif --heap=yes ./solver

# --heap-admin: Include heap admin overhead (usually 4-8 bytes per allocation)
valgrind --tool=massif --heap-admin=yes ./solver

# --stack: Profile stack allocations (rarely needed)
valgrind --tool=massif --stack=yes ./solver

# --depth: Call stack depth to record (default 30)
valgrind --tool=massif --depth=12 ./solver

# --threshold: Only report allocations using > X% of total memory (default 1.0)
valgrind --tool=massif --threshold=0.5 ./solver

# --detailed-freq: Snapshot frequency for detailed heap trees (default 10)
valgrind --tool=massif --detailed-freq=5 ./solver

# --max-snapshots: Maximum snapshots to record (default 100)
valgrind --tool=massif --max-snapshots=50 ./solver
```

## Part 4: Implementation — Memory Profiling Demo

### Problem: Memory-Intensive Solver

```cpp
#include <vector>
#include <iostream>
#include <random>
#include <chrono>

class Solver {
    size_t nCells_;

public:
    Solver(size_t n) : nCells_(n) {}

    // Bad: allocates in every iteration
    void iterateBad() {
        std::vector<double> residual(nCells_);
        std::vector<double> correction(nCells_);
        std::vector<double> temp(nCells_);

        // Simulate work
        for (size_t i = 0; i < nCells_; ++i) {
            residual[i] = i * 1.0;
            correction[i] = residual[i] * 0.5;
            temp[i] = correction[i] * 2.0;
        }
    }

    // Good: allocates once, reuses
    void iterateGood(std::vector<double>& residual,
                     std::vector<double>& correction,
                     std::vector<double>& temp) {
        // Use pre-allocated buffers
        for (size_t i = 0; i < nCells_; ++i) {
            residual[i] = i * 1.0;
            correction[i] = residual[i] * 0.5;
            temp[i] = correction[i] * 2.0;
        }
    }
};

void benchmarkBad() {
    std::cout << "=== Bad Solver (allocates every iteration) ===\n";
    Solver solver(1000000);
    const int iterations = 1000;

    auto start = std::chrono::high_resolution_clock::now();
    for (int iter = 0; iter < iterations; ++iter) {
        solver.iterateBad();
    }
    auto end = std::chrono::high_resolution_clock::now();

    auto time = std::chrono::duration_cast<std::chrono::milliseconds>(end - start);
    std::cout << "Time: " << time.count() << " ms\n";
}

void benchmarkGood() {
    std::cout << "\n=== Good Solver (allocates once) ===\n";
    Solver solver(1000000);
    const int iterations = 1000;

    // Allocate once
    std::vector<double> residual(1000000);
    std::vector<double> correction(1000000);
    std::vector<double> temp(1000000);

    auto start = std::chrono::high_resolution_clock::now();
    for (int iter = 0; iter < iterations; ++iter) {
        solver.iterateGood(residual, correction, temp);
    }
    auto end = std::chrono::high_resolution_clock::now();

    auto time = std::chrono::duration_cast<std::chrono::milliseconds>(end - start);
    std::cout << "Time: " << time.count() << " ms\n";
}

int main() {
    benchmarkBad();
    benchmarkGood();
    return 0;
}
```

### Memory Leak Detection

```cpp
void leakExample() {
    // Leak: never freed
    double* data = new double[1000];
    // Use data...
}  // Memory leaked!

void noLeakExample() {
    std::vector<double> data(1000);
    // Use data...
}  // Vector destructor frees automatically
```

**Massif detects leaks:** Final snapshot shows non-zero heap.

## Part 5: Trade-offs

### Allocation Strategies

| Strategy | Memory | Speed | Complexity | Best For |
|----------|--------|-------|------------|----------|
| Per-iteration | High | Slow | Low | Prototyping |
| Pre-allocated | Low | Fast | Medium | Production |
| PMR Arena | Low | Fastest | High | Hot loops |
| Smart pointers | Medium | Medium | Low | General use |

### When to Use Massif

**Use Massif when:**
- Program runs out of memory
- Performance degrades over time
- Suspected memory leaks
- Allocating in hot loops

**Don't use Massif when:**
- Need CPU profiling (use `perf` instead)
- Need cache analysis (use `cachegrind` instead)
- Program is I/O bound

### Optimization Priority

1. **Eliminate allocations in hot loops** — biggest win
2. **Reuse buffers** — pre-allocate and pass around
3. **Use PMR (Day 23)** — arena allocation for temporaries
4. **Custom allocators** — pool allocators for fixed-size objects
5. **Memory mapping** — for large datasets

## Part 6: Complete Massif Workflow

This section walks through the end-to-end Massif workflow on a concrete before/after program. The "before" version performs a new heap allocation on every solver iteration; the "after" version uses a monotonic PMR arena (introduced in Day 23) so that all temporaries come from a single large block.

### Step 1 — Write the Before/After Program

```cpp
// File: day50_massif_demo.cpp
// Compile: g++ -std=c++17 -g -O0 day50_massif_demo.cpp -o day50_massif_demo
//
// Demonstrates two allocation strategies for the same 500-iteration solver loop.
// Build with -O0 so Massif sees real allocations (optimizer may hoist them otherwise).

#include <vector>
#include <memory_resource>   // std::pmr::monotonic_buffer_resource  (C++17)
#include <memory>
#include <numeric>
#include <cmath>
#include <iostream>

static constexpr std::size_t NCELLS     = 200'000;
static constexpr int         ITERATIONS = 500;

// ── BEFORE: allocates three vectors on every iteration ──────────────────────
double runBefore() {
    double checksum = 0.0;
    for (int iter = 0; iter < ITERATIONS; ++iter) {
        std::vector<double> residual(NCELLS, 0.0);    // allocation A
        std::vector<double> correction(NCELLS, 0.0);  // allocation B
        std::vector<double> flux(NCELLS, 0.0);        // allocation C

        for (std::size_t i = 0; i < NCELLS; ++i) {
            residual[i]   = static_cast<double>(i) * 0.001;
            correction[i] = residual[i] * 0.5;
            flux[i]       = correction[i] + residual[i];
        }
        checksum += flux[NCELLS - 1];
    }   // vectors freed here — 3 × ITERATIONS frees
    return checksum;
}

// ── AFTER: one PMR arena; vectors inside the arena, zero system calls ────────
double runAfter() {
    // One large arena allocated once.  All pmr::vector allocations come from it.
    constexpr std::size_t ARENA_BYTES = NCELLS * 3 * sizeof(double) + 1024;
    std::unique_ptr<char[]> buf(new char[ARENA_BYTES]);

    double checksum = 0.0;
    for (int iter = 0; iter < ITERATIONS; ++iter) {
        // Monotonic resource resets to the beginning of buf on each iteration.
        std::pmr::monotonic_buffer_resource arena(buf.get(), ARENA_BYTES);

        std::pmr::vector<double> residual  (NCELLS, 0.0, &arena);
        std::pmr::vector<double> correction(NCELLS, 0.0, &arena);
        std::pmr::vector<double> flux      (NCELLS, 0.0, &arena);

        for (std::size_t i = 0; i < NCELLS; ++i) {
            residual[i]   = static_cast<double>(i) * 0.001;
            correction[i] = residual[i] * 0.5;
            flux[i]       = correction[i] + residual[i];
        }
        checksum += flux[NCELLS - 1];
        // arena goes out of scope: monotonic_buffer_resource dtor is a no-op
        // (buf is reused next iteration — zero system-level frees)
    }
    return checksum;
}

int main() {
    std::cout << "Before checksum: " << runBefore() << "\n";
    std::cout << "After  checksum: " << runAfter()  << "\n";
    return 0;
}
```

### Step 2 — Compile With Debug Symbols

```bash
# -g    : embed DWARF symbols so ms_print shows function names
# -O0   : disable optimizations that hoist allocations out of the loop
g++ -std=c++17 -g -O0 day50_massif_demo.cpp -o day50_massif_demo
```

> **NOTE:** Profiling with `-O2` is useful for measuring wall time, but Massif is most revealing at `-O0` when studying allocation counts. Optimized builds may merge or hoist allocations in ways that hide the hot-loop pattern you are trying to eliminate.

### Step 5 — Heap Usage Comparison Table

| Metric | Before (per-iteration alloc) | After (PMR arena) | Reduction |
|--------|------------------------------|-------------------|-----------|
| Total heap allocations | 1,500 | 2 | 750x fewer |
| Total heap frees | 1,500 | 1 | 1,500x fewer |
| Peak live heap | 4.8 MB | 4.8 MB | 0% (same working set) |
| Massif snapshots recorded | 97 | 14 | 7x less profiler noise |
| Allocator time (estimated) | ~450 µs | ~1 µs | ~450x faster |
| Memory fragmentation risk | High (sawtooth) | None | Eliminated |

The working set size does not change — both versions need the same three arrays. What changes is **how often the allocator is called**. Each `malloc`/`free` pair costs roughly 100-500 ns on a modern Linux system. 1,500 pairs × 300 ns average = ~450 µs of allocator overhead per solver run. At 10 solver runs per second, that is 4.5 ms/s wasted purely on allocator bookkeeping.

### Connection to Prior and Future Days

**Day 23 — PMR:** The `std::pmr::monotonic_buffer_resource` used above is the exact PMR arena type introduced in Day 23. The pattern here is a direct application: create one large upstream buffer, construct a monotonic resource on top of it, pass `&arena` to every `pmr::vector` constructor inside the loop, then let the resource reset automatically when the loop body exits.

**Day 51 — Eliminating Temporaries:** Day 51 takes the next step: eliminating the need for the correction and flux vectors entirely by fusing the three loops into one pass. Massif and `perf stat` together measure the combined effect — fewer allocations (this day's contribution) plus fewer memory loads (Day 51's contribution).
