## Part 1 — The Problem: Why Misaligned `Field<double>` Kills SIMD

### The CFD Context

A typical incompressible solver spends a large fraction of its time in field operations: dot products for residual norms, element-wise updates in the SIMPLE loop, and flux accumulation over face loops. Each of these is a pure arithmetic loop over a contiguous `double` array — exactly the kind of loop AVX2 is designed to accelerate.

AVX2 processes **4 doubles at once** using 256-bit registers. A single `vmovapd` instruction loads four consecutive doubles in one memory transaction, then `vfmadd231pd` accumulates four products simultaneously. For a field of one million cells, this ideally gives a **4x throughput improvement** over scalar code.

The catch is the word **"a"** in `vmovapd` — it stands for *aligned*. The processor requires that the base address of each 256-bit load is a multiple of 32 bytes. If the address is not aligned, the hardware must perform two separate loads, merge the result, and the instruction triggers a general-protection fault on older microarchitectures (or silently degrades throughput on newer ones).

### The Alignment Problem in Practice

```cpp
// Naive dynamic allocation — alignment is NOT guaranteed.
double* field = new double[1000000];

// What address do we get?
// new double[] calls ::operator new, which calls malloc.
// The C++ standard only guarantees alignment to alignof(max_align_t) = 16 bytes.
// On most Linux systems with glibc, malloc returns 16-byte aligned memory.
// AVX2 requires 32-byte alignment — so this MAY or MAY NOT work.

// This is the danger: code works on one machine, crashes or silently
// produces wrong timings on another where malloc happens to return a
// 16-byte aligned but not 32-byte aligned address.

std::ptrdiff_t addr = reinterpret_cast<std::ptrdiff_t>(field);
bool aligned32 = (addr % 32 == 0);
// aligned32 might be false — the compiler cannot use vmovapd safely!
```

When the address is not 32-byte aligned, the compiler must either:
1. Fall back to `vmovupd` (unaligned load) — works but is slower on older hardware.
2. Emit scalar code — no vectorization at all.
3. Generate a runtime alignment check with a scalar prologue — overhead per call.

The compiler often chooses option 3, which means even if your array happens to be aligned at runtime, you pay for the alignment check on every loop invocation.

### What OpenFOAM Does

OpenFOAM's `Field<T>` uses `List<T>`, which allocates through `new T[]`. As of OpenFOAM v2312, there is **no `alignas` annotation and no aligned allocator** in the default path. OpenFOAM relies on the compiler's auto-vectorizer emitting `vmovupd` (unaligned) instructions and hoping the OS allocator returns sufficiently aligned memory in practice.

This is pragmatic — it keeps the allocator simple and avoids dependencies. For a production HPC solver, it leaves performance on the table. Our custom `AlignedField<T, Align>` addresses this gap.

### Alignment Requirements by SIMD ISA

| ISA | Register width | Doubles per register | Required alignment |
|-----|---------------|---------------------|-------------------|
| SSE2 | 128 bits | 2 | 16 bytes |
| AVX / AVX2 | 256 bits | 4 | 32 bytes |
| AVX-512 | 512 bits | 8 | 64 bytes |

## Part 2 — The Concept: Alignment Fundamentals

### What Is Memory Alignment?

An object of type `T` is **aligned to N bytes** if its address is an exact multiple of N. The CPU's load/store units are optimized for aligned accesses because they map cleanly to cache-line boundaries and SIMD register widths.

```text
Address (decimal)    32-byte aligned?
0                    YES  (0 % 32 == 0)
32                   YES  (32 % 32 == 0)
64                   YES
16                   NO   (16 % 32 == 16)
48                   NO   (48 % 32 == 16)
```

### `alignas` — Static Alignment Guarantee

`alignas(N)` is a C++11 alignment specifier. It can be applied to variables, struct members, and type definitions.

```cpp
// Stack-allocated array with 32-byte alignment.
alignas(32) double buf[1024];

// buf's address is guaranteed to be divisible by 32.
// The compiler inserts padding before buf if necessary.

// Struct with aligned member:
struct alignas(32) AlignedPacket {
    double data[4];
};
// sizeof(AlignedPacket) == 32, alignof(AlignedPacket) == 32.
```

`alignas` only works for **static and automatic storage** (stack, global, class members). For heap-allocated memory, we need a different tool.

### `std::aligned_alloc` — Heap-Aligned Memory

Introduced in C11 and C++17, `std::aligned_alloc(alignment, size)` allocates `size` bytes aligned to `alignment`. The `size` must be a multiple of `alignment`.

```cpp
// Allocate 1000 doubles aligned to 32 bytes.
// Size must be a multiple of alignment (32), so round up.
std::size_t n = 1000;
std::size_t bytes = ((n * sizeof(double) + 31) / 32) * 32;  // round up to 32

double* p = static_cast<double*>(std::aligned_alloc(32, bytes));
if (!p) throw std::bad_alloc{};

// Use p...

std::free(p);  // IMPORTANT: use free(), NOT delete[]
```

On POSIX systems, `posix_memalign` is the older equivalent. On Windows, `_aligned_malloc` / `_aligned_free` must be used instead.

### Aligned vs Unaligned SIMD Load Intrinsics

Intel provides two AVX2 intrinsics for loading 4 doubles:

| Intrinsic | Requirement | Behavior on misalignment |
|-----------|------------|--------------------------|
| `_mm256_load_pd(ptr)` | `ptr` must be 32-byte aligned | General protection fault (pre-Haswell) or penalty |
| `_mm256_loadu_pd(ptr)` | No alignment required | Works always, ~same speed on Haswell and later |

On modern microarchitectures (Haswell 2013+), the performance difference between aligned and unaligned loads is negligible when the data *happens* to be aligned. The key benefit of using `_mm256_load_pd` is that it **communicates intent to the compiler and analyzer**: the compiler can omit the runtime alignment check and prologue scalar loop, generating tighter code.

### How the Compiler Uses Alignment Information

When the compiler sees a loop over a `double*` without alignment information, it generates a conservative pattern:

```text
1. Scalar prologue: advance pointer until aligned to 32 bytes.
2. SIMD body: process 4 doubles per iteration with vmovupd.
3. Scalar epilogue: handle remaining elements.
```

When the compiler *knows* the pointer is 32-byte aligned (via `alignas`, `__builtin_assume_aligned`, or `std::assume_aligned<32>(ptr)`), it can skip step 1 and use `vmovapd` in step 2.

## Part 3 — Implementation: `AlignedField<T, Align>`

This section builds a complete, self-contained field class that manages aligned heap memory. It connects to Day 20 (`std::span` for zero-copy views) and Day 24 (Mesh Topology Storage, where fields are associated with mesh cells).

```cpp
// File: aligned_field.hpp
// Compile: g++ -std=c++20 -O2 -mavx2 -fopt-info-vec -o demo aligned_field_demo.cpp

#pragma once

#include <cstddef>
#include <cstdlib>
#include <cstring>
#include <stdexcept>
#include <span>
#include <type_traits>
#include <new>

// ============================================================
// print_alignment: diagnostic helper
// ============================================================

inline void print_alignment(const char* label, const void* ptr, std::size_t align)
{
    auto addr = reinterpret_cast<std::uintptr_t>(ptr);
    bool ok = (addr % align == 0);
    std::printf("%-30s  addr=0x%zx  %zu-byte aligned: %s\n",
                label, static_cast<std::size_t>(addr), align,
                ok ? "YES" : "NO");
}

// ============================================================
// AlignedField<T, Align>
// A heap-allocated array of T, guaranteed aligned to Align bytes.
// Align must be a power of two and >= alignof(T).
// ============================================================

template<typename T, std::size_t Align = 32>
class AlignedField
{
    static_assert(Align >= alignof(T),
                  "Align must be at least alignof(T)");
    static_assert((Align & (Align - 1)) == 0,
                  "Align must be a power of two");

    T*          data_   = nullptr;
    std::size_t size_   = 0;
    std::size_t bytes_  = 0;   // actual allocation size (rounded up)

    // Round `n * sizeof(T)` up to the next multiple of Align.
    static std::size_t roundedBytes(std::size_t n) noexcept
    {
        std::size_t raw = n * sizeof(T);
        return (raw + Align - 1) & ~(Align - 1);
    }

    void allocate(std::size_t n)
    {
        if (n == 0) { data_ = nullptr; size_ = 0; bytes_ = 0; return; }
        bytes_ = roundedBytes(n);
        void* raw = std::aligned_alloc(Align, bytes_);
        if (!raw) throw std::bad_alloc{};
        data_ = static_cast<T*>(raw);
        size_ = n;
    }

    void deallocate() noexcept
    {
        std::free(data_);
        data_  = nullptr;
        size_  = 0;
        bytes_ = 0;
    }

public:
    // ---- Constructors ----

    AlignedField() = default;

    explicit AlignedField(std::size_t n)
    {
        allocate(n);
        // Value-initialise (zero for arithmetic types).
        if constexpr (std::is_trivially_constructible_v<T>)
            std::memset(data_, 0, bytes_);
        else
            for (std::size_t i = 0; i < n; ++i) new (data_ + i) T{};
    }

    AlignedField(std::size_t n, T value)
    {
        allocate(n);
        for (std::size_t i = 0; i < n; ++i) data_[i] = value;
        // Zero-fill the padding so SIMD epilogue reads safe values.
        for (std::size_t i = n * sizeof(T); i < bytes_; ++i)
            reinterpret_cast<unsigned char*>(data_)[i] = 0;
    }

    // Copy constructor
    AlignedField(const AlignedField& other)
    {
        allocate(other.size_);
        std::memcpy(data_, other.data_, other.size_ * sizeof(T));
    }

    // Move constructor
    AlignedField(AlignedField&& other) noexcept
        : data_(other.data_), size_(other.size_), bytes_(other.bytes_)
    {
        other.data_  = nullptr;
        other.size_  = 0;
        other.bytes_ = 0;
    }

    // Copy assignment
    AlignedField& operator=(const AlignedField& other)
    {
        if (this == &other) return *this;
        if (size_ != other.size_) { deallocate(); allocate(other.size_); }
        std::memcpy(data_, other.data_, other.size_ * sizeof(T));
        return *this;
    }

    // Move assignment
    AlignedField& operator=(AlignedField&& other) noexcept
    {
        if (this == &other) return *this;
        deallocate();
        data_  = other.data_;
        size_  = other.size_;
        bytes_ = other.bytes_;
        other.data_  = nullptr;
        other.size_  = 0;
        other.bytes_ = 0;
        return *this;
    }

    ~AlignedField() { deallocate(); }

    // ---- Accessors ----

    T&       operator[](std::size_t i)       noexcept { return data_[i]; }
    const T& operator[](std::size_t i) const noexcept { return data_[i]; }

    T*       data()       noexcept { return data_; }
    const T* data() const noexcept { return data_; }

    std::size_t size()  const noexcept { return size_; }
    std::size_t bytes() const noexcept { return bytes_; }
    bool        empty() const noexcept { return size_ == 0; }

    // std::span view (connects to Day 20 — zero-copy views)
    std::span<T>       span()       noexcept { return {data_, size_}; }
    std::span<const T> span() const noexcept { return {data_, size_}; }

    // Inform the compiler that data_ is aligned (C++20).
    T* assume_aligned() noexcept
    {
        return std::assume_aligned<Align>(data_);
    }
    const T* assume_aligned() const noexcept
    {
        return std::assume_aligned<Align>(data_);
    }

    // Diagnostic
    void print_info(const char* name) const
    {
        print_alignment(name, data_, Align);
    }
};
```

### Dot Product Using Aligned Memory

```cpp
// File: dot_product.hpp
// AVX2 dot product for AlignedField<double, 32>.
// Falls back to scalar when AVX2 is not available.

#pragma once

#include "aligned_field.hpp"

#ifdef __AVX2__
#  include <immintrin.h>
#endif

// ---- Scalar dot product (reference implementation) ----

double dot_scalar(const double* __restrict__ a,
                  const double* __restrict__ b,
                  std::size_t n) noexcept
{
    double sum = 0.0;
    for (std::size_t i = 0; i < n; ++i)
        sum += a[i] * b[i];
    return sum;
}

// ---- Auto-vectorized dot product ----
// With aligned pointers and std::assume_aligned, the compiler
// can emit vmovapd + vfmadd231pd without a scalar prologue.

double dot_autovec(const AlignedField<double, 32>& a,
                   const AlignedField<double, 32>& b) noexcept
{
    const std::size_t n = a.size();
    // Tell the compiler both pointers are 32-byte aligned.
    const double* __restrict__ pa = std::assume_aligned<32>(a.data());
    const double* __restrict__ pb = std::assume_aligned<32>(b.data());

    double sum = 0.0;
    for (std::size_t i = 0; i < n; ++i)
        sum += pa[i] * pb[i];
    return sum;
}

#ifdef __AVX2__
// ---- Explicit AVX2 dot product (illustrative) ----
// Uses _mm256_load_pd (requires 32-byte alignment).

double dot_avx2(const AlignedField<double, 32>& a,
                const AlignedField<double, 32>& b) noexcept
{
    const std::size_t n = a.size();
    const double* pa = a.data();
    const double* pb = b.data();

    __m256d acc = _mm256_setzero_pd();
    std::size_t i = 0;

    // Process 4 doubles per iteration (aligned loads).
    for (; i + 4 <= n; i += 4)
    {
        __m256d va = _mm256_load_pd(pa + i);   // REQUIRES 32-byte alignment
        __m256d vb = _mm256_load_pd(pb + i);
        acc = _mm256_fmadd_pd(va, vb, acc);    // acc += va * vb
    }

    // Horizontal sum of acc (4 lanes → 1 scalar).
    __m128d lo  = _mm256_castpd256_pd128(acc);
    __m128d hi  = _mm256_extractf128_pd(acc, 1);
    __m128d sum128 = _mm_add_pd(lo, hi);
    __m128d shuf   = _mm_shuffle_pd(sum128, sum128, 1);
    double result;
    _mm_store_sd(&result, _mm_add_pd(sum128, shuf));

    // Scalar tail for remaining elements.
    for (; i < n; ++i)
        result += pa[i] * pb[i];

    return result;
}
#endif
```

### Demo Driver

```cpp
// File: aligned_field_demo.cpp
// Build: g++ -std=c++20 -O2 -mavx2 -fopt-info-vec \
//            -o aligned_demo aligned_field_demo.cpp
// Check for SIMD: objdump -d aligned_demo | grep -E 'vmovapd|vfmadd'

#include "aligned_field.hpp"
#include "dot_product.hpp"

#include <cstdio>
#include <chrono>
#include <cmath>
#include <cstdlib>

// Simple timer helper.
using Clock = std::chrono::high_resolution_clock;
using Ms    = std::chrono::duration<double, std::milli>;

double elapsed_ms(Clock::time_point t0, Clock::time_point t1)
{
    return Ms(t1 - t0).count();
}

int main()
{
    std::printf("=== Day 25: Memory Alignment Demo ===\n\n");

    // ---- 1. Alignment verification ----
    std::printf("--- Alignment Checks ---\n");

    // Stack-allocated aligned array.
    alignas(32) double stack_buf[64];
    print_alignment("alignas(32) stack_buf", stack_buf, 32);

    // Heap via new (no alignment guarantee beyond 16).
    double* heap_plain = new double[64];
    print_alignment("new double[64]",        heap_plain, 32);
    delete[] heap_plain;

    // Heap via AlignedField.
    AlignedField<double, 32> field_a(1'000'000, 1.0);
    AlignedField<double, 32> field_b(1'000'000, 2.0);
    field_a.print_info("AlignedField<double,32> a");
    field_b.print_info("AlignedField<double,32> b");

    std::printf("\n");

    // ---- 2. Correctness check ----
    // dot(ones, twos, N) should equal 2*N.
    double expected = 2.0 * static_cast<double>(field_a.size());
    double got_scalar  = dot_scalar(field_a.data(), field_b.data(), field_a.size());
    double got_autovec = dot_autovec(field_a, field_b);
#ifdef __AVX2__
    double got_avx2 = dot_avx2(field_a, field_b);
#endif

    std::printf("--- Correctness ---\n");
    std::printf("Expected:   %.1f\n", expected);
    std::printf("Scalar:     %.1f  %s\n", got_scalar,
                std::abs(got_scalar  - expected) < 1e-9 ? "OK" : "FAIL");
    std::printf("Autovec:    %.1f  %s\n", got_autovec,
                std::abs(got_autovec - expected) < 1e-9 ? "OK" : "FAIL");
#ifdef __AVX2__
    std::printf("AVX2:       %.1f  %s\n", got_avx2,
                std::abs(got_avx2    - expected) < 1e-9 ? "OK" : "FAIL");
#else
    std::printf("AVX2:       (not compiled, -mavx2 required)\n");
#endif
    std::printf("\n");

    // ---- 3. Timing ----
    const int REPEAT = 200;
    volatile double sink = 0.0;

    // Scalar timing.
    auto t0 = Clock::now();
    for (int r = 0; r < REPEAT; ++r)
        sink = dot_scalar(field_a.data(), field_b.data(), field_a.size());
    auto t1 = Clock::now();
    double ms_scalar = elapsed_ms(t0, t1) / REPEAT;

    // Autovec timing.
    auto t2 = Clock::now();
    for (int r = 0; r < REPEAT; ++r)
        sink = dot_autovec(field_a, field_b);
    auto t3 = Clock::now();
    double ms_autovec = elapsed_ms(t2, t3) / REPEAT;

#ifdef __AVX2__
    // Explicit AVX2 timing.
    auto t4 = Clock::now();
    for (int r = 0; r < REPEAT; ++r)
        sink = dot_avx2(field_a, field_b);
    auto t5 = Clock::now();
    double ms_avx2 = elapsed_ms(t4, t5) / REPEAT;
#endif

    std::printf("--- Timing (N=1,000,000, avg over %d runs) ---\n", REPEAT);
    std::printf("  Scalar:      %.3f ms\n", ms_scalar);
    std::printf("  Autovec:     %.3f ms  speedup=%.1fx\n",
                ms_autovec, ms_scalar / ms_autovec);
#ifdef __AVX2__
    std::printf("  AVX2:        %.3f ms  speedup=%.1fx\n",
                ms_avx2, ms_scalar / ms_avx2);
#endif

    std::printf("\n=== Done ===\n");
    return 0;
}
```

## Part 4 — Profiling / Benchmark

### Verifying SIMD Code Generation

After building, confirm the compiler emitted vectorized instructions:

```bash
# Build with vectorization report (GCC).
g++ -std=c++20 -O2 -mavx2 -fopt-info-vec \
    -o aligned_demo aligned_field_demo.cpp 2>&1 | grep "vectorized"

# Expected output (one line per vectorized loop):
# aligned_field_demo.cpp:67:5: optimized: loop vectorized using 32-byte vectors

# Inspect the generated assembly for AVX instructions.
objdump -d aligned_demo | grep -E 'vmovapd|vfmadd|vmovupd' | head -20
```

The key distinction to look for:

```text
vmovapd   ymm0, YMMWORD PTR [rax]   ← aligned load (what we want)
vmovupd   ymm0, YMMWORD PTR [rax]   ← unaligned load (fallback)
vfmadd231pd ymm0, ymm1, ymm2        ← fused multiply-add (good)
```

If you see `vmovapd` in the inner loop, the alignment guarantee is being honored. If you only see `vmovupd` or scalar `movsd`, the compiler did not trust the alignment.

### Aligned vs Unaligned Performance (Illustrative Numbers)

The following table shows representative measurements on a modern x86-64 system (Intel Core i7-12700, GCC 13, `-O2 -mavx2`). Your results will vary by CPU generation and system load.

| Implementation | N = 1,000,000 doubles | Throughput (GB/s) | Speedup vs scalar |
|----------------|----------------------|-------------------|-------------------|
| Scalar (no SIMD) | ~4.0 ms | ~4.0 | 1.0x |
| Autovec, unaligned pointer | ~1.2 ms | ~13.3 | ~3.3x |
| Autovec, `assume_aligned<32>` | ~1.0 ms | ~16.0 | ~4.0x |
| Explicit AVX2 (`_mm256_load_pd`) | ~1.0 ms | ~16.0 | ~4.0x |

**Key observations:**

1. The autovec path with `std::assume_aligned<32>` matches explicit AVX2 — the compiler is doing the same work.
2. Unaligned autovec is already fast on Haswell+, but still slightly slower due to the generated runtime alignment check.
3. The theoretical peak for AVX2 (4 doubles/cycle at 4 GHz) is ~16 GB/s for a bandwidth-bound dot product — we are hitting it.

### Effect of Alignment on the Scalar Prologue

Without alignment information, GCC generates a vectorized loop with a prologue of up to 3 scalar iterations to reach the next aligned boundary. This overhead is negligible for large N but can be significant for small N (e.g., per-face operations in tight mesh loops with thousands of short arrays).

```text
Without assume_aligned (N=16, repeated 10M times):
  Scalar prologue:  up to 3 iterations every call
  Per-call overhead: measurable at N < 32

With assume_aligned<32>:
  Prologue eliminated entirely
  First instruction is vmovapd at offset 0
```

### Memory Bandwidth Analysis

For a dot product on `Field<double>` with N elements:

| N (elements) | Memory read (2 arrays) | Expected time at 40 GB/s bandwidth |
|-------------|------------------------|-------------------------------------|
| 10,000 | 160 KB | fits in L2 cache, ~0.002 ms |
| 100,000 | 1.6 MB | fits in L3 cache, ~0.04 ms |
| 1,000,000 | 16 MB | main memory, ~0.4 ms |
| 10,000,000 | 160 MB | main memory, ~4 ms |

For real CFD meshes (1M–100M cells), the dot product is **memory-bound**, not compute-bound. Alignment reduces overhead but bandwidth is the true ceiling.

## Summary

**Key takeaways for Day 25:**

1. AVX2 SIMD requires 32-byte aligned data. `new double[]` only guarantees 16-byte alignment.
2. `alignas(32)` enforces alignment for stack and static storage. `std::aligned_alloc` handles heap allocation.
3. `std::assume_aligned<32>(ptr)` informs the compiler of the alignment guarantee, enabling it to emit `vmovapd` and skip the scalar prologue.
4. The explicit AVX2 path (`_mm256_load_pd`) gives the same performance as a well-informed autovectorized loop — both emit the same instructions.
5. OpenFOAM does not use `alignas` in its default `Field<T>` path; it relies on `vmovupd` and OS-provided alignment. A production solver can do better.
