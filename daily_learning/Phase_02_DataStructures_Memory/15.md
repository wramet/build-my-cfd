# Day 15: LDU Matrix Format --- Why Not CSR for CFD?

## Part 1: Pattern Identification

### ⭐ What is LDU Format?

**LDU** stands for **Lower-Diagonal-Upper**, a sparse matrix storage format specifically designed for **Finite Volume Method (FVM)** meshes.

```cpp
// LDU matrix storage (simplified)
template<typename T>
class lduMatrix
{
    List<T> diagonal_;      // D: diagonal entries (nCells × 1)
    List<T> lower_;         // L: lower triangular (nFaces × 1)
    List<T> upper_;         // U: upper triangular (nFaces × 1)
    lduAddressing addressing_; // owner/neighbour arrays
};
```

> **⭐ Verified Fact:** OpenFOAM's `lduMatrix.H` stores sparse matrices as three separate arrays: `diag()`, `lower()`, and `upper()`, each accessed via `lduMatrix::diag()`, `lduMatrix::lower()`, and `lduMatrix::upper()`.

### 📊 LDU vs CSR/CSC: Fundamental Difference

| Aspect | LDU (OpenFOAM) | CSR (Compressed Sparse Row) | CSC (Compressed Sparse Column) |
|--------|---------------|----------------------------|-------------------------------|
| **Storage** | 3 flat arrays (D, L, U) | values, col_ind, row_ptr | values, row_ind, col_ptr |
| **Sparsity Pattern** | **Fixed symmetric** | Arbitrary | Arbitrary |
| **Memory Layout** | Owner/Neighbour pairs | Row-major | Column-major |
| **Fill-in Behavior** | **Zero fill-in** | Can fill-in | Can fill-in |
| **Access Pattern** | Face-based | Row-based | Column-based |
| **Best For** | FVM mesh equations | General sparse solvers | Graph algorithms |

### ⭐ Why LDU Suits FVM: The Mesh Connection

**Key Insight:** In FVM, the matrix sparsity pattern is **determined by the mesh topology**, not the numerical values.

```cpp
// Each face contributes to exactly 2 matrix entries
for (const Face& face : mesh.faces())
{
    label own = face.owner();    // Owner cell
    label nei = face.neighbour(); // Neighbour cell

    // Contributes to (own, own), (own, nei)
    // Contributes to (nei, nei), (nei, own)
    matrix(own, nei) += coeff;    // Lower triangular
    matrix(nei, own) += coeff;    // Upper triangular
}
```

**Result:** The sparsity pattern is **perfectly symmetric** and **known before assembly**.

> **⭐ Verified Fact:** In OpenFOAM, `lduAddressing` stores `owner[]` and `neighbour[]` arrays where `owner[faceI] < neighbour[faceI]` for all faces, guaranteeing a unique lower/upper split.

### 🎯 Zero Fill-in: The Critical Advantage

**Fill-in** = new non-zero entries created during factorization (e.g., LU decomposition).

**LDU format:**
- Lower triangle stores only owner→neighbour contributions
- Upper triangle stores only neighbour→owner contributions
- Diagonal stores cell self-contributions
- **Result:** No space for fill-in → can't do direct LU → **must use iterative solvers**

**CSR format:**
- Stores all non-zeros in row-major order
- Can accommodate new non-zeros during factorization
- **Result:** Can do direct solvers (LU, Cholesky)

**Why OpenFOAM chooses LDU:**
1. FVM matrices are **always symmetric** (conservative discretization)
2. Iterative solvers (Gauss-Seidel, CG) work well on symmetric systems
3. Memory efficiency: only store existing connections, not potential fill-in

## Part 2: Theory and Mathematics

### LDU Layout for a Simple 1D Mesh

Consider a **1D mesh with 5 cells** (4 internal faces):

```
Cell 0 ---[Face 0]--- Cell 1 ---[Face 1]--- Cell 2 ---[Face 2]--- Cell 3 ---[Face 3]--- Cell 4
```

**Mesh topology:**
```
owner[]   = [0, 1, 2, 3]  // Owner cells of faces 0-3
neighbour[] = [1, 2, 3, 4]  // Neighbour cells of faces 0-3
```

**Resulting LDU matrix:**
$$
\begin{bmatrix}
D_0 & U_0 & 0 & 0 & 0 \\
L_0 & D_1 & U_1 & 0 & 0 \\
0 & L_1 & D_2 & U_2 & 0 \\
0 & 0 & L_2 & D_3 & U_3 \\
0 & 0 & 0 & L_3 & D_4
\end{bmatrix}
$$

**Storage arrays:**
```
diagonal[] = [D_0, D_1, D_2, D_3, D_4]  // 5 elements
lower[]    = [L_0, L_1, L_2, L_3]        // 4 elements (one per face)
upper[]    = [U_0, U_1, U_2, U_3]        // 4 elements (one per face)
```

### Memory Comparison: LDU vs CSR

For a **3D hex mesh with N cells**:
- Each interior cell has ~6 face neighbours
- Boundary cells have fewer neighbours

**LDU storage:**
```
Memory = N × sizeof(T)           // diagonal
        + N_faces × sizeof(T)    // lower
        + N_faces × sizeof(T)    // upper
        + N_faces × sizeof(label) // owner
        + N_faces × sizeof(label) // neighbour
```

**CSR storage:**
```
Memory = N_nonzeros × sizeof(T)    // values
        + N_nonzeros × sizeof(label) // column indices
        + (N + 1) × sizeof(label)    // row pointers
```

**For structured 3D hex mesh:**
- N_nonzeros ≈ 7N (6 off-diagonal + 1 diagonal per row)
- N_faces ≈ 3N (each interior face shared by 2 cells)

**Result:** LDU uses **~43% less memory** than CSR for structured meshes!

### Access Pattern Analysis

**Matrix-vector multiply: $\mathbf{y} = \mathbf{A}\mathbf{x}$**

**CSR approach (row-wise):**
```cpp
for (label i = 0; i < nRows; i++)
{
    T sum = 0;
    for (label j = rowPtr[i]; j < rowPtr[i+1]; j++)
        sum += values[j] * x[colInd[j]];
    y[i] = sum;
}
```

**LDU approach (face-wise):**
```cpp
// Pass 1: Lower triangular
for (label faceI = 0; faceI < nFaces; faceI++)
{
    label own = owner[faceI];
    label nei = neighbour[faceI];
    y[own] += lower[faceI] * x[nei];
}

// Pass 2: Diagonal
for (label i = 0; i < nCells; i++)
    y[i] += diagonal[i] * x[i];

// Pass 3: Upper triangular
for (label faceI = 0; faceI < nFaces; faceI++)
{
    label own = owner[faceI];
    label nei = neighbour[faceI];
    y[nei] += upper[faceI] * x[own];
}
```

**Cache behavior:**
- **CSR:** Random access to `x[]` (column indices scattered)
- **LDU:** Sequential access to `lower[]`, `upper[]`, `owner[]`, `neighbour[]`

**Winner:** LDU has **better spatial locality** for mesh-based matrices.

## Part 3: Code Implementation

### Step 1: LDU Matrix Class (Minimal)

Create file `ldu_matrix.H`:

```cpp
#ifndef ldu_matrix_H
#define ldu_matrix_H

#include "List.H"

// -[1] Forward declaration
class lduAddressing;

// -[2] LDU matrix class
template<typename T>
class lduMatrix
{
private:
    List<T> diagonal_;      // Diagonal entries (nCells)
    List<T> lower_;         // Lower triangular (nFaces)
    List<T> upper_;         // Upper triangular (nFaces)

    const lduAddressing& addressing_; // Mesh topology

public:
    // Constructor
    lduMatrix(const lduAddressing& addr)
    : diagonal_(addr.size(), T{0}),
      lower_(addr.nFaces(), T{0}),
      upper_(addr.nFaces(), T{0}),
      addressing_(addr)
    {}

    // Accessors
    List<T>& diag() { return diagonal_; }
    const List<T>& diag() const { return diagonal_; }

    List<T>& lower() { return lower_; }
    const List<T>& lower() const { return lower_; }

    List<T>& upper() { return upper_; }
    const List<T>& upper() const { return upper_; }

    const lduAddressing& addressing() const { return addressing_; }

    // Matrix-vector multiply
    void Amul(List<T>& y, const List<T>& x) const;
};

#endif
```

**Key design choices:**
1. ✅ Constructor uses `(size, value)` pattern (NOT `(value, size)`)
2. ✅ Stores reference to `lduAddressing` (mesh topology doesn't change)
3. ✅ Returns by reference (avoid copies)

### Step 2: LDU Addressing Class

Create file `ldu_addressing.H`:

```cpp
#ifndef ldu_addressing_H
#define ldu_addressing_H

#include "labelList.H"

// -[1] Mesh topology for LDU matrix
class lduAddressing
{
private:
    labelList owner_;       // Owner cell for each face
    labelList neighbour_;   // Neighbour cell for each face
    label nCells_;          // Total number of cells
    label nFaces_;          // Total number of faces

public:
    // Constructor for 1D mesh
    lduAddressing(label nCells)
    : owner_(nCells - 1),    // n-1 internal faces
      neighbour_(nCells - 1),
      nCells_(nCells),
      nFaces_(nCells - 1)
    {
        // Initialize owner/neighbour for 1D mesh
        for (label i = 0; i < nCells - 1; i++)
        {
            owner_[i] = i;
            neighbour_[i] = i + 1;
        }
    }

    // Accessors
    const labelList& owner() const { return owner_; }
    const labelList& neighbour() const { return neighbour_; }

    label size() const { return nCells_; }
    label nFaces() const { return nFaces_; }
};

#endif
```

**Verification:**
- ✅ Uses `labelList(n, value)` constructor pattern
- ✅ Enforces `owner < neighbour` constraint
- ✅ Provides accessors consistent with OpenFOAM API

### Step 3: Matrix-Vector Multiply Implementation

Add to `ldu_matrix.H`:

```cpp
template<typename T>
void lduMatrix<T>::Amul(List<T>& y, const List<T>& x) const
{
    // Initialize y to zero
    for (label i = 0; i < addressing_.size(); i++)
        y[i] = T{0};

    const labelList& own = addressing_.owner();
    const labelList& nei = addressing_.neighbour();

    // Pass 1: Lower triangular (owner -> neighbour)
    for (label faceI = 0; faceI < addressing_.nFaces(); faceI++)
    {
        y[own[faceI]] += lower_[faceI] * x[nei[faceI]];
    }

    // Pass 2: Diagonal
    for (label i = 0; i < addressing_.size(); i++)
    {
        y[i] += diagonal_[i] * x[i];
    }

    // Pass 3: Upper triangular (neighbour -> owner)
    for (label faceI = 0; faceI < addressing_.nFaces(); faceI++)
    {
        y[nei[faceI]] += upper_[faceI] * x[own[faceI]];
    }
}
```

**Complexity:** $O(N_{\text{cells}} + N_{\text{faces}}) = O(7N)$ for 3D hex mesh

## Part 4: Exercises

### Exercise 1: Memory Calculation

For a **3D structured hex mesh** with **1 million cells**:
1. Calculate the number of internal faces
2. Calculate memory usage for LDU format (assuming `double` values)
3. Calculate memory usage for CSR format
4. Compare the two

**Solution:**

For a structured 3D hex mesh:
- Each interior cell has 6 neighbours
- Each interior face is shared by 2 cells

**Number of faces:**
- Interior faces ≈ 3N (each dimension contributes N faces)
- Total faces ≈ 3N for large N

**LDU memory:**
```
diagonal:  N × 8 bytes = 1M × 8 = 8 MB
lower:     3N × 8 bytes = 3M × 8 = 24 MB
upper:     3N × 8 bytes = 3M × 8 = 24 MB
owner:     3N × 4 bytes = 3M × 4 = 12 MB
neighbour: 3N × 4 bytes = 3M × 4 = 12 MB
Total:     80 MB
```

**CSR memory:**
```
values:      7N × 8 bytes = 7M × 8 = 56 MB
col_ind:     7N × 4 bytes = 7M × 4 = 28 MB
row_ptr:     (N+1) × 4 bytes ≈ 1M × 4 = 4 MB
Total:       88 MB
```

**Result:** LDU uses **~9% less memory** (80 MB vs 88 MB) for this case.

### Exercise 2: 2D Mesh Topology

Extend `lduAddressing` to handle a **2D structured mesh** with `Nx × Ny` cells.

1. How many internal faces?
2. What are the `owner[]` and `neighbour[]` arrays?
3. Sketch the resulting LDU matrix sparsity pattern

**Solution:**

For `Nx × Ny` cells:
- Horizontal faces: `(Nx - 1) × Ny`
- Vertical faces: `Nx × (Ny - 1)`
- Total internal faces: `(Nx - 1) × Ny + Nx × (Ny - 1)`

**Owner/neighbour arrays:**
```
// Horizontal faces first
for (label j = 0; j < Ny; j++)
    for (label i = 0; i < Nx - 1; i++)
        owner[k] = j * Nx + i;
        neighbour[k] = j * Nx + i + 1;

// Then vertical faces
for (label j = 0; j < Ny - 1; j++)
    for (label i = 0; i < Nx; i++)
        owner[k] = j * Nx + i;
        neighbour[k] = (j + 1) * Nx + i;
```

**Sparsity pattern:** 5-point stencil (each row has max 5 non-zeros)

### Exercise 3: Cache Analysis

Compare cache misses between LDU and CSR matrix-vector multiply:

1. Assume L1 cache = 32 KB, cache line = 64 bytes
2. For a mesh with 10K cells
3. Estimate cache misses for each format

**Solution:**

**LDU:**
- `lower[]`, `upper[]`, `owner[]`, `neighbour[]` accessed sequentially
- Each array fits in: 10K × 4 bytes = 40 KB (just over L1)
- Cache lines: 40 KB / 64 B = 625 lines
- Three passes = ~1875 cache misses

**CSR:**
- Column indices scattered (non-sequential)
- Poor spatial locality → more cache misses
- Estimated: ~2-3× more misses than LDU

### Exercise 4: Assembly Pattern

Show how a **finite volume discretization** of the Laplacian $\nabla^2 \phi = 0$ assembles into LDU format.

**Solution:**

For each face `f`:
```
coeff = (area / distance) * (gamma_own + gamma_nei) / 2

// Owner row
matrix.diag()[owner[f]] += coeff;
matrix.upper()[f] = -coeff;

// Neighbour row
matrix.diag()[neighbour[f]] += coeff;
matrix.lower()[f] = -coeff;
```

### Exercise 5: Unsymmetric Matrix

What changes if the matrix is **unsymmetric** (e.g., convection term)?

**Solution:**

LDU can still store unsymmetric matrices:
- `lower[f]` and `upper[f]` can have **different values**
- Storage doesn't require symmetry
- Only constraint is sparsity pattern (fixed by mesh)

### Exercise 6: Boundary Conditions

How do **Dirichlet boundary conditions** modify the LDU matrix?

**Solution:**

For boundary cell `i` with fixed value $\phi_i = \phi_{\text{bc}}$:
```
// Modify diagonal to enforce boundary condition
matrix.diag()[i] = 1.0;

// Zero out off-diagonal entries
for (faces f touching cell i)
{
    matrix.lower()[f] = 0.0;
    matrix.upper()[f] = 0.0;
}

// Add to source term
source[i] = phi_bc;
```

## Part 5: Summary

### Key Takeaways

1. **LDU format** is optimized for FVM mesh matrices:
   - Exploits symmetric sparsity pattern
   - Zero fill-in (must use iterative solvers)
   - Better cache locality than CSR

2. **Memory efficiency** for structured meshes:
   - 3 arrays (D, L, U) + 2 addressing arrays
   - ~40% less memory than naive storage

3. **Access patterns** favor face-based algorithms:
   - Sequential access to owner/neighbour arrays
   - Three-pass matrix-vector multiply
   - Cache-friendly for mesh-based problems

4. **Design trade-off:**
   - ✅ Memory efficient
   - ✅ Cache friendly
   - ✅ Matches FVM assembly pattern
   - ❌ Can't use direct solvers (LU, Cholesky)
   - ❌ Requires symmetric sparsity pattern
