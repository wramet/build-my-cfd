# Day 22: Modern Hashing — `std::unordered_map` for Boundary Lookups

## Part 1: Pattern Identification

### Boundary Patch Lookup Problem

In CFD solvers, boundary conditions are applied to mesh patches (surfaces). Each patch has:
- A unique name (e.g., "inlet", "outlet", "walls", "symmetry")
- A boundary condition type (Dirichlet, Neumann, Robin)
- Associated values (fixed temperature, heat flux, etc.)

During solver iterations, we frequently:
1. Look up BC by patch name
2. Apply BC to specific faces
3. Update BC values dynamically

**The naive approach:**
```cpp
std::vector<std::pair<std::string, BoundaryCondition>> patches;
// O(n) lookup requires linear search
```

**The efficient approach:**
```cpp
std::unordered_map<std::string, BoundaryCondition> patchMap;
// O(1) average lookup
```

## Part 2: Theory — Hash Maps

### Hash Table Structure

A hash table uses a hash function to map keys to bucket indices:

$$
\text{bucket\_index} = \text{hash}(\text{key}) \mod \text{num\_buckets}
$$

### Collision Resolution

Two main strategies:
1. **Separate chaining**: Each bucket holds a linked list
2. **Open addressing**: Probe for empty slots

C++'s `std::unordered_map` uses separate chaining.

### Load Factor

$$
\alpha = \frac{n}{m}
$$

Where $n$ = number of elements, $m$ = number of buckets.

- **Optimal load factor**: ~0.75
- **Rehashing**: Automatically occurs when $\alpha > \text{max\_load\_factor}$

### Complexity

| Operation | Average | Worst |
|-----------|---------|-------|
| Insert | $O(1)$ | $O(n)$ |
| Find | $O(1)$ | $O(n)$ |
| Delete | $O(1)$ | $O(n)$ |

### FNV-1a Hash Function for Strings

When you use a `std::string` as a key, the standard library calls `std::hash<std::string>`. The exact implementation is unspecified, but many standard library implementations use a variant of **FNV-1a** (Fowler-Noll-Vo alternate), which is fast and exhibits good avalanche behavior for short strings like patch names.

The FNV-1a formula for a byte sequence $b_0, b_1, \ldots, b_{n-1}$ is:

$$
H_0 = \text{offset\_basis}
$$

$$
H_{i+1} = (H_i \oplus b_i) \times \text{FNV\_prime}
$$

For 64-bit FNV-1a:
- $\text{offset\_basis} = 14695981039346656037$
- $\text{FNV\_prime} = 1099511628211$

The XOR-then-multiply order (versus multiply-then-XOR in FNV-1) makes every input byte immediately affect all high bits through the multiplication, producing better dispersion. For CFD patch names such as `"inlet"`, `"wall_top"`, `"symmetry_z"`, the strings are typically short (4-20 characters) and the FNV-1a avalanche effect ensures they distribute evenly across buckets even when names share long prefixes like `"wall_left"` and `"wall_right"`.

A minimal 64-bit FNV-1a implementation for reference:

```cpp
// ⚠️ For illustration only — std::hash<std::string> is not guaranteed to use this
constexpr uint64_t FNV_OFFSET_BASIS = 14695981039346656037ULL;
constexpr uint64_t FNV_PRIME        = 1099511628211ULL;

uint64_t fnv1a_64(const char* data, size_t len) {
    uint64_t hash = FNV_OFFSET_BASIS;
    for (size_t i = 0; i < len; ++i) {
        hash ^= static_cast<uint64_t>(static_cast<unsigned char>(data[i]));
        hash *= FNV_PRIME;
    }
    return hash;
}
```

**Why string hashing matters for CFD:** In a simulation loop running millions of iterations, the hash computation itself must be fast. Short patch names (under 32 bytes) are often hashed faster than the cache-miss penalty of a failed tree traversal in `std::map`. This is the primary reason `std::unordered_map` wins the benchmark you will run in Part 4.

### Load Factor and Expected Probe Length

With **separate chaining** (the strategy used by `std::unordered_map`), the expected number of elements examined during a successful lookup equals:

$$
E[\text{probes}] = 1 + \frac{\alpha}{2}
$$

During an **unsuccessful** lookup (key not found), the expected probes are:

$$
E[\text{probes\_miss}] = \alpha
$$

The practical impact across different load factor settings:

| Load factor $\alpha$ | Successful lookup (avg probes) | Failed lookup (avg probes) | Memory pressure |
|----------------------|--------------------------------|----------------------------|-----------------|
| 0.25 | 1.13 | 0.25 | Low (75% empty buckets) |
| 0.50 | 1.25 | 0.50 | Moderate |
| 0.75 | 1.38 | 0.75 | Default `std::unordered_map` |
| 1.00 | 1.50 | 1.00 | One chain element per bucket on average |
| 2.00 | 2.00 | 2.00 | Noticeably slower |
| 5.00 | 3.50 | 5.00 | Significantly degraded |

The default `max_load_factor()` for `std::unordered_map` is 1.0. Rehashing doubles the bucket count when $\alpha$ would exceed this. For CFD registries where patch names are registered once and then queried millions of times, setting `max_load_factor(0.5)` before population reduces lookup time by keeping chains shorter, at the cost of roughly double the bucket memory — usually a worthwhile trade-off.

### Robin Hood Hashing

An alternative open-addressing strategy called **Robin Hood hashing** reduces variance in probe sequence length by stealing slots from "rich" elements (those that landed close to their home bucket) and giving them to "poor" elements (those displaced far from home). The key invariant is:

> A displaced element always has a probe distance at least as large as the element it displaces.

This is the collision resolution strategy used in some implementations of `std::unordered_map` (notably GCC's `libstdc++` in certain build configurations) and is the default in `absl::flat_hash_map` and `tsl::robin_map`. Robin Hood hashing produces lower worst-case lookup times compared to naive linear probing, making it attractive for real-time CFD applications. The full implementation is beyond this session's scope, but understanding the concept explains why third-party hash maps from the Abseil or TSL libraries often outperform the standard one in CPU-bound workloads.

## Part 3: C++ Mechanics

### Basic `std::unordered_map` Usage

```cpp
#include <unordered_map>
#include <string>
#include <iostream>

void basicUsage() {
    // Create hash map
    std::unordered_map<std::string, double> patchMap;

    // Insert elements
    patchMap["inlet"] = 10.0;
    patchMap["outlet"] = 0.0;
    patchMap.insert({"wall", 273.15});

    // Lookup
    if (patchMap.find("inlet") != patchMap.end()) {
        std::cout << "Inlet temperature: " << patchMap["inlet"] << "\n";
    }

    // Iterate
    for (const auto& [name, value] : patchMap) {
        std::cout << name << ": " << value << "\n";
    }
}
```

### Custom Hash Function

struct CaseInsensitiveHash {
    size_t operator()(const std::string& key) const {
        size_t hash = 0;
        for (char c : key) {
            hash = hash * 31 + std::tolower(c);
        }
        return hash;
    }
};

struct CaseInsensitiveEqual {
    bool operator()(const std::string& a, const std::string& b) const {
        return std::equal(a.begin(), a.end(), b.begin(), b.end(),

                             return std::tolower(c1) == std::tolower(c2);
                         });
    }

### Pre-allocating Buckets with `reserve()`

Every insertion into an `std::unordered_map` may trigger a **rehash** if the load factor would be exceeded. A rehash allocates a new bucket array, re-hashes every existing element, and frees the old array — an $O(n)$ operation that also invalidates all iterators and references.

In CFD registry code where the number of patches is known at startup (it is read from the mesh file), you should call `reserve()` before inserting any elements:

```cpp
#include <unordered_map>
#include <string>

void demonstrateReserve() {
    const size_t expectedPatches = 128;  // read from mesh metadata

    std::unordered_map<std::string, double> patchMap;

    // Reserve enough buckets so that no rehash occurs during population
    // reserve(n) ensures at least n elements can be inserted
    // before a rehash, given the current max_load_factor
    patchMap.reserve(expectedPatches);

    std::cout << "Bucket count after reserve: "
              << patchMap.bucket_count() << "\n";

    // All inserts below are guaranteed rehash-free
    patchMap["inlet"]     = 300.0;
    patchMap["outlet"]    = 290.0;
    patchMap["wall_top"]  = 273.15;
    patchMap["symmetry"]  = 0.0;
}
```

The relationship between `reserve(n)` and `bucket_count()` is:

$$
\text{bucket\_count()} \geq \lceil n \; / \; \text{max\_load\_factor()} \rceil
$$

So `reserve(128)` with `max_load_factor() = 1.0` guarantees at least 128 buckets (rounded up to the next power of two in most implementations, giving 128 or 256 depending on the library).

### Inspecting the Internal State

Three methods let you observe the hash map's internal state at runtime, which is useful for performance diagnostics in long CFD runs:

```cpp
void inspectHashMap() {
    std::unordered_map<std::string, double> patchMap;
    patchMap.reserve(64);

    patchMap["inlet"]  = 300.0;
    patchMap["outlet"] = 290.0;
    patchMap["wall"]   = 273.15;

    // bucket_count(): total number of buckets allocated
    std::cout << "Buckets:          " << patchMap.bucket_count()  << "\n";

    // load_factor(): current n/m ratio
    std::cout << "Load factor:      " << patchMap.load_factor()   << "\n";

    // max_load_factor(): rehash threshold (default 1.0)
    std::cout << "Max load factor:  " << patchMap.max_load_factor() << "\n";

    // Diagnose hot buckets — useful if you suspect poor hash distribution
    for (size_t b = 0; b < patchMap.bucket_count(); ++b) {
        if (patchMap.bucket_size(b) > 2) {
            std::cout << "Hot bucket " << b
                      << " has " << patchMap.bucket_size(b)
                      << " elements — possible hash collision cluster\n";
        }
    }
}
```

Monitoring `load_factor()` at the end of mesh initialization is a cheap sanity check that the registry is not accidentally being filled beyond its reserved capacity.

### Heterogeneous Lookup with `std::string_view`

A subtle performance problem occurs when querying a `std::unordered_map<std::string, V>` with a string literal or a `const char*`. The standard library constructs a temporary `std::string` — allocating heap memory — just to compute the hash and compare keys, even though the underlying data is already in a contiguous buffer.

```cpp
// PROBLEM: temporary std::string is heap-allocated on every call
// even though "inlet" is a compile-time literal
auto it = patchMap.find("inlet");   // constructs std::string("inlet")
```

In C++17 and later, you can use `std::string_view` to avoid this allocation by providing a **transparent hasher and equality comparator**. The trick is to make both the hasher and the equal functor templated so they accept any string-like type:

```cpp
#include <unordered_map>
#include <string>
#include <string_view>
#include <functional>

// Transparent (heterogeneous) string hash
struct StringViewHash {
    using is_transparent = void;   // opt-in marker required by the standard

    size_t operator()(std::string_view sv) const {
        return std::hash<std::string_view>{}(sv);
    }
    size_t operator()(const std::string& s) const {
        return std::hash<std::string_view>{}(s);
    }
    size_t operator()(const char* p) const {
        return std::hash<std::string_view>{}(p);
    }
};

// Transparent equality
struct StringViewEqual {
    using is_transparent = void;   // opt-in marker required by the standard

    bool operator()(std::string_view a, std::string_view b) const {
        return a == b;
    }
    // The comparator accepts any combination of string/string_view/char*
    // because std::string and const char* both convert implicitly to string_view
};

// Declare the map with transparent hasher and equal
using HeterogeneousPatchMap = std::unordered_map<
    std::string,          // key stored as std::string (owns memory)
    double,               // mapped value
    StringViewHash,       // transparent hasher
    StringViewEqual       // transparent equality
>;

void demonstrateHeterogeneousLookup() {
    HeterogeneousPatchMap patchMap;
    patchMap["inlet"]  = 300.0;
    patchMap["outlet"] = 290.0;

    // These lookups require NO heap allocation for the key
    std::string_view sv = "inlet";
    auto it1 = patchMap.find(sv);           // zero-copy: string_view lookup
    auto it2 = patchMap.find("outlet");     // zero-copy: char* -> string_view

    if (it1 != patchMap.end()) {
        std::cout << "Inlet (sv lookup): " << it1->second << "\n";
    }
    if (it2 != patchMap.end()) {
        std::cout << "Outlet (literal lookup): " << it2->second << "\n";
    }
}
```

**Why this matters in CFD:** During a transient solver loop, patch name lookups may run inside a time-stepping loop that executes $10^5$ to $10^6$ iterations. If each iteration allocates and frees a temporary `std::string` per boundary condition applied, the cumulative allocator pressure is measurable. The transparent comparator pattern eliminates those allocations entirely, keeping boundary lookup in the hot path truly $O(1)$ with no hidden allocator cost.

**Note:** The `is_transparent` typedef is the opt-in tag that tells `std::unordered_map::find` to accept heterogeneous arguments. Without it, `find` only accepts `const Key&`, so the overload accepting `string_view` would not be instantiated.

### PatchRegistry Class

The class below extends the minimal version from Part 1 with the three additions motivated by the theory sections: bucket pre-allocation, an `allPatches()` view, and heterogeneous `string_view` lookup.

```cpp
// Transparent hasher and equality (same as defined in Part 3)
struct StringViewHash {
    using is_transparent = void;
    size_t operator()(std::string_view sv) const {
        return std::hash<std::string_view>{}(sv);
    }
    size_t operator()(const std::string& s) const {
        return std::hash<std::string_view>{}(s);
    }
    size_t operator()(const char* p) const {
        return std::hash<std::string_view>{}(p);
    }
};

struct StringViewEqual {
    using is_transparent = void;
    bool operator()(std::string_view a, std::string_view b) const {
        return a == b;
    }
};

class PatchRegistry {
private:
    std::unordered_map<std::string,
                       BoundaryCondition,
                       StringViewHash,
                       StringViewEqual> patches_;

public:
    // --- Capacity management ---

    // Pre-allocate buckets so that inserting n patches causes no rehash.
    // Call this immediately after reading the mesh patch count.
    void reserveCapacity(size_t n) {
        patches_.reserve(n);
    }

    // --- Mutation ---

    void addPatch(const std::string& name, const BoundaryCondition& bc) {
        patches_[name] = bc;
    }

    void updateBC(const std::string& name, double newValue) {
        patches_[name].value = newValue;
    }

    // --- Query ---

    // Heterogeneous lookup: accepts std::string, std::string_view, const char*
    // with zero heap allocation for string_view and char* arguments.
    const BoundaryCondition& getBC(std::string_view name) const {
        auto it = patches_.find(name);
        if (it == patches_.end()) {
            throw std::runtime_error("Patch not found: " + std::string(name));
        }
        return it->second;
    }

    bool hasPatch(std::string_view name) const {
        return patches_.find(name) != patches_.end();
    }

    // --- View of all patches ---

    // Returns a const reference to the underlying map.
    // Callers can range-over it to enumerate every registered patch.
    const std::unordered_map<std::string,
                             BoundaryCondition,
                             StringViewHash,
                             StringViewEqual>& allPatches() const {
        return patches_;
    }

    // --- Diagnostics ---

    size_t size()        const { return patches_.size(); }
    size_t bucketCount() const { return patches_.bucket_count(); }
    float  loadFactor()  const { return patches_.load_factor(); }
};
```

### FieldRegistry Template Class

In a real CFD solver, you need one registry per physical field: temperature, pressure, velocity, turbulent kinetic energy, etc. Rather than duplicating `PatchRegistry` for each scalar or vector type, define a single template:

```cpp
#include <array>
#include <stdexcept>
#include <string>
#include <string_view>
#include <unordered_map>

// A simple 3-component vector for velocity / flux fields
struct Vector3d {
    double x = 0.0, y = 0.0, z = 0.0;
};

// Generic field registry: maps patch name -> field value of type T
// T can be double (scalar), Vector3d (vector), or any copyable type.
template<typename T>
class FieldRegistry {
private:
    std::unordered_map<std::string,
                       T,
                       StringViewHash,
                       StringViewEqual> data_;

public:
    void reserveCapacity(size_t n) { data_.reserve(n); }

    void set(const std::string& patchName, const T& value) {
        data_[patchName] = value;
    }

    // Zero-allocation lookup via string_view
    const T& get(std::string_view patchName) const {
        auto it = data_.find(patchName);
        if (it == data_.end()) {
            throw std::runtime_error("Field not found on patch: "
                                     + std::string(patchName));
        }
        return it->second;
    }

    bool has(std::string_view patchName) const {
        return data_.find(patchName) != data_.end();
    }

    // Iterate over all patch-value pairs
    auto begin() const { return data_.begin(); }
    auto end()   const { return data_.end(); }
    size_t size() const { return data_.size(); }
};
```

**Multi-field registry pattern used in CFD solvers:**

```cpp
void runSolverStep(const std::vector<std::string>& patchNames) {
    // One registry per physical field — all share the same patch name keys
    FieldRegistry<double>  tempRegistry;   // temperature [K]
    FieldRegistry<double>  pressRegistry;  // pressure [Pa]
    FieldRegistry<Vector3d> velRegistry;   // velocity [m/s]

    // Pre-allocate all registries before population
    const size_t n = patchNames.size();
    tempRegistry.reserveCapacity(n);
    pressRegistry.reserveCapacity(n);
    velRegistry.reserveCapacity(n);

    // Populate (typically from mesh boundary data)
    for (const auto& name : patchNames) {
        tempRegistry.set(name,  300.0);
        pressRegistry.set(name, 101325.0);
        velRegistry.set(name,   {1.0, 0.0, 0.0});
    }

    // Per-iteration lookups — all zero-allocation via string_view
    std::string_view inletSV = "inlet";
    double     T_inlet = tempRegistry.get(inletSV);
    double     P_inlet = pressRegistry.get(inletSV);
    Vector3d   U_inlet = velRegistry.get(inletSV);

    std::cout << "Inlet T=" << T_inlet
              << "  P=" << P_inlet
              << "  Ux=" << U_inlet.x << "\n";
}
```

This pattern is representative of how production solvers like OpenFOAM organize boundary data: each field (`volScalarField`, `volVectorField`) maintains its own boundary condition list, and patch lookups by name occur in the boundary condition evaluation loop inside each time step.

### Benchmark: unordered_map vs map

```cpp
void runBenchmark() {
    const size_t nPatches = 10000;
    const size_t nLookups = 1000000;

    // Create test data
    std::vector<std::string> patchNames;
    for (size_t i = 0; i < nPatches; ++i) {
        patchNames.push_back("patch_" + std::to_string(i));
    }

    // Benchmark unordered_map
    {
        std::unordered_map<std::string, BoundaryCondition> umap;
        for (const auto& name : patchNames) {
            umap[name] = BoundaryCondition(BCType::Dirichlet, 300.0);
        }

        auto start = std::chrono::high_resolution_clock::now();

        std::random_device rd;
        std::mt19937 gen(rd());
        std::uniform_int_distribution<> dist(0, nPatches - 1);

        volatile double sum = 0;
        for (size_t i = 0; i < nLookups; ++i) {
            const auto& bc = umap[patchNames[dist(gen)]];
            sum += bc.value;
        }

        auto end = std::chrono::high_resolution_clock::now();
        auto umapTime = std::chrono::duration_cast<std::chrono::milliseconds>(end - start);

        std::cout << "unordered_map: " << umapTime.count() << " ms\n";
    }

    // Benchmark std::map
    {
        std::map<std::string, BoundaryCondition> omap;
        for (const auto& name : patchNames) {
            omap[name] = BoundaryCondition(BCType::Dirichlet, 300.0);
        }

        auto start = std::chrono::high_resolution_clock::now();

        std::random_device rd;
        std::mt19937 gen(rd());
        std::uniform_int_distribution<> dist(0, nPatches - 1);

        volatile double sum = 0;
        for (size_t i = 0; i < nLookups; ++i) {
            const auto& bc = omap[patchNames[dist(gen)]];
            sum += bc.value;
        }

        auto end = std::chrono::high_resolution_clock::now();
        auto omapTime = std::chrono::duration_cast<std::chrono::milliseconds>(end - start);

        std::cout << "std::map:        " << omapTime.count() << " ms\n";
        std::cout << "Speedup:         " << (double)omapTime.count() / umapTime.count() << "x\n";
    }
}

int main() {
    std::cout << "=== Day 22: Modern Hashing ===\n";
    runBenchmark();
    return 0;
}
```

## Part 5: Trade-offs

### unordered_map vs map

| Aspect | `unordered_map` | `map` |
|--------|-----------------|------|
| **Lookup** | $O(1)$ average | $O(\log n)$ |
| **Ordering** | Unordered | Sorted by key |
| **Iteration** | Unpredictable order | Sorted order |
| **Memory** | More overhead (bucket array) | Less overhead |
| **Use case** | Fast lookups | Range queries, ordered traversal |

### When to Use Each

**Use `std::unordered_map` when:**
- Fast lookup is critical
- Ordering doesn't matter
- Key type has good hash distribution
- Many lookups, few iterations

**Use `std::map` when:**
- Need sorted iteration
- Need range queries (`lower_bound`, `upper_bound`)
- Key has poor hash function
- Ordered output required

### When Third-Party Hash Maps Are Better

`std::unordered_map` has two well-known structural limitations that matter in production CFD code:

1. **Pointer-based separate chaining**: Each bucket holds a pointer to a linked list node. The nodes are individually heap-allocated. This means that even a successful $O(1)$ lookup involves at least one pointer dereference to a non-contiguous heap location — a cache miss on large maps.

2. **Guaranteed iterator stability**: The standard requires that insertions do not invalidate existing iterators or references (only rehashing does). To satisfy this, node-based storage is required. This design choice prevents the more cache-friendly **flat** (open-addressing) layout.

When your CFD boundary registry has more than a few thousand entries and lookup latency is the bottleneck, consider:

- **`absl::flat_hash_map`** (Google Abseil): Open addressing with quadratic probing, SIMD-accelerated group scanning, and inline value storage. Typical 2-4x faster than `std::unordered_map` for read-heavy workloads. Requires the Abseil library.

- **`tsl::robin_map`** (Tessil): Open addressing with Robin Hood displacement. Supports heterogeneous lookup natively via `find_as()`. Header-only, single-file include. Good choice when you cannot add Abseil as a dependency.

### Iterator Invalidation Rules and CFD Implications

**`std::unordered_map` invalidation rules:**

| Operation | Iterators invalidated | References invalidated |
|-----------|-----------------------|------------------------|
| `insert` / `emplace` (no rehash) | None | None |
| `insert` / `emplace` (triggers rehash) | All | All |
| `erase` | Only erased element | Only erased element |
| `reserve` / `rehash` | All | All |

The key hazard in CFD code is calling `reserve()` or inserting into the registry while holding a reference or iterator obtained earlier:

```cpp
PatchRegistry reg;
reg.reserveCapacity(32);
reg.addPatch("inlet", BoundaryCondition(BCType::Dirichlet, 300.0));

// DANGER: take a reference before further inserts
const BoundaryCondition& ref = reg.getBC("inlet");

// If the next insert causes a rehash, 'ref' becomes a dangling reference.
// With reserveCapacity(32), inserts #2 through #32 are safe.
// Insert #33 would rehash and invalidate 'ref'.
reg.addPatch("outlet", BoundaryCondition(BCType::Neumann, 0.0));

// Safe here (no rehash occurred) — but fragile if nPatches grows
std::cout << ref.value << "\n";
```

**Safe pattern:** Complete all insertions before taking any references. The `reserveCapacity()` method in `PatchRegistry` is designed to enforce this — call it with the total expected patch count before the population loop.

**Contrast with `absl::flat_hash_map`:** Flat hash maps with open addressing have even stricter rules — any insert can invalidate all iterators and references regardless of whether a rehash occurs. This makes the "hold a reference across inserts" anti-pattern dangerous regardless of the reserved capacity. The design tradeoff is that flat maps are faster to iterate due to contiguous memory.

### Extended Comparison: Three Hash Map Options

| Property | `std::unordered_map` | `absl::flat_hash_map` | `tsl::robin_map` |
|----------|----------------------|-----------------------|------------------|
| **Layout** | Node-based (heap per element) | Flat (one array) | Flat (one array) |
| **Collision strategy** | Separate chaining | SIMD group probing | Robin Hood open addressing |
| **Lookup speed** | Baseline | 2-4x faster (read-heavy) | 1.5-3x faster |
| **Iterator stability on insert** | Stable (no rehash) | Never stable | Never stable |
| **Heterogeneous lookup** | C++20 `is_transparent` | Built-in | `find_as()` method |

**Practical recommendation for this course:** Use `std::unordered_map` with `reserve()` and transparent comparators (as built in Part 4). Measure.
