# Day 24: Mesh Topology Storage — Memory Footprint Analysis

## Part 1: Pattern Identification

### Comparing Storage Formats for Mesh Topology

Mesh connectivity (which cells connect to which faces) can be stored in multiple ways:

1. **Nested vectors**: `std::vector<std::vector<int>>`
2. **Flat compact**: Values array + Offsets array (Day 21)
3. **CSR format**: Compressed Sparse Row (from numerical linear algebra)

Each representation trades off memory, access speed, and implementation complexity. For large meshes (millions of cells), this decision significantly impacts:
- Total memory usage
- Cache performance during mesh operations
- Ease of serialization
- NUMA-aware allocation on multi-socket systems

The choice of data structure is not merely an academic exercise. A poorly chosen topology representation for a 10-million-cell mesh can consume 400+ MB more RAM than the optimal format, and access patterns that defeat the CPU cache hierarchy can slow a solver iteration by 3-5x compared to cache-friendly alternatives.

### Quick Comparison Table

| Property | Nested | Compact | CSR |
|----------|--------|---------|-----|
| Memory overhead | High (24 B/cell) | Low (4 B/cell) | Low (4 B/cell) |
| Cache locality | Poor | Excellent | Excellent |
| Modification | Easy | Hard | Hard |
| Sparse solver compat | No | Partial | Yes |
| Build complexity | Low | Medium | Medium |

## Part 2: Theory — Three Representations

### Representation 1: Nested Vectors

```cpp
std::vector<std::vector<int>> nested;
// nested[i] = {face_indices_for_cell_i}
```

**Memory**: High overhead (24 bytes per inner vector on a 64-bit system)
**Access**: `nested[cellId][faceIndex]`
**Pros**: Simple to use, easy to grow
**Cons**: Poor cache locality, high fragmentation, 24-byte overhead per row

Each `std::vector<int>` object stores three pointers/sizes internally: a pointer to the heap allocation, the current size, and the capacity. On a 64-bit system this is exactly 24 bytes per inner vector, regardless of how many faces the cell has. A cell with 1 face wastes 20 of those 24 bytes in overhead.

### Representation 2: Flat Compact (Day 21)

```cpp
std::vector<int> values;   // All face indices concatenated
std::vector<int> offsets;  // offsets[i] = start of cell i in values
```

**Memory**: Minimal overhead (only offsets array, 4 bytes per cell)
**Access**: Via `std::span` slice — zero-copy view of a contiguous sub-range
**Pros**: Excellent cache locality, low memory
**Cons**: Requires slice construction, costly to insert/delete rows

### Representation 3: CSR (Compressed Sparse Row)

```cpp
std::vector<int> col_indices;  // Column (face) indices
std::vector<int> row_ptr;      // row_ptr[i] = start of row i in col_indices
```

**Memory**: Same as compact (offsets = row_ptr)
**Access**: Standard sparse matrix format
**Pros**: Compatible with sparse linear algebra libraries (PETSc, Eigen, cuSPARSE)
**Cons**: More complex iteration pattern for non-linear-algebra use cases

CSR and Compact are structurally identical. The difference is naming convention: `row_ptr` / `col_indices` is the sparse matrix vocabulary, while `offsets` / `values` is the mesh-adjacency vocabulary.

### Quantitative Memory Analysis for 1 Million Cells

Consider a mesh with 1,000,000 cells, each having an average of 5 faces (a common value for 3D hexahedral/polyhedral meshes).

**Total face-index slots:** 5,000,000 integers = 5,000,000 × 4 bytes = **20 MB of raw data**.

#### Nested vectors:

```
Outer vector of std::vector<int> objects:
  1,000,000 cells × 24 bytes = 24 MB (vector metadata alone)

Inner allocations (face data):
  5,000,000 ints × 4 bytes = 20 MB

Total: ~44 MB
```

The 24 MB overhead is a fixed tax on the number of cells, independent of connectivity density.

#### Compact / CSR format:

```
Offsets (or row_ptr) array:
  (1,000,000 + 1) entries × 4 bytes ≈ 4 MB

Values (or col_indices) array:
  5,000,000 × 4 bytes = 20 MB

Total: ~24 MB
```

**Memory savings by switching from nested to compact/CSR: ~20 MB per million cells.**

For a 10-million-cell large-eddy simulation mesh this becomes ~200 MB in savings — enough to fit an additional field variable or finer grid in the same RAM budget.

### Why std::vector Has a 24-Byte Header

On a typical 64-bit Linux system with libstdc++:

```
sizeof(std::vector<int>) = 24 bytes

Layout (conceptual):
  [0..7]   pointer to heap data  (8 bytes)
  [8..15]  current size          (8 bytes)
  [16..23] current capacity      (8 bytes)
```

Even an empty `std::vector<int>` occupies these 24 bytes on the stack/in the parent container. The actual heap allocation for data comes on top.

### Programmatic Memory Estimation

The following helper functions provide in-process memory accounting without an external profiler:

```cpp
size_t estimateMemoryNested(const std::vector<std::vector<int>>& nested) {
    // Outer container: capacity * sizeof(std::vector<int>) = capacity * 24 bytes
    size_t total = nested.capacity() * sizeof(std::vector<int>);
    for (const auto& vec : nested) {
        // Each inner vector's heap allocation: capacity * sizeof(int)
        total += vec.capacity() * sizeof(int);
    }
    return total;
}

size_t estimateMemoryCompact(const std::vector<int>& values,
                              const std::vector<int>& offsets) {
    return values.capacity() * sizeof(int) +
           offsets.capacity() * sizeof(int);
}
```

**Note:** These functions measure the logical data only. The `std::vector` objects themselves sit on the stack or inside the struct — add `sizeof(std::vector<int>)` for each top-level vector if you want the full picture.

### std::span for Zero-Copy Slice Access

C++20 `std::span` allows a caller to receive a view over a contiguous sub-range of the compact representation without copying any data:

```cpp
// Add this method to CompactTopology
std::span<const int> cellFaces(size_t cellId) const {
    // Return a non-owning view into the values array
    // starting at offsets[cellId], with length = face count for this cell
    return {values.data() + offsets[cellId],
            static_cast<size_t>(offsets[cellId + 1] - offsets[cellId])};
}
```

Usage at the call site:

```cpp
CompactTopology topo(nested);

// No copy — span is just a (pointer, length) pair
std::span<const int> faces = topo.cellFaces(42);
for (int faceId : faces) {
    // process face
}
```

Key properties of `std::span`:
- Size: typically 16 bytes (pointer + length) — no heap allocation
- Lifetime: the span is only valid while the underlying `CompactTopology` is alive
- Iterator support: works with range-for, algorithms, and structured bindings

## Part 4: Implementation

### Three Topology Representations

```cpp
#include <vector>
#include <span>
#include <chrono>
#include <iostream>
#include <random>
#include <numeric>
#include <iomanip>
#include <cassert>

// -----------------------------------------------------------------------
// Free function: estimate heap bytes used by a nested vector structure
// -----------------------------------------------------------------------
size_t estimateMemoryNested(const std::vector<std::vector<int>>& nested) {
    size_t total = nested.capacity() * sizeof(std::vector<int>);
    for (const auto& vec : nested) {
        total += vec.capacity() * sizeof(int);
    }
    return total;
}

// -----------------------------------------------------------------------
// Representation 1: Nested vectors
// -----------------------------------------------------------------------
struct NestedTopology {
    std::vector<std::vector<int>> cellFaces;

    NestedTopology(size_t nCells, int /*avgFaces*/) {
        cellFaces.resize(nCells);
        std::mt19937 gen(42);  // Fixed seed for reproducibility
        std::uniform_int_distribution<> faceDist(3, 7);

        for (size_t i = 0; i < nCells; ++i) {
            int nFaces = faceDist(gen);
            cellFaces[i].resize(nFaces);
            for (int j = 0; j < nFaces; ++j) {
                // BUG FIX: was `&faces - &cellFaces[0] * 10 + j`
                // which multiplied a pointer by 10 (operator precedence error).
                // Correct: use the loop index i to compute a deterministic value.
                cellFaces[i][j] = static_cast<int>(i) * 10 + j;
            }
        }
    }

    size_t memoryBytes() const {
        return estimateMemoryNested(cellFaces);
    }
};

// -----------------------------------------------------------------------
// Representation 2: Flat compact storage
// -----------------------------------------------------------------------
struct CompactTopology {
    std::vector<int> values;   // All face indices, concatenated
    std::vector<int> offsets;  // offsets[i] = start of cell i

    // Construct from a nested representation
    explicit CompactTopology(const std::vector<std::vector<int>>& nested) {
        size_t nCells = nested.size();
        offsets.resize(nCells + 1);
        offsets[0] = 0;

        for (size_t i = 0; i < nCells; ++i) {
            offsets[i + 1] = offsets[i] + static_cast<int>(nested[i].size());
        }

        size_t totalFaces = static_cast<size_t>(offsets.back());
        values.resize(totalFaces);

        size_t pos = 0;
        for (const auto& faces : nested) {
            for (int f : faces) {
                values[pos++] = f;
            }
        }
    }

    // Build from owner/neighbour arrays (OpenFOAM-style mesh data)
    void buildFromOwnerNeighbour(
        const std::vector<int>& owner,      // owner[faceId]     = owning cell
        const std::vector<int>& neighbour,  // neighbour[faceId] = neighbour cell (-1 for boundary)
        size_t nCells)
    {
        // Step 1: count faces per cell
        std::vector<int> count(nCells, 0);
        for (int c : owner) {
            if (c >= 0) ++count[static_cast<size_t>(c)];
        }
        for (int c : neighbour) {
            if (c >= 0) ++count[static_cast<size_t>(c)];
        }

        // Step 2: build offsets via prefix sum
        offsets.resize(nCells + 1);
        offsets[0] = 0;
        for (size_t i = 0; i < nCells; ++i) {
            offsets[i + 1] = offsets[i] + count[i];
        }

        // Step 3: fill values using a running write pointer per cell
        size_t totalSlots = static_cast<size_t>(offsets.back());
        values.resize(totalSlots);
        std::vector<int> writePos(offsets.begin(), offsets.begin() + nCells);

        for (size_t faceId = 0; faceId < owner.size(); ++faceId) {
            int oc = owner[faceId];
            if (oc >= 0) {
                values[static_cast<size_t>(writePos[static_cast<size_t>(oc)]++)] =
                    static_cast<int>(faceId);
            }
            int nc = neighbour[faceId];
            if (nc >= 0) {
                values[static_cast<size_t>(writePos[static_cast<size_t>(nc)]++)] =
                    static_cast<int>(faceId);
            }
        }
    }

    // Zero-copy view of faces for a single cell (C++20 std::span)
    std::span<const int> cellFaces(size_t cellId) const {
        return {values.data() + offsets[cellId],
                static_cast<size_t>(offsets[cellId + 1] - offsets[cellId])};
    }

    size_t memoryBytes() const {
        return values.capacity()  * sizeof(int) +
               offsets.capacity() * sizeof(int);
    }

    size_t nCells() const {
        return offsets.empty() ? 0u : offsets.size() - 1;
    }
};

// -----------------------------------------------------------------------
// Representation 3: CSR (Compressed Sparse Row)
// -----------------------------------------------------------------------
class CSRTopology {
    std::vector<int> col_indices;  // Face (column) indices
    std::vector<int> row_ptr;      // row_ptr[i] = start of cell i

public:
    explicit CSRTopology(const std::vector<std::vector<int>>& nested) {
        size_t nCells = nested.size();
        row_ptr.resize(nCells + 1);
        row_ptr[0] = 0;

        for (size_t i = 0; i < nCells; ++i) {
            row_ptr[i + 1] = row_ptr[i] + static_cast<int>(nested[i].size());
        }

        size_t totalFaces = static_cast<size_t>(row_ptr.back());
        col_indices.resize(totalFaces);

        size_t pos = 0;
        for (const auto& faces : nested) {
            for (int f : faces) {
                col_indices[pos++] = f;
            }
        }
    }

    // Expose row access for the benchmark
    int rowStart(size_t cell) const { return row_ptr[cell]; }
    int rowEnd(size_t cell)   const { return row_ptr[cell + 1]; }
    int colAt(size_t idx)     const { return col_indices[idx]; }

    size_t memoryBytes() const {
        return col_indices.capacity() * sizeof(int) +
               row_ptr.capacity()     * sizeof(int);
    }
};
```

### Building Topology from Owner/Neighbour Arrays

OpenFOAM stores mesh connectivity as two integer arrays per face:
- `owner[faceId]` — the cell on the "owner" side (always defined)
- `neighbour[faceId]` — the cell on the "neighbour" side (-1 for boundary faces)

The `buildFromOwnerNeighbour` method above converts this OpenFOAM-style layout into compact storage via a two-pass algorithm:

**Pass 1 — Count:** iterate over owner and neighbour arrays, incrementing a per-cell counter.

**Pass 2 — Fill:** build the prefix-sum offsets, then iterate again, writing each face index into the appropriate slot using a per-cell write cursor.

This is an O(nFaces) algorithm with O(nCells) auxiliary storage for the write cursors.

```cpp
// Minimal usage example
void exampleBuildFromMesh() {
    // A simple 3-cell, 4-face mesh (linear chain)
    std::vector<int> owner     = {0, 1, 2, 0};  // face 3 is a boundary face of cell 0
    std::vector<int> neighbour = {1, 2, -1, -1}; // -1 = boundary

    CompactTopology topo;
    topo.buildFromOwnerNeighbour(owner, neighbour, 3);

    // Inspect cell 1: should see faces 0 and 1
    for (int f : topo.cellFaces(1)) {
        std::cout << "cell 1 has face " << f << "\n";
    }
}
```

### Benchmark: Memory and Access Time

```cpp
void runBenchmark() {
    const size_t nCells    = 1'000'000;  // 1 million cells
    const int    avgFaces  = 5;
    const int    iterations = 100;

    // ----------------------------------------------------------------
    // Build nested topology with random connectivity (seed=42)
    // ----------------------------------------------------------------
    std::vector<std::vector<int>> nested;
    nested.resize(nCells);
    std::mt19937 gen(42);
    std::uniform_int_distribution<> faceDist(3, 7);

    for (size_t i = 0; i < nCells; ++i) {
        int nFaces = faceDist(gen);
        nested[i].resize(nFaces);
        for (int j = 0; j < nFaces; ++j) {
            nested[i][j] = static_cast<int>(i) * 10 + j;
        }
    }

    // ----------------------------------------------------------------
    // Construct other representations
    // ----------------------------------------------------------------
    CompactTopology compact(nested);
    CSRTopology     csr(nested);

    // ----------------------------------------------------------------
    // Memory comparison
    // ----------------------------------------------------------------
    double nestedMB  = estimateMemoryNested(nested) / (1024.0 * 1024.0);
    double compactMB = compact.memoryBytes()         / (1024.0 * 1024.0);
    double csrMB     = csr.memoryBytes()             / (1024.0 * 1024.0);

    std::cout << "=== Memory Footprint (1M cells, avg 5 faces) ===\n";
    std::cout << std::fixed << std::setprecision(1);
    std::cout << "Nested:   " << nestedMB  << " MB\n";
    std::cout << "Compact:  " << compactMB << " MB\n";
    std::cout << "CSR:      " << csrMB     << " MB\n";
    std::cout << "Savings (nested vs compact): "
              << (nestedMB - compactMB) << " MB\n\n";

    // ----------------------------------------------------------------
    // Access time benchmark
    // ----------------------------------------------------------------
    std::cout << "=== Access Time Benchmark (" << iterations << " passes) ===\n";
    volatile long long sum = 0;

    // Nested access
    {
        auto t0 = std::chrono::high_resolution_clock::now();
        for (int iter = 0; iter < iterations; ++iter) {
            for (size_t cell = 0; cell < nCells; ++cell) {
                for (int face : nested[cell]) {
                    sum += face;
                }
            }
        }
        auto t1   = std::chrono::high_resolution_clock::now();
        auto ms   = std::chrono::duration_cast<std::chrono::milliseconds>(t1 - t0).count();
        std::cout << "Nested:   " << ms << " ms\n";
    }

    // Compact access via std::span
    {
        auto t0 = std::chrono::high_resolution_clock::now();
        for (int iter = 0; iter < iterations; ++iter) {
            for (size_t cell = 0; cell < nCells; ++cell) {
                for (int face : compact.cellFaces(cell)) {
                    sum += face;
                }
            }
        }
        auto t1 = std::chrono::high_resolution_clock::now();
        auto ms = std::chrono::duration_cast<std::chrono::milliseconds>(t1 - t0).count();
        std::cout << "Compact (span): " << ms << " ms\n";
    }

    // CSR access
    {
        auto t0 = std::chrono::high_resolution_clock::now();
        for (int iter = 0; iter < iterations; ++iter) {
            for (size_t cell = 0; cell < nCells; ++cell) {
                int rStart = csr.rowStart(cell);
                int rEnd   = csr.rowEnd(cell);
                for (int i = rStart; i < rEnd; ++i) {
                    sum += csr.colAt(static_cast<size_t>(i));
                }
            }
        }
        auto t1 = std::chrono::high_resolution_clock::now();
        auto ms = std::chrono::duration_cast<std::chrono::milliseconds>(t1 - t0).count();
        std::cout << "CSR:      " << ms << " ms\n";
    }

    std::cout << "\nChecksum (prevent dead-code elimination): " << sum << "\n";
}

int main() {
    std::cout << "=== Day 24: Mesh Topology Storage ===\n\n";
    runBenchmark();
    return 0;
}
```

## Part 5: Trade-offs

### Cache-Line Analysis

Modern CPUs load memory in 64-byte cache lines. Understanding how topology data fits into cache lines is critical for predicting access performance.

```
Cache line capacity:
  64 bytes / 4 bytes per int = 16 ints per cache line
```

**Compact/CSR layout — cell with 5 faces:**

```
values array (contiguous):
  [..., f0, f1, f2, f3, f4, ...]
         ^--- all 5 faces fit in one 64-byte cache line
```

A cell with up to 16 faces can have all its face indices loaded in a single cache-line fetch. For typical polyhedral cells (5–12 faces), this means the hot loop body runs entirely from L1 cache after the first access.

**Nested layout — cell i:**

```
cellFaces[i] holds a std::vector<int> with a pointer to a heap allocation.
The heap allocation was made at an arbitrary time during construction,
so it may be anywhere in the 44 MB heap.

Access sequence:
  1. Load cellFaces[i]                 -> pointer dereference (likely L1 hit)
  2. Load *cellFaces[i].data()         -> pointer dereference (likely L2/L3 miss)
  3. Load face values from that address -> may or may not be in cache
```

For sequential cell traversal (i = 0, 1, 2, ...) in nested storage, consecutive cells are almost certainly in different heap pages, so cache misses are the norm rather than the exception.

**Quantified impact (approximate, 1M cells, 100 iterations):**

| Representation | Cache misses (est.) | Access time |
|----------------|---------------------|-------------|
| Nested | ~500M L2 misses | ~1800 ms |
| Compact | ~30M L2 misses | ~900 ms |
| CSR | ~30M L2 misses | ~900 ms |

The 2x speedup of compact/CSR over nested comes almost entirely from eliminating pointer-chasing cache misses.

### NUMA Considerations

On multi-socket servers (common in HPC cluster nodes), memory is divided between NUMA nodes. A CPU on socket 0 has fast access to NUMA node 0 memory and slow (2-3x latency) access to NUMA node 1 memory.

**Nested vectors and NUMA:**

Construction of nested topology allocates each inner `std::vector` independently. In a multi-threaded build (OpenMP parallel for over cells), allocations may land on any NUMA node depending on which thread runs `new`. This produces scattered allocations across NUMA nodes.

```
Socket 0 (Thread 0 builds cells 0..499k):
  cellFaces[0]..cellFaces[499k] data -> NUMA node 0

Socket 1 (Thread 1 builds cells 500k..999k):
  cellFaces[500k]..cellFaces[999k] data -> NUMA node 1

Sequential read on socket 0:
  cells 0..499k  -> fast (local NUMA)
  cells 500k..999k -> slow (remote NUMA access)
```

**Compact/CSR and first-touch policy:**

The Linux kernel uses a "first-touch" NUMA policy: a page is allocated on the NUMA node of the thread that first writes to it. For compact storage with a single `values` array, initializing the array sequentially on socket 0 places all data on NUMA node 0.

For a NUMA-aware implementation, initialize with a parallel loop that mirrors the access pattern:

```cpp
// NUMA-aware initialization of compact topology values
// Assumes OpenMP and that threads are pinned to NUMA nodes
#pragma omp parallel for schedule(static)
for (size_t i = 0; i < nCells; ++i) {
    int start = offsets[i];
    int end   = offsets[i + 1];
    for (int k = start; k < end; ++k) {
        values[static_cast<size_t>(k)] = static_cast<int>(i) * 10 + (k - start);
    }
}
```

With this pattern, thread T on socket S writes the cells it will later read, ensuring first-touch places those pages on NUMA node S.

**Nested vectors and NUMA remediation is difficult.** Even if you allocate inner vectors carefully, a later reallocation (e.g., `push_back` causing a copy to a new heap page) can silently migrate data to a different NUMA node. Compact/CSR avoids reallocation after the initial build.

### Which Minimizes Memory?

**Winner: Compact and CSR (tied)**

Both use a single offsets/row_ptr array plus values. For 1M cells with average 5 faces:
- Nested: ~44 MB
- Compact/CSR: ~24 MB
- **Savings: ~20 MB per million cells**

For 10M cells: ~200 MB saved — enough to hold two additional scalar fields on the same mesh.

### Which Minimizes Access Time?

**Winner: Compact/CSR (faster due to cache locality)**

Sequential traversal through compact/CSR data hits the L1/L2 cache for face index loads. Nested storage requires a pointer dereference per cell, causing frequent L2/L3 cache misses that stall the CPU pipeline.

### Recommendations

| Scenario | Recommended |
|----------|-------------|
| General mesh storage | Compact (Day 21) |
| Integration with sparse solvers | CSR |
| Frequent mesh modifications | Nested (easier to modify) |
| Memory-constrained systems | Compact/CSR |
| Multi-socket NUMA servers | Compact + first-touch init |
| OpenFOAM-style input data | `buildFromOwnerNeighbour` |

### Bug Note: Operator Precedence in Pointer Arithmetic

The original `NestedTopology` constructor contained:

```cpp
// BUGGY — operator precedence multiplies a pointer by 10
faces[j] = &faces - &cellFaces[0] * 10 + j;
```

In C++, `*` binds more tightly than `-`, so `&cellFaces[0] * 10` attempts to multiply a pointer by the integer 10. This is undefined behavior — pointers cannot be multiplied by scalars. The correct expression uses the cell index `i` from the enclosing loop:

```cpp
// FIXED — i is the cell index (size_t), cast to int for the computation
cellFaces[i][j] = static_cast<int>(i) * 10 + j;
```

The `static_cast<int>` converts `i` from `size_t` to `int` before the multiplication, avoiding potential overflow for large `i` on platforms where `int` is 32 bits while `size_t` is 64 bits. For 1M cells, `i * 10` reaches 10,000,000 which is well within `INT_MAX` (2,147,483,647), but the cast documents the intent explicitly.
