## Part 1: Core Theory - Boundary Condition Fundamentals

### 1.1 What are Boundary Conditions and Why They Matter

In computational fluid dynamics, boundary conditions represent the interaction between our computational domain and the external world. They are crucial because:

1. **Well-posedness**: A PDE requires boundary conditions to have a unique, stable solution
2. **Physical reality**: They encode the physical behavior at domain boundaries
3. **Solution convergence**: Proper BCs ensure numerical stability
4. **Computational efficiency**: Appropriate BCs reduce solution time

For our R410A evaporator simulation, boundary conditions will control:
- Liquid refrigerant entry conditions
- Two-phase mixture exit behavior
- Heat transfer at tube walls
- Symmetry conditions for axisymmetric flow

### 1.2 Mathematical Classification of Boundary Conditions

Boundary conditions are mathematically classified into three fundamental types:

#### Dirichlet Boundary Conditions
- **Definition**: Specify the value of the solution at the boundary
- **Mathematical form**: $u = g(\mathbf{x})$ on $\Gamma_D$
- **Physical interpretation**: Fixed value (e.g., inlet temperature, wall velocity)
- **OpenFOAM equivalent**: `fixedValueFvPatchField<Type>`

#### Neumann Boundary Conditions
- **Definition**: Specify the normal derivative of the solution at the boundary
- **Mathematical form**: $\frac{\partial u}{\partial n} = h(\mathbf{x})$ on $\Gamma_N$
- **Physical interpretation**: Fixed flux (e.g., heat flux, pressure gradient)
- **OpenFOAM equivalent**: `fixedGradientFvPatchField<Type>`

#### Robin Boundary Conditions
- **Definition**: Specify a linear combination of value and normal derivative
- **Mathematical form**: $u + \beta \frac{\partial u}{\partial n} = g(\mathbf{x})$ on $\Gamma_R$
- **Physical interpretation**: Mixed conditions (e.g., convective heat transfer)
- **OpenFOAM equivalent**: `mixedFvPatchField<Type>`

### 1.3 Physical Interpretation in CFD

#### Fixed Value (Dirichlet)
- **When to use**: When the boundary value is known from physical considerations
- **Examples**:
  - Inlet velocity for liquid R410A: $\mathbf{U} = (U_{inlet}, 0, 0)$
  - Wall temperature: $T = T_{wall}$
  - Outlet pressure: $p = p_{outlet}$
- **Implementation in OpenFOAM**:

#### Fixed Gradient (Neumann)
- **When to use**: When the flux across the boundary is known
- **Examples**:
  - Heat flux at wall: $\frac{\partial T}{\partial n} = \frac{q''}{k}$
  - Pressure gradient at inlet
- **Implementation in OpenFOAM**:

  ```cpp
  };
  ```

The mathematical implementation of fixed gradient condition is:

⭐ **Fixed Gradient Formula:**
$$
x_p = x_c + \frac{\nabla(x)}{\Delta}
$$

where:
- $x_p$ = patch face values
- $x_c$ = internal field values at adjacent cells
- $\nabla(x)$ = user-specified gradient
- $\Delta$ = inverse distance from patch face centre to cell centre

#### Mixed (Robin)
- **When to use**: When both value and flux are involved, typically in heat transfer
- **Examples**:
  - Convective heat transfer: $-k\frac{\partial T}{\partial n} = h(T - T_{\infty})$
  - Radiation boundary conditions
- **Implementation in OpenFOAM**:

⭐ **Mixed BC Formula:**
$$
x_p = w \cdot x_{ref} + (1-w) \cdot \left(x_c + \frac{\nabla_{ref}}{\Delta}\right)
$$

where:
- $w$ = weight field (0 = pure gradient, 1 = pure value)
- $x_{ref}$ = reference value
- $\nabla_{ref}$ = reference normal gradient

### 1.4 Patch-Normal Gradient (snGrad) Concept

The patch-normal gradient is a fundamental concept in OpenFOAM's boundary condition implementation. It represents the gradient of a field normal to the patch face.

#### Mathematical Definition
The patch-normal gradient is defined as:
$$
\frac{\partial u}{\partial n} = \nabla u \cdot \mathbf{n}
$$
where $\mathbf{n}$ is the unit normal vector to the patch face.

#### Implementation in OpenFOAM
For zero-gradient boundary conditions:
```cpp
// File: openfoam_temp/src/finiteVolume/fields/fvPatchFields/basic/zeroGradient/zeroGradientFvPatchField.H
// Lines: 135-141
tmp<Field<Type>> zeroGradientFvPatchField<Type>::snGrad() const
{
    return tmp<Field<Type>>
    (
        new Field<Type>(this->size(), Zero)
    );
}
```

This returns zero because the gradient normal to the patch is zero (the field value is extrapolated from the internal field).

⭐ **Mixed BC snGrad Implementation:**
$$
\nabla_{\perp} x = w \cdot (x_p - x_c) \cdot \Delta + (1-w) \cdot \nabla_{ref}
$$

This formula shows how the mixed BC interpolates between a gradient condition (when $w=0$) and a value condition (when $w=1$).

## Part 2: Physical Challenge - BC Selection for Evaporator Tube

### 2.1 Inlet BC: Subcooled Liquid R410A Entry

For the inlet of our R410A evaporator tube, we have several fields that require boundary conditions:

#### Velocity Field
- **Type**: Dirichlet (fixed value)
- **Reason**: Known inlet flow rate
- **Implementation**: $\mathbf{U} = (U_{inlet}, 0, 0)$
- **OpenFOAM**: `fixedValueFvPatchField<vector>`

#### Volume Fraction Field
- **Type**: Dirichlet (fixed value)
- **Reason**: Subcooled liquid enters with $\alpha_{liquid} = 1.0$
- **Implementation**: $\alpha_{liquid} = 1.0$, $\alpha_{vapor} = 0.0$
- **OpenFOAM**: `fixedValueFvPatchField<scalar>`

#### Temperature Field
- **Type**: Dirichlet (fixed value)
- **Reason**: Known inlet temperature of subcooled liquid
- **Implementation**: $T = T_{inlet}$ (typically 280-285K for R410A)
- **OpenFOAM**: `fixedValueFvPatchField<scalar>`

#### Pressure Field
- **Type**: Zero gradient (Neumann)
- **Reason**: Pressure adjusts based on internal flow
- **Implementation**: $\frac{\partial p}{\partial n} = 0$
- **OpenFOAM**: `zeroGradientFvPatchField<scalar>`

### 2.2 Outlet BC: Handling Reverse Flow and Pressure Specification

Outlet boundary conditions are challenging because we don't know a priori what the flow conditions will be at the exit.

#### Velocity Field
The best choice for velocity at the outlet is `inletOutlet`:
- **Operation**:
  - Positive flux (outflow): Zero gradient
  - Negative flux (reverse flow): Fixed value (inletValue)
- **Implementation**:
  ```cpp

  // Lines: 81-84
  class inletOutletFvPatchField

  {
  public:
    //- Runtime type information
    TypeName("inletOutlet");
  };
  ```

#### Pressure Field
- **Type**: Dirichlet (fixed value)
- **Reason**: Known outlet pressure for evaporator
- **Implementation**: $p = p_{outlet}$ (typically 1.0-1.2 MPa gauge)
- **OpenFOAM**: `fixedValueFvPatchField<scalar>`

#### Volume Fraction Field
- **Type**: Zero gradient (Neumann)
- **Reason**: Unknown exit composition
- **Implementation**: $\frac{\partial \alpha}{\partial n} = 0$
- **OpenFOAM**: `zeroGradientFvPatchField<scalar>`

### 2.3 Wall BC: No-slip, Fixed Temperature vs Fixed Heat Flux

Wall boundary conditions are critical for heat transfer simulation.

#### Velocity Field
- **Type**: Dirichlet (fixed value)
- **Condition**: No-slip condition
- **Implementation**: $\mathbf{U} = (0, 0, 0)$
- **OpenFOAM**: `fixedValueFvPatchField<vector>`
- **Physical meaning**: Fluid velocity matches wall velocity (zero for stationary wall)

#### Temperature Field
Two common options:

**Option 1: Fixed Temperature**
- **Type**: Dirichlet (fixed value)
- **Implementation**: $T = T_{wall}$
- **OpenFOAM**: `fixedValueFvPatchField<scalar>`
- **Use case**: Known wall temperature (e.g., heated tube)

**Option 2: Fixed Heat Flux**
- **Type**: Neumann (fixed gradient)
- **Implementation**: $\frac{\partial T}{\partial n} = \frac{q''}{k}$
- **OpenFOAM**: `fixedGradientFvPatchField<scalar>`
- **Use case**: Known heat input rate

### 2.4 Axis BC: Symmetry for 2D Axisymmetric Formulation

For axisymmetric simulations, we need special boundary conditions on the axis:

- **Type**: Symmetry
- **Implementation**:
  - Normal velocity: $U_n = 0$
  - Tangential velocity: Zero gradient
  - All scalar fields: Zero gradient
- **OpenFOAM**: `symmetryFvPatchField<Type>`

The symmetry condition ensures:
1. No flow across the axis
2. Smooth variation along the axis
3. Mathematical consistency with axisymmetric coordinates

### 2.5 Well-posedness: Ensuring Unique Solution

A boundary value problem is well-posed if:
1. A solution exists
2. The solution is unique
3. The solution depends continuously on the data

For our R410A evaporator:
- **Existence**: Ensure boundary conditions are physically realistic
- **Uniqueness**: Avoid conflicting BCs (e.g., both value and gradient specified)
- **Stability**: Choose BCs that don't cause numerical oscillations

Common well-posedness issues:
- Over-specification (multiple BCs on same field)
- Under-specification (missing BCs)
- Inconsistent BCs (e.g., inflow with reversed velocity)

### 3.3 Mixed (Robin) BC: valueFraction Weighting

The mixed boundary condition is the most flexible and mathematically interesting. It combines Dirichlet and Neumann conditions using a weighting factor.

⭐ **Mixed BC Evaluate Implementation:**
```cpp
// File: openfoam_temp/src/finiteVolume/fields/fvPatchFields/basic/mixed/mixedFvPatchField.C
// Lines: 193-201
void mixedFvPatchField<Type>::evaluate(const Pstream::commsTypes commsType)
{
    Field<Type>::operator=
    (
        valueFraction_*refValue_
      + (1.0 - valueFraction_)
       *(this->patchInternalField() + refGrad_/this->patch().deltaCoeffs())
    );
}
```

This implements the formula:
$$
x_p = w \cdot x_{ref} + (1-w) \cdot \left(x_c + \frac{\nabla_{ref}}{\Delta}\right)
$$

where:
- `valueFraction_` = $w$ (the weighting factor)
- `refValue_` = $x_{ref}$ (reference value)
- `refGrad_` = $\nabla_{ref}$ (reference gradient)
- `deltaCoeffs()` = $\Delta$ (inverse distance)

⭐ **Mixed BC snGrad Implementation:**
```cpp
// File: openfoam_temp/src/finiteVolume/fields/fvPatchFields/basic/mixed/mixedFvPatchField.C
// Lines: 208-217
tmp<Field<Type>> mixedFvPatchField<Type>::snGrad() const
{
    return tmp<Field<Type>>
    (
        new Field<Type>
        (
            valueFraction_*(refValue_ - this->patchInternalField())
           *this->patch().deltaCoeffs()
          + (1.0 - valueFraction_)*refGrad_
        )
    );
}
```

This gives:
$$
\nabla_{\perp} x = w \cdot (x_p - x_c) \cdot \Delta + (1-w) \cdot \nabla_{ref}
$$

### 3.4 Derived BC: inletOutlet for Flux-Based Switching

The `inletOutlet` boundary condition demonstrates how to create complex behavior by switching between different boundary modes based on flow direction.

⭐ **inletOutlet Implementation:**
```cpp
// File: openfoam_temp/src/finiteVolume/fields/fvPatchFields/derived/inletOutlet/inletOutletFvPatchField.C
// Key logic for flux-based switching
void inletOutletFvPatchField<Type>::updateCoeffs()
{
    if (this->updated())
    {
        return;
    }

    // Get the flux field (usually phi)
    const Field<Type>& phi = this->db().lookupObject<Field<Type>>(phiName_);

    // Determine if flow is in or out
    const scalarField& outsideFlux = phi.boundaryField()[this->index()];

    // Set valueFraction based on flux direction
    valueFraction_ = pos(outsideFlux);

    // Set reference value for reverse flow
    refValue_ = inletValue_;

    // Set reference gradient for forward flow
    refGrad_ = Zero;

    fvPatchField<Type>::updateCoeffs();
}
```

The logic is:
1. **Positive flux** (outflow): `valueFraction_ = 1` (pure value condition)
2. **Negative flux** (inflow): `valueFraction_ = 0` (pure gradient condition)

### 3.5 Matrix Coefficients: InternalCoeffs vs BoundaryCoeffs

When solving linear systems, boundary conditions contribute to the matrix coefficients. OpenFOAM provides two virtual methods for this:

#### valueInternalCoeffs() and valueBoundaryCoeffs()
These methods contribute to the matrix equation for field values:
```cpp
// Typical pattern in fvPatchField<Type>
virtual tmp<Field<Type>> valueInternalCoeffs() const = 0;
virtual tmp<Field<Type>> valueBoundaryCoeffs() const = 0;
```

For Dirichlet conditions:
- `valueInternalCoeffs()` = diagonal coefficient (usually 1)
- `valueBoundaryCoeffs()` = source term (the fixed value)

For Neumann conditions:
- `valueInternalCoeffs()` = diagonal coefficient (related to delta)
- `valueBoundaryCoeffs()` = source term (related to gradient)

#### gradientInternalCoeffs() and gradientBoundaryCoeffs()
These methods contribute to gradient calculations:
```cpp
virtual tmp<Field<Type>> gradientInternalCoeffs() const = 0;
virtual tmp<Field<Type>> gradientBoundaryCoeffs() const = 0;
```

The matrix coefficient system ensures:
1. Conservation of mass, momentum, and energy
2. Proper coupling between boundary and internal field
3. Numerical stability through consistent discretization
