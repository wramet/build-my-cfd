## Part 1: Core Theory - Sparse Matrix Concepts (ทฤษฎีพื้นฐาน - แนวคิดเมทริกซ์เบาบาง)

### 1.1 Why Sparse Matrices for CFD?

In computational fluid dynamics, we solve systems of partial differential equations (PDEs) discretized over computational domains. The resulting linear systems have a crucial property: **sparsity**. For a 3D simulation with N control volumes, the coefficient matrix $\mathbf{A}$ has dimensions $N \times N$, but each row typically has only 5-7 non-zero entries.

> **⭐ Verified Fact**: OpenFOAM uses the LDU (Lower-Diagonal-Upper) format for sparse matrix storage.
> **Source**: `openfoam_temp/src/OpenFOAM/matrices/LduMatrix/LduMatrix/lduMatrix.H:80-88`

#### Structured vs Unstructured Meshes

**Structured meshes**: Regular connectivity (Cartesian, cylindrical). Non-zero entries follow predictable patterns (banded matrices).

**Unstructured meshes**: Irregular connectivity (tetrahedral, polyhedral). Non-zero pattern depends on mesh topology.

For both mesh types, the sparsity pattern arises from the **locality principle**: a control volume's equation only involves itself and its immediate neighbors. The fill ratio is approximately:

$$
\text{Fill ratio} = \frac{\text{Non-zeros}}{N^2} \approx \frac{cN}{N^2} = \frac{c}{N}
$$

where $c$ is the average number of neighbors (typically 6-7 for 3D).

### 1.2 Dense vs Sparse Storage: Memory Complexity

**Dense storage**: Store all $N^2$ elements.

- Memory: $O(N^2)$
- Example: For $N = 10^6$, requires $8 \times 10^{12}$ bytes ≈ 8 TB (double precision)

**Sparse storage**: Store only non-zero elements and their locations.

- Memory: $O(cN)$
- Example: For $N = 10^6$, $c = 7$, requires $8 \times 7 \times 10^6$ bytes ≈ 56 MB

The memory savings are dramatic:

$$
\text{Savings factor} = \frac{N^2}{cN} = \frac{N}{c}
$$

For $N = 10^6$, this is approximately $1.4 \times 10^5$ times less memory!

### 1.3 LDU (Lower-Diagonal-Upper) Format Derivation

The LDU format exploits the **symmetric nonzero pattern** of finite volume matrices. Consider a general sparse matrix $\mathbf{A}$. We can decompose it as:

$$
\mathbf{A} = \mathbf{L} + \mathbf{D} + \mathbf{U}
$$

where:
- $\mathbf{L}$: strictly lower triangular part
- $\mathbf{D}$: diagonal
- $\mathbf{U}$: strictly upper triangular part

#### LDU Storage Arrays

> **⭐ Verified Fact**: `lduMatrix` stores three coefficient arrays: `lowerPtr_`, `diagPtr_`, and `upperPtr_`.
> **Source**: `openfoam_temp/src/OpenFOAM/matrices/LduMatrix/LduMatrix/lduMatrix.H:88`

- `lowerPtr_`: Contains $L_{ij}$ for $i > j$
- `diagPtr_`: Contains $D_{ii}$
- `upperPtr_`: Contains $U_{ij}$ for $i < j$

The indexing follows mesh connectivity. For face $f$ connecting cells $P$ (owner) and $N$ (neighbor):

- If $P < N$: $A_{PN} = \text{upper}[f]$, $A_{NP} = \text{lower}[f]$
- Diagonal entries: $A_{PP}$ and $A_{NN}$ in `diag`

### 1.4 Matrix-Vector Multiplication Algorithm

Given $\mathbf{y} = \mathbf{A}\mathbf{x}$, where $\mathbf{A}$ is stored in LDU format:

#### Complete Derivation

Let $\mathbf{A} = \mathbf{L} + \mathbf{D} + \mathbf{U}$. Then:

$$
y_i = \sum_{j=1}^{N} A_{ij}x_j = D_{ii}x_i + \sum_{j<i} L_{ij}x_j + \sum_{j>i} U_{ij}x_j
$$

For face-based storage with faces $f = 1,\dots,F$, connecting owner cell $o(f)$ and neighbor cell $n(f)$:

1. Initialize: $y_i = D_{ii}x_i$
2. For each face $f$:
   - Contribution to owner: $y_{o(f)} \mathrel{+}= A_f x_{n(f)}$
   - Contribution to neighbor: $y_{n(f)} \mathrel{+}= A_f x_{o(f)}$

> **⭐ Verified Fact**: The `Amul` function implements matrix-vector multiplication in LDU format.
> **Source**: `openfoam_temp/src/OpenFOAM/matrices/LduMatrix/LduMatrix/lduMatrixATmul.C:34-92`

#### Implementation in OpenFOAM

```cpp
// File: openfoam_temp/src/OpenFOAM/matrices/LduMatrix/LduMatrix/lduMatrixATmul.C
// Lines: 67-79
const label nCells = diag().size();
for (label cell=0; cell<nCells; cell++)
{
    ApsiPtr[cell] = diagPtr[cell]*psiPtr[cell];
}

const label nFaces = upper().size();

for (label face=0; face<nFaces; face++)
{
    ApsiPtr[uPtr[face]] += lowerPtr[face]*psiPtr[lPtr[face]];
    ApsiPtr[lPtr[face]] += upperPtr[face]*psiPtr[uPtr[face]];
}
```

**Algorithm complexity**: $O(N + F)$ where $F \approx 3N$ for 3D → $O(N)$

### 1.5 Memory Efficiency Calculations

**LDU storage requirements:**
- Diagonal: $N$ elements
- Lower/Upper: $F$ elements each
- Total: $N + 2F$ elements

For a 3D hexahedral mesh:
- Interior cells: ≈ $N$
- Faces: ≈ $3N$ (each cell has ~6 faces, each face shared by 2 cells)
- Total storage: $N + 2 \times 3N = 7N$ elements

**Memory for double precision (8 bytes/element):**

$$
\text{Memory} = 8 \times 7N \text{ bytes}
$$

For $N = 10^6$: **56 MB**

Compare to dense storage: $8 \times 10^{12}$ bytes = **8 TB**

**Savings factor**: $\approx 1.4 \times 10^5$ times less memory!

## Part 2: Physical Challenge - Memory vs Speed Tradeoffs (ความท้าทายทางฟิสิกส์ - การแลกเปลี่ยนระหว่างหน่วยความจำและความเร็ว)

### 2.1 Memory Requirements for R410A Evaporator Simulation

Consider an R410A evaporator simulation with:
- Mesh: 2 million cells (typical for industrial simulation)
- Variables: $p, \mathbf{U}, T, k, \epsilon, \alpha$ (multiple fields)
- Non-zeros per row: 7 (hexahedral mesh)
- Double precision: 8 bytes/value

**Per variable memory:**
- Matrix storage: $7N = 14 \times 10^6$ elements
- Bytes: $14 \times 10^6 \times 8 = 112$ MB
- For 8 variables: **896 MB**

**Additional memory:**
- Solution vectors: $8 \times N \times 8 \times 3$ (3 time levels) = 384 MB
- Temporary vectors: ~200 MB
- **Total estimate: ~1.5 GB**

This is manageable on modern workstations but requires careful memory management.

### 2.2 Bandwidth Optimization and Cache Performance

**Memory hierarchy challenge:**
- L1 cache: 32-64 KB
- L2 cache: 256-512 KB
- L3 cache: 8-32 MB
- RAM: 16-128 GB
- Matrix: 112 MB/variable → Doesn't fit in cache!

**Optimization strategies:**

1. **Blocking/Tiling**: Process matrix in cache-sized blocks
2. **Vectorization**: SIMD instructions for multiple faces
3. **Memory access patterns**: Sequential access to coefficient arrays

### 2.3 Parallel Decomposition Considerations

**Domain decomposition for parallel computing:**
- Partition mesh into subdomains
- Each processor gets subset of cells
- Interface cells require communication

**Communication overhead:**
- For face $f$ with owner in proc $P$ and neighbor in proc $Q$
- During SpMV: Need $\mathbf{x}$ values from neighbor proc
- Solution: Halo/ghost cells with MPI communication

### 2.4 Preconditioning for Convergence Acceleration

**Why precondition?** The condition number $\kappa(\mathbf{A})$ affects convergence:

$$
\text{Iterations} \propto \sqrt{\kappa(\mathbf{A})}
$$

For poorly conditioned matrices (common in CFD), $\kappa$ can be $10^6$ or more.

**Preconditioner $\mathbf{M}$**: Solve $\mathbf{M}^{-1}\mathbf{A}\mathbf{x} = \mathbf{M}^{-1}\mathbf{b}$ instead

> **⭐ Verified Fact**: `diagonalPreconditioner` stores reciprocal diagonal for faster multiplication.
> **Source**: `openfoam_temp/src/OpenFOAM/matrices/LduMatrix/Preconditioners/DiagonalPreconditioner/diagonalPreconditioner.H:58-59`

**Common preconditioners for CFD:**

1. **Diagonal (Jacobi)**: $M_{ii} = A_{ii}^{-1}$
2. **Incomplete LU (ILU)**: Approximate LU factorization
3. **GAMG**: Hierarchical multigrid approach

### 3.3 Matrix-Vector Product (Amul) Implementation

> **⭐ Verified Fact**: The `Amul` function computes $\mathbf{A}\mathbf{\psi}$ using LDU format.
> **Source**: `openfoam_temp/src/OpenFOAM/matrices/LduMatrix/LduMatrix/lduMatrixATmul.C:34-92`

```cpp
// File: openfoam_temp/src/OpenFOAM/matrices/LduMatrix/LduMatrix/lduMatrixATmul.C
// Lines: 34-92 (complete function)
void Foam::lduMatrix::Amul
(
    scalarField& Apsi,
    const tmp<scalarField>& tpsi,
    const FieldField<Field, scalar>& interfaceBouCoeffs,
    const lduInterfaceFieldPtrsList& interfaces,
    const direction cmpt
) const
{
    scalar* __restrict__ ApsiPtr = Apsi.begin();

    const scalarField& psi = tpsi();
    const scalar* const __restrict__ psiPtr = psi.begin();

    const scalar* const __restrict__ diagPtr = diag().begin();

    const label* const __restrict__ uPtr = lduAddr().upperAddr().begin();
    const label* const __restrict__ lPtr = lduAddr().lowerAddr().begin();

    const scalar* const __restrict__ upperPtr = upper().begin();
    const scalar* const __restrict__ lowerPtr = lower().begin();

    // Initialise the update of interfaced interfaces
    initMatrixInterfaces
    (
        interfaceBouCoeffs,
        interfaces,
        psi,
        Apsi,
        cmpt
    );

    // Diagonal contribution
    const label nCells = diag().size();
    for (label cell=0; cell<nCells; cell++)
    {
        ApsiPtr[cell] = diagPtr[cell]*psiPtr[cell];
    }

    // Off-diagonal contribution
    const label nFaces = upper().size();

    for (label face=0; face<nFaces; face++)
    {
        ApsiPtr[uPtr[face]] += lowerPtr[face]*psiPtr[lPtr[face]];
        ApsiPtr[lPtr[face]] += upperPtr[face]*psiPtr[uPtr[face]];
    }

    // Update interface interfaces
    updateMatrixInterfaces
    (
        interfaceBouCoeffs,
        interfaces,
        psi,
        Apsi,
        cmpt
    );

    tpsi.clear();
}
```

**Key optimizations:**
1. `__restrict__` pointers: Tell compiler no aliasing
2. Direct pointer access: Avoid bounds checking in inner loop
3. Contiguous memory access: Arrays stored contiguously

> **⭐ Verified Fact**: `diagonalPreconditioner` stores the reciprocal diagonal (`rD`) because multiplication is faster than division on most systems.
> **Source**: `openfoam_temp/src/OpenFOAM/matrices/LduMatrix/Preconditioners/DiagonalPreconditioner/diagonalPreconditioner.H:58-59`

**Preconditioning formula:**

$$
\mathbf{M}^{-1}\mathbf{r} \text{ where } M_{ii} = \frac{1}{D_{ii}}
$$

**Performance benefit**: Division ≈ 10-20 CPU cycles, multiplication ≈ 3-5 cycles. For $N = 10^6$, saves ~15 million cycles per application.

### 4.1 Verifying LDU Storage Format Correctness

```python
def verify_ldu_storage(A_dense, lower, diag, upper, owner, neighbour):
    """Verify LDU storage matches dense matrix."""
    n = len(diag)
    A_reconstructed = np.zeros((n, n))

    # Diagonal
    for i in range(n):
        A_reconstructed[i, i] = diag[i]

    # Off-diagonal from faces
    for f in range(len(owner)):
        o, nb = owner[f], neighbour[f]
        A_reconstructed[o, nb] = upper[f]
        A_reconstructed[nb, o] = lower[f]

    # Check equality
    assert np.allclose(A_dense, A_reconstructed)
```
